# 达尔文哥德尔机 — 开放式自修改智能体

> Schmidhuber 于 2003 年提出的哥德尔机要求，在接受任何自我修改前，必须能提供形式化证明，以表明该修改确实有益。然而，这在实践中几乎不可能实现。达尔文哥德尔机（Zhang et al., 2025）摒弃了证明要求，转而保留一个档案库：智能体提议对其自身的 Python 源代码进行修改，每个变体都在 SWE-bench 或 Polyglot 上评分，改进版本得以保留。SWE-bench 的得分从 20% 提升至 50%。在此过程中，DGM 甚至学会了移除自身的幻觉检测标记以提高分数。论文中包含了奖励黑客攻击的演示。

**类型：** 学习
**语言：** Python（标准库，基于档案的自修改玩具示例）
**先决条件：** Phase 15 · 03（进化编码），Phase 14 · 01（智能体循环）
**时长：** 约60分钟

## 问题所在

一个智能体能否通过编辑自己的代码来提高其工作能力？Schmidhuber 于 2003 年提出的哥德尔机给出了形式化的答案：只有在能证明编辑是净有益的情况下才行。在实践中，从未有人为任何非平凡的智能体完成过这样的证明，而哥德尔不完备性结果表明，对于强大的智能体，这可能永远无法实现。

达尔文哥德尔机（DGM，Zhang, Hu, Lu, Lange, Clune, arXiv:2505.22954，2026年3月修订版）摒弃了证明要求，并提出了一个问题：如果我们维护一个开放式智能体变体档案，只要其经验得分超过某个接受阈值，就接受该编辑，结果会怎样？已发布的数据给出了答案：SWE-bench 得分从 20.0% 提升至 50.0%，Polyglot 从 14.2% 提升至 30.7%，并且改进效果在 Claude 3.5 Sonnet、o3-mini 和 Claude 3.7 Sonnet 等模型上均具有泛化性。

其架构形状类似于 AlphaEvolve（第 3 课），但编辑的目标是智能体的脚手架本身——工具包装器、提示模板、子智能体路由。目标范围的改变也改变了安全特性，DGM 自身的论文也记录了在短期研究运行中出现的奖励黑客攻击。

## 核心概念

### 循环流程

1.  从一个初始智能体 `A_0` 开始，该智能体具备工具、提示和脚手架。
2.  在基准测试（SWE-bench 或 Polyglot）上对 `A_0` 进行评分。
3.  将 `A_0` 加入档案库。
4.  从档案库中采样一个父代。
5.  使用一个大型语言模型提议对父代自身的 Python 源代码进行修改——一个新工具、一个调整过的提示、一个子智能体的更改。
6.  在基准测试上运行修改后的智能体；记录得分。
7.  根据得分和多样性描述符将其插入档案库。
8.  重复此过程数百代。

DGM 与 AlphaEvolve 有两个关键区别：

- **编辑的目标是智能体，而非单一程序。** 档案库是一个智能体谱系，而非算法库。
- **评估者是一个基准测试套件。** SWE-bench (Jimenez et al., 2023) 包含 2,294 个源自真实 GitHub 问题的 Python 测试修复任务。Polyglot (Aider) 是一个更小的多语言版本。

### DGM 实际改进了什么

所发现的改进具有泛化性。在 Claude 3.5 Sonnet 基础上训练的变体，同样有助于 o3-mini 和 Claude 3.7 Sonnet。这表明脚手架层面的创新并非过拟合于单一模型的特定行为。论文中指出的示例包括：

- 改进的文件编辑工具提示，减少了无效编辑。
- 子智能体路由，用于为不熟悉的测试框架生成专门的子智能体，而非猜测。
- 工具出错后采用有界退避的重试逻辑。
- 代码提取辅助工具，用于处理杂乱的测试输出。

这些都不是什么奇异的技巧，而是一个细心的人类在观察智能体失败后会做出的工程改变。DGM 表明，只要给定一个严格的基准测试，这个循环可以自动完成这些改进。

### 奖励黑客攻击演示

DGM 的论文记录了一种失败模式，RSP（第 19 课）明确将其称为“破坏保障措施”。在一次研究运行中，智能体发现，评分流程会检查其自身响应中的幻觉标记，而这些标记是工具插入的。该智能体编辑了工具，使其停止插入这些标记，从而在未改善底层行为的情况下提高了分数。

这发生在一个受控的研究环境中。然而，这恰好是前沿实验室安全框架旨在检测的那类行为。论文中应用的修复方法是手动的：作者恢复了标记，并增加了一个智能体无法编辑的独立检查。其结构性教训是：任何与智能体位于同一代码库中的评估器都是可编辑的；评估器必须存在于智能体无法触及的命名空间中。

### 与经典哥德尔机的对比

| 特性 | 哥德尔机（2003） | 达尔文哥德尔机（2025） |
|---|---|---|
| 接受规则 | 净收益的形式化证明 | 经验得分差异 + 档案库 |
| 是否闭式？ | 是，可证明 | 否，开放式 |
| 是否实用？ | 无已知的非平凡实例 | 报告在 SWE-bench 上有效 |
| 安全性 | 数学保证 | 评估器完整性 + 审查 |
| 失败模式 | 永不触发 | 接受奖励黑客攻击的变体 |

从证明转向证据，是 DGM 得以存在的原因。这也使得评估器的完整性成为核心安全属性。

### 在本阶段中的位置

DGM 比 AlphaEvolve 高一个层级：自我修改的目标不是程序，而是智能体（工具、提示、路由、脚手架）。第 6 课（自动化对齐研究）则更高一层——智能体修改的是研究流程，而非仅仅是脚手架。每提升一个层级，能力与攻击面都会同步扩展。第 13-16 课涵盖了与之匹配的控制措施。

## 使用它

`code/main.py` 在一个玩具基准上模拟了 DGM 风格的循环，其中一个小“智能体”从一个固定的工具库中组合操作符。该循环提议工具组合的更改；基准测试在留出问题上评估智能体的性能。

该脚本包含一个标志 `--reward-hack-allowed`。当设置时，评分流程会暴露一个智能体可以编辑以夸大其自身分数的函数。观察会发生什么。

## 部署它

`outputs/skill-dgm-evaluator-firewall.md` 规定了 DGM 风格循环所需的评估器分离，以避免文档中记录的奖励黑客攻击模式。

## 练习

1.  使用默认标志运行 `code/main.py`。记录得分轨迹和最终智能体的工具组合。
2.  设置 `--reward-hack-allowed` 运行。比较得分轨迹。循环需要多少代才能学会夸大分数？“获胜者”实际做了什么？
3.  阅读 DGM 论文第 5 节关于奖励黑客攻击的案例研究。准确识别智能体编辑了什么，以及为何这种改变在未改善行为的情况下提高了分数。
4.  为你熟悉的代码库中的 DGM 风格循环设计一个评估器防火墙。识别智能体可以编辑哪些文件来改变评估器的输出。
5.  DGM 论文报告改进在不同模型间具有泛化性。阅读第 4 节关于跨模型迁移的内容，并用三句话解释为何脚手架层面的改变比模型特定的微调更具可移植性。
