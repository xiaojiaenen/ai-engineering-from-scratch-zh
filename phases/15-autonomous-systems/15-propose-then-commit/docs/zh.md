# 人机协作：提议-提交模式

> 2026年关于人机协作的共识非常具体。它并非"智能体询问，用户点击同意"。而是提议-提交模式：提议的操作会通过幂等键持久化存储；向审核者展示时包含意图、数据血缘、涉及的权限、影响半径和回滚计划；仅在获得明确确认后才提交；执行后还需验证以确认副作用确实发生。LangGraph 的 `interrupt()` 配合 PostgreSQL 检查点、微软智能体框架的 `RequestInfoEvent` 以及 Cloudflare 的 `waitForApproval()` 都实现了相同结构。典型的失败模式是橡皮图章式审批：未加审核便点击"同意"。文档记载的缓解措施是要求审核者回答清单问题的问答式确认。

**类型：** 学习  
**语言：** Python（标准库，带幂等性的提议-提交状态机）  
**前置要求：** 第15阶段 · 第12节（持久化执行），第15阶段 · 第14节（触发警报）  
**时间：** 约60分钟

## 问题背景

智能体执行某个操作。用户需要决定：批准还是不批准。如果决定瞬间完成，那很可能不是真正的审核。如果决定是结构化的，虽然缓慢但更可靠。工程问题在于如何使结构化审核成为阻力最小的路径。

2023年时期的人机协作模式是同步提示："智能体要向X发送内容为Y的邮件——同意吗？"用户点击同意。所有人觉得系统很安全。实际上这种界面极易被橡皮图章式审批：用户快速批准，批准操作无法有效预测风险，当智能体出错时，审计日志显示用户已记不清的长期批准历史。

2026年的模式——提议-提交——将人机协作构建在持久化基础上，附加结构化元数据，并要求明确提交。每个主流智能体开发包都提供实现：LangGraph 的 `interrupt()`、微软智能体框架的 `RequestInfoEvent`、Cloudflare 的 `waitForApproval()`。API名称不同；核心结构相同。

## 核心概念

### 提议-提交状态机

1. **提议。** 智能体生成操作提议。持久化存储（PostgreSQL、Redis、持久化对象）。包含：
   - 意图（智能体执行此操作的原因）
   - 数据血缘（导致此提议的源数据）
   - 涉及的权限（作用域/文件/端点）
   - 影响半径（最坏情况的影响范围）
   - 回滚计划（提交后如何撤销）
   - 幂等键（每个提议唯一；重新提交返回相同记录）
2. **展示。** 审核者看到包含所有元数据的提议。审核者必须是人类（非智能体自我审核）。
3. **提交。** 获得明确确认。操作执行。
4. **验证。** 执行后读回并确认副作用。若验证失败，系统进入已知错误状态并触发警报。

### 幂等性键

若无幂等性键，瞬态故障后的重试可能导致批准操作双重执行。具体案例：用户批准"从A转账100美元到B"。网络闪断。工作流重试。用户仅批准一次但转账执行两次。幂等性键将批准与单一、唯一的副作用绑定；第二次执行成为无操作。

这与Stripe和AWS API使用的幂等性模式相同。微软智能体框架文档明确建议将其用于智能体审批。

### 持久化：为何审批比进程更持久

审批等待室是智能体不拥有的状态片段。工作流已暂停（第12课）。当审批到达时，工作流从精确断点恢复。这就是LangGraph将 `interrupt()` 与PostgreSQL检查点（而非内存状态）配对的原因——两天后的审批仍能找到完整的工作流。

### 橡皮图章式审批与问答式缓解

人机协作的默认界面（"同意"/"拒绝"按钮）导致缺乏真实审核的快速批准。文档记载的缓解措施：问答式清单要求对特定问题给出肯定回答后才能启用同意按钮。具体形式：

- "您理解此操作涉及哪些资源吗？ [ ]"
- "您已验证影响半径可接受吗？ [ ]"
- "若此操作失败，您有回滚计划吗？ [ ]"

这不是形式主义——而是强制功能。无法勾选方框的审核者要么要求澄清（升级处理）要么拒绝（安全默认）。Anthropic的智能体安全研究明确将清单驱动的人机协作列为橡皮图章式审批模式的缓解措施。

### 何种操作需要人机协作

并非所有操作都需要提议-提交。2026年指导原则：

- **重大操作**（始终需要人机协作）：不可逆写入、金融交易、对外通信、生产数据库变更、破坏性文件系统操作。
- **可逆操作**（有时需要人机协作）：本地文件编辑、预发布环境变更、带明确回滚的可逆写入。
- **读取与检查**（从不需要人机协作）：读取文件、列出资源、调用只读API。

### 执行后验证

"提交已运行"不等同于"副作用已发生"。网络分区和竞态条件可能导致工作流认为成功而后端未持久化。验证步骤在提交后重新读取目标资源进行确认。这与数据库事务中的 `RETURNING` 子句或AWS的 `GetObject`（在 `PutObject` 之后）模式相同。

### 欧盟人工智能法案第14条

第14条要求欧盟高风险AI系统进行有效人类监督。"有效"非装饰性。监管语言明确排除橡皮图章式模式。微软智能体治理工具包的合规文档指出，带问答确认的提议-提交模式是能通过第14条审查的结构。

## 实践应用

`code/main.py` 使用Python标准库实现了提议-提交状态机。持久化存储为JSON文件。幂等性键是（thread_id, action_signature）的哈希值。驱动程序模拟三种情况：正常审批流程、瞬态故障后的重试（不得双重执行）、橡皮图章式默认与问答式流程的对比。

## 部署指南

`outputs/skill-hitl-design.md` 审查提议的人机协作工作流是否符合提议-提交结构，并标记缺失的元数据、幂等性、验证或问答确认层。

## 练习

1. 运行 `code/main.py`。确认已批准提议的重试使用持久化记录而非重新执行。现在修改幂等性键使其包含时间戳，展示重试导致的双重执行。

2. 扩展提议记录以包含 `rollback` 字段。模拟一次验证步骤失败的执行。展示回滚自动触发。

3. 阅读微软智能体框架的 `RequestInfoEvent` 文档。找出该API包含而示例引擎缺少的一个元数据字段。添加该字段并解释其防护的风险。

4. 为特定操作设计问答式清单（例如"发布到公开Twitter账户"）。审核者必须回答哪三个问题？为什么是这三个？

5. 选择一种同步"同意？"提示即可满足需求的场景（无需持久化存储）。解释原因，并说明您接受的风险类别。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|---|---|---|
| 提议-提交 | "两阶段审批" | 持久化提议 + 明确提交 + 验证 |
| 幂等性键 | "重试安全令牌" | 每个提议唯一；二次执行无操作 |
| 数据血缘 | "数据来源" | 导致提议的具体源内容 |
| 影响半径 | "最坏情况" | 操作出错时的影响范围 |
| 橡皮图章 | "快速批准" | 未经真实审核即点击"同意" |
| 问答确认 | "强制清单" | 审核者必须明确回答特定问题 |
| RequestInfoEvent | "微软智能体框架原语" | 带结构化元数据的持久化人机协作请求 |
| `interrupt()` / `waitForApproval()` | "框架原语" | LangGraph / Cloudflare 的等效实现 |
