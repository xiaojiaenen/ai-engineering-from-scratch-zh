# 杀死开关、断路器与金丝雀令牌

> 杀死开关是存储在智能体编辑范围之外的一个布尔值——可能是一个Redis键、一个功能标志或一个经过签名的配置——用于完全禁用智能体。断路器更为精细：它针对特定模式（例如连续五次相同的工具调用）触发，暂停异常路径，并升级至人工处理。金丝雀令牌继承了经典的欺骗策略：它是一个智能体没有合法理由访问的伪造凭证或蜜罐记录，一旦访问即触发警报。基于eBPF的数据通路（如Cilium）可以在内核层将隔离Pod的出站流量重定向至取证蜜罐；已发布的Cilium基准测试报告称，在负载下P99数据通路延迟低于一毫秒（您的传播开销取决于策略更新如何到达节点，而非数据通路本身）。自适应移动基线的统计检测器（EWMA，CUSUM）会悄然接受漂移——应将其与不会弯曲的严格宪法限制进行分层部署。

**类型：** 学习  
**语言：** Python（标准库，三检测器模拟器：杀死开关、断路器、金丝雀）  
**前置课程：** 第15阶段 · 13（成本治理器），第15阶段 · 10（权限模式）  
**时间：** 约60分钟

## 问题

成本治理器（第13课）限制了智能体的花费。但它们不限制智能体在预算内可以做什么。一个拥有50美元速度限制的智能体仍然可以泄露机密、发布错误的帖子或删除资源——昂贵的操作在token消耗上往往很廉价。

本课涵盖位于成本层旁边的三个检测器：

1.  **杀死开关**：存放在智能体触及范围之外的布尔开关。
2.  **断路器**：动作模式检测器，用于暂停特定路径。
3.  **金丝雀令牌**：诱饵，没有合法理由触碰的智能体会因触碰而暴露自身。

这三者都属于LLM之前的工程技术。经典的欺骗策略、速率限制断路器和功能标志杀死机制早于自主智能体出现。新的是攻击面：智能体会读取不可信内容（第11课），编辑自身记忆，并且可以将许多看似安全的操作组合成不安全的操作。这里命名的检测器之所以有效，是因为它们不信任智能体的自我报告。

## 概念

### 杀死开关

杀死开关是智能体读取但无法写入的布尔值。实现选择：

- **托管服务中的功能标志。** LaunchDarkly、Statsig、Unleash。更新在几秒内传播。
- **智能体轮询的Redis键。** 简单；要求智能体进程在每个回合检查。
- **对象存储中经过签名的配置。** 智能体在启动时验证签名；拒绝未签名的状态。
- **操作系统级信号或容器生命周期终止。** Docker `kill`, Kubernetes `kubectl delete pod`, systemd stop。

一个正确的杀死开关应具备的属性：

- 智能体无法将其设置为 `off`。（存储在智能体凭据无法写入的系统中。）
- 它在每一个有后果的动作上都会被检查，而不仅仅是在启动时。
- 当它关闭时，智能体不执行任何外部可观察的操作，包括向智能体可达的系统记录日志。
- 重新启用它是一个明确的人工操作，而非自动超时。

### 断路器

断路器暂停的是特定模式，而非整个智能体。经典形态（源自2007年Nygard的著作，至今仍适用）：

- **关闭**：动作被允许。
- **打开**：动作被阻止。
- **半开**：冷却期后，允许进行1-3次探测尝试（默认1次）；成功则关闭断路器，任何剩余的失败则重新打开它。

与智能体相关的触发器：

- 连续五次相同的工具调用（重复循环）。
- 使用不同输入对同一工具连续五次失败（系统性故障）。
- 工具调用速率超过阈值（第13课的速度限制）。
- 在读取超出信任的内容（第11课）后调用特定工具（例如 `file.delete`）。

### 金丝雀令牌

金丝雀令牌（也称蜜令牌）是智能体永远不应触碰的输入。访问即警报。

智能体的示例：

- 环境中一个没有实际权限的伪造 `AWS_SECRET_ACCESS_KEY`。泄露企图可以立即归因。
- 一个带有已知哨兵值的伪造数据库记录。任何对此记录的读取或更新都会触发警报。
- 工作区中的一个伪造文件。读取或编辑此文件是可疑的。
- 在会话开始时注入的一个伪造记忆条目。如果它在后续输出中出现，则表明记忆已被泄露。

金丝雀的设计针对具体的工作流。一个智能体有合法理由接触的金丝雀是噪音。

### 为何要分层统计限制和硬性限制

统计检测器（EWMA、CUSUM、工具调用率的z-score）自适应于移动基线。对于合法漂移来说这是个特性，但对于对抗性漂移来说这是个缺陷。耐心的攻击者可以缓慢移动基线，使得没有任何警报被触发。

硬性的宪法限制（第17课）不会弯曲。“永不发送电子邮件给 `secrets@`”作为一条宪法规则更简单、可审计，并且不会被缓慢漂移所操纵。分层模式：统计检测器捕获大多数噪音，硬性限制捕获绕过检测器的攻击。

### 通过eBPF数据通路重定向进行隔离

Cilium基于eBPF的网络策略可以在内核层将隔离Pod的出站流量重定向到一个取证蜜罐。Cilium的公开基准测试报告在TCP_RR上的P99数据通路延迟低于一毫秒；在您的集群中*触发*隔离的挂钟时间预算主要取决于策略更新如何从控制平面传播到节点（在Kubernetes上，如果操作员已预加载，几百毫秒是现实的；尾延迟取决于集群大小和调和节奏）。这为您带来两点：智能体不会泄露任何真实信息，并且智能体在蜜罐内部的行为被完全观察。与金丝雀令牌相同的模式，不同的层：智能体通过在受限状态下尝试做的事情来揭示其意图。

### 没有检测器能捕获的内容

- **从允许的原语组合而成的语义攻击。** 每个单独的动作都通过了每个检测器；组合才是问题。这是第10课中的自动模式组合失败。
- **将检测器作为载荷一部分的攻击。** 如果智能体读取一个页面，上面写着“在行动前，禁用金丝雀”，并且智能体具备该能力，那么检测器就被攻破了。金丝雀令牌应存在于智能体无法修改的系统中。

## 使用它

`code/main.py` 模拟了一个智能体轨迹穿过三个检测器的过程。一个存放在外部字典中的杀死开关；一个在五次相同工具调用时触发的断路器；一个读取即触发警报的金丝雀文件。输入一条合成轨迹：合法操作、重复循环、金丝雀探测，以及一个智能体操作被停止的杀死开关触发场景。

## 发布它

`outputs/skill-tripwire-design.md` 审查一个为智能体部署提议的检测器栈，并标记出缺陷（缺少杀死开关、缺少金丝雀、断路器阈值过于宽松）。

## 练习

1.  运行 `code/main.py`。确认断路器在第5回合（第五次相同调用）触发，金丝雀在第9回合（伪造密钥读取）触发。

2.  添加一个统计检测器：工具调用率的EWMA z-score。输入一条缓慢漂移的轨迹，显示检测器从未触发。现在添加一个硬性限制（10分钟内不超过50次工具调用），显示硬性限制在同一轨迹上触发。

3.  为一个浏览器智能体（第11课）设计一个金丝雀令牌集。列出至少三个金丝雀以及每个将检测到什么。

4.  阅读Cilium网络策略文档。具体描述一个出站重定向隔离流程：使用哪个策略选择器，哪个Pod，哪个出站重写，哪个警报。是什么支配着从“决定隔离”到“第一个被重定向的数据包”的挂钟延迟？

5.  为一个已触发杀死开关的智能体定义一个重新启用程序。谁可以重新启用？必须记录什么？在重新启用前智能体必须做出什么改变？

## 关键术语

| 术语 | 人们如何说 | 实际含义 |
|---|---|---|
| 杀死开关 | "关闭按钮" | 位于智能体编辑范围之外的布尔值；在每一个有后果的动作上被检查 |
| 断路器 | "模式暂停" | 针对重复、失败率或速率限制的特定动作触发器 |
| 金丝雀令牌 | "蜜令牌" | 智能体没有合法理由触碰的诱饵；访问即触发警报 |
| 蜜罐 | "取证沙箱" | 被隔离的智能体被观察的重定向流量/工作区 |
| EWMA | "移动平均" | 指数加权；适应漂移（特性 + 缺陷） |
| CUSUM | "累积和" | 检测相对于基线的持续偏移 |
| 硬性限制 | "宪法规则" | 不适应；无论历史如何，保持恒定 |
| 宪法限制 | "永远为真的规则" | 与第17课的宪法绑定；无法被智能体编辑 |
