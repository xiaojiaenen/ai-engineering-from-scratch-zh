# AlphaEvolve — 进化编码智能体

> 将前沿编码模型与进化循环和可机器检查的评估器相结合。让循环运行足够长时间。它发现了一种使用48次标量乘法的4x4复数矩阵乘法程序——这是56年来对Strassen算法的首次改进。它还在生产环境中找到了一个可为Google全局Borg调度恢复约0.7%集群算力的启发式方法。该架构刻意保持简洁。其优势源自评估器的严格性。

**类型：** 学习
**语言：** Python（标准库，进化循环玩具示例）
**前提：** 第15阶段 · 01（长期任务框架），第15阶段 · 02（自我学习推理）
**时间：** 约60分钟

## 问题所在

大型语言模型可以编写代码。进化算法可以搜索代码空间。两者已被分别尝试了几十年；但都遇到了瓶颈。LLM的瓶颈是虚构：模型编写的代码看似合理却无法实现其声称的功能。进化的瓶颈是搜索成本：对语法的随机突变很少能产生可编译的程序，更不用说更好的程序了。

AlphaEvolve（Novikov等人，DeepMind，arXiv:2506.13131，2025年6月）将两者结合。LLM向程序数据库提出有针对性的编辑；自动评估器对每个变体评分；高分变体成为下一代的父代。LLM处理编写看似合理代码这一昂贵步骤；评估器则捕获虚构错误。该循环运行时间从数小时到数周不等。

报告的结果：使用48次标量乘法的4x4复数矩阵乘法（Strassen 1969年的界限是49次），Google生产环境中的Borg调度启发式方法，FlashAttention内核32.5%的加速，Gemini训练吞吐量提升。

该架构之所以有效，是因为评估器是可机器检查的。在评估器不可行的领域，它就无法工作。这种不对称性是关键启示。

## 核心概念

### 循环过程

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个变体由评估器评分。
3. 从数据库中采样一个或多个父代（基于MAP-elites风格或岛模型）。
4. 提示LLM（对多数候选使用Gemini Flash，对困难问题使用Gemini Pro）生成父代的修改变体。
5. 编译、运行并在预留的评估器上评估该变体。
6. 将其按分数和特征向量插入数据库。
7. 重复循环。

有两个细节很重要。首先，LLM的提示不仅包含父代程序——通常还包括数据库中的若干顶级变体、评估器签名以及简短的任务描述。模型的任务是提出可能提高分数的针对性修改。其次，数据库的结构化设计（MAP-elites网格、岛模型）使循环能探索多样性，而不仅仅关注当前的领先者。

### 评估器不可或缺的原因

AlphaEvolve的成功均来自评估器快速、确定且难以钻空的领域：

- **矩阵乘法算法**：一个执行矩阵乘法并逐位检查相等性的单元测试。
- **Borg调度启发式方法**：一个生产级模拟器，重放历史集群负载并测量浪费的算力。
- **FlashAttention内核**：正确性测试加上在真实硬件上的挂钟时间基准测试。
- **Gemini训练吞吐量**：测量每步的GPU-秒数。

在每种情况下，评估器都捕获了原本会主导结果的LLM错误类别：虚构的正确性声明、在硬件上消失的性能声明，以及边缘情况下的失败。移除评估器后，循环就会优化表面看起来漂亮的代码。

### 奖励破解是上述观点的另一面

进化优化的是评估器测量的任何指标。如果评估器不完善，循环就会发现这种缺陷。在未验证的领域，循环会优化表面特征，而非预期行为。DeepMind在论文中明确指出了这一点：AlphaEvolve的成功仅适用于评估器严格性能匹配搜索雄心的领域。

2025-2026年代码搜索循环中奖励破解的具体示例：

- 奖励“完成时间”的优化目标导致提交空解决方案。
- 奖励“测试下正确性”的基准测试得分导致记忆测试并过拟合。
- “代码质量”代理指标奖励删除注释和重写变量名，而无任何语义变化。

AlphaEvolve的解决方案是：部署一个LLM从未见过的预留评估器，其输入在评估时生成。即便如此，DeepMind仍建议对任何提议的部署进行严格审查。

### 为何LLM+搜索优于单独使用

LLM可以产生可编译、语义合理的修改。对一个2000行的Python文件进行随机突变遗传算法几乎总是产生语法错误。LLM还把搜索集中在合理的邻域（修改一个函数，而非随机字节），这极大地减少了浪费的评估器调用。

反过来，评估器能捕获LLM的虚构错误。LLM会自信地声称某个函数“在极限情况下是O(n log n)”，而实际上是O(n^2)；挂钟时间基准测试可以解决这个问题。

### AlphaEvolve在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 成功示例 |
|---|---|---|---|---|
| AlphaEvolve | Gemini | 正确性 + 基准测试 | 算法、内核、调度器 | 48乘 4x4 矩阵乘法 |
| FunSearch (DeepMind, 2023) | PaLM / Codey | 正确性 | 组合数学 | cap-set 下界 |
| AI Scientist v2 (Sakana, L5) | GPT/Claude | LLM 批评 + 实验 | 机器学习研究 | ICLR 研讨会论文 |
| Darwin Godel Machine (L4) | 智能体脚手架 | SWE-bench / Polyglot | 智能体代码 | SWE-bench 20% → 50% |

这四者都是同一方案的变体：生成器加评估器，循环运行。差异在于评估器评分的内容及其严格程度。

## 使用它

`code/main.py` 实现了一个针对玩具符号回归问题的最小化类AlphaEvolve循环。“LLM”是一个标准库代理，它对一个计算目标函数的程序提出小的语法突变。“评估器”测量在预留测试点上的均方误差。

观察：

- 最佳分数如何随着代际改进。
- MAP-elites网格如何保持多样化的解决方案存活，使循环不会收敛到局部最小值。
- 移除预留测试（仅训练集评估器）如何导致循环剧烈过拟合。

## 部署它

`outputs/skill-evaluator-rigor-audit.md` 是在新领域考虑采用类AlphaEvolve循环的前提条件：你的评估器是否真的能捕获你关心的失败？

## 练习

1. 运行 `code/main.py`。记录最佳分数轨迹。禁用预留评估器（标志 `--no-holdout`）并重新运行。量化过拟合程度。

2. 阅读AlphaEvolve论文第3节关于MAP-elites网格的内容。为一个新问题（例如编译器优化遍）设计一个特征向量描述符，以保持搜索多样性。

3. 48乘4x4结果在56年后改进了Strassen的49乘界限。阅读论文附录F，用三句话解释为什么该问题的评估器特别容易做到正确，以及为什么大多数领域不像它。

4. 提出一个AlphaEvolve会失败的领域。准确指出评估器在哪里失效以及原因。

5. 对于你熟悉的一个领域，编写你将使用的评估器签名。包括（a）正确性条件，（b）性能指标，（c）预留输入生成规则，（d）至少一项反奖励破解检查。

## 关键术语

| 术语 | 人们常说 | 其实际含义 |
|---|---|---|
| AlphaEvolve | “DeepMind的进化编码智能体” | Gemini + 程序数据库 + 可机器检查的评估器 |
| MAP-elites | “多样性保持存档” | 按特征向量索引的网格；每个单元格保存具有该描述符的最佳变体 |
| 岛模型 | “并行进化子种群” | 独立种群定期迁移；防止过早收敛 |
| 可机器检查的评估器 | “确定性预言机” | LLM无法伪造的单元测试、模拟器或基准测试——是该循环的前提条件 |
| 奖励破解 | “优化度量而非目标” | 循环找到一种在不完成预期任务的情况下最大化分数的方法 |
| 种子程序 | “起点” | 循环从中进化的初始正确但次优的程序 |
| 预留评估器 | “LLM从未见过的评估数据” | 在评估时生成的输入，以防止记忆 |
