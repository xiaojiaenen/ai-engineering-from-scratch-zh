# 长期运行后台智能体：持久化执行

> 生产环境的长周期智能体不会在`while True`中运行。每个LLM调用都成为具有检查点、重试和重放机制的活动。Temporal的OpenAI Agents SDK集成已于2026年3月正式发布。Claude Code Routines（Anthropic）可运行计划任务中的Claude Code调用，而无需持久化本地进程。会话在人类输入时暂停，可跨部署存活，并能从基于`thread_id`标识的最新检查点恢复。这种新体验背后是经典模式——工作流编排——仅新增一个要素：LLM调用作为非确定性活动必须在恢复时确定性重放。

**类型：** 学习指南
**语言：** Python（标准库，最小化持久化执行状态机）
**前置要求：** 第15阶段·10（权限模式），第15阶段·01（长周期智能体）
**耗时：** 约60分钟

## 问题场景

假设一个智能体持续运行四小时。它调用三个工具，两次提示用户，并进行四十次LLM调用。运行中途，宿主机重启了。会发生什么？

- 在朴素`while True`循环中：所有数据丢失。运行完全重启。三个工具调用（具有实际副作用）会再次执行。用户将再次收到已批准事项的提示。四十次LLM调用会被重新计费。
- 采用持久化执行时：运行将从最近检查点恢复。已完成的活动不会重新执行；其结果会从持久化日志中重放。用户无需重新批准已批准事项。已进行的LLM调用不会重复计费。

这正是工作流引擎（Temporal、Cadence、Uber的Cherami）已沿用十年的模式。新的突破在于LLM调用现在成为一种活动——非确定性、成本高、具有副作用——且完美适配该模式。

本课核心主题：长周期可靠性会衰减（METR观察到“35分钟性能退化”——成功率随时间跨度呈近似平方衰减）。持久化执行支持超过可靠性曲线的运行时长，若设计得当可安全降级，若设计不当则会导致故障。

## 核心概念

### 活动、工作流与重放

- **工作流**：确定性编排代码。定义活动序列、分支结构和等待机制。必须保持确定性，以便从事件日志重放时不会产生意外偏差。
- **活动**：非确定性、可能失败的工作单元。如LLM调用、工具调用、文件写入、HTTP请求。每个活动会记录输入参数和（完成后）输出结果。
- **事件日志**：持久化存储后端。记录每次活动的启动、完成、失败、重试，以及每个工作流决策。
- **重放**：恢复时，工作流代码从头重新运行；所有已完成活动直接返回日志记录的结果而不实际执行。仅未完成的活动会被实际执行。

这与React基于虚拟DOM的重渲染、Git从提交重建工作树的模式相同。编排器的确定性是实现低成本持久化的关键。

### LLM调用为何契合此模式

LLM调用具有：
- 非确定性（温度>0；即使温度为0也会跨模型版本漂移）
- 高成本（金钱和延迟）
- 可能失败（速率限制、超时）
- 具有副作用（若调用工具）

这正是活动的典型特征。将每个LLM调用封装为活动可获得：指数退避重试、跨重启检查点，以及用于调试的可重放追踪。

### 基于`thread_id`的检查点机制

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects和Claude Code Routines都采用了相同的API形态：`thread_id`（或等效标识）定义会话范围；每次状态转换持久化到后端（默认PostgreSQL，开发环境用SQLite，缓存用Redis）；恢复时读取最新检查点。

后端选择至关重要：
- **PostgreSQL**：持久化、可查询、跨部署存活。LangGraph默认选项。
- **SQLite**：仅限本地开发；跨主机会丢失数据。
- **Redis**：快速但非持久化，除非配置AOF/快照。
- **Cloudflare Durable Objects**：透明分布式；基于唯一键作用域；可存活数小时至数周。

### 人类输入作为一等状态

提议-然后提交模式（第15课）需要持久化的“等待人类输入”状态。工作流暂停，外部队列暂存待处理请求，审批后将从精确断点恢复。缺乏持久化时这是尽力而为；具备持久化后，隔夜审批可至，工作流将在次日接续。

### 35分钟性能退化

METR观察到所有被测智能体类别在持续运行超过约35分钟后均表现出可靠性衰减。任务时长加倍会导致失败率近似四倍增长。持久化执行无法解决此问题；但它允许运行超过可靠性曲线的时长。安全模式是将持久化与需要重新进入时HITL的新鲜检查点结合，并采用预算熔断机制（第13课）限制总计算量。

### 持久化执行不适用场景

- 运行时间少于数分钟且无需人类输入。开销大于收益。
- 严格只读的信息检索任务。
- 正确性要求必须在单次上下文窗口内端到端完成的任务（部分推理任务；部分一次性生成）。

## 实践应用

`code/main.py`用标准库Python实现了最小化持久化执行引擎，支持：

- `@activity`装饰器：将输入输出记录到JSON事件日志
- 工作流函数：编排活动序列
- `run_or_replay(workflow, event_log)`函数：重放已完成活动而不实际执行

驱动程序模拟三活动工作流，中途崩溃演示：(a)朴素重试如何重新执行全部活动，(b)重放如何仅执行缺失活动。

1. 运行`code/main.py`。观察朴素重试与重放在活动执行次数上的差异。改变崩溃点并展示重放次数相应变化。

2. 将玩具引擎改造为显式使用`thread_id`。模拟两个并发会话共享引擎，确认其事件日志不会冲突。

3. 选取玩具引擎中的某个活动。引入非确定性因素（在工作流决策中使用系统时钟时间戳）。演示重放时产生的偏差。说明实际引擎如何处理（副作用注册、`Workflow.now()`API）。

4. 阅读LangChain“生产深度智能体背后的运行时”文章。列举运行时持久化的所有状态，并说明每种状态覆盖的故障模式。

5. 为6小时自主编码任务设计检查点策略。在何处设置检查点？崩溃恢复流程如何？哪些操作需要新鲜HITL？

## 核心术语

| 术语 | 常见说法 | 实际含义 |
|---|---|---|
| 工作流 | “智能体脚本” | 确定性编排代码；可从事件日志重放 |
| 活动 | “一个步骤” | 非确定性单元（LLM调用、工具调用）；前后均记录 |
| 事件日志 | “后端存储” | 每次状态转换的持久化记录 |
| 重放 | “恢复” | 重新运行工作流；已完成活动返回日志结果而不执行 |
| 检查点 | “存档点” | 基于thread_id的持久化状态；恢复时最新优先 |
| thread_id | “会话密钥” | 限定持久化状态范围的标识符 |
| 35分钟性能退化 | “可靠性衰减” | METR：成功率随时间跨度呈近似平方衰减 |
| 非确定性 | “重放漂移” | 系统时钟、随机数、LLM输出；必须注册为副作用 |
