# 检查点与回滚

> 每次图状态转换都会被持久化。当一个工作节点崩溃时，它的租约会过期，另一个工作节点会从最新的检查点处接续工作。Cloudflare Durable Objects 可以维持状态数小时乃至数周。提案-然后提交（第15课）为每个操作定义了一个回滚计划。操作后验证完成了闭环。欧盟AI法案第14条强制要求对高风险系统进行有效的人工监督——在实践中，这意味着检查点必须可查询，回滚必须经过演练，并且审计追踪必须能在一次部署后依然存在。一个尖锐的失败模式是：如果没有幂等性键和前置条件检查，在瞬时故障后的重试可能会导致一个已被批准的操作被重复执行。操作后验证正是为了捕获这种情况。

**类型：** 学习
**语言：** Python（标准库，检查点与回滚状态机）
**先决条件：** 第15阶段 · 12（持久化执行），第15阶段 · 15（提案-然后提交）
**时间：** ~60分钟

## 问题所在

持久化执行（第12课）使得崩溃的智能体能够恢复。提案-然后提交（第15课）使得已批准的操作可审计。本课将它们结合起来：当一个已批准的操作部分执行、崩溃然后恢复时，会发生什么？回滚何时执行？针对的是什么状态？

现实系统中对此的实现方式各不相同：

- **LangGraph** 将每次图状态转换都检查点化到 PostgreSQL。当工作节点崩溃时，租约会释放，另一个工作节点会从最新的检查点处恢复工作。工作流会在 `interrupt()` 处暂停，该状态本身也会被持久化。
- **Cloudflare Durable Objects** 可以按键维持状态数小时乃至数周。将计算与已批准操作的存储位置放在一起。
- **Microsoft Agent Framework** 在工作流 API 中暴露 `Checkpoint` 原语；重放加上幂等性可以覆盖重试。

在所有情况下，真正有效的组合是：幂等性键（防止重复执行） + 前置条件检查（状态仍与我们批准时的状态一致） + 操作后验证（副作用确实发生了） + 验证失败时回滚。

## 核心概念

### 每次转换都会持久化

图状态转换是任何将工作流从一个命名状态移动到另一个状态的步骤。朴素的实现只在特定提交点进行持久化；生产实现则会持久化每次转换。其代价（多一些写入）相对于可靠性收益（重放可以落到任何地方，租约恢复更精确）来说很小。

### 租约恢复

当一个工作节点崩溃时，工作流不会丢失；租约（一个表明此工作节点正在执行此次运行的短暂声明）只是简单地过期。另一个工作节点接续最新的检查点并恢复。租约机制使得生产系统能够在滚动部署中幸存，而不会丢失正在进行的工作。

### 幂等性加上前置条件

仅有幂等性是不够的。考虑这种情况：一个工作流被批准“在余额 > $1000 时从 A 转账 $100 给 B”。工作流被提交，在执行中途崩溃，然后恢复。如果只检查了幂等性键，且执行恢复，转账会执行一次（正确）。但考虑在崩溃和恢复之间，A 的余额通过另一个不同的工作流降到了 $500。幂等性检查仍然通过；但前置条件不再满足。如果没有前置条件检查，我们就会导致透支。

每个重要的操作都需要两者兼具：

- **幂等性键**：防止重复执行。
- **前置条件检查**：确认状态仍与批准时的状态一致。

### 操作后验证

“工具返回了 200”并不是验证。真正的验证是重新读取目标状态并确认副作用确实发生了。模式包括：

- 数据库更新：`UPDATE ... RETURNING *`，然后断言返回的行与预期状态匹配。
- 发送邮件：提交后检查已发送邮件文件夹中是否存在该消息 ID。
- 写文件：读回文件并计算其哈希值。
- API 调用：对目标资源进行后续的 `GET`。

如果验证失败，工作流就处于已知的不良状态。回滚机制启动。

### 回滚计划

提案-然后提交（第15课）中的每个重要操作都附带一个回滚计划。类型包括：

- **带内回滚**：直接逆转副作用（`INSERT` 之后 `DELETE`，发送之后 `Send-correction-email`）。
- **补偿性事务**：一个新的操作来抵消原操作（标准的 SAGA 模式）。
- **带外回滚**：警示人工，暂停工作流，保留不良状态以供调查。

空操作回滚（“我们无法撤销此操作”）必须在提案中说明。没有回滚的操作在提交时需要更强的人机交互（第15课的挑战-响应）。

### 欧盟AI法案第14条操作性解读

第14条要求对高风险系统进行“有效的人工监督”。在操作层面，实施者将其解读为：

- 检查点可被审计员查询。
- 回滚经过演练（至少端到端测试过一次）。
- 审计追踪能在一次部署后幸存（检查点后端非临时性）。
- 失败的验证会被警示，而不是静默记录。

一个在提交中途崩溃、恢复并完成副作用却没有验证 + 回滚路径的工作流，无法通过第14条的测试。

### 尖锐的失败模式：双重执行

该领域最常见的生产事故：

1. 操作被批准，幂等性键为 k。
2. 提交开始，执行，返回 200。
3. 工作流在持久化“已提交”状态之前崩溃。
4. 工作流恢复；看到“已批准但未提交”；重新执行。
5. 副作用触发两次。

缓解措施：在执行前持久化一个“进行中”的意图，使用幂等性键执行，然后仅在操作后验证成功后才标记为“已提交”。如果操作触发了但状态写入失败，你知道需要验证并（如果有必要）重新触发。如果状态写入成功但操作失败，你通过恢复路径进行验证并恰好触发一次。

## 练习

1.  运行 `code/main.py`。验证四种场景。对于提交期间崩溃的情况，确认在重试过程中操作恰好触发一次。

2.  修改“先标记为完成，然后去做”的模式，使状态写入在操作之后触发。重新运行崩溃场景。测量有多少重复的操作被触发。

3.  为一个特定的生产操作（例如，“发布到 Slack 频道”）设计一个回滚计划。将其分类为带内、补偿或带外。说明选择理由。

4.  选择一个你了解的工作流。标识出每个状态转换。标记每个转换的持久化要求（需要持久化 / 不需要持久化）。计算出你当前没有进行持久化的数量。

5.  回滚演练测试：设计一个端到端测试，该测试运行一个真实的工作流，使其崩溃，并确认回滚路径被触发。该测试应断言什么？

## 关键术语

| 术语 | 人们的说法 | 其实际含义 |
|---|---|---|
| Checkpoint | “保存点” | 每次图状态转换都持久化到持久存储 |
| Lease | “工作者声明” | 一个表明工作者正在执行某次运行的短暂声明；崩溃时过期 |
| Precondition | “状态门控” | 断言状态仍与已批准的操作一致 |
| Post-action verify | “重读检查” | 确认副作用确实在目标系统中发生 |
| In-band rollback | “直接撤销” | 使用逆操作逆转副作用 |
| Compensating transaction | “SAGA撤销” | 一个抵消原操作的新操作 |
| Mark-as-done-first | “状态写入顺序” | 在从提交返回前持久化已提交状态 |
| Article 14 | “欧盟AI法案人工监督” | 操作层面：可查询的检查点、演练过的回滚、可审计的追踪 |
