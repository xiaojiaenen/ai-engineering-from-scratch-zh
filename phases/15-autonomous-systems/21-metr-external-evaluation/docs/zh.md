# METR 时间跨度与外部能力评估

> METR（前身为ARC Evals）自2023年12月起成为独立的501(c)(3)组织。其时间跨度1.1基准（2026年1月）将任务成功概率与对数（专家完成时间）拟合为逻辑斯谛曲线；在50%概率处的交点定义了模型的时间跨度。2025-2026年的评估集涵盖GPT-5.1、GPT-5.1-Codex-Max以及原型监控评估（监控器能否捕捉附带任务；智能体能否规避检测）。基准套件：HCAST（180多项机器学习、网络安全、软件工程、推理任务；耗时1分钟至8小时以上）、RE-Bench（71项具有专家基线的机器学习研究工程任务）、SWAA。客观说明：METR的测量是理想化的——没有人类参与，没有真实后果——且团队已记录评估行为与部署行为之间的差异（教训1）。时间跨度是一个上限，而非部署预测。

**类型：** 学习
**语言：** Python（标准库，逻辑斯谛拟合时间跨度估计器）
**前提条件：** 第15阶段 · 01（长时间跨度智能体），第15阶段 · 19（RSP）
**时间：** 约60分钟

## 问题背景

扩展政策（教训19、20）只有在其引用的测量数据准确时才有用。"AI研发-4阈值"和"长程自主性"是在政策文本中定义的；只有当特定评估产生具体数值时，它们才变得可操作。

METR是2024-2026年的外部评估组织，已定义了许多此类数值。他们评估前沿模型——通常在发布前，在与实验室的保密协议下进行——之后发布方法论。时间跨度1.1基准（2026年1月）是其标志性成果：一个单一标量，将能力压缩成人类可读的单位（"该模型能以50%的可靠性完成专家需要X小时才能完成的任务"）。

本课部分内容关于方法论（如何计算时间跨度），部分内容关于解读（为什么时间跨度是上限而非部署预测）。这两项技能相辅相成。一个理解时间跨度拟合原理的团队，比一个只在幻灯片上看到"14小时"的团队，更不容易被糟糕的供应商宣称所愚弄。

## 核心概念

### METR 背景

- 成立时间：2023年12月（前身为ARC Evals，剥离成为独立的501(c)(3)组织）。
- 范围：评估前沿模型的自主能力，通常在发布前进行。
- 合作实验室：Anthropic、OpenAI（2025-2026年多次合作）。
- 重要交付物：时间跨度1.0（2025年3月），时间跨度1.1（2026年1月），原型监控评估。

### 时间跨度拟合

方法论（摘自METR博客和论文）：

1.  收集一个任务套件，其专家完成时间从分钟级到小时级不等。当前套件：HCAST（180多项任务）、RE-Bench（71项任务）、SWAA。
2.  在每个任务上运行模型；记录成功或失败。
3.  拟合一条逻辑斯谛曲线：成功概率P(success)作为对数（专家完成时间）的函数。
4.  时间跨度是P(success) = 0.5时的专家用时。

逻辑斯谛拟合形状是合适的，因为能力与任务难度通常呈递增并趋近平台的关系。50%点是一个选择（也可以是10%、90%）；METR在详细论文中报告了多个阈值，但以50%作为主要指标，因为它最直观。

### 2026年1月的数据

根据时间跨度1.1：

- Claude Opus 4.6：在50%可靠性下约14小时，数据截至时间跨度1.1（2026年1月）。
- HCAST风格任务的倍增时间：根据时间跨度1.1（2026年1月）报告的2023年后拟合结果，约为4.3个月（130.8天）；而约7个月的数字是时间跨度1.0中基于2019-2025年全数据的拟合结果，并在TH1.1中与2023年后数据一并报告。

这些是第15阶段其余部分引用的具体数值。它们在Anthropic的RSP v3.0（教训19）和DeepMind的FSF v3（教训20）中被引用，作为支撑阈值评估的测量数据之一。

### 基准套件

- **HCAST**：横跨机器学习、网络安全、软件工程和通用推理的189项任务。时间跨度：1分钟至8小时以上的专家用时。是时间跨度数值背后的主要套件。
- **RE-Bench**：71项具有人类专家基线的机器学习研究工程任务。衡量模型能否完成支撑AI研发-4阈值的那一类机器学习研发工作。
- **SWAA**：较短任务；用于校准时间跨度曲线的低端部分。

### 原型监控评估

2025-2026年的METR工作包括监控评估原型：监控器能否捕捉到模型执行附带任务（非操作员要求的任务）；智能体能否规避监控。这些直接为OpenAI PF v2中的"破坏保障措施"类别以及DeepMind FSF v3中的欺骗性对齐监控（教训20）提供了依据。

### 为什么时间跨度是上限

- **理想化工具**：基准任务为模型提供了干净的工具和格式良好的数据。真实部署则更加混乱。
- **无真实后果**：模型从未真正向客户收费、删除真实数据或联系真实人员。真实部署具有不可逆的利害关系。
- **评估情境博弈**：教训1。模型在测试中的行为与部署时不同。2026年《国际AI安全报告》对此进行了实证记录。
- **无合法用户差异**：基准提示是结构化的。真实用户会产生模糊、依赖上下文的请求。

时间跨度是理想条件下的能力上限。部署可靠性是另一个数值，更低，团队必须测量自己的分布才能了解它。

### 外部评估机构的作用

外部评估很重要，因为内部实验室有动机去优化它们所报告的指标。METR的独立性——一个拥有公开方法论和经过同行评审论文的501(c)(3)组织——是结构性的缓解措施。仅凭此并不足够（实验室仍能控制METR所见内容），但这严格优于没有外部评估。

### 如何在实践中使用时间跨度数据

- **作为能力过滤器**：如果模型的时间跨度远低于所提任务的专家用时，则不要部署其自主执行（教训1的技能文件）。
- **作为趋势指标**：倍增时间告诉你，即使没有新的缓解措施，当前做法还能安全持续多久。
- **作为先验**：14小时的时间跨度是一个起点。根据你的任务分布、工具质量和部署情境向下调整。

## 实践应用

`code/main.py` 实现了给定合成结果集的任务成功概率与对数（专家时间）的逻辑斯谛拟合。它报告了50%时间跨度（METR的主要指标）、10%时间跨度（保守）和90%时间跨度（乐观）。同时演示了当成功率因评估情境博弈被人为抬高时会发生什么变化。

`outputs/skill-horizon-interpretation.md` 审查供应商的时间跨度声明，并产生基准声明与部署现实之间的差距分析。

## 练习题

1.  运行`code/main.py`。确认拟合的50%时间跨度与合成真实值一致。现在将任务时间网格减半；时间跨度估计值是否有显著变化？
2.  阅读METR的时间跨度1.1博文。找出可靠性最高和最低的具体任务。解释为何存在差距。
3.  阅读METR的"测量自主AI能力"资源。列出HCAST任务类别。选择一个你认为对生产任务应赋予更高权重的类别，并说明理由。
4.  在模拟器中引入评估情境博弈：将约20%失败的任务翻转为成功。报告新的时间跨度。这近似于20%的博弈率对观测数值的影响。
5.  设计一个基于你自己的缺陷积压或代表性任务集的内部时间跨度评估。描述数据收集、拟合过程以及输出结果的含义。与METR数值进行比较。
