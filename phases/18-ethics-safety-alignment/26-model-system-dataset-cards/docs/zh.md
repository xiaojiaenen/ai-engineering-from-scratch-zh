# 模型卡、系统卡与数据表

> 三种文档格式构筑了AI透明度体系。模型卡（Mitchell等人，2019年）—— 模型的“营养标签”：涵盖训练数据、量化分类分析、伦理考量、注意事项；但Hugging Face上仅0.3%的模型卡记录了伦理考量（Oreamuno等人，2023年）。数据集的数据表（Gebru等人，2018年，CACM）—— 包含动机、构成、收集过程、标注、分发、维护；类比电子元件数据表。数据卡（Pushkarna等人，Google，2022年）—— 模块化分层细节（望远镜级、潜望镜级、显微镜级），作为面向多元读者的边界对象。2024-2025年新进展：通过大语言模型自动生成（CardGen，Liu等人，2024年）；模型卡详细度与Hugging Face下载量最高提升29%相关（Liang等人，2024年）；可验证声明（Laminator，Duddu等人，2024年）；新增碳足迹/水足迹的可持续性报告（Jouneaux等人，2025年7月）；欧盟/ISO监管性卡片正在涌现。系统卡（Sidhpurwala，2024年；Meta系统级透明度；“信任蓝图”arXiv:2509.20394）—— 覆盖安全能力、提示词注入防护、数据窃取检测、与人类价值观对齐的端到端AI系统文档。

**类型:** 构建
**语言:** Python（标准库，模型卡 + 数据表 + 系统卡生成器）
**先决条件:** 第18阶段·第18课（安全框架），第18阶段·第24课（法规）
**时间:** 约60分钟

## 学习目标

- 描述Mitchell等人（2019年）原始模型卡及Gebru等人（2018年）数据表。
- 描述数据卡的望远镜/潜望镜/显微镜分层结构。
- 描述系统卡及其端到端覆盖范围。
- 列举三项2024-2025年新进展（自动化生成、可验证声明、可持续性报告）。

## 问题所在

监管框架（第24课）与实验室安全政策（第18课）均要求文档记录。文档格式从模型专属（模型卡）演进至数据集专属（数据表），再到系统专属（系统卡）。每种格式解决不同范围的透明度问题。2024-2025年的自动化与可验证声明研究，针对的是长期存在的采用率问题。

## 核心概念

### 模型卡（Mitchell等人，2019年）

章节：
- 模型详情。
- 预期用途。
- 因子（评估中相关的人口统计学或环境因子）。
- 指标。
- 评估数据。
- 训练数据。
- 定量分析（按因子分类）。
- 伦理考量。
- 注意事项与建议。

采用率问题：Oreamuno等人（2023年）对Hugging Face模型卡的审计发现，仅0.3%记录了伦理考量。

### 数据集的数据表（Gebru等人，2018年）

类比电子元件数据表。章节：
- 动机（为何创建该数据集）。
- 构成（包含什么内容）。
- 收集过程（如何组装）。
- 标注（如适用）。
- 用途（预期、禁止、风险）。
- 分发。
- 维护。

发表于CACM 2021。数据表是上游文档；模型卡依赖于数据表的准确性。

### 数据卡（Pushkarna等人，Google，2022年）

模块化分层细节。三个缩放级别：
- **望远镜级。** 面向非专家的高层摘要。
- **潜望镜级。** 面向ML实践者的中层概述。
- **显微镜级。** 面向审计员的详细特征级文档。

边界对象框架：不同读者从同一文档中提取不同信息。

### 系统卡

范围：端到端AI系统，包括模型 + 安全栈 + 部署环境。章节通常包括：
- 安全能力。
- 提示词注入防护。
- 数据窃取检测。
- 与声明的人类价值观对齐。
- 事件响应。

Sidhpurwala（2024年）与Meta系统级透明度研究。“信任蓝图”（arXiv:2509.20394）将系统卡正式定义为模型卡在部署层的补充。

### 2024-2025年新进展

- **CardGen（Liu等人，2024年）。** 通过大语言模型自动生成模型卡；报告显示其在标准化的Mitchell 2019字段上比许多人工编写的模型卡具有更高客观性。
- **下载量相关性（Liang等人，2024年）。** 详细的模型卡与Hugging Face上最高29%的下载率提升相关——采用压力现在是市场驱动，而不仅是合规驱动。
- **Laminator（Duddu等人，2024年）。** 通过硬件TEE / 密码学签名实现可验证声明——使模型卡能够携带声明证明，而不仅仅是声明本身。
- **可持续性（Jouneaux等人，2025年7月）。** 新增碳、水及计算能耗足迹；新兴ISO标准。
- **监管性卡片。** 欧盟AI法案（第24课）GPAI实践守则透明度章节将模型卡作为合规产物。

### 在第18阶段中的位置

第24-25课是监管与CVE层。第26课是文档层。第27课是训练数据治理，即数据表的上游。第28课是产生卡片所引用评估的研究生态系统。

## 实践应用

`code/main.py` 为一个玩具部署生成一个最小化的模型卡、数据表和系统卡。每个都遵循规范的章节结构。您可以检查格式并比较三种范围。

## 部署交付

本课产出 `outputs/skill-card-audit.md`。给定一个模型卡、数据表或系统卡，它将审计章节覆盖范围、数值分类情况以及是否存在可验证声明。

## 练习

1. 运行 `code/main.py`。检查生成的卡片。识别薄弱的章节（仅含占位符）并说明需要什么证据来加强它们。

2. 扩展模型卡，加入对两个人口统计学群体的量化分类分析（第20课）。

3. 阅读Oreamuno等人（2023年）关于0.3%采用率的研究。提出一个对模型卡规范的结构性修改，以提高伦理考量的采用率。

4. Laminator（Duddu等人，2024年）使用TEE进行可验证声明。设计一个模型卡字段，用于携带评估结果的密码学证明，并描述验证者的角色。

5. 为您过去的一个项目或一个假设的部署编写一个系统卡（系统卡，而非模型卡）。识别对第三方审计员价值最高的章节。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 模型卡 | “米切尔卡” | Mitchell等人（2019年）提出的ML模型标准文档 |
| 数据表 | “盖布数据表” | Gebru等人（2018年）提出的数据集标准文档 |
| 数据卡 | “普什卡纳卡” | Google（2022年）提出的模块化分层数据文档 |
| 系统卡 | “部署卡” | 覆盖安全栈的端到端AI系统文档 |
| 边界对象 | “不同读者，同一文档” | 数据卡的框架：同一文档服务于多元受众 |
| 可验证声明 | “Laminator声明” | 附加在文档声明上的密码学或TEE证明 |
| 可持续性字段 | “碳/水足迹” | 2025年新兴的用于环境核算的字段 |
