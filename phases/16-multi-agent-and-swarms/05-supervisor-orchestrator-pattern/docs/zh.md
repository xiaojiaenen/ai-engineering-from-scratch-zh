# 监督者/工作者-工作者模式

> 一个主导智能体进行规划和委派；专业化的工作者在并行上下文中执行任务并汇报结果。这是 Anthropic Research 系统（以 Claude Opus 4 为主导智能体，Sonnet 4 为子智能体）背后的模式，在内部研究评估中相比单一智能体的 Opus 4 提升了 +90.2%。Anthropic 的工程帖子指出，BrowseComp 上 80% 的方差可以仅通过 token 使用量来解释——多智能体的优势主要源于每个子智能体都能获得全新的上下文窗口。本课程将从基础原语构建监督者模式，并涵盖 2026 年生产部署中积累的工程经验。

**类型：** 学习 + 构建
**语言：** Python (标准库, `threading`)
**前置课程：** 第 16 阶段 · 04 (基础模型)
**时间：** ~75 分钟

## 问题

研究是单一智能体系统失败的典型任务。你问“2023 到 2026 年间多智能体系统发生了什么变化？”单一智能体按顺序阅读五篇论文，将上下文的一半填满它们的文本，然后必须综合推理所有内容。当它读到第五篇时，已经忘记了第一篇的内容。它无法并行化。

监督者模式解决了这个问题：一个主导智能体规划搜索，将每个子问题委派给一个工作者，然后进行综合。每个工作者为一个细分问题获得自己的 20 万 token 窗口。主导智能体从不看原始论文——只看到工作者的摘要。

Anthropic 的生产级 Research 系统报告显示，相比单一的 Opus 4，在内部研究评估中取得了 +90.2% 的提升。同一帖子指出，BrowseComp 上 80% 的方差可以仅通过 *token 使用量* 来解释。为每个子智能体提供全新上下文是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   Lead       │  plans, decomposes,
                 │  (Opus 4)    │  synthesizes
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         fresh       fresh        fresh
         context     context      context
```

主导智能体从不阅读原始材料。工作者们在主导智能体综合之前也看不到彼此的工作。每个箭头都是一次带有细分产出物的交接。

### 为什么它有效

三个机制：

1.  **每个子智能体拥有全新上下文。** 一个探索“FIPA-ACL 传承”的工作者，不会承载主导智能体在规划时已经花掉的 4 万个 token。它为一个问题获得一个 20 万 token 的窗口。
2.  **通过提示词进行专业化。** 主导智能体的提示词是“分解与综合”，而不是“研究”。每个工作者的提示词是细分的：“找出 X 方面的变化”。专注的提示词产生专注的输出。
3.  **并行性。** 工作者并发运行。总耗时大约是 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。

### 工程经验（Anthropic 2025）

Anthropic 的帖子列出了几条在 2026 年仍然适用的生产级经验：

- **根据查询复杂度调整投入。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导智能体必须评估这一点，而不是由调用者决定。
- **先广后深。** 首先分解为广泛的子问题，然后如果答案需要深入，再为每个子问题生成更多工作者。
- **彩虹部署。** 智能体是长时间运行且有状态的。传统的蓝绿部署不适用。Anthropic 使用彩虹式部署：在旧版本排空时逐步推出新版本。
- **Token 使用量主导。** 多智能体使用的 token 大约是单一智能体的 15 倍。仅当任务价值能证明成本合理时才运行它。

### LangGraph 的转向

LangGraph 最初发布了一个 `langgraph-supervisor` 库，其中包含一个高级 `create_supervisor` 辅助函数。在 2025 年，LangChain 将建议改为直接通过工具调用实现监督者模式，因为工具调用对 *监督者能看到什么*（上下文工程）有更多控制。该库仍然可用；文档现在推荐工具调用的形式。

### 失败模式

- **主导智能体虚构计划。** 如果主导智能体生成的子问题没有分解真实问题，工作者就会对错误的目标进行精确研究。
- **工作者过度探索。** 如果没有明确的范围边界，工作者会偏离分配的子问题，并污染综合步骤。
- **综合冲突。** 两个工作者返回了矛盾的事实。主导智能体必须要么重新提问（增加一轮），要么明确指出分歧。默默地选择一方是最坏的失败：用户永远不知道存在分歧。

### 何时不适合使用监督者

- **顺序任务。** 如果步骤 2 字面上需要步骤 1 的输出，并行性就没有收益。请使用流水线（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单一智能体处理它们更快且更便宜。在生成工作者之前，使用主导智能体的“调整投入”检查。
- **严格的确定性。** 监督者使用 LLM 选择的委派。当审计/回放比适应性更重要时，静态图更好。

## 构建它

`code/main.py` 使用 `threading` 实现了一个包含三个并行工作者的监督者。主导智能体将查询分解为子问题，工作者在每个子问题上并发运行，然后主导智能体进行综合。没有真实的 LLM——工作者被脚本化以模拟获取和摘要。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假的摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者，等待它们完成，并进行综合。

运行：

```
python3 code/main.py
```

输出显示了计划、带有开始/结束时间戳的并行工作者轨迹，以及最终的综合结果。你可以看到总耗时的优势：三个耗时 0.3 秒的工作者运行了大约 0.35 秒，而不是 0.9 秒。

## 使用它

`outputs/skill-supervisor-designer.md` 接受一个用户查询并生成一个监督者模式的设计：主导系统提示词、工作者角色、子问题分解规则以及综合模板。在构建新的研究型智能体系统之前使用它。

## 部署检查清单

部署监督者模式前的检查清单：

- **模型配对。** 主导智能体使用推理级模型（Opus 类，`o3` 类）。工作者使用更快、更便宜的模型（Sonnet，`o4-mini`）。
- **工作者超时。** 任何运行时间超过中位运行时间 2 倍的工作者会被终止；主导智能体要么以更窄的范围重新生成，要么在没有它的情况下继续。
- **每个工作者的 Token 上限。** 硬性限制（例如，预期综合输入量的 10 倍）可以防止失控的工作者耗尽预算。
- **可观测性。** 追踪主导智能体的计划、每个工作者的工具调用以及综合过程。这是任何事后调试的基础。
- **彩虹式部署。** 有状态的长时间运行智能体需要渐进的版本过渡，而不是热替换。

## 练习

1.  运行 `code/main.py`，然后将主导智能体修改为生成 5 个工作者而不是 3 个。观察总耗时效果。在这个演示中，当工作者数量达到多少时，生成开销会超过并行节省的时间？
2.  实现工作者超时：终止任何运行超过 0.5 秒的工作者，并让主导智能体综合剩余的结果。你需要哪些可观测性信息来知道一个工作者被中断了？
3.  在主导智能体的综合步骤中添加冲突检测：如果两个工作者返回矛盾的答案，主导智能体记录分歧而不是选择其中一个。如何在不调用 LLM 的情况下检测矛盾？
4.  阅读 Anthropic 的 Research 系统工程帖子。列出这个玩具演示若要在生产中运行需要采用的三种实践。
5.  比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用推荐。哪一种能让你更好地控制监督者所看到的内容？为什么 Anthropic 在综合时明确只传递子答案，而不是工作者的原始上下文？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|------------|----------|
| Supervisor | “主导智能体” | 一个负责规划、委派和综合的编排智能体。它自己不做具体工作。 |
| Worker | “子智能体” | 一个由监督者调用的、具有狭窄范围和自身上下文窗口的专注型智能体。 |
| Orchestrator-worker | “监督者模式” | 同一事物，不同名称。2026 年的文献两者都用。 |
| Fresh context | “干净的窗口” | 一个工作者的上下文从其系统提示词和分配的问题开始，而不是来自主导智能体的历史记录。 |
| Rainbow deployment | “渐进式部署” | 长时间运行的有状态智能体需要版本化的排空替换，而不是蓝绿部署。 |
| Token dominance | “上下文是变量” | 根据 Anthropic 的数据，研究评估中 80% 的方差来自使用的总 token 量，而不是模型选择。 |
| Scale effort | “根据复杂度匹配智能体数量” | 主导智能体评估查询难度，并相应生成 1 个或 10+ 个工作者。 |
| Synthesis conflict | “工作者意见不一” | 两个工作者返回了矛盾的事实；主导智能体必须揭示分歧，而不是默默地选择一个。 |
