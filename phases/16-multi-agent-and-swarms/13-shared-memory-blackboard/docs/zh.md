# 共享内存与黑板模式

> 2026年多智能体系统中并存着两种主要架构：**消息池**（所有成员可见全部消息，如AutoGen GroupChat或MetaGPT）和**订阅式黑板**（智能体仅订阅相关事件，如CA-MCP或Matrix框架）。这两者是多智能体系统中唯一具有状态的组件——这意味着所有有趣的缺陷都藏在这里。典型的故障模式是**内存污染**：某个智能体虚构了一个“事实”，其他智能体将其视为已验证信息，导致准确性逐渐衰减，这种衰减比系统崩溃更难调试。本课程将使用标准库从零构建这两种架构，注入污染攻击，并展示生产环境中真正有效的三种缓解措施。

**类型：** 学习+构建  
**语言：** Python（标准库，`threading`）  
**前置条件：** 第16阶段第4课（基础模型），第16阶段第9课（并行集群网络）  
**时间：** 约75分钟

## 问题背景

多智能体系统需要让智能体共享事实信息。一种方案是“在消息中传递所有内容”——但这等同于通过额外拷贝重新发明共享状态。另一种方案是“给每个成员提供全局日志”——但全局日志会无限增长且易受污染。第三种方案是“为每个智能体投影视图”——可扩展但需要复杂模式设计。

当某个智能体产生幻觉并将幻觉写入共享状态时，所有读取该状态的下游智能体都会将幻觉视为事实。当人类察觉时，推理链已深入五层，而根本原因可能是系统写入的第三条消息。调试多智能体系统的准确性衰减比调试崩溃更困难。

这就是内存污染。它是MAST分类法（Cemri等，arXiv:2503.13657）中记录第二多的故障类别，且具有结构性特征：任何缺乏溯源机制和不可写验证器的共享内存设计最终都会出现此问题。

## 核心概念

### 两种主要拓扑结构

**全消息池**：每个智能体读取所有消息。AutoGen GroupChat和MetaGPT采用此模式。简单透明、便于检查，但无法扩展至超过10个智能体，因为每个智能体的上下文都会被其他智能体的工作内容填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**订阅式黑板**：智能体声明对特定主题的关注；基础设施仅路由相关消息。CA-MCP（arXiv:2601.11595）和去中心化框架Matrix（arXiv:2511.21686）采用此模式。扩展性更强，但需要预先设计模式定义才能使订阅具有实际意义。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各自适用场景

- **全消息池**在以下情况胜出：智能体数量少（<10）、异构性强、对话生命周期短。当所有人都能看到全部信息时，追溯发言内容非常简单。
- **黑板模式**在以下情况胜出：智能体数量多、角色同质但实例众多（集群）、对话长期运行。路由机制可节省token消耗并减少上下文污染。

生产系统常采用混合架构：顶层使用小型全消息池（规划层），下层使用黑板（工作层）。

### 内存污染场景示例

三个智能体协同完成研究任务：智能体A负责检索，智能体B负责总结，智能体C负责分析。

1. A获取网页并向共享状态写入消息：“研究报告显示准确率提升42%。”
2. 实际获取的网页内容为“提升4.2%”。A虚构了小数点。
3. B读取共享状态后写入：“报告显著提升42%准确率（来源：A）。”
4. C读取共享状态后写入：“建议采用——42%的提升具有变革性意义。”
5. 最终报告引用了从未存在的42%数据。

没有智能体崩溃，没有测试失败。系统“正常工作”。通过共享状态，幻觉从一个智能体的上下文传播至所有下游智能体的推理过程。

### 结构性缺陷的本质

在无共享状态的情况下，智能体A的幻觉仅存在于其自身上下文中。下游智能体会重新获取或重新推导，可能发现错误。在原始共享状态下，A的上下文成为所有人的上下文，幻觉被“洗白”为事实。

问题不在于共享状态本身，而在于**缺乏溯源机制和独立验证器的共享状态**。三种缓解措施可应对此问题：

1. **每次写入都记录溯源信息**：共享状态中的每个条目记录写入者、时间、触发提示词及（如适用）引用的原始来源。下游智能体根据溯源信息谨慎读取。
2. **版本化写入；视为仅追加**：修正是取代旧条目的新条目，而非原地更新。完整审计轨迹得以保留。
3. **至少保留一个无法写入共享状态的智能体**：只读验证器智能体采样条目，重新获取来源并标记不一致之处。由于无法写入消息池，因此不会被消息池污染。

### 黑板模式的先驱（Hayes-Roth，1985）

黑板模式比LLM智能体早出现四十年。Hayes-Roth（1985年《控制用黑板架构》）描述了专家知识源观察全局黑板、贡献部分解决方案并触发其他知识源的机制。2026年的黑板（CA-MCP、Matrix）本质相同模式，只是将LLM智能体作为知识源，JSON数据作为部分解决方案。早期文献已记载了现代系统重新发现的写入竞争、机会控制和一致性解决方案。

### 投影视图与完整视图

纯黑板为所有订阅者提供相同的投影视图（主题范围限定）。更激进的设计是**智能体专属投影视图**：每个智能体获得根据其角色定制的视图。LangGraph的状态归约器是2026年的标准实现——归约函数将全局状态折叠为角色特定切片。

智能体专属投影视图扩展性更强但需要模式定义。若无模式定义，每个智能体都需要在提示词中重建临时投影。

### 写入竞争模式

多个智能体同时写入是并发问题，而非LLM特有问题。三种有效模式：

- **顺序写入者（单生产者）**：所有写入通过一个协调智能体序列化。简单但存在瓶颈。
- **带版本的乐观并发**：每个条目带有版本号；写入者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区**：不同智能体负责不同主题。无跨主题竞争。需要预设分区边界。

2026年多数框架默认采用顺序写入者模式，因为LLM调用足够慢，竞争罕见，瓶颈影响有限。

### 不可写验证器

最关键的缓解措施是只读验证器。实现规则：

- 验证器与团队共享状态（读取黑板或消息池）
- 验证器无共享状态写入权限——仅能写入独立验证通道
- 验证器独立获取写入条目中引用的来源，标记不一致处
- 验证器输出路由至人类或独立决策智能体，永不反馈至消息池

若不分离，验证器输出将成为消息池新条目，导致污染的消息池污染验证器，进而污染其验证结果。

## 构建实现

`code/main.py`使用Python标准库实现两种拓扑结构，并包含简易污染攻击演示及三种缓解措施。

- `MessagePool` — 线程安全的仅追加日志，支持完整读取
- `Blackboard` — 基于主题的发布/订阅，支持智能体专属订阅
- `ProvenanceEntry` — 每次写入记录（写入者、时间戳、提示哈希、来源URI）
- `PoisoningScenario` — 执行三智能体研究任务，智能体A虚构小数点，输出最终报告
- `Verifier` — 只读智能体重新获取来源并标记不一致，在有验证器场景下运行相同任务

运行：

```
python3 code/main.py
```

预期输出：
- 第1次运行（无验证器）：虚构的42%传播至最终报告
- 第2次运行（有验证器）：验证器标记不一致，消息池标记为“已标记”，最终报告包含勘误声明

## 生产部署

对于任何共享内存设计：

- 每次写入都记录溯源信息：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`
- 保持日志仅追加。修正条目为引用被取代条目的新条目
- 部署至少一个具备独立来源访问权限的只读验证器智能体
- 将验证器输出路由至独立通道，而非反馈至共享池
- 记录被取代条目占总写入量的比例——比例上升是幻觉模式的早期迹象

## 核心术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 消息池 | "共享聊天记录" | 所有智能体读取的仅追加日志。完全透明但扩展性差 |
| 黑板 | "共享工作区" | 基于主题的发布/订阅。智能体订阅相关主题。扩展性更强 |
| 溯源 | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示词、来源 |
| 内存污染 | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其视为事实 |
| 仅追加 | "无原地更新" | 修正为取代旧条目的新条目。保留审计轨迹 |
| 不可写验证器 | "独立审计员" | 重新获取来源并标记不一致的只读智能体 |
| 投影 | "范围限定视图" | 根据全局状态为每个智能体计算的视图。LangGraph归约器是典型实现 |
| 知识源 | "专家智能体" | Hayes-Roth在1985年对黑板参与者的称谓 |
