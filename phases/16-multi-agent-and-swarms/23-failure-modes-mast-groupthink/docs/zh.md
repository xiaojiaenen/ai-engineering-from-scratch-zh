# 故障模式 — MAST、群体思维、单体文化、级联错误

> 2026年的参考分类法是 **MAST**（Cemri 等，NeurIPS 2025，arXiv:2503.13657），源自7个最先进开源多智能体系统（MAS）的1642条执行轨迹，显示 **41–86.7%的失败率**。三个根本类别：**规格问题** (41.77%) — 角色模糊、任务定义不明确；**协调失败** (36.94%) — 通信中断、状态不同步；**验证缺口** (21.30%) — 缺失验证、缺少质量检查。**群体思维**系列问题（arXiv:2508.05687）增加了：单体文化崩溃（相同基础模型 → 相关性错误）、从众偏差（智能体相互强化彼此的错误）、心智理论缺陷、混合动机动态、级联可靠性失败。级联错误示例：重试风暴，其中支付失败触发订单重试，进而触发库存重试，最终压垮库存服务（几秒内负载增加10倍 — 需要断路器）。记忆污染：一个智能体的幻觉进入共享记忆，下游智能体将其视为事实；准确性逐渐衰减，使得根本原因诊断非常痛苦。**STRATUS**（NeurIPS 2025）报告，通过专门的检测/诊断/验证智能体，缓解成功率提高了1.5倍。本课将故障模式视为首要的工程目标。

**类型:** 学习
**语言:** Python (标准库)
**先决条件:** 第16阶段 · 13（共享记忆），第16阶段 · 14（共识与拜占庭容错），第16阶段 · 15（投票与辩论拓扑）
**时间:** ~75分钟

## 问题

多智能体系统在真实任务上的失败率高达41-86.7%（Cemri等人于2025年在7个开源MAS上测量得出）。这无法通过"只是添加更多智能体"来调试。失败具有结构性原因。MAST分类法提供了分类。本课将每个类别映射到具体的检测、诊断和缓解模式，以便让这些数字看起来不再随意。

2026年的生产实践是将故障模式作为设计输入。只有在您能指出每个MAST类别并说出所部署的缓解措施后，您的架构才"足够好"。

## 概念

### MAST 类别

**规格问题（41.77%的故障）。** 智能体的任务定义不够精确。示例：

- 角色模糊：两个智能体都认为自己是审阅者。
- 任务规格不明确："总结这个"，而用户想要特定角度。
- 成功标准隐含：智能体无法判断自己是否成功。

缓解措施：
- 编写明确的角色契约。每个智能体的提示说明它*做什么以及不做什么*。
- 每个任务的验收测试。在智能体开始之前，定义"完成状态看起来像X"。
- 发射前规格检查：一个单独的智能体在派发前审查任务定义。

**协调失败（36.94%）。** 通信或状态中断。

示例：
- 两个智能体在没有同步的情况下更新共享状态。
- 智能体之间的消息丢失（队列故障、超时）。
- 状态漂移：智能体A认为任务已完成；智能体B仍在执行。

缓解措施：
- 具有乐观并发控制的版本化共享状态。
- 关键消息的明确确认（重试直到确认）。
- 定期的状态同步检查点；尽早检测漂移。

**验证缺口（21.30%）。** 没有对输出进行独立检查。

示例：
- 一个智能体声称成功；无人验证。
- 智能体链中每个都信任前一个的输出。
- 对涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立的验证智能体（第13课）。只读，独立的源访问权限。
- 明确的交接契约："A的输出必须在B开始前通过检查器C的检查。"
- 结果日志记录，用于事后分析。

### 群体思维系列问题（arXiv:2508.05687）

当智能体同质化或相互模仿时，会出现五种相关的失败模式：

**单体文化崩溃。** 相同的基础模型或训练数据 → 相关性错误。当三个智能体共享一个大语言模型（LLM）时，它们也共享其幻觉。

**从众偏差。** 智能体会向声音最大或最自信的同伴靠拢，即使对方是错的。

**心智理论缺陷。** 智能体未能建模彼此的信念；协调瓦解（第18课）。

**混合动机动态。** 具有部分对齐激励的智能体会漂移到妥协中间点，这无法满足任何人。

**级联可靠性失败。** 一个组件的错误模式会触发依赖组件中的错误模式。

### 级联错误示例 — 重试风暴

一个典型的2026年事故模式：

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

解决方案是经典的：**断路器**。当下游错误率超过阈值时，用缓存或默认结果进行短路。加上每个请求有上限的重试预算。

断路器是少数几种您可以直接从分布式系统借鉴并无需修改的多智能体故障缓解措施之一。

### 记忆污染（重温）

来自第13课：一个智能体的幻觉成为共享记忆的事实；下游智能体基于被污染的事实进行推理。用MAST的术语来说，这是在共享记忆层的一个验证缺口。

症状是准确性逐渐衰减。您不会得到崩溃；您得到的是缓慢的漂移，很难定位根本原因。

缓解措施：仅追加日志、来源、不可写的验证器。已在第13课中涵盖。

### STRATUS — 用于故障检测的专门智能体

STRATUS（NeurIPS 2025）报告，当您部署以下智能体时，缓解成功率提高了1.5倍：

- **检测智能体。** 观察症状模式（高分歧、重试峰值、准确性漂移）。
- **诊断智能体。** 给定症状，从MAST分类法中推断可能的根本原因。
- **验证智能体。** 在应用缓解措施后，检查症状是否清除。

这是SRE风格的事件响应，应用于智能体系统。这三个角色都可以是具有专门提示的LLM智能体。

### 故障模式审计

2026年的最佳实践是每年（或每次主要发布前）进行故障模式审计：

1. **跟踪样本。** 收集约1000条真实执行轨迹。
2. **分类。** 对于每条轨迹的失败，映射到MAST + 群体思维类别。
3. **计算按类别划分的失败率。** 哪些类别在您的系统中占主导？
4. **对缓解措施进行排名。** 哪种修复能消除最多的失败？
5. **选择2-3项缓解措施。** 实施；下个季度重新审计。

这种纪律比具体选择更重要。没有审计，失败会混入噪音中，永远不会得到系统性的解决。

### 当系统静默失败时

最危险的故障类别是静默的正确性失败。一个大声失败的系统（崩溃、异常、警报）可以被监控。一个产生看似合理但错误输出的系统无法通过异常日志检测到。这就是为什么尽管验证缺口按数量计算仅占21.30%，但按单次故障计算却是最昂贵的类别。

投资于：
- 基于样本的人工审查。
- 黄金数据集回归测试。
- 在重要输出上进行跨智能体交叉检查。

### 故障与缓慢故障

有些故障是即时的；有些是缓慢的。即时故障（超时、模式不匹配、认证错误）检测成本低。缓慢故障（记忆污染、单体文化漂移、角色模糊）检测和预防成本高。

2026年的工程举措：为缓慢故障的代理指标进行仪表化，以便在漂移变成可见错误之前捕获它。共识率、重试率、输出长度分布以及连续智能体版本之间的编辑距离都是有用的代理指标。

## 构建它

`code/main.py` 实现了：

- `FailureTaxonomy` — 将模拟的事件分类到 MAST + 群体思维类别。
- `CircuitBreaker` — 经典模式；当错误率超过阈值时打开。
- `RetryStormSimulator` — 展示级联故障；切换断路器的开启/关闭。
- `DetectionAgent` — 脚本化的 STRATUS 风格症状匹配器。

运行：

```
python3 code/main.py
```

预期输出：
- 无断路器时的重试风暴：库存错误激增（模拟）。
- 有断路器时：限制在阈值；提供降级模式响应。
- 检测智能体标记该模式并指明其MAST类别。

## 使用它

`outputs/skill-mast-auditor.md` 在一个多智能体系统上运行 MAST 风格的故障模式审计。轨迹 → 分类 → 缓解措施排名。

## 发布它

生产中的故障模式纪律：

- **每季度进行一次 MAST 审计。** 不是每年一次。类别会随着系统发展而变化。
- **无处不在的断路器。** 对任何依赖服务的每个出站调用。默认开放阈值设在5-10%的错误率。
- **黄金数据集。** 小型、高质量、经过人工审核。每周针对它们进行回归测试。
- **STRATUS 三元组。** 检测 + 诊断 + 验证智能体监控生产。先从检测智能体开始；当症状嘈杂时再增加诊断。
- **故障预算。** 为按类别划分的故障率设定明确的SLO。超出预算将触发停止发布的对话。

## 练习

1. 运行 `code/main.py`。确认断路器限制了重试风暴。改变故障阈值并观察权衡。
2. 实现一个**缓慢故障代理指标**：3个并行智能体之间的共识率。当它急剧下降时，触发警报。通过逐渐关联智能体输出来模拟单体文化漂移。
3. 阅读 Cemri 等人（arXiv:2503.13657）。选择他们研究的7个 MAS 系统中的一个，并映射其前3个故障类别。这些与 MAST 预测的相比如何？
4. 阅读群体思维论文（arXiv:2508.05687）。识别五种模式中哪种在生产中最难检测。提出一个代理指标。
5. 为您熟悉的一个特定多智能体系统设计一个 STRATUS 风格的检测-诊断-验证三元组。检测监视哪些症状？诊断建议哪些缓解措施？验证如何确认它们有效？
