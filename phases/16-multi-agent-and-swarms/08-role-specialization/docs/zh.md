# 角色专业化 — 规划者、批评者、执行者、验证者

> 2026年最常见的多智能体分解模式：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的标准操作程序（SOPs）——产品经理、架构师、项目经理、工程师、QA工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过“聊天链”串联设计师、程序员、审查员、测试员，并运用“沟通式去幻觉”（智能体明确请求缺失的细节）。验证者至关重要：Cemri等人（MAST, arXiv:2503.13657）表明，每个多智能体失败都可追溯到验证的缺失或失效。普华永道（PwC）报告称，在CrewAI中添加结构化验证循环后，准确率提升了7倍（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**先决条件：** Phase 16 · 04（原始模型），Phase 16 · 05（监督者）
**时间：** ~60分钟

## 问题

通用的多智能体系统产出通用的结果。群聊中的三个程序员写出三种风味的同质平庸代码。你可以增加更多智能体、更多轮次，但依然无法跨越质量门槛。

解决方法不是更多智能体——而是*不同的*智能体。分配不同的角色。赋予批评者规划者所没有的工具。给予验证者一个客观的测试套件。这样系统就有了基于事实的内部修正，而不仅仅是并行猜测。

## 概念

### 四种经典角色

**规划者（Planner）。** 读取目标，生成步骤列表或规格说明。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）。** 每次读取计划的一个步骤，生成产物。工具：实际工作工具（代码编译器、命令行、API客户端）。输出：产物。

**批评者（Critic）。** 对照规划者的意图，评估执行者的输出。工具：对产物的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者（Verifier）。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败及证据。

批评者是主观的、有主见的，通常基于大语言模型。验证者是客观的、确定性的，通常基于代码。它们不是同一角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理** 撰写 PRD。
- **架构师** 产出系统设计。
- **项目经理** 拆分任务。
- **工程师** 实施。
- **QA 工程师** 运行测试。

每个角色都有严格的输入/输出模式。角色提示说明了该角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 的表述——确定性的 SOP 将一个大语言模型团队变成了一个可预测的流水线。

### ChatDev 的沟通式去幻觉

ChatDev 增加了一个关键动作：当执行者需要计划中未包含的某个具体细节时，它会在继续之前明确向设计师询问。这避免了大语言模型凭空捏造细节的典型失败。

实现方式：角色提示中包含“当你需要未获得的具体信息时，在产出输出前，通过名字向相关角色提问。”

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 次多智能体执行失败。21.3% 是验证缺口——系统交付了无人检查的答案。其余的 79% 往往可追溯到“存在一个检查但静默失败或从未运行”。验证是承重角色。

普华永道报告（CrewAI 部署，2025 年）称，添加结构化验证循环使准确率从 10% 提升到 70%。一个角色带来了 7 倍的收益。

### 批评者 vs 验证者

- **批评者**是一个大语言模型，审查产物以评估其质量。主观。可能被看似合理的文本所迷惑。
- **验证者**是一个运行在产物上的确定性程序。客观。提供通过/失败及证据。

两者都要用。批评者能发现验证者无法言说的品味问题。验证者能发现批评者看不到的错误，因为这些错误只在运行时显现。

### 反模式

你系统中的每个角色都是大语言模型，每个角色的输出都是“看起来不错”。典型的 MAST 失败模式。至少添加一个验证者，其通过/失败由代码而非大语言模型决定。

### 框架映射

- **CrewAI** —— `Agent(role, goal, backstory)` 是典型的专业化表面。
- **LangGraph** —— 节点可以有专门的提示；边强化流水线。
- **AutoGen** —— 带有简短名字的、特定于角色的可对话智能体，位于一个群聊中。
- **OpenAI Agents SDK** —— 角色专业化智能体之间的移交工具。

## 构建它

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

- **规划者** 产出规格说明。
- **执行者** 生成代码字符串。
- **批评者**（大语言模型模拟）标记明显问题。
- **验证者** 在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行者生成正确代码（批评者 + 验证者都通过），一次执行者生成偏离规格的代码（批评者因其看似合理而错过了错误，验证者因测试失败而捕获了它）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md` 接收一个任务并产出角色阵容（3-5 个角色）、每个角色的输入/输出模式以及验证者检查。在将智能体接入框架之前使用此工具。

## 交付它

检查清单：

- **至少一个确定性验证者。** 绝不能全部由大语言模型承担。
- **每个角色有明确的 I/O 模式。** 规划者返回规格说明，而非散文；执行者读取该模式。
- **沟通式去幻觉。** 执行者在信息缺失时必须询问规划者；绝不捏造。
- **批评者/验证者顺序。** 先运行批评者（便宜，捕获设计问题），再运行验证者（慢，捕获错误）。
- **循环预算。** 在上报给人类之前，最多进行 2 轮批评者-执行者修订。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批评者错过的错误。添加一个静态分析检查（计算 `return` 的出现次数）作为额外的验证者。它捕获了哪些运行时测试无法发现的问题？
2. 添加第 5 个角色：“需求分析师”，将用户愿望转化为规划者可处理的规格说明。应向其传递哪些沟通式去幻觉请求？
3. 阅读 MetaGPT 第 3 节（“智能体”）。列出 MetaGPT 5 个角色各自的输入/输出模式。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。找出沟通式去幻觉在何处打断了原本会无限循环的流程。
5. 普华永道的 7 倍准确率提升来自验证循环。假设三个添加验证者无济于事的任务——在这些任务中，正确性的确定性检查是不可能或极其昂贵的。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 角色专业化 | “不同的智能体，不同的工作” | 为规划者/执行者/批评者/验证者角色调优的不同系统提示。 |
| SOP 模式 | “编码的标准操作程序” | MetaGPT 的框架：每个角色的严格 I/O 模式将一个团队变成流水线。 |
| 沟通式去幻觉 | “先问再编” | ChatDev 模式：执行者在细节缺失时询问规划者，而非自行编造。 |
| 批评者 | “大语言模型审查员” | 主观的、有主见的审查员。能抓住品味问题。可能被看似合理的文本迷惑。 |
| 验证者 | “确定性检查” | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。不会被迷惑。 |
| 验证缺口 | “无人检查” | 21.3% 的 MAST 失败。交付了未经检查的答案，而该检查本可捕获错误。 |
| 修订循环 | “批评者打回” | 批评者拒绝触发执行者带着反馈重新运行。需要预算。 |
| 全大语言模型反模式 | “看起来不错” | 每个角色都是大语言模型，没有确定性检查。典型的 MAST 失败。 |
