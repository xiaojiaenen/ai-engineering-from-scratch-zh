# 并行 / 群体 / 网络化架构

> 与监督者模式对比：无中心决策者。智能体读取共享事件总线，异步拾取任务，将结果写回。LangGraph 显式支持用于去中心化动态环境的“群体架构”。Matrix (arXiv:2511.21686) 将控制流和数据流均表示为通过分布式队列传递的序列化消息，以消除编排器瓶颈。这里的取舍是明确的：用确定性和可追溯性换取可扩展性。群体适用于存在许多独立子问题的任务；不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python (标准库, `threading`, `queue`)
**前提条件：** 第16阶段 · 05 (监督者模式), 第16阶段 · 04 (原始模型)
**时间：** 约75分钟

## 问题

监督者可以扩展到几个工作节点。那成百上千个呢？监督者本身成为瓶颈：关于“谁做什么”的每个决策都要通过一个智能体。一个缓慢的计划步骤会拖慢整个系统。

群体架构颠覆了设计思路。不是由中心规划者分派任务，而是由工作节点从共享队列中拾取任务。协调机制被内置于事件总线的语义中。没有编排器；系统可以一直扩展，直到队列成为瓶颈。

## 概念

### 形态

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排器。每个工作节点重复：拉取任务、处理、写入结果（可选地将后续任务入队）。

### 群体适用的场景

- **大量独立任务。** 爬取、转换、分类。任务之间互不依赖。
- **可变时长的工作。** 如果某些任务耗时100毫秒，而另一些耗时10秒，群体可以自动平衡负载——快速的工作节点会接着领取下一份工作。而监督者需要预估任务时长。
- **吞吐量优先于确定性。** 你关心总完成时间，而非严格的顺序执行。

### 群体不适用的场景

- **有序工作流。** 如果步骤3需要步骤2的输出，群体可能会导致步骤3在步骤2完成前就被触发。
- **需要全局规划的任务。** 复杂的研究问题受益于规划器。一群独立的研究员会产生零散的事实，而不是一份连贯的报告。
- **调试困难。** 由于没有中心日志且工作是异步的，复现一个bug的成本很高。

### Matrix (arXiv:2511.21686)

Matrix 是2025年的一篇论文，它将群体模式推向了其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中心协调器。容错性来自消息持久性。可扩展性是消息代理的问题，而非系统本身。

贡献：一种编程模型，其中多智能体协调的核心问题变成了“这个智能体订阅哪个消息主题？”而不是“监督者接下来选择哪个智能体？”。这使得系统看起来像一个发布/订阅事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档将“群体架构”明确描述为多智能体模式之一：智能体是节点，但边构成带有循环的有向图，并且任何节点都可以从任务池中被激活。工作节点根据条件从可用任务中拾取，而非由监督者分配。

### 故障模式：饥饿与热点

如果所有工作节点都拉取最快可用的任务，那么长时间运行的任务将永远不会被拾取，直到它们是队列中仅剩的任务。这是经典的队列饥饿现象。

缓解措施：
- **带老化机制的优先级队列** (随着等待时间增加而提高任务优先级)。
- **工作节点专业化**：部分工作节点只接受“长耗时”任务。
- **反压**：限制快速任务进入队列的速度。

### 与基于内容路由的联系

群体模式与基于内容的路由（第22课）天然契合。可以不使用通用队列，而是为每种消息类型设置一个专用队列。专业的工作节点只订阅其对应的类型。这是可扩展到数千智能体的消息总线架构的基础。

## 构建它

`code/main.py` 实现了一个由4个工作线程组成的群体，它们从一个共享的 `queue.Queue` 中拾取任务。任务具有可变时长（有些快，有些慢）。该示例进行了对比：

- **顺序基线：** 一个工作节点串行处理所有任务。
- **固定分配：** 每个任务预分配给特定工作节点（监督者模式）。
- **群体：** 工作节点从共享队列中拾取任务。

群体自动平衡负载；而固定分配模式下，当某个工作节点被分配到耗时任务时，其他快速工作节点会处于闲置状态。

运行：

```
python3 code/main.py
```

输出显示了每个工作节点的任务计数（群体分配不均匀但最优）和墙钟时间。

## 使用它

`outputs/skill-swarm-fit.md` 评估一个任务应该使用群体模式还是监督者模式。输入参数包括：任务独立性、时长方差、顺序要求、可调试性需求。

## 部署

检查清单：

- **带老化机制的优先级队列。** 防止长任务饥饿。
- **工作节点幂等性。** 如果工作节点在运行中崩溃，任务可能被拾取多次。工作节点必须是幂等的。
- **持久队列。** 生产环境中使用 Kafka、Redis Streams 或基于数据库的队列。`queue.Queue` 仅是内存队列。
- **每个任务的可观测性。** 每个任务都有跟踪ID；每个工作节点用此ID记录开始/结束日志。
- **反压。** 如果队列增长速度超过工作节点的处理速度，则减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长任务负载下，群体比顺序模式快多少？比固定分配快多少？
2. 添加一个优先级队列变体（使用 `queue.PriorityQueue`）。根据任务的“重要性”字段分配优先级。观察在持续负载下，低优先级任务是否会发生饥饿。
3. 实现一个热点检测器：当任一工作节点处理的任务数是最慢工作节点的3倍时记录日志。这对任务时长分布意味着什么？
4. 阅读 Matrix 论文 (arXiv:2511.21686) 的摘要和第3节。识别 Matrix 所接受的一个具体取舍（可扩展性收益）和它所放弃的一个方面（可追溯性、确定性）。
5. 将群体示例转换为使用存储 (task_type, payload) 元组的 `queue.Queue`，并且工作节点只订阅特定的类型。当任务异构时，什么样的路由规则是合理的？

## 关键术语

| 术语 | 通俗说法 | 确切含义 |
|------|----------------|------------------------|
| 群体架构 | “去中心化智能体” | 工作节点从共享队列拾取任务；无中心编排器。 |
| 事件总线 | “智能体订阅主题” | 根据类型或内容将任务路由给工作节点的消息代理。 |
| 饥饿 | “任务永远不运行” | 由于高优先级任务持续到达，低优先级任务永远无法被拾取。 |
| 热点 | “一个工作节点不堪重负” | 负载不均衡，其中一个工作节点获得了大部分任务。 |
| 反压 | “让生产者慢下来” | 当队列满时，向上游发出信号以停止生产的机制。 |
| 幂等工作节点 | “可以安全重试” | 一个任务被处理两次会产生相同的结果。这是必需的，因为工作节点可能在运行中崩溃。 |
| 持久队列 | “崩溃后能存活” | 由磁盘或复制存储支持的队列；当工作节点崩溃时任务不会丢失。 |
| Matrix 框架 | “完全消息传递的群体” | 数据流和控制流都是分布式队列上的序列化消息。 |
