# 多智能体原始模型

> 2026年发布的每个多智能体框架——AutoGen、LangGraph、CrewAI、OpenAI Agents SDK、Microsoft Agent Framework——都是四维设计空间中的一个点。仅四个原始要素，别无其他：智能体、交接、共享状态、编排器。本课程从零开始构建它们，在所有四个要素上运行一个玩具系统，然后将每个主要框架映射到同一坐标轴上，让你能通过一段话读懂任何新发布的框架。

**类型：** 学习  
**语言：** Python（标准库）  
**前提：** 第14阶段（智能体工程），第16阶段·01（为何需要多智能体）  
**时间：** 约60分钟

## 问题

每六个月就有一个新的多智能体框架发布。2023年的AutoGen。2024年的CrewAI。2024年的LangGraph和OpenAI Swarm。2025年4月的Google ADK。2026年2月的Microsoft Agent Framework RC。每篇新闻稿都声称自己是"正确的抽象"。

如果你尝试逐个学习，你会精疲力竭。API看起来各不相同。文档对"智能体"是什么各执一词。一个框架称其共享内存为"黑板"，另一个称其为"消息池"，第三个称其为"StateGraph"。你开始怀疑这个领域只是在不断折腾。

事实并非如此。在营销背后，这四个原始要素是稳定的。学一次，就能通过一段话读懂每个新框架。

## 概念

### 四个原始要素

1.  **智能体** —— 一个系统提示词加上一个工具列表。无状态；每次运行都从其系统提示词和当前消息历史开始。
2.  **交接** —— 将控制权从一个智能体结构化地转移给另一个。从机制上说，是一个返回新智能体的工具调用，或一个遵循条件的图边。
3.  **共享状态** —— 多个智能体可以读取（有时可写入）的任何数据结构。消息池、黑板、键值存储、向量记忆。
4.  **编排器** —— 决定谁接下来发言的角色。选项包括：一个显式图（确定性的）、一个LLM发言选择器（软性的）、最后发言者的交接调用（OpenAI Swarm），或队列上的调度器（Swarm架构）。

这就是整个设计空间。每个框架为每个轴选择默认值；其余只是表面语法。

### 2026年每个框架如何映射

| 框架 | 智能体 | 交接 | 共享状态 | 编排器 |
|------|--------|------|----------|--------|
| OpenAI Swarm / Agents SDK | `Agent(instructions, tools)` | 工具返回智能体 | 调用者的问题 | LLM的下一次交接调用 |
| AutoGen v0.4 / AG2 | `ConversableAgent` | GroupChat上的发言选择器 | 消息池 | 选择器函数（LLM或轮询） |
| CrewAI | `Agent(role, goal, backstory)` | `Process.Sequential / Hierarchical` | 任务输出链 | 经理LLM或固定顺序 |
| LangGraph | 节点函数 | 图边 + 条件 | `StateGraph` 归约器 | 图，确定性的 |
| Microsoft Agent Framework | 智能体 + 编排模式 | 模式特定 | 线程/上下文 | 模式特定 |
| Google ADK | 智能体 + A2A卡 | A2A任务 | A2A工件 | 主机决定 |

表面差异看起来很大。底层：相同的四个旋钮。

### 为什么这很重要

一旦你看到了这些原始要素，框架比较就变成了一个简短的检查清单：

-   编排器是信任LLM进行路由（Swarm）还是在代码中固定路由（LangGraph）？
-   共享状态是全历史（GroupChat）还是投影的（StateGraph归约器）？
-   智能体能否修改彼此的提示词（CrewAI经理）还是只能进行交接（Swarm）？

这三个问题回答了80%的问题：哪个框架适合给定问题。你不再寻找"最好的多智能体框架"，而是开始为你真正在意的轴进行设计。

### 无状态洞察

除了共享状态，每个原始要素都是无状态的。智能体是（提示词，工具）的函数。交接是一个函数调用。编排器是一个调度器。**系统中唯一有状态的东西是共享状态。** 所有有趣的bug都在那里：记忆污染（第15课）、消息排序、版本控制、写入争用。

隐藏共享状态的框架（Swarm）将问题推给调用者。集中管理共享状态的框架（LangGraph检查点，AutoGen池）使其可检查，但将协调成本转移到共享状态实现上。

### 单个原始要素的剖析

#### 智能体

```
Agent = (system_prompt, tools, model, optional_name)
```

没有记忆。没有状态。两个具有相同系统提示词和工具的智能体是可互换的。所有看起来像单个智能体状态的东西，实际上都在共享状态或交接协议中。

#### 交接

```
Handoff = (from_agent, to_agent, reason, payload)
```

三种主导实现：

-   **函数返回** —— 工具返回下一个智能体。这是OpenAI Swarm模式。智能体在其工具模式中携带路由信息。
-   **图边** —— LangGraph。边是声明式的。LLM产生一个值；一个条件选择下一个节点。
-   **发言选择** —— AutoGen GroupChat。一个选择器函数（有时本身是一个LLM调用）读取池并选择下一个发言者。

#### 共享状态

```
SharedState = { messages: [], artifacts: {}, context: {} }
```

至少是一个消息列表。通常更多：结构化工件（CrewAI任务输出）、类型化上下文（LangGraph归约器）、外部记忆（MCP，向量数据库）。

两种拓扑：**全池**（每个智能体看到所有消息）和**投影的**（智能体看到角色范围的视图）。全池简单但扩展性差。投影池可扩展但需要预先设计模式。

#### 编排器

```
Orchestrator = ({state, last_speaker}) -> next_agent
```

四种风格：

-   **静态** —— 图在构建时固定（LangGraph确定性的，CrewAI顺序执行）。
-   **LLM选择** —— 一个LLM读取池并选择下一个发言者（AutoGen，CrewAI层级式）。
-   **交接驱动** —— 当前智能体通过调用一个交接工具来决定（Swarm）。
-   **队列驱动** —— 工作者从共享队列中拉取任务；没有明确的下一个发言者（Swarm架构，Matrix）。

### 框架之间有什么变化

一旦原始要素固定下来，剩余的设计决策就是：

-   **记忆策略** —— 瞬时 vs 持久化检查点（LangGraph检查点器）。
-   **安全边界** —— 谁可以批准交接（人在回路中）。
-   **成本核算** —— 每个智能体的token预算。
-   **可观察性** —— 跟踪交接，持久化状态以供重放。

所有这些都可以在原始要素之上实现。它们都不是新的原始要素。

## 构建它

`code/main.py` 用约150行标准库Python代码实现了这四个原始要素。没有真实的LLM——每个智能体是一个脚本化策略，以便将重点保持在协调结构上。

该文件导出：

-   `Agent` —— 一个数据类，包含名称、系统提示词、工具、策略函数。
-   `Handoff` —— 一个返回新智能体的函数。
-   `SharedState` —— 一个线程安全的消息池。
-   `Orchestrator` —— 三种变体：`StaticOrchestrator`、`HandoffOrchestrator`、`LLMSelectorOrchestrator`（模拟）。

该演示通过所有三种编排器类型运行相同的三智能体流水线（研究 -> 撰写 -> 审查），并在最后打印消息池。你可以看到输出仅在 *谁选择下一个* 方面有所不同；智能体和共享状态在所有运行中都是相同的。

运行它：

```
python3 code/main.py
```

预期输出：三次编排器运行，每种模式一次。每次都会打印最终的消息池。如果研究者决定提前结束，交接驱动的运行会达到更少的智能体——这是LLM路由权衡的缩影。

## 练习

1.  使用不同的智能体策略运行`code/main.py`三次。观察编排器选择如何改变运行哪些智能体。
2.  实现第四种编排器类型：队列驱动型，智能体轮询共享状态以获取工作。可能发生什么死锁，你如何检测它？
3.  获取LangGraph快速入门（https://docs.langchain.com/oss/python/langgraph/workflows-agents）并将其重写为四个原始要素。LangGraph的哪些抽象是1:1映射，哪些是便利包装？
4.  阅读OpenAI Swarm手册（https://developers.openai.com/cookbook/examples/orchestrating_agents）。确定Swarm使哪个原始要素最符合人体工程学，以及它将哪一个推给了调用者。
5.  在此表中找到一个完全隐藏共享状态的框架。解释当智能体需要在不重新读取历史记录的情况下跨交接协调时，会发生什么故障。

## 关键术语

| 术语 | 人们怎么说 | 它的实际含义 |
|------|------------|--------------|
| 智能体 | "一个带工具的LLM" | 一个`(system_prompt, tools, model)`三元组。无状态。 |
| 交接 | "控制权转移" | 一个结构化调用，用于命名下一个智能体和可选的有效载荷。三种实现：函数返回、图边、发言选择。 |
| 共享状态 | "记忆"/"上下文" | 多智能体系统中唯一有状态的部分。消息池或黑板。 |
| 编排器 | "协调者" | 决定谁接下来运行的角色。静态图、LLM选择器、交接驱动或队列驱动。 |
| 原始要素 | "抽象" | 每个框架参数化的四个轴之一。不是框架功能。 |
| 消息池 | "共享聊天历史" | 全历史共享状态。易于推理，扩展性差。 |
| 投影状态 | "范围视图" | 共享状态中角色特定的视图。可扩展，需要模式设计。 |
| 发言选择 | "谁接下来发言" | 编排器模式，其中一个函数（通常是LLM）从组中选择下一个智能体。 |
