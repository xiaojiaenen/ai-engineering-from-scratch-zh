# FIPA-ACL与言语行为的历史传承

> 在MCP和A2A之前，FIPA-ACL早已存在。2000年，IEEE智能物理代理基金会批准了一种代理通信语言，包含二十种言语行为、两种内容语言以及一系列交互协议——合同网、订阅/通知、条件请求。它因本体开销过大而逐渐淡出行业视野，但大语言模型驱动的多智能体系统复兴正悄然重新实现相同理念，只是去除了形式化语义：JSON合约替代了言语行为，自然语言替代了本体体系。本节课程将严肃解读FIPA-ACL，助你辨别2026年协议设计中哪些是旧瓶新酒，哪些是真正创新，以及当前浪潮将重新发现哪些2000年代已解决的问题。

**类型：** 学习
**语言：** Python（标准库）
**前置要求：** 阶段16 · 01（为何需要多智能体系统）
**时间：** 约60分钟

## 问题背景

2026年的智能体协议领域异常活跃：MCP处理工具调用，A2A处理智能体间协作，ACP面向企业审计，ANP构建去中心化信任，NLIP支持自然语言内容，此外还有CA-MCP及二十余项研究提案。每项规范都自称奠基性标准。

客观来看，它们大多是在重新发现一个已有二十年历史的决策路径。奥斯汀（1962）和塞尔（1969）的言语行为理论提出了“话语即行动”。KQML（1993）将其转化为线路协议。FIPA-ACL（2000年批准）形成了参考标准：二十种言语行为、SL0/SL1内容语言、合同网与订阅-通知交互协议。JADE和JACK曾是Java参考平台。该体系因本体开销过重且Web技术胜出而在2010年左右衰落。

当你审视MCP的`tools/call`、A2A的任务生命周期或CA-MCP的共享上下文存储时，你看到的正是FIPA决策体系的轻量化、JSON原生重构。了解历史能让你洞察两点：哪些“新创新”实为旧酒新瓶，哪些旧失败模式正被新规范重新发现。

## 核心概念

### 言语行为理论精要

奥斯汀注意到某些句子并非描述世界——而是改变世界：“我承诺”“我请求”“我宣判”。他将这些称为施事话语。塞尔将其形式化为五类：断言类、指令类、承诺类、表达类、宣告类。KQML（Finin等人，1993）将其应用于软件智能体：消息由言语行为（动作）和内容（动作所涉对象）组成。FIPA-ACL弥补了KQML的缺陷并标准化了二十种言语行为。

### FIPA二十种言语行为（部分列表）

| 言语行为 | 意图 |
|---|---|
| `inform` | “告知你命题P为真” |
| `request` | “请求你执行动作X” |
| `query-if` | “询问命题P是否为真” |
| `query-ref` | “查询变量X的值” |
| `propose` | “提议我们执行X” |
| `accept-proposal` | “接受该提议” |
| `reject-proposal` | “拒绝该提议” |
| `agree` | “同意执行X” |
| `refuse` | “拒绝执行X” |
| `confirm` | “确认命题P为真” |
| `disconfirm` | “否认命题P” |
| `not-understood` | “消息解析失败” |
| `cfp` | “针对X发起提案征集” |
| `subscribe` | “当X变化时通知我” |
| `cancel` | “取消正在进行的X” |
| `failure` | “尝试执行X但失败” |

完整列表参见`fipa00037.pdf`（FIPA ACL消息结构规范）。重点不在于记忆——而在于其中每一项都对应着大语言模型协议最终会重新添加的原语。

### 标准FIPA-ACL消息结构

```
(inform
  :sender       agent1@platform
  :receiver     agent2@platform
  :content      "((price IBM 83))"
  :language     SL0
  :ontology     finance
  :protocol     fipa-request
  :conversation-id   conv-42
  :reply-with   msg-17
)
```

七个字段承载协议信封，一个字段（`content`）承载有效载荷。其余字段正是你每次为JSON协议添加重试、线程和本体支持时重新设计的内容。

### 两大遗留平台

**JADE**（Java代理开发框架，1999–2020年代）是使用最广泛的FIPA合规运行时。代理继承基类，交换ACL消息，在容器内运行，通过“行为”进行协调。其交互协议库内置了合同网、订阅-通知、条件请求、提议-接受等模式。

**JACK**（代理导向软件公司，商业产品）在FIPA消息基础上强调BDI（信念-愿望-意图）推理。更形式化，但采用率较低。

两者均在Web技术栈吞没多智能体应用场景后衰落。MCP和A2A正是2026年的运行时“容器”。

### FIPA衰落的原因

- **本体开销过高。** FIPA要求共享本体才能解析`content`。达成共识本体是长达数年的标准化过程。而Web仅使用HTTP + JSON。
- **形式化语义无人问津。** SL（语义语言）提供了严格的真值条件，但多数生产系统采用自由格式内容，无视形式化规范。
- **工具链锁定。** JAD仅限Java；JACK为商业软件。多语言技术团队都绕道而行。
- **互联网技术栈胜出。** REST、JSON-RPC、gRPC逐步取代了ACL的传输层。

### 大语言模型复兴实为FIPA精简版

对比FIPA的`request`与MCP的`tools/call`：

```
(request                                {
  :sender  agent1                         "jsonrpc": "2.0",
  :receiver tool-server                   "method":  "tools/call",
  :content "(lookup stock IBM)"           "params":  {"name":"lookup_stock",
  :ontology finance                                   "arguments":{"symbol":"IBM"}},
  :conversation-id c42                    "id": 42
)                                        }
```

相同信封，不同语法。两者都承载：发送方、接收方、意图、有效载荷、关联ID。两者相比都不是革命性突破——它们只是对同一设计的不同权衡取舍。

Liu等人2025年综述（《智能体互操作协议综述：MCP、ACP、A2A、ANP》，arXiv:2505.02279）明确阐述了这种谱系关系：MCP对应工具使用言语行为，A2A对应对等智能体言语行为，ACP对应审计追踪言语行为，ANP对应去中心化身份扩展。新规范是ACL的后裔，采用JSON语法和更宽松的语义。

### 权衡取舍的坦诚表述

**FIPA提供而现代规范舍弃的：**

- 形式化语义——可证明`inform`蕴含发送方相信内容为真
- 标准化的言语行为目录——无需反复争论“是否需要`cancel`？”
- 交互协议模式库——合同网、订阅-通知、提议-接受——均具备已验证的正确性

**现代规范提供而FIPA缺乏的：**

- JSON原生载荷，兼容所有现代工具
- 自然语言内容，大语言模型可直接解释而无需手工编写本体
- Web技术栈传输层（HTTP、SSE、WebSocket）
- 通过自描述文档进行能力发现（MCP的`listTools`、A2A的智能体卡片）

更宽松的意图语义带来更简便的实现。这就是本质的权衡。

### 值得移植的交互协议

FIPA包含约15种交互协议。以下三种值得移植到大语言模型多智能体系统：

1. **合同网协议（CNP）。** 管理器发布`cfp`（提案征集）；竞标方响应`propose`；管理器接受/拒绝。这是任务市场的经典模式（阶段16 · 16 协商）。
2. **订阅/通知。** 订阅方发送`subscribe`；当主题变化时发布方发送`inform`。这是2026年所有事件总线的原型。
3. **条件请求。** “当条件Y满足时执行X”。带前置条件的延迟动作。其2026年对应物是持久化工作流引擎中的延迟任务（阶段16 · 22 生产环境扩展）。

每种模式都可无缝映射到现代消息队列、HTTP+轮询或SSE流式传输。

### 放弃本体将引发的问题

若无共享本体，智能体将从自然语言内容推断含义。2026年记录的失败模式是**语义漂移**：两个智能体对相同词汇（`"customer"`）理解存在细微差异，接收方智能体基于错误解释行动，且无模式校验器捕获该问题。FIPA的本体要求本可在解析阶段拒绝该消息。

无需完整本体的缓解方案：

- 对`content`使用JSON Schema——在线路上拒绝结构性错误
- 类型化工件（A2A）——拒绝错误模态
- 信封中明确的言语行为——即使内容为自然语言也能使意图明确

### 2026年规范与言语行为谱系对应

| 现代规范 | FIPA对应物 | 保留特性 | 舍弃特性 |
|---|---|---|---|
| MCP `tools/call` | `request` | 显式意图、关联ID | 形式化语义、本体 |
| MCP `resources/read` | `query-ref` | 显式意图、关联ID | 形式化语义 |
| A2A任务生命周期 | 合同网 + 条件请求 | 异步生命周期、状态转换 | 形式化完备性保证 |
| A2A流式事件 | 订阅/通知 | 异步推送 | 类型化谓词订阅 |
| CA-MCP共享上下文 | 黑板（Hayes-Roth 1985） | 多写入者共享内存 | 逻辑一致性模型 |
| NLIP | 自然语言内容 | 大语言模型原生 | 模式约束 |

纵览表格，模式清晰：保留结构原语，舍弃形式化体系，让大语言模型弥合歧义。

## 动手实践

`code/main.py`实现了一个纯标准库的FIPA-ACL转换器。它编解码标准ACL信封，展示每条MCP/A2A消息如何简化为相同的七个字段。演示内容包括：

- 将五条MCP风格和A2A风格消息编码为FIPA-ACL
- 将FIPA-ACL解码回现代等价形式
- 使用`cfp`、`propose`、`accept-proposal`、`reject-proposal`运行一个管理器与三个竞标方的玩具合同网协商

执行：

```
python3 code/main.py
```

输出是并排追踪：每条现代消息同时显示其2026年JSON形式与FIPA-ACL形式，随后是合同网投标的往返过程。相同的协议原语在往返中得以保留；仅语法不同。

## 生产化建议

不要恢复FIPA-ACL。恢复其检查清单：

- 每条消息的意图原语（言语行为）是什么？
- 请求-响应和取消操作是否有关联ID？
- 是否有明确的内容语言（JSON-RPC、纯文本、结构化类型化工件）？
- 交互协议是否为一级概念，还是从头重新实现合同网？
- 当两个智能体对内容含义产生分歧时会发生什么（语义漂移）？

在将任何新协议投入生产前，请为这五个问题编写文档。
