# 交接与例程 —— 无状态编排

> OpenAI 的 Swarm（2024 年 10 月）将多智能体编排提炼为两个原语：**例程**（作为系统提示的指令 + 工具）和**交接**（一个返回另一个智能体的工具）。无需状态机，无需分支 DSL——大语言模型通过调用正确的交接工具来路由。OpenAI Agents SDK（2025 年 3 月）是其生产级后继者。Swarm 本身仍是概念上最清晰的参考——其全部源代码仅几百行。这个模式之所以流行，是因为其 API 接口大致是：“智能体 = 提示词 + 工具；交接 = 返回智能体的函数”。局限：无状态，因此记忆是调用方的责任。

**类型：** 学习 + 实践
**语言：** Python（标准库）
**先决条件：** 阶段 16 · 04（原始模型）
**时间：** ~60 分钟

## 问题

每个多智能体框架都要求你学习其 DSL：LangGraph 的节点和边，CrewAI 的船员和任务，AutoGen 的 GroupChat 和管理器。这些 DSL 是真正的抽象，但它们让事情变得比必要的更重。

Swarm 推向了相反的方向：使用模型已有的工具调用能力。交接变成了工具调用。编排器是当前持有对话的任何智能体。状态机隐含在智能体的系统提示中。

## 概念

### 两个原语

**例程。** 一个定义智能体角色和可用工具的系统提示。将其视为一组有作用域的指令：“你是一个分诊智能体；如果用户询问退款，请转交给退款智能体。”

**交接。** 智能体可以调用的一个工具，该工具返回一个新的 Agent 对象。Swarm 运行时检测到 Agent 返回值，并为下一轮切换活跃智能体。

这就是全部的抽象。

```
def transfer_to_refunds():
    return refund_agent  # Swarm sees Agent return → switch active agent

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分诊智能体的系统提示使其根据用户消息选择正确的交接。大语言模型的工具调用完成路由。

### 为什么它流行

- **小型 API。** 只需学习两个概念。
- **利用模型已有的能力。** 工具调用在各个提供商中已经是生产级的。
- **无状态机负担。** 你无需描述图；智能体的提示描述了它们将交接给谁。

### 无状态的权衡

Swarm 在运行之间是明确无状态的。框架在一次运行期间保留消息历史，但它不持久化任何东西。记忆、连续性、长期运行的任务——都是调用方的责任。

在生产中（OpenAI Agents SDK，2025 年 3 月），这是主要的变化之一：SDK 增加了内置会话管理、护栏和追踪，同时保留了交接原语。

### Swarm/交接适用场景

- **分诊模式。** 前线智能体将用户路由给专家。
- **基于技能的交接。** “如果任务需要编码，就调用编码员；如果需要研究，就调用研究员。”
- **简短、有界的对话。** 客户支持、常见问题到工单、简单工作流。

### Swarm 遇到困难的场景

- **具有共享记忆的长会话。** 交接将对话状态重置为新智能体的提示词加上历史。没有调用方管理的记忆，就没有跨智能体的持久状态。
- **并行执行。** 交接是串行的——活跃智能体会切换。并行性需要调用方协调多个 Swarm 运行。
- **审计和回放。** 无状态运行很难精确重放；大语言模型的交接选择不是确定性的。

### OpenAI Agents SDK（2025 年 3 月）

这个生产级后继者增加了：

- **会话状态。** 跨运行的持久线程。
- **护栏。** 输入/输出验证钩子。
- **追踪。** 记录每次工具调用和交接。
- **交接过滤器。** 控制交接时传递的上下文。

交接原语得以保留；围绕它增加了生产级的人机工程学。

### Swarm 与 GroupChat 对比

两者都使用大语言模型驱动的路由，但它们在 **谁选择下一个** 方面不同：

- GroupChat：一个选择器（函数或大语言模型）从外部选择下一个发言者。
- Swarm：当前智能体通过调用交接工具来选择其继任者。

Swarm 是“智能体决定下一步”；GroupChat 是“管理者决定下一步”。Swarm 的决策存在于活跃智能体的工具调用中；GroupChat 的决策存在于 `GroupChatManager`。

## 动手构建

`code/main.py` 从头实现了 Swarm：一个 Agent 数据类，一个交接机制（工具返回 Agent），以及一个检测智能体切换的运行循环。

演示：一个分诊智能体路由到退款、销售或支持专家。每个专家有自己的工具。运行循环会打印每次交接。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-handoff-designer.md` 为给定任务设计一个交接拓扑：存在哪些智能体，它们可以调用哪些交接，传递什么上下文。

## 部署它

检查清单：

- **交接日志。** 每次交接都写入一个包含来源智能体、目标智能体和上下文快照的追踪事件。
- **上下文传递规则。** 决定交接时移动什么：完整历史记录（成本高）、最后 N 条消息，或摘要。
- **交接护栏。** 交接给具有不同工具权限的专家必须经过身份验证——否则提示注入可能强制进行不必要的交接。
- **循环检测。** 两个智能体互相交接是一种常见的故障；用简单的最近 K 次环形检查来检测。
- **回退智能体。** 如果交接目标不存在，则回退到一个安全的默认值。

## 练习

1. 运行 `code/main.py`，分诊到退款智能体。确认第二轮的活跃智能体是退款。
2. 添加一个循环检测规则：如果相同的两个智能体连续交接了 3 次，则强制退出。设计回退方案。
3. 阅读 OpenAI Agents SDK 文档中关于交接过滤器的部分。实现一个“交接时总结”版本：即将退出的智能体在即将接管的智能体接手之前，将上下文压缩成要点摘要。
4. 比较 Swarm 交接与 GroupChatManager 选择器。哪种模式使提示注入更严重，为什么？
5. 阅读 Swarm cookbook (https://developers.openai.com/cookbook/examples/orchestrating_agents)。识别 Swarm 做出的一个明确设计决策，该决策被 OpenAI Agents SDK 改变或保留。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|------------|----------|
| 例程 | “智能体提示词” | 系统提示 + 工具列表。定义角色和可用交接。 |
| 交接 | “转移给另一个智能体” | 活跃智能体可以调用的一个工具，它返回一个新的 Agent。运行时切换活跃智能体。 |
| 无状态 | “运行之间没有记忆” | Swarm 不持久化任何东西；记忆是调用方的责任。 |
| 活跃智能体 | “现在谁在说话” | 当前持有对话的智能体。交接会改变这个状态。 |
| 上下文传递 | “交接时移动什么” | 关于传入智能体能看到什么历史记录的策略：完整、最后 N 条或摘要。 |
| 交接循环 | “智能体乒乓” | 两个智能体不断互相交接的故障模式。 |
| OpenAI Agents SDK | “生产级 Swarm” | 2025 年 3 月的后继者；在交接原语之上添加了会话、护栏和追踪。 |
| 交接过滤器 | “转移时的门控” | SDK 功能，用于在交接边界检查和修改上下文。 |
