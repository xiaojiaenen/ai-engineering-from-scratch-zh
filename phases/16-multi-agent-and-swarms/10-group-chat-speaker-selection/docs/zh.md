# 群聊与说话者选择

> AutoGen的GroupChat与AG2的GroupChat在N个代理间共享同一个对话；一个选择函数（LLM、轮询或自定义）决定下一个发言者。这是涌现式多智能体对话的原型——代理在静态图中不知道自己的角色，只是对共享消息池做出反应。AutoGen v0.2的GroupChat语义在AG2分支中得以保留；AutoGen v0.4将其重写为事件驱动的Actor模型。微软于2026年2月将AutoGen转入维护模式，并将其与Semantic Kernel合并为**微软智能体框架**（RC 2026年2月发布）。GroupChat原语在AG2和微软智能体框架中都得以保留——一次学习，处处可用。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**先修课程：** 阶段16 · 04（原语模型）
**时间：** 约60分钟

## 问题

当工作流程已知时，静态图（如LangGraph）非常出色。真实的对话并非静态：有时程序员会询问审阅者，有时会询问研究员，有时会询问撰稿人。为所有可能的交互硬编码会导致分支爆炸。你需要的是*代理对共享消息池做出反应*，并由某个函数决定下一个发言者。

这正是AutoGen的GroupChat所做的事情。

## 概念

### 结构形态

```
              ┌─── shared pool ────┐
              │   m1  m2  m3  ...  │
              └─────────┬──────────┘
                        │ (everyone reads all)
      ┌───────┬─────────┼─────────┬───────┐
      ▼       ▼         ▼         ▼       ▼
    Agent A  Agent B  Agent C  Agent D  Selector
                                           │
                                           ▼
                                  "next speaker = C"
```

每个代理都能看到每条消息。在每一轮，系统会调用一个选择函数来决定下一个发言者。

### 三种选择函数类型

**轮询。** 固定循环。确定性。线性扩展到N个代理，但忽略上下文——即使主题是法律审查，程序员也可能获得发言权。

**LLM选择。** 调用一个LLM，它读取近期的消息池并返回最佳的下一位发言者。具有上下文感知能力但速度慢：每轮都会增加一次LLM调用。这是AutoGen的默认方式。

**自定义。** 一个包含任意逻辑的Python函数。典型用法：带回退规则的LLM选择（例如，“在程序员发言后总是给验证者发言权”）。

### 可对话代理API

```
agent = ConversableAgent(
    name="coder",
    system_message="You write Python.",
    llm_config={...},
)
chat = GroupChat(agents=[coder, reviewer, tester], messages=[])
manager = GroupChatManager(groupchat=chat, llm_config={...})
```

`GroupChatManager`持有着选择器。当一个代理完成发言后，管理器会调用选择器，选择器返回下一个代理。循环持续直到满足终止条件。

### 终止

三种常见模式：

- **最大轮次。** 对总轮数设定硬上限。
- **“TERMINATE”标记。** 代理可以发出一个哨兵消息；管理器在检测到该消息时停止。
- **目标达成检查。** 一个轻量级的验证器在每轮运行，当任务完成时停止对话。

### AutoGen → AG2分裂与微软智能体框架合并

2025年初，微软开始围绕事件驱动的Actor模型对AutoGen（v0.4）进行重大重写。社区将AutoGen v0.2的GroupChat语义分支为AG2，保留了早期采用者已集成的API。

2026年2月，微软宣布AutoGen将进入维护模式，事件驱动的Actor模型被合并到**微软智能体框架**（2026年2月RC，现已与Semantic Kernel合并）。GroupChat概念在这两个发展路径中都得以保留；实现细节有所不同。AG2是兼容v0.2代码的首选上游项目。

### GroupChat适用场景

- **涌现式对话。** 你不想预先连接好所有可能的下一个发言者。
- **角色混合任务。** 程序员询问研究员，研究员询问档案员，档案员又反过来询问程序员。流程不是DAG（有向无环图）。
- **探索性问题解决。** 想象成“头脑风暴会议”，而不是“装配线”。

### 失效场景

- **严格确定性。** LLM选择器可能不一致。相同的提示，不同的运行，产生不同的下一位发言者。
- **谄媚级联。** 代理倾向于将发言权交给最自信的人。需要通过明确提示来对抗。
- **上下文膨胀。** 每个代理都读取每条消息；经过10轮后，上下文会变得巨大。使用投影（第15课）来限定视图范围。
- **热点说话者。** 一个代理因选择器偏爱其专长而主导对话。引入说话者平衡作为选择器特性。

### 群聊 vs. 主管

相同的原语，不同的默认设置：

- 主管：一个代理规划，其他代理执行。选择器是“询问规划者下一步做什么”。
- 群聊：所有代理都是对等的；选择器是基于共享消息池的一个函数。

两者都使用第04课的四个原语。群聊默认使用LLM选择的编排方式和完整的共享状态池。

## 实际使用

`outputs/skill-groupchat-selector.md` 为一个给定任务配置GroupChat选择器——轮询、LLM选择还是自定义，以及使用哪些选择器输入（近期消息、代理专长、轮次计数）。

## 交付清单

- **最大轮次上限。** 务必设置。对于典型任务，设为10-20。
- **说话者平衡指标。** 跟踪每个代理的发言次数；当不平衡超过阈值时发出警报。
- **终止标记。** `TERMINATE` 或一个专门的验证者代理。
- **投影或范围化内存。** 大约10条消息后，考虑为每个代理提供范围化的视图，以防止上下文膨胀。
- **选择器日志记录。** 对于LLM选择的变体，记录选择器的输入及其选择。否则调试将无法进行。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|------------|----------|
| GroupChat | “代理在一个聊天室里” | 共享消息池 + 选择函数。AutoGen / AG2原语。 |
| 说话者选择 | “谁下一个发言” | 选择下一个代理的函数。轮询、LLM选择或自定义。 |
| GroupChatManager | “会议主持人” | AutoGen组件，拥有选择器并循环处理轮次。 |
| ConversableAgent | “基础代理” | AutoGen基类；能够发送和接收消息的代理。 |
| 终止标记 | “停止词” | 结束聊天的哨兵字符串（通常是`TERMINATE`）。 |
| 热点说话者 | “一个代理主导” | 选择器持续选择同一个代理的失败模式。 |
| 上下文膨胀 | “池无限增长” | 每个代理读取所有先前消息；上下文随轮次增长。 |
| 投影 | “范围化视图” | 针对共享池的角色特定视图，以防上下文膨胀。 |
