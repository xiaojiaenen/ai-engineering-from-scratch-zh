# 生产扩展 — 队列、检查点与持久性

> 将多智能体系统扩展到数千并发运行需要**持久执行**。LangGraph 的运行时在每个超步骤后写入检查点，键值为 `thread_id`（默认使用 Postgres）；工作进程崩溃会释放租约，另一个工作进程将恢复。智能体可以无限期休眠以等待人类输入。**MegaAgent**（arXiv:2408.09955）为每个智能体运行了一个生产者-消费者队列，包含三种状态（空闲/处理/响应）和两层协调（组内聊天 + 组间管理员聊天）。**纤程/异步**优于每任务一线程用于 LLM 流式处理：线程 99% 的时间都在空闲等待 token，而纤程在 I/O 上协作式让出。反方观点：Ashpreet Bedi 的《扩展智能体软件》主张，在负载证明需要之前，**FastAPI + Postgres + 别无其他**就足够了——简单架构比预期走得更远。本课程构建一个持久检查点日志、一个带状态转换的每智能体工作队列、一个异步与线程对比演示，并最终落地务实的“从简开始”规则。

**类型：** 学习 + 构建
**语言：** Python（标准库，`asyncio`, `sqlite3`）
**先决条件：** 第16阶段 · 09（并行群体网络），第16阶段 · 13（共享内存）
**时间：** 约75分钟

## 问题

一个原型多智能体系统在单个笔记本上运行，包含三个智能体和一个内存中事件循环。你转向生产环境：

- 智能体有时会运行数小时（长时间研究、人机交互等待）。
- 工作进程会崩溃。重启会丢失状态。
- 峰值负载是平均负载的10倍；你需要水平扩展。
- 用户按智能体运行次数付费；你需要精确一次语义进行计费。

内存中事件循环无法实现这些功能。你需要一个底层的持久执行层。2026 年的典型选项包括：

1. 带检查点的工作流引擎（Temporal，LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每智能体生产者-消费者模式）。
4. 手工构建的 FastAPI + Postgres（Bedi 的论点）。

本课程将构建每种的微缩版本。

## 概念

### 持久执行，该模式

持久执行引擎在每个“步骤”（LangGraph 术语中称为“超步骤”）后持久化完整的程序状态。崩溃时：

```
worker crashes mid-step
  -> lease timeout
  -> another worker picks up the thread_id
  -> resumes from last checkpoint
  -> no duplicate side effects
```

实现此功能的要求：

- **可序列化状态。** 所有智能体状态必须可持久化。包含活跃数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体会产生相同的动作（或推迟到外部确定性预言机进行 LLM 调用）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的，或使用去重键。

LangGraph 在每个超步骤后写入检查点；Temporal 在每个活动后写入；Restate 使用事件溯源日志。三者都实现了相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化字典；每个超步骤向检查点表写入一行。恢复时，运行时从最后一个检查点重放，而不是从头开始。智能体可以 `interrupt()` 等待人类输入；运行时会持久化并释放工作进程。当输入到达时，任何工作进程都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每智能体队列

arXiv:2408.09955 描述了一个扩展实验：一个集群中数千个并发智能体。架构：

```
agent i:
  state ∈ {Idle, Processing, Response}
  in_queue   <- messages addressed to agent i
  out_queue  -> replies + side effects

coordinators:
  intra-group chat  (agents in the same group)
  inter-group admin chat  (high-level routing)
```

两层协调使得组内对话可以密集进行，而组间保持稀疏——这是保持数千智能体成本线性增长的模式。

### 异步 vs 每任务线程

LLM 调用是 I/O 密集型的。一个等待下一个 token 的线程 99% 的时间都是空闲的。每个线程大约消耗 1MB 内存；在 10,000 个并发调用时，仅栈就需要 10GB。

纤程（Python `asyncio`、Go goroutines、Rust `tokio`）在 I/O 上协作式让出。相同的 10,000 个调用可以轻松地装进进程。在 LLM 智能体规模下，异步不是优化——它是架构。

例外：CPU 密集型的后处理（嵌入、分词器技巧）仍然需要线程或进程。将你的 I/O 层与 CPU 层分开。

### Bedi 的反方观点

《扩展智能体软件》（Ashpreet Bedi，2026）认为，大多数团队在测量负载之前就过度工程化。务实的默认方案：

- FastAPI + Postgres。
- 每次智能体运行是一行；状态通过乐观并发原地更新。
- 后台任务通过 `pg_notify` 或简单的 Celery 工作进程。
- 在应用代码中设置重试策略。

对于管理任务上约 100 次以下并发智能体运行的负载，这通常就是你所需要的。当你测量到它失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，才采用持久执行框架。过早采用会把时间浪费在无回报的仪式上。

### 精确一次语义

对于付费的智能体运行，你需要“精确一次有效”（至少一次交付 + 幂等消费者）。工程要点：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用首先写入表，然后由单独的进程执行它们。两步都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，调度补偿。

这些是数据库工程模式，不是 LLM 特有的。LLM 的额外成本仅在于 LLM 调用缓慢；其余都是标准的分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用“彩虹部署”：多个版本的智能体运行时并发运行，这样长时间运行的智能体就不必在每次代码部署时被终止。在流量切片上金丝雀测试新版本；当其智能体完成时退役旧版本。

这是长时间运行有状态系统的标准做法；2026 年的调整是智能体可以存活数小时，因此部署周期必须适应。

### 典型生产检查清单

- 持久状态（检查点、快照或发件箱 + 可重放日志）。
- 幂等副作用。
- 用于 LLM 调用的异步 I/O 层。
- 至少一次交付与去重。
- 用于有状态工作负载的彩虹/金丝雀部署。
- 可观察性：每智能体追踪、超步骤审计、重试计数器。

## 构建它

`code/main.py` 实现了：

- `CheckpointStore` — 基于 SQLite 的检查点日志，以线程 ID 为键。每个超步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟运行中途崩溃；第二个工作进程从最后一个检查点恢复。
- `AgentQueue` — 每个智能体的空闲/处理/响应状态机，带有一个小型工作队列。
- `demo_async_vs_threads()` — 通过 asyncio 和线程运行 500 个并发模拟的“LLM 调用”；报告挂钟时间和峰值内存（近似值）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本在 < 1 秒内处理 500 个并发调用；线程版本需要几秒钟，并且每并发单位使用的内存量高出几个数量级。

## 交付它

典型的生产加固：

- **从简开始（Bedi 的规则）。** 在你测量到失败之前，使用 FastAPI + Postgres。
- **在优化前全面监控。** 每次运行的延迟直方图、每步骤时间、重试计数、失败分类。
- **副作用使用发件箱模式。** 尤其是支付和外部 API 调用。
- **彩虹部署。** 在部署期间永远不要终止进行中的智能体运行。
- **当你遇到特定问题时采用持久执行引擎（Temporal / LangGraph / Restate）：** 长达数小时的人机交互等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层使用异步。** 仅对 CPU 密集型后处理使用线程。

## 练习

1. 运行 `code/main.py`。确认检查点恢复有效；测量异步与线程并发差异。
2. 实现一个**发件箱**表：每个工具调用先写入发件箱，然后由单独的 goroutine/任务执行。通过运行工具调用两次来验证幂等性。
3. 模拟**彩虹部署**：两个并发的运行时版本；将一半的新 thread_id 路由到每个版本；确认旧版本上进行中的线程未被中断。
4. 阅读 LangGraph 的运行时文档（如下链接）。识别运行时的哪些功能在手工构建的 FastAPI + Postgres 版本中需要最长的时间来复制。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理员聊天）是明确的。概述你将如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| 持久执行 | "持久化程序状态" | 引擎在每个超步骤后写入状态；崩溃恢复是确定性的。 |
| 超步骤 | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性 | "重试安全" | 重复一次副作用产生的结果与一次尝试相同。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；单独的执行器执行并标记完成。 |
| 至少一次交付 | "可能有重复" | 消息队列语义；去重键使消费者有效一次。 |
| 彩虹部署 | "重叠版本" | 长时间运行工作负载期间多个运行时版本并发。 |
| 异步纤程 | "协作式让出" | 用户模式并发；对于 I/O 密集型负载，比线程更廉价。 |
| 检查点 | "状态快照" | 超步骤边界处的序列化状态；恢复的关键。 |
