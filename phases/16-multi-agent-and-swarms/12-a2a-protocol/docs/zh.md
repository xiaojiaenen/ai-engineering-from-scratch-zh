# A2A — 智能体间通信协议

> 谷歌于2025年4月发布A2A协议；截至2026年4月，规范详见 https://a2a-protocol.org/latest/specification/，已有150多家机构支持。A2A是MCP（第13课）的横向补充：MCP处理纵向关系（智能体↔工具），A2A则实现对等通信（智能体↔智能体）。它定义了智能体卡（发现机制）、带生成物（文本/结构化数据/视频）的任务、非透明任务生命周期及认证机制。生产系统越来越多地将MCP与A2A结合使用。谷歌云在2025-2026年间已将A2A支持集成至Vertex AI智能体构建器。

**类型：** 学习+实践
**语言：** Python（标准库，`http.server`，`json`）
**前置要求：** 阶段16 · 04（基础模型）
**时长：** 约75分钟

## 问题描述

您的智能体需要调用另一个系统上的智能体，该如何实现？可以暴露HTTP端点、定义自定义JSON模式，并期望对方遵循——但每对智能体都需要定制化集成。

A2A正是为此场景设计的通用通信协议。它提供标准发现机制、标准任务模型、标准传输协议和标准生成物格式。如同HTTP+REST将服务作为一等公民，A2A将智能体作为一等公民。

## 核心概念

### 四要素

**智能体卡。** 位于`/.well-known/agent.json`的JSON文档，描述智能体的名称、技能、端点、支持模态及认证要求。发现机制通过读取该卡实现。

```
GET https://agent.example.com/.well-known/agent.json
→ {
    "name": "code-review-agent",
    "skills": ["review-python", "review-typescript"],
    "endpoints": {
      "tasks": "https://agent.example.com/tasks"
    },
    "auth": {"type": "bearer"},
    "modalities": ["text", "structured"]
  }
```

**任务。** 工作单元。具有生命周期的异步有状态对象：`submitted → working → completed / failed / canceled`。客户端发送任务后，通过轮询或订阅获取更新。

**生成物。** 任务产生的结果类型，包括文本、结构化JSON、图像、视频、音频。生成物具有类型标识，使不同模态成为一等数据。

**非透明生命周期。** A2A不限定远端智能体解决任务的具体方式。客户端仅观察状态转换和生成物；实现方可自由选择任何框架。

### MCP/A2A分工

- **MCP**（第13课）：智能体↔工具。智能体通过JSON-RPC读写工具服务器，默认无状态。
- **A2A**：智能体↔智能体。对等协议；双方均为具有自主推理能力的智能体。

生产级多智能体系统通常两者兼用。A2A对等方可调用其本地的MCP工具。这种分工使两个关注点保持清晰。

### 发现流程

```
Client                     Agent server
  ├──GET /.well-known/agent.json──>
  <──Agent Card JSON─────────────
  ├──POST /tasks {skill, input}──>
  <──201 task_id, state=submitted
  ├──GET /tasks/{id}──────────────>
  <──state=working, 42% done──────
  ├──GET /tasks/{id}──────────────>
  <──state=completed, artifacts──
```

或使用流式方式：订阅`/tasks/{id}/events`的SSE连接以获取推送更新。

### 认证机制

A2A支持三种常见模式：

- **承载令牌** — OAuth2或不透明令牌
- **双向TLS** — 组织间互相验证身份
- **签名请求** — 对载荷进行HMAC计算

认证方式在智能体卡中声明；客户端发现后依此执行。

### 截至2026年4月已有150+机构支持

企业级应用推动了A2A的规模化。关键突破在于：A2A成为企业智能体系统跨越信任边界的标准方式。谷歌云在Vertex AI智能体构建器中实现了A2A支持；微软智能体框架也支持该协议；主流框架（LangGraph、CrewAI、AutoGen）均提供A2A适配器。

### A2A的优势领域

- **跨组织调用。** A公司智能体调用B公司智能体。若无A2A，每对组合都需要定制协议。
- **异构框架兼容。** LangGraph智能体可调用CrewAI智能体，再调用自定义Python智能体。A2A实现标准化通信。
- **类型化生成物。** 视频结果、结构化JSON、音频均为一等数据类型。
- **长时间运行任务。** 非透明生命周期+轮询机制使数小时的任务也能顺畅处理。

### A2A的局限性

- **延迟敏感的微调用。** A2A生命周期为异步模式。亚毫秒级智能体间通信不适合此协议，建议使用直接RPC。
- **紧密耦合的进程内智能体。** 若两个智能体运行在相同Python进程中，A2A的HTTP往返开销过大。
- **小型团队。** 规范开销真实存在；仅内部使用的智能体可能无需此形式化协议。

### A2A与ACP、ANP、NLIP对比

2024-2026年间出现多个相关规范：

- **ACP**（IBM/基金会）— A2A的前身，适用范围更窄
- **ANP**（智能体网络协议）— 侧重对等发现，优先去中心化
- **NLIP**（Ecma自然语言交互协议，2025年12月标准化）— 自然语言内容类型

截至2026年4月，A2A是采用最广泛的对等协议。详见arXiv:2505.02279（Liu等人《智能体互操作协议综述》）的对比分析。

## 实践构建

`code/main.py`使用`http.server`和JSON实现了A2A最小服务器和客户端。服务器：

- 暴露`/.well-known/agent.json`
- 接受`POST /tasks`
- 管理任务状态
- 在`GET /tasks/{id}`时返回生成物

客户端：

- 获取智能体卡
- 提交任务
- 轮询直至完成
- 读取生成物

运行命令：

```
python3 code/main.py
```

脚本在后台线程启动服务器，然后运行客户端进行测试。您将看到完整流程：发现、提交、轮询、获取生成物。

## 部署清单

- **固定规范版本。** A2A仍在演进；智能体卡应声明协议版本
- **幂等任务创建。** 重复提交（网络重试）应仅产生单个任务
- **生成物模式声明。** 声明智能体返回的数据结构；消费者应进行验证
- **速率限制+认证。** A2A面向公网；需实施标准Web安全措施
- **失败任务死信队列。** 定期检查模式以识别重复出现的故障类型

## 练习

1. 运行`code/main.py`，确认客户端能发现服务器并接收正确生成物
2. 为服务器添加第二个技能（如"摘要"），更新智能体卡。编写根据任务类型选择技能的客户端
3. 实现SSE流式端点：`/tasks/{id}/events`用于推送状态变更。客户端需要哪些不同处理？
4. 阅读A2A规范（https://a2a-protocol.org/latest/specification/），找出规范要求但本演示未实现的三个方面
5. 比较A2A（智能体卡发现）与MCP（通过`listTools`的服务器能力列表）。自描述智能体与能力探测之间如何权衡？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| A2A | "智能体间通信" | 跨系统智能体间调用的对等协议，谷歌2025年推出 |
| 智能体卡 | "智能体名片" | 位于`/.well-known/agent.json`的JSON文档，描述技能、端点、认证信息 |
| 任务 | "工作单元" | 具有生命周期的异步有状态对象；完成时生成产物 |
