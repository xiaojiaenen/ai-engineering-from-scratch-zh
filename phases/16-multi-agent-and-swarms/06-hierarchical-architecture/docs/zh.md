# 分层架构及其失败模式

> 分层即监督者嵌套。经理代理管理子经理，子经理管理工作者。CrewAI `Process.hierarchical` 是教科书版本：`manager_llm` 动态分配任务并验证输出。LangGraph 的等效实现是 `create_supervisor(create_supervisor(...))`。当任务本质上是组织架构图时，这是自然的模式。但它也是最有可能陷入管理循环的模式——经理代理分配任务不当、误解子代理输出，或未能达成共识。顺序模式常常优于它。

**类型：** 学习 + 构建
**语言：** Python (标准库)
**先决条件：** 阶段 16 · 05 (监督者模式)
**时间：** 约 60 分钟

## 问题

一旦监督者模式融会贯通，自然的下一步是“如果工作者本身就是监督者呢？”。团队下有小组；公司里有部门再下设部门。分层架构映射了这一点。

问题在于：LLM 经理与人类经理不同。人类经理对其下属的能力有稳定的先验认知。LLM 经理每次推理都基于其上下文中的一切信息重新思考组织结构。上下文中微小的漂移，就可能导致整棵树错误分配工作。

## 概念

### 形态

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点进行规划、分配和综合。只有叶节点执行实际工作。

### 适用场景

- **清晰的组织映射。** 如果真实任务是部门化的（“法务部审查文档，财务部审查文档，工程部审查文档，然后为高管总结”），层级结构是明确的。
- **局部总结。** 每个子经理在顶层经理看到之前，先综合其团队的输出。顶层经理看到的是三个子经理的总结，而不是十五个工作者的输出。

### 失效模式

2026 年的事后复盘不断发现三种失败模式：

1. **任务分配错误。** 经理理解目标，幻想出一种分解方式，并将任务分配给了错误的子经理。由于子经理会忠实地处理分配给它的工作，错误直到顶层综合时才暴露出来——这已经距离人类本可以发现错误的地方相差了一个层级。
2. **输出误解。** 子经理返回“无法验证声明 X”。顶层经理总结为“声明 X 未得到确认”。含义在每个层级都发生漂移。
3. **共识循环。** 两个子经理意见不合；顶层经理要求他们协调；他们重新向下分配；工作者重新运行；子经理返回略微不同的答案；陷入循环。CrewAI 的 `Process.hierarchical` 通过步骤限制来防止这种情况，但这个限制本身现在是一个超参数。

### 决定性问题

顺序（线性流水线）模式 vs 分层模式：你的任务是否真的拥有独立的子团队，还是只是一个伪装成树的线性流程？如果是后者，请使用顺序模式。如果是前者，请使用分层模式，但要预算明确的协调规则。

### CrewAI 的实现

`Process.hierarchical` 将一个经理 LLM 连接到专家团队上。经理：

- 接收顶层任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定接受、重新分配或迭代。

文档：https://docs.crewai.com/en/introduction （在核心概念下查找“分层流程”）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为一个不透明的节点。这在调试方面比 CrewAI 更清晰（你可以分别单步执行每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor

## 构建它

`code/main.py` 运行一个三级层级结构：

- 顶层经理：将任务分解为“工程”和“法务”两个分支，
- 工程子经理：分解为“前端”和“后端”工作者，
- 法务子经理：一个工作者。

演示对比了正常路径（所有人达成一致）与一个**扰动路径**：顶层经理的分解错误地将“法务”标记为“财务”，并观察错误级联——子经理忠实地执行财务工作，顶层综合者报告财务发现，原始的法务问题悬而未决。

运行：

```
python3 code/main.py
```

输出显示了两种路径，清晰并列对比“询问内容”与“交付内容”。

## 应用它

`outputs/skill-hierarchy-fitness.md` 评估给定任务应使用分层、顺序还是扁平监督者模式。输入：任务描述、组织结构、协调预算。输出：模式建议以及需要防范的具体失败模式。

## 部署它

如果要部署分层架构：

- **限制树深度为 2 层。** 三层已经足以让大部分错误无法被观测。
- **明确的协调预算。** 设置顶层经理必须做出决定前的最大轮次。通常为 2。
- **每次综合都附带来源。** 每个节点的总结必须注明是基于哪些叶节点的输出生成的。
- **监控分解漂移。** 记录经理每一步的分解方式；与用户查询进行比对。如果分解不再覆盖查询，则触发警报。

## 练习

1. 运行 `code/main.py` 并比较正常路径与扰动路径。经过多少层级的经理传递后，顶层输出会完全偏离用户的问题？
2. 增加第三级（顶层 → 子经理 → 子子经理 → 工作者）。测量随着深度增加，扰动路径是自我纠正还是完全发散的频率。
3. 在每个子经理处实现一个“金丝雀”工作者，它始终被问及原始用户问题（不做修改）。使用金丝雀的答案来检测分解漂移。当金丝雀与综合答案不一致时，经理应如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体防护措施（步骤限制、manager_llm 约束），并描述其针对哪种失败模式。
5. 比较嵌套的 LangGraph 监督者与 CrewAI 分层模式。哪种模式更容易检测协调循环？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------|----------|
| 分层 (Hierarchical) | “组织架构图模式” | 监督者管理监督者；只有叶节点执行实际工作。 |
| 经理 LLM (Manager LLM) | “老板” | 在内部节点进行分解、分配和验证的 LLM。 |
| 分解漂移 (Decomposition drift) | “老板跑题了” | 顶层经理的分解不再覆盖原始问题。 |
| 协调循环 (Reconciliation loop) | “无尽的会议” | 子经理意见不合；顶层重新分配；工作者重新运行；循环直至预算耗尽。 |
| 深度-2 上限 (Depth-2 ceiling) | “别超过 2 层” | 经验性防护措施：3 层以上会丧失可观测性。 |
| 金丝雀问题 (Canary question) | “每一层的真相基准” | 一个工作者始终被问及原始查询（不做修改），用于检测漂移。 |
| 来源链 (Provenance chain) | “谁说了什么” | 从每次综合回溯到产生它的叶节点输出。 |
