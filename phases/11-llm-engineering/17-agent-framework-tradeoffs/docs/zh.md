# 代理框架权衡 — LangGraph vs CrewAI vs AutoGen vs Agno

> 每个框架都兜售相同的演示（研究代理生成报告），并隐藏相同的缺陷（状态模式与编排层冲突）。选择其抽象模型匹配你问题形态的框架；其他一切皆是重复编写的胶水代码。

**类型：** 学习
**语言：** Python
**前置要求：** 第11阶段 · 09（函数调用），第11阶段 · 16（LangGraph）
**时间：** ~45分钟

## 问题所在

你面临一个需要多次LLM调用的任务。它可能是一个研究工作流（规划、搜索、总结、引用）。也可能是一个代码审查流程（解析差异、评审、修补、验证）。或许是一个多轮对话助手，需要预订航班、撰写邮件并提交报销报告。于是你选择了一个框架。

三天后，你发现该框架的抽象存在漏洞。CrewAI赋予你角色设定，但当"研究员"需要将结构化方案传递给"撰写者"时便会冲突。AutoGen提供代理间的对话能力，但缺乏一流的状态管理，导致你的检查点不过是对话日志的pickle序列化。LangGraph提供状态图，却强制你在明确代理行为前就命名每个转换。Agno提供单一代理原语，但当你尝试扇出至三个并发工作者时便捉襟见肘。

解决方案并非"选择最佳框架"，而是将框架的核心抽象与你问题的形态相匹配。本课将绘制这幅对应地图。

## 核心概念

![代理框架矩阵：核心抽象 vs 问题形态](../assets/framework-matrix.svg)

四大框架主导2026年的技术版图。它们的核心抽象并不相同。

| 框架 | 核心抽象 | 最佳适用场景 | 最差适用场景 |
|-----------|------------------|----------|-----------|
| **LangGraph** | `StateGraph` — 类型化状态、节点、条件边、检查点器。 | 具有显式状态和人机回环中断的工作流；需要时间旅行调试的生产级代理。 | 拓扑未知的松散、角色驱动型头脑风暴。 |
| **CrewAI** | `Crew` — 角色（目标、背景故事）、任务、流程（顺序或层级）。 | 具有短期线性/层级规划的角色扮演或人格驱动工作流。 | 任何超出团队轮换历史的状态化需求；复杂分支逻辑。 |
| **AutoGen** | `ConversableAgent` 对话组 — 两个或多个代理轮流对话直至满足退出条件。 | 思考过程源于对话的多代理*对话*（师生、提议者-评审者、演员-审阅者）。 | 具有已知DAG的确定性工作流；任何需要跨重启持久化状态的需求。 |
| **Agno** | `Agent` — 单个LLM + 工具 + 记忆，可组合成团队。 | 快速构建的单代理及轻量团队；强大的多模态能力与内置存储驱动。 | 具有自定义归约器的深度显式分支图。 |

### "抽象"的实际含义

框架的核心抽象是你在白板上勾勒架构时绘制的概念。

- **LangGraph** → 你绘制一个图。节点代表步骤，边代表转换，每个节点的状态对象都是类型化的。心智模型是一个状态机。
- **CrewAI** → 你绘制一个组织结构图。每个角色都有职位描述，由经理分配任务。心智模型是一个由专家组成的小团队。
- **AutoGen** → 你绘制一个Slack私聊窗口。两个代理相互发信息；如需仲裁者，第三个代理会加入。心智模型是对话。
- **Agno** → 你绘制一个挂载工具的方框。并排放置方框即组成团队。心智模型是"开箱即用的代理"。

### 状态问题

状态是大多数框架选择在生产环境中失效的关键。

- **LangGraph。** 类型化状态（`TypedDict` 或 Pydantic 模型），逐字段归约器，一流的检查点器（SQLite/Postgres/Redis）。恢复、中断和时间旅行皆为免费功能。*（参见第11阶段 · 16）*
- **CrewAI。** 状态通过 `context` 字段在任务间以字符串形式流动，或通过 `output_pydantic` 进行结构化传递。开箱即用不含可持久化的团队专属存储；若团队需跨重启存活，需自行添加。
- **AutoGen。** 状态即聊天历史及任何用户定义的 `context`。对话记录可持久化；任意工作流状态若不编写适配器则无法保留。
- **Agno。** 通过 `storage=` 将内置存储驱动（SQLite, Postgres, Mongo, Redis, DynamoDB）挂载至 `Agent` —— 对话会话与用户记忆自动持久化。这不是完整的图检查点器，而是会话存储。

### 分支问题

任何非平凡的代理都会产生分支。谁决定分支至关重要。

- **LangGraph** — 由你决定，通过条件边。路由是一个具有命名分支的Python函数。分支在编译后的图中是一等公民；检查点器会记录所选取的分支。
- **CrewAI** — 在层级模式下由经理决定；在顺序模式下，你在构建时决定。路由隐含在任务列表中；除了经理的提示词外，没有一等的"if"语句。
- **AutoGen** — 由代理通过对话决定。分支从"谁接下来发言"中涌现。`GroupChatManager` 选择下一个发言者；你可以手写 `speaker_selection_method`，但默认由LLM驱动。
- **Agno** — 由代理决定下一步调用哪个工具。团队拥有协调者/路由器/协作模式；超越此范围的分支则是开发者的责任。

### 可观测性问题

- **LangGraph** — 通过LangSmith或任何OTel导出器集成OpenTelemetry。每个节点转换都是一个跟踪跨度；检查点同时作为可重放的跟踪记录。LangSmith是首选官方方案；Langfuse/Phoenix也有适配器。
- **CrewAI** — 自2025年末起支持一流的OpenTelemetry；与Langfuse、Phoenix、Opik、AgentOps集成。
- **AutoGen** — 通过 `autogen-core` 集成OpenTelemetry；AgentOps和Opik有连接器。跟踪粒度是每代理-每消息，而非每节点。
- **Agno** — 内置 `monitoring=True` 标志加上OpenTelemetry导出器；与Langfuse紧密集成以跟踪会话。

### 成本与延迟

所有四个框架都会增加每次调用的开销（框架逻辑、验证、序列化）。大致的开销递增顺序为：Agno ≈ LangGraph < CrewAI ≈ AutoGen。差异主要取决于框架执行了多少额外的LLM路由。CrewAI的层级经理消耗token来决定下一步执行者；AutoGen的 `GroupChatManager` 同样如此。LangGraph仅在您编写 `llm.invoke` 的地方消耗token。Agno的单代理路径则很轻量。

当每次运行成本至关重要时，优先选择显式路由（LangGraph的边，AutoGen的 `speaker_selection_method`）而非LLM选择的路由。

### 互操作性

- **LangGraph** ↔ **LangChain** 工具、检索器、LLM。一流MCP适配器（工具作为MCP服务器导入）。
- **CrewAI** ↔ 工具继承自 `BaseTool`；LangChain工具、LlamaIndex工具和MCP工具均可适配。通过 `allow_delegation=True` 实现团队间委托。
- **AutoGen** → `FunctionTool` 封装任何Python可调用对象；提供MCP适配器。与AG2生态在代理对代理模式上紧密耦合。
- **Agno** → `@tool` 装饰器或BaseTool子类；MCP适配器；工具可在代理和团队间共享。

## 技能要点

> 你能用一句话解释，为何某个特定框架适合某个特定代理问题。

构建前检查清单：

1. **描绘形态。** 这是一个图（类型化状态，命名转换）？一个角色扮演（专家们交接工作）？一个对话（代理交谈直至完成）？一个配备工具的单一代理？
2. **决定分支由谁掌控。** 开发者决定分支 → LangGraph。经理代理决定 → CrewAI层级模式。对话涌现 → AutoGen。工具调用决定 → Agno。
3. **检查状态预算。** 你需要从检查点恢复吗？时间旅行？运行中的人机中断？若需要，LangGraph是默认选择；Agno会话可覆盖对话作用域的状态。
4. **检查成本预算。** LLM选择的路由每轮次会额外消耗token。如果代理每天运行数千次，请优先选择显式路由。
5. **估算框架开销。** 每个框架都是额外依赖。如果任务只是两次LLM调用和一个工具，用30行普通Python代码编写；没有框架比没有框架更便宜。

在能画出图、组织结构图、对话或代理方框之前，切勿贸然引入框架。切勿选择一个迫使你与其状态模型斗争以满足实际需求的框架。

## 决策矩阵

| 问题形态 | 首选框架 | 原因 |
|---------------|---------------------|-----|
| 具有类型化状态、人工审批、长时间运行的工作流DAG | LangGraph | 一流的状态管理、检查点器、中断、时间旅行。 |
| 具有不同角色的研究/写作流程 | CrewAI（顺序）或 LangGraph 子图 | 每个任务一个角色在CrewAI中表达成本低；当分支变得复杂时，可用LangGraph扩展。 |
| 提议者-评审者或师生对话 | AutoGen | 两人聊天是其原生形态。 |
| 具备工具、会话、记忆的单一代理 | Agno | 最精简的设置，内置存储和记忆。 |
| 具有归约器的数千并行扇出 | LangGraph + `Send` | 唯一拥有一流并行分发原语的框架。 |
| 快速原型设计，不绑定框架 | 纯Python + 提供者SDK | 没有框架就是最快的框架。 |

## 练习

1. **简单。** 取相同任务——"研究Anthropic的总部，撰写200字简介，引用来源"——分别在LangGraph（四个节点：规划、搜索、撰写、引用）和CrewAI（三个角色：研究员、撰写者、编辑）中实现。报告每次运行的token成本和代码行数。
2. **中等。** 在AutoGen（研究员 ↔ 撰写者对话，编辑通过 `GroupChat` 加入）和Agno（单个代理配备 `search_tools` 和 `write_tools`，加上会话存储）中构建相同任务。根据 (a) 每次运行成本，(b) 崩溃后恢复能力，(c) 在撰写步骤前注入人工审批的能力，对四种实现进行排名。
3. **困难。** 构建一个决策树脚本 `pick_framework.py`，输入简短的问题描述（JSON：`{has_typed_state, has_roles, has_dialogue, has_parallel_fanout, needs_resume}`），返回带有一句话理由的建议。在你自己设计的六个案例上进行验证。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|-----------------|-----------------------|
| 编排 | "代理如何协作" | 决定哪个节点/角色/代理接下来运行的层。 |
| 持久化状态 | "重启后恢复" | 能够在进程终止后存活的状态，附加到检查点或会话存储。 |
| LLM选择路由 | "让模型决定" | 每轮次由规划LLM选择下一步；灵活但每次决策都消耗token。 |
| 显式路由 | "开发者决定" | 由Python函数或静态边选择下一步；成本低且可审计。 |
| 团队 | "CrewAI团队" | 绑定为单一可运行单元的角色 + 任务 + 流程（顺序或层级）。 |
| 群聊 | "AutoGen的多代理对话" | 一个由N个代理参与、带发言选择器的托管对话。 |
| 团队 (Agno) | "多代理Agno" | 对一组代理进行路由/协调/协作的模式。 |
| 状态图 | "LangGraph的图" | 类型化状态、节点、条件边、检查点器原语。 |
