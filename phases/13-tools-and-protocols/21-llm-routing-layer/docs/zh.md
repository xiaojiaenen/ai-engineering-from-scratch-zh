# LLM 路由层 — LiteLLM、OpenRouter、Portkey

> 供应商锁定成本高昂。不同的工具调用工作负载适合不同的模型。路由网关提供统一的API接口、重试、故障转移、成本跟踪和防护栏。2026年主导着三种原型：LiteLLM（开源自托管）、OpenRouter（托管SaaS）、Portkey（生产级，于2026年3月开源）。本课阐述决策标准，并演练一个标准库路由网关。

**类型：** 学习
**语言：** Python（标准库、路由 + 故障转移 + 成本跟踪器）
**先决条件：** 阶段13 · 02（函数调用），阶段13 · 17（网关）
**时间：** ~45分钟

## 学习目标

- 区分自托管、托管和生产级路由选项。
- 实现一个回退链，在提供程序故障时按定义的优先级顺序进行重试。
- 跨提供程序跟踪每请求成本和令牌使用量。
- 根据特定生产约束在LiteLLM、OpenRouter和Portkey之间做出决策。

## 问题场景

供应商路由至关重要的场景：

1.  **成本。** Claude Sonnet的价格是Haiku的三倍。对于分类任务，Haiku就足够了；对于综合任务，Sonnet物有所值。按请求路由。
2.  **故障转移。** OpenAI有一段时间服务不佳。每个请求都失败。你希望自动回退到Anthropic，而无需重新部署。
3.  **延迟。** 实时聊天UI需要快速的首令牌时间。批量摘要生成器则不需要。按延迟SLA路由。
4.  **合规性。** 欧盟用户必须留在欧盟区域。按区域路由。
5.  **实验。** 在同一工作负载上对两个模型进行A/B测试。按测试分组路由。

为每个集成手动编写所有这些代码是重复的。路由网关提供一个兼容OpenAI的API，并处理其余部分。

## 核心概念

### 兼容OpenAI的代理模式

大家都使用OpenAI格式。路由网关暴露 `/v1/chat/completions`，接受OpenAI模式，并在内部将请求代理到Anthropic / Gemini / Cohere / Ollama / 其他任何后端。客户端无需关心。

### 模型别名

你的代码不使用 `claude-3-5-sonnet-20251022`，而是使用 `our_smart_model`。网关将别名映射到实际模型。当Anthropic发布Claude 4时，你在服务器端更改别名；你的代码无需任何改动。

### 回退链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义此回退链。重试次数计入预算，以避免回退级联导致成本爆炸。

### 语义缓存

相同或近乎相同的提示词会命中缓存，而不是访问提供程序。在重复的代理循环中可节省30%至60%的成本。键基于嵌入；近乎相同的提示词共享一个缓存槽位。

### 防护栏

网关级别：
- **PII（个人身份信息）编辑。** 在发送提示词前通过正则表达式或基于ML的通道。
- **策略违规。** 拒绝包含禁止内容的提示词。
- **输出过滤器。** 清理补全内容以防止信息泄露。

Portkey和Kong都提供了有主见的防护栏。LiteLLM则将它们设为可选。

### 每密钥速率限制

一个API密钥 = 一个团队。每密钥预算可防止一个团队耗尽共享配额。大多数网关都支持此功能。

### 自托管与托管的权衡

| 因素 | LiteLLM（自托管） | OpenRouter（托管） | Portkey（生产级） |
|------|----------------------|----------------------|----------------------|
| 代码 | 开源，Python | 托管SaaS | 开源（2026年3月）+ 托管 |
| 设置 | 部署代理 | 注册账号 | 两者皆可 |
| 提供程序 | 100+ | 300+ | 100+ |
| 计费 | 使用你自己的密钥 | OpenRouter信用额度 | 使用你自己的密钥 |
| 可观测性 | OpenTelemetry | 仪表板 | 完整的OTel + PII编辑 |
| 最适合 | 需要完全控制的团队 | 快速原型设计 | 需要合规性的生产环境 |

当你拥有SRE团队并希望数据主权时，LiteLLM胜出。当你想要单一订阅且无需基础设施时，OpenRouter胜出。当你需要开箱即用的防护栏和合规性时，Portkey胜出。

### 成本跟踪

每个请求都携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以每模型每令牌价格（从网关维护的价格表中获取）。按用户/团队/项目聚合。

### MCP与路由结合

网关可以同时路由LLM调用**和**MCP采样请求。当采样请求的`modelPreferences`偏好特定模型时，网关会将其翻译到正确的后端。这就是阶段13 · 17（MCP网关）和本课的路由网关有时会合并为一个服务的地方。

### 路由策略

- **静态优先级。** 列表中的第一个；出错时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 选择满足延迟/质量要求的最便宜模型。
- **延迟感知。** 选择最近N分钟内最快的模型。
- **任务感知。** 提示分类器将编码任务路由到一个模型，摘要任务路由到另一个模型。

## 实践应用

`code/main.py` 用大约150行代码实现了一个路由网关：接受OpenAI格式的请求，翻译为各提供程序的存根，运行优先级回退链，跟踪每请求成本，并在输入上应用PII编辑。运行它进行三个场景测试：正常请求、主提供程序故障触发回退、PII泄露被编辑捕获。

需要关注的地方：
- `ROUTES` 字典：别名 -> 按优先级排序的具体提供程序列表。
- 回退循环在遇到5xx错误时重试。
- 成本跟踪器将令牌使用量乘以每模型费率。
- PII编辑器在转发前清除SSN（社会安全号码）格式的模式。

## 练习

1.  运行 `code/main.py`。触发故障场景；确认回退成功落到第二个提供程序，并且成本正确归属。
2.  添加语义缓存：提示词的SHA256值作为查找键；缓存命中则立即返回。测量在重复调用上的成本节省。
3.  添加一个提示分类器，将“code ...”提示路由到偏好智能的别名，将“summarize ...”提示路由到偏好速度的别名。
4.  设计每团队预算：每个团队有月度支出上限；一旦达到上限，网关即拒绝请求。选择执行粒度（按请求或按窗口）。
5.  并排阅读LiteLLM、OpenRouter和Portkey的文档。指出每个网关独有的、另外两个没有的一项功能。

## 关键术语

| 术语 | 人们的说法 | 其实际含义 |
|------|----------------|------------------------|
| 路由网关 | "LLM代理" | 位于多个提供程序之上的统一API层 |
| 兼容OpenAI | "说OpenAI模式" | 接受 `/v1/chat/completions` 格式，翻译为任意后端 |
| 模型别名 | "our_smart_model" | 你代码中的名称，由网关映射到具体模型 |
| 回退链 | "重试列表" | 在故障时尝试的有序提供程序列表 |
| 语义缓存 | "提示词嵌入缓存" | 键是提示词的嵌入；近似重复共享缓存命中 |
| 防护栏 | "输入/输出过滤器" | 编辑PII，拒绝策略违规 |
| 每密钥速率限制 | "团队预算" | 限定于某个API密钥的配额 |
| 成本跟踪 | "每请求支出" | 聚合令牌使用量 x 每模型价格 |
| LiteLLM | "开源代理" | 可自托管的开源路由网关 |
| OpenRouter | "托管SaaS" | 基于信用额度计费的托管网关 |
| Portkey | "生产选项" | 内置防护栏的开源 + 托管方案 |
