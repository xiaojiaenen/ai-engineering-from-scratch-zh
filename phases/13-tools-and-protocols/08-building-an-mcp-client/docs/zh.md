# 构建 MCP 客户端 —— 发现、调用与会话管理

> 大多数 MCP 内容都提供服务器端教程，对客户端部分则一笔带过。客户端代码才是硬核编排所在：进程生成、能力协商、跨多服务器的工具列表合并、采样回调、重连机制以及命名空间冲突解决。本课程将构建一个多服务器客户端，将三个不同的 MCP 服务器提升到一个扁平的工具命名空间中供模型使用。

**类型：** 构建
**语言：** Python（标准库，多服务器 MCP 客户端）
**前置要求：** 阶段 13 · 07（构建 MCP 服务器）
**时长：** 约 75 分钟

## 学习目标

- 生成一个 MCP 服务器作为子进程，完成 `initialize`，并发送 `notifications/initialized`。
- 维护每个服务器的会话状态（能力、工具列表、最后收到的通知 ID）。
- 将多个服务器的工具列表合并到一个命名空间，并处理冲突。
- 将工具调用路由到拥有该工具的服务器，并重组响应。

## 问题所在

一个真实的代理主机（Claude Desktop、Cursor、Goose、Gemini CLI）会同时加载多个 MCP 服务器。用户可能同时运行文件系统服务器、Postgres 服务器和 GitHub 服务器。客户端的工作是：

1.  生成每个服务器。
2.  分别与每个服务器进行握手。
3.  在每个服务器上调用 `tools/list` 并将结果扁平化。
4.  当模型发出 `notes_search` 时，在合并后的命名空间中查找并路由到正确的服务器。
5.  非阻塞地处理来自任何服务器的通知（`tools/list_changed`）。
6.  传输失败时重新连接。

手动实现所有这些是“玩具级”与“可用级”应用的分界线。官方 SDK 封装了这些功能，但理解其心智模型至关重要。

## 核心概念

### 子进程生成

使用 `subprocess.Popen` 并配合 `stdin=PIPE, stdout=PIPE, stderr=PIPE`。设置 `bufsize=1` 并使用文本模式进行逐行读取。每个服务器是一个进程；客户端为每个服务器持有一个 `Popen` 句柄。

### 每服务器会话状态

一个 `Session` 对象为每个服务器保存：

- `process` — Popen 句柄。
- `capabilities` — 服务器在 `initialize` 时声明的内容。
- `tools` — 最近一次 `tools/list` 的结果。
- `pending` — 请求 ID 到等待响应的 promise/future 的映射。

请求本质上是异步的；在服务器 B 进行调用期间向服务器 A 发送的 `tools/call` 必须不能阻塞。可以使用带队列的线程或 asyncio。

### 合并命名空间

当客户端看到聚合的工具列表时，名称可能冲突。两个服务器可能都暴露了 `search`。客户端有三种选择：

1.  **按服务器名加前缀。** `notes/search`，`files/search`。清晰但不够美观。
2.  **静默的先到先得。** 后来的服务器的 `search` 覆盖先前的。有风险；隐藏了冲突。
3.  **冲突拒绝。** 拒绝加载第二个服务器；通知用户。对安全敏感的主机最安全。

Claude Desktop 使用按服务器名加前缀。Cursor 使用冲突拒绝并给出明确错误。VS Code MCP 也采用按服务器名加前缀。

### 路由

合并后，一个调度表将 `tool_name -> session` 映射。模型发出一个按名称的调用；客户端找到会话并向该服务器的 stdin 写入 `tools/call` 消息，然后等待响应。

### 采样回调

如果服务器在 `initialize` 时声明了 `sampling` 能力，它可能会发送 `sampling/createMessage` 要求客户端运行其 LLM。客户端必须：

1.  在采样完成前，阻塞对该服务器的进一步请求，或者如果其实现支持并发则进行流水线处理。
2.  调用其 LLM 提供商。
3.  将响应发送回服务器。

课程 11 涵盖了端到端的采样。本课程为完整性提供了其存根。

### 通知处理

`notifications/tools/list_changed` 意味着重新调用 `tools/list`。`notifications/resources/updated` 意味着如果资源正在使用中则重新读取。通知不应产生响应 —— 不要尝试确认它们。

一个常见的客户端 bug：在流中存在通知时，阻塞了对 `tools/call` 的读取循环。使用一个后台读取线程，将每条消息推入队列；主线程从队列中取出并分发。

### 重连

传输可能失败：服务器崩溃、操作系统终止了进程、stdio 管道断开。客户端在 stdout 上检测到 EOF 并将会话视为死亡。选项：

-  静默地重启服务器并重新握手。适用于纯只读服务器。
-  将失败表面化给用户。适用于具有用户可见会话的有状态服务器。

阶段 13 · 09 涵盖了可流式传输的 HTTP 重连语义；stdio 相对更简单。

### 心跳与会话 ID

可流式 HTTP 使用 `Mcp-Session-Id` 头。Stdio 没有会话 ID —— 进程标识就是会话。心跳 ping 是可选的；stdio 管道在不活动时不会断开。

## 动手实践

`code/main.py` 生成三个模拟的 MCP 服务器作为子进程，与每个服务器进行握手，合并它们的工具列表，并将工具调用路由到正确的服务器。这些“服务器”实际上是运行着玩具响应器（无真实 LLM）的其他 Python 进程。运行它可以观察到：

-  三次初始化，每次都有各自的能力集。
-  三个 `tools/list` 的结果合并为一个包含 7 个工具的命名空间。
-  基于工具名称的路由决策。
-  通过命名空间前缀防止的冲突。

查看要点：

-  `Session` 数据类清晰地保存了每个服务器的状态。
-  后台读取线程从 stdout 上无阻塞地取出每一行，不阻塞主线程。
-  调度表是一个简单的 `dict[str, Session]`。
-  冲突处理是显式的：当两个服务器声明相同名称时，后者被重命名并加上前缀。

## 交付成果

本课程产出 `outputs/skill-mcp-client-harness.md`。给定一个声明式的 MCP 服务器列表（名称、命令、参数），该技能生成一个 harness，用于生成服务器、合并工具列表，并提供一个具有冲突解决功能的路由函数。

## 练习

1.  运行 `code/main.py` 并观察服务器生成日志。用 SIGTERM 终止其中一个模拟服务器进程，观察客户端如何检测 EOF 并将会话标记为死亡。
2.  实现命名空间前缀。当两个服务器都暴露 `search` 时，将第二个重命名为 `<server>/search`。更新调度表并验证工具调用是否正确路由。
3.  为服务器重启添加连接池风格的退避：连续失败时指数退避，上限为 30 秒，在三次失败后向用户发出通知。
4.  设计一个支持 100 个并发 MCP 服务器的客户端。什么数据结构可以替代简单的调度字典？（提示：用于前缀命名空间的 trie，以及一个每服务器工具数量的度量。）
5.  将客户端移植到官方 MCP Python SDK。该 SDK 封装了 `stdio_client` 和 `ClientSession`。代码应从约 200 行缩减到约 40 行，同时保留多服务器路由功能。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------|---------|
| MCP 客户端 | "代理主机" | 生成服务器并编排工具调用的进程 |
| 会话 | "每服务器状态" | 能力、工具列表和待处理请求的簿记 |
| 合并命名空间 | "一个工具列表" | 所有活动服务器上的扁平工具名称集 |
| 命名空间冲突 | "两个服务器同一工具" | 客户端必须前缀、拒绝或采用先到先得策略处理重复项 |
| 路由 | "谁得到这个调用？" | 从工具名称到拥有该工具的服务器的分发 |
| 后台读取器 | "非阻塞的 stdout" | 将服务器 stdout 排空到队列的线程或任务 |
| 采样回调 | "LLM即服务" | 客户端处理来自服务器的 `sampling/createMessage` 的处理器 |
| `notifications/*_changed` | "原始变异体" | 通知客户端必须重新发现或重新读取的信号 |
| 重连策略 | "服务器何时死亡" | 传输失败时的重启语义 |
| Stdio 会话 | "进程=会话" | 没有会话 ID；子进程生命周期即会话生命周期 |
