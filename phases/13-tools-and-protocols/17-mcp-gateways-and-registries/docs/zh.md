# MCP 网关与注册中心 — 企业控制平面

> 企业不能让每个开发者随意安装MCP服务器。网关集中了认证、RBAC、审计、限流、缓存和工具投毒检测功能，然后将合并后的工具表面作为单一的MCP端点暴露出去。官方MCP注册中心（由Anthropic、GitHub、PulseMCP和Microsoft共同管理，支持命名空间验证）是规范的上游源。本课程将说明网关的定位，演示一个最小化实现，并概述2026年的供应商格局。

**类型：** 学习  
**语言：** Python（标准库，最小化网关）  
**先决条件：** 第13阶段·15（工具投毒），第13阶段·16（OAuth 2.1）  
**时间：** 约45分钟

## 学习目标

- 解释MCP网关所处的位置（位于MCP客户端和多个后端MCP服务器之间）。
- 实现网关的五个核心职责：认证、RBAC、审计、限流、策略执行。
- 在网关层强制执行固定工具哈希的清单。
- 区分官方MCP注册中心与元注册中心（如Glama、MCPMarket、MCP.so、Smithery、LobeHub）。

## 问题背景

一家财富500强企业有30个已批准的MCP服务器、5000名开发者、合规与审计要求，以及一个需要集中策略的安全团队。让每个开发者在其IDE中随意安装服务器是不可行的。

网关模式：

1.  网关作为单一的**可流式传输的HTTP端点**运行，开发者连接到它。
2.  网关持有每个后端MCP服务器的凭证。
3.  每个开发者的请求都通过网关自身的OAuth进行认证和权限范围限定。
4.  网关将调用路由到后端服务器，并应用策略。
5.  所有调用都被记录以供审计。

Cloudflare MCP门户、Kong AI网关、IBM ContextForge、MintMCP、TrueFoundry、Envoy AI网关等都在2025-2026年发布了网关或网关功能。

与此同时，官方MCP注册中心作为规范的上游启动：经过策划、命名空间验证、使用反向DNS命名的服务器，网关可以从中拉取。元注册中心（如Glama、MCPMarket、MCP.so、Smithery、LobeHub）则聚合来自多个来源的服务器。

## 概念解析

### 网关的五大职责

1.  **认证。** 使用OAuth 2.1识别开发者；映射到用户角色。
2.  **RBAC。** 按用户的策略：哪些服务器、哪些工具、哪些权限范围。
3.  **审计。** 每次调用都记录谁、做了什么、何时、结果。
4.  **限流。** 按用户/工具/服务器设置上限，防止滥用。
5.  **策略。** 拒绝投毒的描述，执行“双重规则”，编辑（脱敏）个人身份信息。

### 网关作为单一端点

对开发者来说，网关看起来像一个MCP服务器。内部则路由到N个后端。会话ID（第13阶段·09）在边界处被重写。

### 凭证保管

开发者永远不会看到后端令牌。网关持有它们（或代理到持有它们的身份提供商）。一个在网关上拥有 `notes:read` 的开发者，可以借助网关自身的后端凭证，传递性地访问笔记MCP服务器——但这仅限于在策略约束传递访问权限的情况下。

### 网关层的工具哈希固定

网关持有一个已批准工具描述的清单（SHA256哈希）。在发现时，它会获取每个后端的 `tools/list`，将哈希与清单进行比较，并移除任何描述发生变更的工具。这是第13阶段·15中的“地毯式撤回”防御在中心位置的应用。

### 策略即代码

高级网关使用OPA/Rego、Kyverno或Styra来表达策略。像“用户 `alice` 只能在组织 `acme` 的仓库上调用 `github.open_pr`”这样的规则被声明式编码。简单的网关使用手工编写的Python。两种形式都是有效的。

### 会话感知路由

当用户的会话涉及多个服务器混合时，网关进行多路复用：开发者的单个MCP会话包含N个后端会话，每个服务器一个。来自任何后端的通知都通过网关路由到开发者的会话。

### 命名空间合并

网关合并来自所有后端的工具命名空间，通常在冲突时添加前缀。 `github.open_pr`, `notes.search`。这使得路由清晰明确。

### 注册中心

- **官方MCP注册中心（`registry.modelcontextprotocol.io`）。** 在Anthropic、GitHub、PulseMCP、Microsoft的管理下启动。命名空间验证（反向DNS：`io.github.user/server`）。经过基本质量预筛选。
- **Glama.** 以搜索为中心的元注册中心，聚合多个来源。
- **MCPMarket.** 偏向商业的目录，有供应商列表。
- **MCP.so.** 社区目录；开放提交。
- **Smithery.** 类似包管理器的安装流程。
- **LobeHub.** 集成在其LobeChat应用中的UI注册中心。

企业网关默认从官方注册中心拉取，允许管理员从元注册中心策划添加内容，并拒绝任何未固定的内容。

### 反向DNS命名

官方注册中心要求公共服务器使用反向DNS名称：`io.github.alice/notes`。命名空间可防止抢注，并使信任委托更清晰。

## 使用方法

`code/main.py` 提供了一个约150行的最小化网关：通过伪造的Bearer令牌认证用户，持有按用户的RBAC策略，将请求路由到两个后端MCP服务器，将每次调用写入审计日志，强制执行限流，并拒绝任何描述哈希与固定清单不匹配的后端工具。

查看重点：

- `RBAC` 字典，以 `user_id` 为键，包含允许的 `server_tool` 条目。
- `AUDIT_LOG` 是一个仅追加的事件列表。
- 限流使用按用户的令牌桶。
- 固定清单是一个 `server::tool -> hash` 字典。

## 练习

1.  运行 `code/main.py`。以一个允许的用户身份发起调用；然后以一个不允许的用户身份；再发起一次超过限流的突发请求。验证这三种情况。

2.  添加一个策略，在返回结果给客户端之前，编辑（脱敏）其中的PII。使用一个简单的正则表达式匹配SSN格式的字符串；注意其局限性（如电子邮件、电话号码）。

3.  扩展审计日志，以发出OpenTelemetry GenAI跨度。第13阶段·20涵盖了具体的属性。

4.  为一个拥有50名开发者、使用五个后端（笔记、GitHub、PostgreSQL、Jira、Slack）的团队设计一个RBAC策略。谁对哪个后端有只读权限？谁有写权限？

5.  从头到尾阅读Cloudflare企业MCP文章。找出一个Cloudflare提供而这个标准库网关没有的功能。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 网关 | "MCP代理" | 位于客户端和后端之间的集中化服务器 |
| 凭证保管 | "后端令牌留在服务器端" | 开发者永远看不到上游令牌 |
| 会话感知路由 | "多后端会话" | 网关为每个开发者会话多路复用N个后端会话 |
| 工具哈希固定 | "已批准的清单" | 每个已批准工具描述的SHA256；在中心位置阻止“地毯式撤回” |
| RBAC | "按用户的策略" | 基于角色的工具和服务器访问控制 |
| 策略即代码 | "声明式规则" | 在网关执行的OPA/Rego、Kyverno、Styra策略 |
| 审计日志 | "谁、什么、何时" | 用于合规的仅追加事件日志 |
| 限流 | "按用户的令牌桶" | 按分钟设置上限以防止滥用 |
| 官方MCP注册中心 | "规范上游" | `registry.modelcontextprotocol.io`，命名空间验证 |
| 反向DNS命名 | "注册中心命名空间" | `io.github.user/server` 约定 |
