# 工具接口 —— 为何智能体需要结构化输入输出

> 语言模型生成的是令牌，程序执行的是动作。连接二者的关键便是工具接口：一份契约，允许模型请求执行动作，而宿主程序负责具体执行。2026年的所有技术栈——无论是OpenAI、Anthropic和Gemini上的函数调用；还是MCP的`tools/call`；亦或是A2A的任务部件——本质上都是对同一个四步循环的不同编码。本节课程将明确这个循环，并展示运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，无LLM）
**先决条件：** 阶段11（LLM补全API）
**时间：** 约45分钟

## 学习目标

- 解释为何一个只能生成文本的LLM，其自身无法对现实世界执行操作。
- 绘制四步工具调用循环（描述 → 决策 → 执行 → 观察），并明确每一步由谁负责。
- 将一个工具描述编写为三个部分：名称、JSON Schema 输入和一个确定性的执行器函数。
- 区分纯工具和产生副作用的工具，并说明这种区分对安全性的重要性。

## 问题所在

LLM输出的是下一个令牌的概率分布，这是其全部的输出界面。如果你询问聊天模型“现在班加罗尔的天气如何”，它可能会写出一个看似合理的句子，但它无法实际调用天气API。这个句子可能碰巧正确，也可能已经是三天前的旧信息。

填补这个缺口正是工具接口的目的。宿主程序——你的智能体运行时、Claude Desktop、ChatGPT、Cursor或一个自定义脚本——向模型宣告一组可调用的工具。当模型判断需要执行某个动作时，它会生成一个结构化的负载，指明一个工具及其参数。宿主解析该负载，真实地运行该工具，并将结果反馈给模型。这个循环持续进行，直到模型判断不再需要调用工具为止。

这份契约的首个版本于2023年6月随OpenAI的"functions"参数发布。Anthropic在Claude 2.1中引入了`tool_use`代码块。几个月后，Gemini增加了`functionDeclarations`。如今，每个提供商都暴露了相同的形态：输入一个JSON-Schema类型化的工具列表，输出一个JSON负载的工具调用。模型上下文协议（2024年11月）将此契约泛化，使得一个工具注册表可以服务所有模型。A2A（2026年4月，v1.0）则将相同的原语应用于智能体间的委托任务。

四步循环是所有这些实现背后的不变核心。阶段13中的其他内容都是对其的详细阐述。

## 核心概念

### 第一步：描述

宿主通过三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。`get_weather`，而不是"天气相关的东西"。
- **描述。** 一段自然语言简述。"当用户询问特定城市的当前天气状况时使用。不要用于历史数据。"
- **输入模式。** 一个JSON Schema对象（2020-12草案），描述工具的参数。

模型接收这个列表。现代提供商使用特定于提供商的模板将这些声明序列化到系统提示中，因此作为调用者的你只需处理结构化表单。

### 第二步：决策

根据用户的消息和可用的工具，模型会从三种行为中选择一种。

1. **直接回答**，以文本形式。无工具调用。
2. **调用一个或多个工具。** 发出结构化的调用对象。在`parallel_tool_calls: true`模式下（OpenAI和Gemini默认，Anthropic可选），模型可以在一个回合中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以生成一个类型化的`refusal`块，而不是调用。

工具调用负载包含三个稳定字段：一个调用`id`，一个工具`name`，以及一个JSON`arguments`对象。这个id的存在是为了让宿主能够将后续的结果与特定的调用关联起来，当并行调用乱序返回时这一点很重要。

### 第三步：执行

宿主接收到调用，根据声明的模式验证参数，然后运行执行器。无效参数意味着模型虚构了一个字段或使用了错误的类型——这是弱模型上非常常见的故障模式。生产环境的宿主在遇到无效参数时会做三件事之一：快速失败并将错误暴露给模型，使用约束解析器修复JSON，或者将验证错误包含在提示中重试模型。

执行器本身是普通的代码。Python、TypeScript、一个shell命令、一个数据库查询。它产生一个结果，通常是一个字符串，但可以是任何JSON值或结构化内容块（在MCP中是文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主将工具结果附加到对话中（作为一个带有匹配`id`的`tool`角色消息），并重新调用模型。模型现在在上下文中拥有了工具输出，可以生成最终答案或请求更多调用。这个过程持续进行，直到模型停止发出调用，或者宿主达到迭代次数的安全限制。

### 信任分离

工具按安全重要性分为两种。

- **纯工具。** 只读、确定性、无副作用。`get_weather`, `search_docs`, `get_current_time`。可以安全地推测性调用。
- **重要工具。** 修改状态、花费金钱、接触用户数据。`send_email`, `delete_file`, `execute_trade`。必须进行门控。

Meta在2026年为智能体安全制定的"两人规则"规定，一个回合最多只能组合以下两项：不可信输入、敏感数据、重要操作。工具接口正是你执行这条规则的地方——通过拒绝调用、要求用户确认或提升权限范围。完整的安全部分请见阶段13 · 15，智能体级别的权限策略请见阶段14 · 09。

### 循环所处的位置

| 上下文 | 谁描述 | 谁决策 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP服务器 | LLM（通过MCP客户端） | MCP服务器 |
| A2A | 智能体卡片发布者 | 调用方智能体 | 被调用方智能体 |
| 网页浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无论在哪里，都是相同的四个步骤。列名会变；结构不变。

### 为什么不直接提示模型输出JSON？

"要求模型以JSON格式回复"是函数调用之前的模式。在前沿模型上，这种方法大约有5%到15%的失败率，在更小的模型上失败率则高得多。故障模式包括缺少花括号、尾随逗号、虚构字段和错误类型。随后你就需要JSON修复流程、重试，或者使用受约束的解码器。

原生函数调用更好，原因有三。首先，提供商对模型进行了针对特定调用形状的端到端训练，因此在严格模式下，有效JSON率可提升至98%到99%。其次，调用负载位于其自身的协议槽中，而不是在自由文本内部——因此工具调用永远不会泄露到用户可见的回复中。第三，提供商通过受约束的解码（OpenAI的严格模式、Anthropic的`tool_use`、Gemini的`responseSchema`）来强制模式合规性。输出保证是验证通过的。

### 断路器

当模型停止发出调用，或者宿主达到最大回合数时，循环终止。生产环境的宿主会将此设置在5到20个回合之间。超过这个范围，你几乎可以肯定陷入了一个模型无法退出的循环。Claude Code默认20；OpenAI助手默认10；Cursor的智能体模式默认25。

另一种情况——无界循环——大约每六个月就会作为"智能体一夜之间花费了400美元API调用"的复盘报告出现。不要在不设置上限的情况下发布产品。

## 动手实践

`code/main.py`运行了无需LLM的四步循环。一个假的"决策器"函数通过模式匹配用户消息来模拟模型；执行器、模式验证器和观察步骤的骨架是真实的。运行它以查看带有可打印中间状态的完整请求/响应编排，然后在后续课程中，用任何真实的提供商替换掉假的决策器。

需要关注的内容：

- 工具注册表为每个工具保存三个字段：名称、描述、模式，以及一个执行器引用。
- 验证器是一个用标准库编写的、最小子集的JSON Schema（类型、必需、枚举、最小/最大值）。阶段13 · 04会提供一个更完整的版本。
- 循环将迭代次数限制在五次。生产环境的智能体正是需要这样的断路器。

## 练习

1.  向`code/main.py`添加一个名为`get_stock_price(ticker)`的第四个工具。将其描述编写为："当用户根据股票代码查询当前股价时使用。不要用于历史价格或市场摘要。"运行该骨架，并确认假的决策器会将提及股票代码的查询路由到新工具。

2.  破坏模式验证器。传递一个调用，其`arguments`对象缺少一个必需字段，确认宿主在执行前拒绝了它。然后传递一个带有多余未知字段的调用。做出决定：宿主应该拒绝还是忽略？用安全论据证明你的选择。

3.  将骨架中的每个工具分类为纯工具或重要工具。为需要的工具在注册表条目中添加一个`consequential: true`标志，并更改循环，以便每当选择了一个重要工具时，打印一行"将与用户确认"。这就是每个生产环境宿主所需的确认门控的形态。

4.  在纸上绘制四步循环，并用上面的提供商列填充表格，针对你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT或自定义技术栈）。交叉参考阶段13 · 06中特定于MCP的变体。

5.  从头到尾阅读OpenAI的函数调用指南。找出一个存在于请求中，但如本文所述的四步循环中不存在的字段。解释它增加了什么，以及为什么它很方便但不是必需的。

## 关键术语

| 术语 | 人们常说的 | 其实际含义 |
|------|----------------|------------------------|
| 工具 (Tool) | "模型可以调用的一个东西" | 一个由名称 + JSON-Schema类型化输入 + 执行器函数组成的三元组 |
| 函数调用 (Function calling) | "原生工具使用" | 提供商层面的API支持，用于发出结构化的工具调用而非散文 |
| 工具调用 (Tool call) | "模型的行动请求" | 模型发出的一个包含`id`, `name`, `arguments`的JSON负载 |
| 工具结果 (Tool result) | "工具返回的东西" | 执行器的输出，包装在一个带有匹配id的`tool`角色消息中 |
| 并行工具调用 (Parallel tool calls) | "同时发起多个调用" | 在一个模型回合中包含多个调用对象，独立且可按id排序 |
| 严格模式 (Strict mode) | "保证有效的JSON" | 受约束的解码，强制模型的输出根据声明的模式进行验证 |
| 纯工具 (Pure tool) | "只读工具" | 无副作用；可安全重跑 |
| 重要工具 (Consequential tool) | "操作工具" | 修改外部状态；需要门控、审计或用户确认 |
| 四步循环 (Four-step loop) | "工具调用周期" | 描述 → 决策 → 执行 → 观察 |
| 宿主 (Host) | "智能体运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
