# Reflexion：基于语言的强化学习

> 基于梯度的强化学习需要数千次尝试和一个 GPU 集群才能修复一个故障模式。Reflexion（Shinn 等人，NeurIPS 2023）用自然语言做到了：每次失败尝试后，智能体写下反思，存入情景记忆，并在下次尝试时以该记忆为条件。这就是 Letta 的睡眠时计算、Claude Code 的 CLAUDE.md 学习记录以及 pro-workflow 的 learn-rule 背后的模式。

**类型：** 构建
**语言：** Python（标准库）
**前提条件：** 第 14 阶段 · 01（智能体循环），第 14 阶段 · 02（ReWOO）
**时间：** 约 60 分钟

## 学习目标

- 列举 Reflexion 的三个组件（执行者、评估者、自省者）以及情景记忆的作用。
- 使用标准库实现一个 Reflexion 循环，包含二元评估器、反思缓冲区和全新的重新尝试。
- 针对给定任务，在标量、启发式和自评估反馈源之间进行选择。
- 解释为什么语言强化能捕获基于梯度的强化学习需要数千次尝试才能修复的错误。

## 问题所在

一个智能体任务失败了。在标准的强化学习中，你需要运行数千次尝试，计算梯度，更新权重。这既昂贵又缓慢，而且大多数生产型智能体没有针对每个失败的训练预算。

Reflexion（Shinn 等人，arXiv:2303.11366）提出了一个不同的问题：如果智能体只是思考它为什么失败，并带着这个思考再次尝试，会怎样？没有权重更新。没有梯度。只有存储在不同尝试之间的自然语言。

结果：在 ALFWorld 上，它击败了 ReAct 和其他非微调基线。在 HotpotQA 上，它优于 ReAct。在代码生成（HumanEval/MBPP）方面，它在当时达到了最先进的水平。所有这些都没有使用任何梯度步骤。

## 概念

### 三个组件

```
Actor         : generates a trajectory (ReAct-style loop)
Evaluator     : scores the trajectory — binary, heuristic, or self-eval
Self-Reflector: writes a natural-language reflection on the failure
```

加上一个数据结构：

```
Episodic memory: list of prior reflections, prepended to the next trial's prompt
```

一次尝试运行执行者。评估者对其进行评分。如果分数低，自省者生成一个反思（“我选错了工具，因为我把问题误解为关于 X，而实际上它是关于 Y”）。反思被存入情景记忆。下一次尝试从头开始，但可以看到之前的反思。

### 三种评估器类型

1. **标量** —— 一个外部二元信号。ALFWorld 成功或失败。HumanEval 测试通过或失败。最简单，信号最强。
2. **启发式** —— 预定义的故障特征。“如果智能体连续两次产生相同的操作，标记为卡住。”“如果轨迹超过 50 步，标记为效率低下。”
3. **自评估** —— 大语言模型为自己的轨迹评分。当没有真实标签时需要。信号较弱；可与工具验证（第 05 课 —— CRITIC）良好配合。

2026 年的默认做法是混合使用：有标量信号时用标量，没有时用自评估，启发式规则作为安全护栏。

### 为什么这能泛化

Reflexion 与其说是一种新算法，不如说是一个命名的模式。几乎每个生产型“自我修复”智能体都运行着某个变体：

- Letta 的睡眠时计算（第 08 课）：一个独立的智能体反思过去的对话并写入记忆块。
- Claude Code 的 `CLAUDE.md` / “保存记忆”模式：反思被捕获为学习记录，添加到未来的会话中。
- pro-workflow 的 `/learn-rule` 命令：纠正被明确捕获为规则。
- LangGraph 的反思节点：一个对输出评分并在需要时路由到优化的节点。

它们都源于同一个见解：自然语言是一个足够丰富的媒介，可以在运行之间传递“我从失败中学到了什么”。

### 何时有效，何时无效

Reflexion 在以下情况有效：

- 有清晰的失败信号（测试失败、工具错误、答案错误）。
- 任务类别可重现（可以再次提出相同类型的问题）。
- 反思有改进轨迹的空间（足够的操作预算）。

Reflexion 在以下情况没有帮助：

- 智能体第一次尝试就成功了。
- 失败是外部原因（网络中断、工具损坏）——反思“网络中断”对未来运行没有帮助。
- 反思变成了迷信——存储了关于一次偶然失败的叙述。

2026 年的陷阱：记忆腐化。反思不断积累；有些已过时或错误；随着情景缓冲区增大，重新运行变得更慢。缓解措施：定期压缩（第 06 课）、对反思设置存活时间，或使用独立的睡眠时清理智能体（Letta）。

## 构建它

`code/main.py` 在一个玩具谜题上实现了 Reflexion：生成一个总和为目标的三元素列表。执行者输出候选列表；评估者检查总和；自省者就出错之处写一行反思。反思被存入情景记忆，供下次尝试使用。

组件：

- `Actor` —— 一个脚本化策略，当它看到反思时会改进。
- `Evaluator.binary()` —— 对目标总和进行通过/失败检查。
- `SelfReflector` —— 生成一行关于失败的诊断。
- `EpisodicMemory` —— 一个带有存活时间语义的有界列表。

运行它：

```
python3 code/main.py
```

跟踪显示三次尝试。尝试 1 失败，一条反思被存储，尝试 2 看到反思后有所改进但仍然失败，尝试 3 成功。与基线运行（无反思）进行比较——它一直卡在尝试 1 的答案上。

## 使用它

LangGraph 将反思作为一个节点模式提供。Claude Code 的 `/memory` 命令和 pro-workflow 的 `/learn-rule` 将情景缓冲区外部化为一个 markdown 文件。Letta 的睡眠时计算在空闲时运行自省者，因此主要智能体保持低延迟。OpenAI Agents SDK 没有直接提供 Reflexion；你需要使用一个根据分数拒绝轨迹的自定义防护栏和一个跨运行持续存在的记忆 `Session` 来构建它。

## 交付它

`outputs/skill-reflexion-buffer.md` 创建并维护一个情景缓冲区，具有反思捕获、存活时间和去重功能。给定一个任务类别和一个失败，它生成一个对下次尝试真正有帮助的反思（而不是通用的“更加小心”）。

## 练习

1.  从二元评估器切换到返回距离度量（离目标有多远）的标量评估器。它是否收敛得更快？
2.  为反思添加 10 次尝试的存活时间。过了这个时间点，旧的反思是有害还是有帮助？
3.  实现启发式评估器：如果相同的操作重复，标记该次尝试为卡住。这如何与自省者交互？
4.  使用一个忽略反思的对抗性执行者运行 Reflexion。最低限度的反思提示工程需要什么才能迫使执行者注意到它们？
5.  阅读 Reflexion 论文关于 AlfWorld 的第 4 节。从概念上重现 130% 的成功率提升：与原版 ReAct 相比，关键差异是什么？

## 关键术语

| 术语 | 人们常说 | 其实际含义 |
|------|----------|------------|
| Reflexion | “自我纠正” | Shinn 等人 2023 — 执行者、评估者、自省者加上情景记忆 |
| 语言强化 | “无梯度学习” | 自然语言反思添加到下次尝试的提示中 |
| 情景记忆 | “每任务反思” | 针对一个任务类别的先前反思的有界缓冲区 |
| 标量评估器 | “二元成功信号” | 来自真实标签的通过/失败或数值分数 |
| 启发式评估器 | “基于模式的检测器” | 预定义的故障特征（例如卡住循环、步骤过多） |
| 自评估器 | “LLM 对自身轨迹做裁判” | 当没有真实标签时的低信号后备方案 —— 配合工具验证使用 |
| 记忆腐化 | “过时的反思” | 情景缓冲区充满过时条目；通过压缩/存活时间修复 |
| 睡眠时反思 | “异步自反思” | 在非关键路径上运行自省者，使主要智能体保持快速 |
