# Anthropic的工作流模式：简单优于复杂

> Schluntz和Zhang（Anthropic，2024年12月）区分了工作流（预定义路径）和智能体（动态工具使用）。五种工作流模式覆盖了大多数情况。从直接API调用开始，仅在步骤无法预测时添加智能体。

**类型：** 学习 + 实践
**语言：** Python（标准库）
**前提：** 第14阶段 · 01（智能体循环）
**时间：** 约60分钟

## 学习目标

- 说出Anthropic的五种工作流模式：提示链、路由、并行化、编排器-工作者、评估器-优化器。
- 解释智能体与工作流的区别，以及各自的工程成本。
- 确定何时选择工作流而非智能体（反之亦然）。
- 使用标准库对脚本化LLM实现所有五种模式。

## 问题所在

团队倾向于为需要单次函数调用的问题使用多智能体框架。其成本是真实的：框架增加了遮蔽提示、隐藏控制流并引发过早复杂性的层次。Schluntz和Zhang在2024年12月的帖子是行业中最受引用的反驳：从简单开始，仅在物有所值时才增加复杂性。

## 核心概念

### 工作流与智能体

- **工作流。** 通过预定义的代码路径编排LLM和工具。工程师掌控图结构。
- **智能体。** LLM动态指导自身工具并执行自身步骤。模型掌控图结构。

两者各有其位。工作流更廉价、更快速且更易于调试。智能体能够解决开放式问题，但使失败模式更难推理。

### 增强型LLM

所有五种模式的基础：一个集成了三种能力的LLM——搜索（检索）、工具（操作）、记忆（持久化）。任何API调用均可使用这些能力。

### 五种模式

1.  **提示链。** 第1次调用的输出是第2次调用的输入。适用于任务具有清晰线性分解的情况。步骤之间可设置可编程门控。

2.  **路由。** 一个分类器LLM选择要调用的下游LLM或工具。适用于类别不同的输入需要不同处理的情况（一级支持 vs 退款 vs 故障报告 vs 销售咨询）。

3.  **并行化。** 并发运行N个LLM调用，聚合结果。两种形式：分段处理（不同数据块）和投票机制（相同提示，N次运行，取多数/综合）。

4.  **编排器-工作者。** 一个编排器LLM动态决定运行哪些工作者（也是LLM）并综合其输出。类似于智能体循环，但编排器不会无限循环。

5.  **评估器-优化器。** 一个LLM提出答案，另一个LLM评估它。迭代直到评估器通过。这是自我改进（第05课）的泛化。

### 工作流优于智能体的场景

- **可预测的任务。** 如果你能列举步骤，就应该这样做。
- **成本敏感的任务。** 工作流的步骤数量有界；智能体可能失控。
- **合规性要求高的任务。** 审计人员希望阅读图结构，而不是从轨迹推断。

### 智能体优于工作流的场景

- **开放式研究。** 当下一步取决于上一步的返回结果时。
- **可变长度任务。** 耗时数分钟到数小时，且步骤数量未知。
- **新兴领域。** 当你尚不知道正确的工作流时——先探索，后固化。

### 上下文工程的伴生学科

“AI智能体的有效上下文工程”（Anthropic 2025）形式化了相邻学科：20万token的上下文窗口是一个预算，而非容器。决定包含什么、何时压缩、何时让上下文增长。在第14阶段关于上下文压缩的课程（本课程中第14阶段更早的第06课，重新编号前）中有详细讨论。

## 实践构建

`code/main.py`针对一个`ScriptedLLM`实现了所有五种工作流模式：

- `prompt_chain(input, steps)` — 顺序执行。
- `route(input, classifier, handlers)` — 分类 + 分发。
- `parallel_vote(prompt, n, aggregator)` — N次运行，聚合结果。
- `orchestrator_workers(task, workers)` — 编排器选择工作者。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` — 循环直到通过。

运行它：

```
python3 code/main.py
```

每种模式都会打印其执行轨迹。每种模式的代码行数约为10-15行；框架的成本则以数千行计。

## 如何使用

- 大多数任务使用直接API调用。
- 仅在模式确实需要持久化状态（LangGraph）、角色模型并发（AutoGen v0.4）或角色模板化（CrewAI）时才使用框架。
- 当你需要Claude代码工具链的形态但不想重新构建时，可以考虑Claude智能体SDK。

## 部署实践

`outputs/skill-workflow-picker.md`为给定任务描述选择正确的模式，包括决策理由，以及在工作流不足时向智能体演进的重构路径。

## 练习

1.  实现带有置信度阈值的路由。低于阈值 -> 升级至人工处理。对于一级支持用例，阈值应设在哪里？
2.  为`parallel_vote`添加超时机制。当某个调用挂起时会发生什么？如何聚合有缺失投票的结果？
3.  将`evaluator_optimizer`改进为老虎机算法：跨迭代保留前两个输出，这样晚出现的好结果不会被晚出现的坏结果覆盖。
4.  结合提示链与路由：一个路由器从三个链中选择一个。衡量token成本，与使用单个大型提示的替代方案对比。
5.  选择你生产环境中的一个功能。绘制工作流图。计算步骤数。智能体在这里真的会更好吗？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------|----------|
| 工作流 | “预定义流程” | 工程师掌控的LLM和工具调用图结构 |
| 智能体 | “自主AI” | 模型掌控的图结构；动态工具指导 |
| 增强型LLM | “带工具的LLM” | LLM + 搜索 + 工具 + 记忆；原子单元 |
| 提示链 | “顺序调用” | 第N次调用的输出是第N+1次调用的输入 |
| 路由 | “分类器分发” | 选择哪个链/模型处理输入 |
| 并行化 | “扇出” | N个并发调用；通过分段或投票聚合 |
| 编排器-工作者 | “调度器智能体” | 编排器LLM动态选择专家LLM |
| 评估器-优化器 | “提议者 + 裁判” | 迭代直到评估器通过；自我改进的泛化 |
