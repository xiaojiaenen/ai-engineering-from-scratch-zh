# 技能库与终身学习 (Voyager)

> Voyager (Wang et al., TMLR 2024) 将可执行代码视为技能。技能具有命名、可检索、可组合、并通过环境反馈进行优化的特性。这是 Claude Agent SDK 技能、skillkit 以及 2026 年技能库模式的参考架构。

**类型：** 构建
**语言：** Python (标准库)
**先决条件：** 第 14 阶段 · 07 (MemGPT)，第 14 阶段 · 08 (Letta Blocks)
**时间：** ~75 分钟

## 学习目标

- 说出 Voyager 的三个组成部分——自动课程、技能库、迭代提示机制——及其各自的作用。
- 解释为什么 Voyager 将动作空间设为代码，而非原始命令。
- 实现一个基于标准库的技能库，包含注册、检索、组合和基于失败的优化。
- 将 Voyager 的模式映射到 2026 年的 Claude Agent SDK 技能和 skillkit 生态系统。

## 问题所在

在每个会话中从头重建所有能力的智能体犯了三个错误：

1.  **浪费 token。** 每个任务都重新引发相同的推理过程。
2.  **丢失进展。** 在会话 A 中学习到的纠正不会转移到会话 B。
3.  **无法处理长时序组合。** 复杂的任务需要能力层次结构；单次提示无法表达它们。

Voyager 的解决方案是：将每个可复用能力视为一个存储在库中的命名代码块，可通过相似性检索，可与其他技能组合，并通过执行反馈进行优化。

## 概念解析

### 三个组成部分

Voyager (arXiv:2305.16291) 围绕以下结构构建智能体：

1.  **自动课程。** 一个基于好奇心的提议者根据智能体当前的技能集和环境状态选择下一个任务。探索是自下而上的。
2.  **技能库。** 每个技能都是可执行代码。任务成功时会添加新技能。技能通过查询描述相似性进行检索。
3.  **迭代提示机制。** 遇到失败时，智能体会收到执行错误、环境反馈和自我验证输出，然后对该技能进行优化。

Minecraft 评估 (Wang et al., 2024)：独特物品数量增加 3.3 倍，石制工具制作速度加快 8.5 倍，铁制工具制作速度加快 6.4 倍，地图遍历距离延长 2.3 倍（与基线相比）。这些数字是 Minecraft 特定的，但该模式可迁移。

### 动作空间 = 代码

大多数智能体发出原始命令。Voyager 发出 JavaScript 函数。一个技能是：

```
async function craftIronPickaxe(bot) {
  await mineIron(bot, 3);
  await mineStick(bot, 2);
  await placeCraftingTable(bot);
  await craft(bot, 'iron_pickaxe');
}
```

由子技能组合而成。基于描述和嵌入进行存储和键控。作为程序而非提示检索。

这就是 2026 年 Claude Agent SDK 的技能：一个命名的、可检索的代码块加上智能体按需加载的指令。

### 技能检索

新任务“制作一个钻石镐”。智能体：

1.  嵌入任务描述。
2.  查询技能库，获取前 k 个最相似的技能。
3.  检索 `craftIronPickaxe`、`mineDiamond`、`placeCraftingTable` 等。
4.  使用检索到的原始技能和新逻辑组合成新技能。

这就是 MCP 资源（第 13 阶段）和 Agent SDK 技能所实现的模式：在知识/代码表面进行检索，范围限定于当前任务。

### 迭代优化

Voyager 的反馈循环：

1.  智能体编写一个技能。
2.  技能在环境中运行。
3.  返回三个信号之一：`success`、`error`（带堆栈跟踪）、`self-verification failure`。
4.  智能体使用该信号作为上下文重写技能。
5.  循环直到成功或达到最大轮次。

这是 Self-Refine（第 5 课）在代码生成中的应用，结合了基于环境的验证。CRITIC（第 5 课）是相同的模式，但使用外部工具作为验证器。

### 课程与探索

Voyager 的课程模块会提议类似“在湖边建造一个避难所”的任务，这基于智能体已拥有但尚未尝试过的能力。提议者使用环境状态和技能清单来选择一个略高于当前能力水平的任务——探索的“甜蜜点”。

对于生产型智能体，这转化为一个“缺失什么”的操作符：给定当前技能库和一个领域，我们尚未覆盖哪些技能？团队通常将其作为课程审查手动实现。

### 这个模式可能出错的地方

-   **技能库腐烂。** 同一个技能被添加了 10 次，但描述略有不同。在写入时添加去重；检索时只返回一个。
-   **组合技能漂移。** 父技能依赖于一个被优化过的子技能。对技能进行版本控制；固定到 v1 的父技能不会自动获取 v3 的改进。
-   **检索质量。** 当技能库增长到几百个时，基于技能描述的向量检索质量会下降。需要补充标签过滤器和硬性约束（“只检索带有 `category=tooling` 的技能”）。

## 动手构建

`code/main.py` 实现了一个基于标准库的技能库：

-   `Skill` —— 名称、描述、代码（字符串形式）、版本、标签、依赖项。
-   `SkillLibrary` —— 注册、搜索（基于 token 重叠）、组合（依赖项的拓扑排序）和优化（更新时版本号递增）。
-   一个脚本化的智能体，它注册了三个原始技能，组合了第四个，遭遇失败，并进行了优化。

运行它：

```
python3 code/main.py
```

跟踪记录显示了库写入、检索、组合、一次失败的执行以及一次 v2 优化——完整展现了 Voyager 的循环过程。

## 如何应用

-   **Claude Agent SDK 技能** (Anthropic) —— 2026 年的参考实现：每个技能包含描述、代码和指令；在智能体会话期间按需加载。
-   **skillkit** (npm: skillkit) —— 为 32 种以上 AI 编码智能体提供的跨智能体技能管理。
-   **自定义技能库** —— 领域特定（为数据智能体准备 SQL 技能，为基础设施智能体准备 Terraform 技能）。Voyager 模式可以简化应用。
-   **OpenAI Agents SDK `tools`** —— 在低端实现；每个工具都是一个轻量级技能。

## 发布部署

`outputs/skill-skill-library.md` 为任何目标运行时生成一个 Voyager 式的技能库，内置注册、检索、版本控制和优化功能。

## 练习

1.  在 `compose()` 中添加一个依赖循环检测器。当技能 A 依赖于 B，而 B 又依赖于 A 时会发生什么？报错还是警告？
2.  实现按技能进行版本固定。当父技能组合子技能 `crafting@1` 时，对 `crafting@2` 的优化不得静默升级父技能。
3.  用 sentence-transformers 嵌入（或 BM25 标准库实现）替换 token 重叠检索。在一个包含 50 个技能的玩具库上测量 recall@5。
4.  添加一个“课程”智能体：给定当前库和一个领域描述，提议 5 个缺失的技能。每周调用一次。
5.  阅读 Anthropic 的 Claude Agent SDK 技能文档。将玩具库移植到该 SDK 的技能架构。可发现性方面有什么变化？

## 关键术语

| 术语 | 人们通常怎么说 | 它的实际含义 |
|------|----------------|------------------------|
| 技能 (Skill) | “可复用的能力” | 命名的代码块 + 描述，可通过相似性检索 |
| 技能库 (Skill library) | “智能体关于如何做的记忆” | 技能的持久化存储，可搜索和组合 |
| 课程 (Curriculum) | “任务提议者” | 由当前能力差距驱动的自下而上的目标生成器 |
| 组合 (Composition) | “技能 DAG” | 技能调用技能；执行时进行拓扑排序 |
| 迭代优化 (Iterative refinement) | “自我纠正循环” | 环境反馈 + 错误 + 自我验证反馈到下一个版本 |
| 动作空间即代码 (Action-space-as-code) | “程序化动作” | 发出函数而非原始命令，以实现时间扩展行为 |
| 写入时去重 (Dedup on write) | “技能坍缩” | 近乎重复的描述合并为一个规范技能 |
