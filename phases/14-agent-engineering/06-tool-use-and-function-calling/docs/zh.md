# 工具使用与函数调用

> Toolformer（Schick 等人，2023年）开创了自监督工具标注。伯克利函数调用排行榜 V4（Patil 等人，2025年）设定了2026年的基准：40% 智能体化、30% 多轮对话、10% 实时、10% 非实时、10% 幻觉检测。单轮对话已解决。记忆、动态决策和长周期工具链尚未解决。

**类型：** 构建
**语言：** Python（标准库）
**先修课程：** 阶段14 · 01（智能体循环），阶段13 · 01（函数调用深入解析）
**时间：** 约60分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：仅保留执行能降低下一 token 损失的工具标注。
- 列举 BFCL V4 的五个评估类别及其各自衡量标准。
- 使用标准库实现一个具有模式验证、参数强制转换和执行沙箱的工具注册表。
- 诊断2026年的三个开放问题：长周期工具链接、动态决策和记忆。

## 问题背景

早期的工具使用提出：模型能否预测正确的函数调用？现代的工具使用则提出：模型能否在具有记忆、部分可观测性、工具故障恢复能力且不会幻觉出不存在工具的情况下，跨40个步骤链接工具？

Toolformer 建立了基准：模型可以通过自监督学习何时调用工具。BFCL V4 定义了2026年的评估目标。两者之间的差距正是生产级智能体所处的空间。

## 核心概念

### Toolformer（Schick 等人，NeurIPS 2023）

思路：让模型用自己的预训练语料库标注候选 API 调用。对每个候选调用，执行它。仅当包含工具结果能降低下一 token 的损失时，才保留该标注。在过滤后的语料库上进行微调。

涵盖的工具：计算器、QA 系统、搜索引擎、翻译器、日历。自监督信号纯粹是关于工具是否有助于预测文本——无需人工标签。

规模化结果：工具使用随规模涌现。较小的模型会因工具标注而受损；较大的模型则会受益。这就是为什么2026年的前沿模型内建了强大的工具使用能力，而大多数7B模型需要显式的工具使用微调才能可靠运行。

### 伯克利函数调用排行榜 V4（Patil 等人，ICML 2025）

BFCL 是2026年事实上的评估标准。V4 的构成：

- **智能体化（40%）** — 完整的智能体轨迹：记忆、多轮对话、动态决策。
- **多轮对话（30%）** — 带有工具链的交互式对话。
- **实时（10%）** — 用户提交的真实提示（分布更难）。
- **非实时（10%）** — 合成的测试用例。
- **幻觉检测（10%）** — 检测何时不应调用任何工具。

V3 引入了基于状态的评估：在工具序列执行后，检查 API 的实际状态（例如“文件是否已创建？”），而不是匹配工具调用的抽象语法树。V4 增加了网络搜索、记忆和格式敏感性类别。

2026年的关键发现：单轮函数调用几乎已解决。失败集中在记忆（跨轮次保持上下文）、动态决策（基于先前结果选择工具）、长周期链接（20+步后发生偏移）以及幻觉检测（当没有工具适用时拒绝调用）。

### 工具模式

每个提供者都有一个模式。它们在细节上有所不同，但共享相同的结构：

```
name: string
description: string (what it does, when to use it)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述是核心——模型通过它来选择正确的工具。糟糕的工具描述是导致选错工具的首要原因。

### 参数验证

不要信任任何工具调用。进行验证：

1. **类型强制转换。** 模型可能在模式指定 int 的地方返回字符串 "5"。如果明确无误则进行转换；否则拒绝。
2. **枚举验证。** 如果模式指定 `status in {"open", "closed"}` 而模型输出 `"in_progress"`，则拒绝并返回描述性错误。
3. **必需字段。** 缺少必需字段 -> 立即将错误观察返回给模型，而不是崩溃。
4. **格式验证。** 日期、电子邮件、URL——使用具体的解析器进行验证，而不是正则表达式。

每次验证失败都应返回一个结构化观察，以便模型可以使用正确的形状重试。

### 并行工具调用

现代提供者在一轮助手回合中支持并行工具调用。循环如下：

1. 模型发出3个具有不同 `tool_use_id` 的工具调用。
2. 运行时执行它们（如果独立则并行执行）。
3. 每个结果作为 `tool_result` 块通过 `tool_use_id` 关联后返回。

工程规则：将关联 ID 视为核心要素。交换它们会导致错误的工具调用结果路由。

### 沙箱

工具执行是沙箱边界。详见第09课。简短版本：每个工具应指定读/写表面、网络访问、超时、内存上限。通用的 `run_shell(cmd)` 是危险信号；具体的 `git_status()` 更安全。

## 构建实现

`code/main.py` 实现了一个生产级形状的工具注册表：

- JSON Schema 子集验证器（仅使用标准库）。
- 工具注册，包含描述、输入模式、超时和执行器。
- 参数强制转换和枚举验证。
- 带有关联 ID 的并行工具分发。
- 作为结构化字符串的错误观察。

运行它：

```
python3 code/main.py
```

跟踪记录显示一个小型智能体在一轮中调用了三个工具，其中一个故意格式错误的调用被拒绝，并返回了模型可以据此行动的描述性错误。

## 使用它

每个提供者都有自己的工具模式——Anthropic、OpenAI、Gemini、Bedrock。如果需要多提供者支持，请使用转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。BFCL 是参考基准——如果工具使用对产品至关重要，请在发布前针对它测试你的智能体。

## 交付它

`outputs/skill-tool-registry.md` 为给定的任务域生成工具目录、模式和注册表。包含描述质量检查（每个工具的描述是否告诉模型何时使用它？）。

## 练习

1. 添加一个"无操作"工具，让模型可以明确拒绝使用任何其他工具。在类似 BFCL 的幻觉测试上进行衡量。
2. 实现 int 转字符串和 float 转字符串的参数强制转换。强制转换在何处开始隐藏真正的错误？
3. 添加每个工具的超时和熔断器（连续3次失败后拒绝该工具60秒）。这对模型的恢复方式有何改变？
4. 阅读 BFCL V4 描述。选择一个类别（例如"多轮对话"），用你的智能体运行10个示例提示。报告通过率。
5. 将标准库验证器移植到 Pydantic 或 Zod。Pydantic/Zod 发现了哪些玩具验证器遗漏的问题？

## 关键术语

| 术语 | 人们怎么说 | 它的实际含义 |
|------|----------|------------|
| 函数调用 | "工具使用" | 带有已验证模式的结构化输出工具调用 |
| Toolformer | "自监督工具标注" | Schick 2023 — 保留结果能降低下一 token 损失的工具调用 |
| BFCL | "伯克利函数调用排行榜" | 2026年基准：40% 智能体化、30% 多轮对话、10% 实时、10% 非实时、10% 幻觉检测 |
| 工具模式 | "给模型的函数签名" | 名称、描述、参数的 JSON Schema |
| tool_use_id | "关联 ID" | 将工具调用与其结果关联；对并行分发至关重要 |
| 幻觉检测 | "知道何时不该调用" | V4 类别：当没有工具适用时拒绝调用 |
| 参数强制转换 | "字符串到整数修复" | 针对可预见模式不匹配的狭窄修复；如果模棱两可则拒绝 |
| 沙箱 | "工具执行边界" | 每个工具的读/写表面、网络、超时、内存上限 |
