# ReWOO 与 Plan-and-Execute：解耦的规划

> ReAct 在一个流中交错思考与行动。ReWOO 将它们分离：先进行一个完整的大规划，然后执行。使用 token 减少 5 倍，在 HotpotQA 上准确率提升 4%，并且可以将规划器蒸馏到 7B 模型中。Plan-and-Execute 将其泛化；Plan-and-Act 将其扩展到网页导航。

**类型：** 构建
**语言：** Python (标准库)
**前置条件：** 阶段 14 · 01 (智能体循环)
**时间：** ~60 分钟

## 学习目标

- 解释 ReWOO 的规划器/工作者/求解器分工为何比 ReAct 的交错循环更节省 token 并提高鲁棒性。
- 实现一个计划有向无环图 (DAG)、一个按依赖顺序执行的执行器，以及一个组合工作者输出的求解器——全部使用标准库。
- 基于 2026 年的“五种工作流模式”框架 (Anthropic) 决定任务应采用“先规划后执行”还是交错 ReAct 方式。
- 识别何时需要 Plan-and-Act 的合成规划数据来处理长时域网页或移动任务。

## 问题所在

ReAct 的思考-行动-观察交错循环简单灵活，但每次工具调用都必须携带完整的先前上下文——包括之前的每次思考。token 使用量随深度呈二次方增长。更糟的是：当工具在循环中失败时，模型必须根据错误观察重新推导整个计划。

ReWOO (Xu 等人，arXiv:2305.18323，2023年5月) 注意到这一点并做出了一个赌注：预先规划整个计划，并行获取证据，最后组合答案。一次 LLM 调用进行规划，N 次工具调用获取证据（可并行），一次 LLM 调用进行求解。这种权衡牺牲了灵活性（计划是静态的），但换来了显著的 token 效率和更清晰的失败模式。

## 核心概念

### 三个角色

```
Planner:  user_question -> [plan_dag]
Workers:  [plan_dag]     -> [evidence]        (tool calls, possibly parallel)
Solver:   user_question, plan_dag, evidence -> final_answer
```

规划器产生一个 DAG。每个节点命名一个工具、其参数以及它依赖的先前节点（如 `#E1`、`#E2` 的引用）。工作者按拓扑顺序执行节点。求解器将所有内容组合起来。

### 为何 token 减少 5 倍

ReAct 的提示词长度随步骤数线性增长。在第 10 步时，提示包含思考 1 加上行动 1 加上观察 1 加上思考 2 加上行动 2 加上观察 2，依此类推。每个中间步骤还冗余地包含了原始提示。

ReWOO 只需支付一个规划器提示（较大）、N 个小的工作者提示（每个只是工具调用，无链式结构）和一个求解器提示。在 HotpotQA 上，论文测量 token 减少约 5 倍，同时准确率绝对值提高 4%。

### 为何更鲁棒

如果工作者 3 在 ReAct 中失败，循环必须在中途从错误中推理。在 ReWOO 中，工作者 3 返回一个错误字符串；求解器在原始计划的上下文中看到它，并能优雅降级。失败定位是针对节点的，而非针对步骤。

### 规划器蒸馏

论文的第二个结果：因为规划器看不到观察结果，你可以用来自 175B 教师模型的规划器输出微调一个 7B 模型。小模型处理规划；推理时不需要大模型。这现在已是标准做法——许多 2026 年的生产智能体使用小规划器和大执行器，反之亦然。

### Plan-and-Execute (LangChain, 2023)

LangChain 团队在 2023 年 8 月的文章中将 ReWOO 泛化为一个模式名称：Plan-and-Execute。预先的规划器发出一个步骤列表，执行器运行每个步骤，可选的重规划器可以在观察结果后进行修订。这比 ReAct 更接近 ReWOO（重规划器将观察结果带回规划），但保留了 token 节省。

### Plan-and-Act (Erdogan 等人，arXiv:2503.09572，ICML 2025)

Plan-and-Act 将该模式扩展到长时域的网页和移动智能体。关键贡献是合成规划数据：一个标记轨迹生成器产生训练数据，其中计划是显式的。用于微调规划器模型，使其在类似 WebArena 的任务上保持 30-50 步以上有效工作，而单条 ReAct 轨迹在此类任务中会失去连贯性。

### 如何选择

| 模式 | 适用场景 |
|---------|------|
| ReAct | 短任务、未知环境、需要反应式异常处理 |
| ReWOO | 结构化任务、工具已知、对 token 敏感、证据可并行获取 |
| Plan-and-Execute | 类似 ReWOO，但可在部分执行后重新规划 |
| Plan-and-Act | 长时域任务 (>30步)、网页/移动/计算机操作 |
| 思维树 (Tree of Thoughts) | 搜索值得付出代价（第 04 课） |

Anthropic 的 2024 年 12 月指南：从最简单的开始。如果任务是一次工具调用加一个总结，就不要构建 ReWOO。如果任务是一个 40 步的研究任务，就不要只使用 ReAct。

## 构建它

`code/main.py` 实现了一个玩具 ReWOO：

- `Planner` — 一个脚本化策略，根据提示发出计划 DAG。
- `Worker` — 通过注册表分派每个节点的工具调用。
- `Solver` — 脚本化组合，读取证据并产生最终答案。
- 依赖解析 — `#E1` 这样的引用会被先前的工作者输出替换。

该演示回答“法国首都的人口是多少，四舍五入到百万？”使用两步计划：(1) 查找首都，(2) 查找人口，然后求解。

运行它：

```
python3 code/main.py
```

轨迹首先显示完整计划，然后显示工作者结果，最后显示求解器组合。将 token 数量（我们打印大致字符数）与 ReAct 风格的交错运行进行比较——ReWOO 在此类结构化任务上胜出。

## 使用它

LangGraph 将 Plan-and-Execute 作为配方提供（`create_react_agent` 用于 ReAct，自定义图用于 plan-execute）。CrewAI 的 Flows 直接编码该模式：你预先定义任务，Flow DAG 执行它们。Plan-and-Act 的合成数据方法主要仍在研究中；运行时模式（显式计划 DAG）通过 LangGraph 和 CrewAI Flows 在生产中提供。

## 部署它

`outputs/skill-rewoo-planner.md` 根据工具目录，从用户请求生成 ReWOO 计划 DAG。它在交给执行器之前验证计划（无环、所有引用已解析、所有工具存在）。

## 练习

1.  为独立的计划节点并行化工作者执行。在一个具有 2 个并行组的 6 节点 DAG 上，这能带来什么收益？
2.  添加一个重规划器节点，如果任何工作者返回错误则触发。使 ReWOO 成为 Plan-and-Execute 的最小改动是什么？
3.  用小模型（7B 级）替换 `Planner`，并将 `Solver` 保留在前沿模型上。比较端到端质量——在哪些地方分工会失效？
4.  阅读 ReWOO 论文中关于规划器蒸馏的第 4 节。在概念上重现 175B -> 7B 的结果：你需要什么训练数据，以及如何评估计划质量？
5.  将玩具移植到 Plan-and-Act 的轨迹形状：计划是一个序列，而非 DAG。哪些权衡会发生变化？

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|----------------|------------------------|
| ReWOO | “无观察推理” | 先规划，然后并行获取证据，再求解——规划提示中无观察 |
| Plan-and-Execute | “LangChain 的 plan-execute 模式” | 带有可选重规划器的 ReWOO |
| Plan-and-Act | “扩展的 plan-execute” | 显式的规划器/执行器分离，使用合成计划训练数据处理长时域任务 |
| 证据引用 | “#E1, #E2, ...” | 计划节点占位符，在分派时用先前工作者输出替换 |
| 规划器蒸馏 | “小规划器，大执行器” | 在大型教师模型的规划器轨迹上微调小模型 |
| Token 效率 | “更少往返次数” | 论文中 HotpotQA 上比 ReAct 减少 5 倍 token |
| DAG 执行器 | “拓扑分派器” | 按依赖顺序运行计划节点；在每一层级并行执行 |
