# 记忆块与睡眠时间计算（Letta）

> MemGPT 于 2024 年更名为 Letta。其 2026 年的演进引入了两个理念：离散的功能性记忆块（模型可直接编辑）和睡眠时间代理（在主代理空闲时异步整合记忆）。这就是将记忆规模扩展到单一对话之外的方法。

**类型：** 构建
**语言：** Python（标准库）
**先决条件：** 阶段 14 · 07（MemGPT）
**时间：** 约 75 分钟

## 学习目标

- 说出 Letta 使用的三个内存层级（核心层、回忆层、归档层）及其各自作用。
- 解释记忆块模式：人类块、人格块以及作为一等类型化对象的用户定义块。
- 描述什么是睡眠时间计算，为何它不在关键路径上，以及为何它能运行比主代理更强大的模型。
- 实现一个脚本化的双代理循环，其中主代理负责响应，睡眠时间代理在轮次之间整合记忆块。

## 问题所在

MemGPT（第 07 课）解决了虚拟内存的控制流问题。随之出现了三个生产环境问题：

1.  **延迟。** 每个内存操作都位于关键路径上。如果代理需要在用户等待时进行修剪、摘要或协调，尾部延迟就会激增。
2.  **记忆腐化。** 写操作不断累积。已被推翻的事实仍然存在。检索淹没在过时的内容中。
3.  **结构丢失。** 扁平化的归档存储无法表达“人类块始终在提示中；人格块始终在提示中；任务块根据会话交换”。

Letta (letta.com) 是 2026 年的重写版本。记忆块使结构显性化；睡眠时间计算将整合操作移出关键路径。

## 概念

### 三个层级

| 层级 | 范围 | 存放位置 | 写入者 |
|------|------|----------|--------|
| 核心层 | 始终可见 | 主提示词内部 | 代理工具调用 + 睡眠时间重写 |
| 回忆层 | 对话历史 | 可检索 | 自动轮次日志记录 |
| 归档层 | 任意外部事实 | 向量数据库 + KV存储 + 图数据库 | 代理工具调用 + 睡眠时间摄入 |

核心层就是 MemGPT 的核心记忆。回忆层是带有被逐出尾部的对话缓冲区。归档层是外部存储。这种划分清理了 MemGPT 两层结构中的过载问题。

### 记忆块

记忆块是核心层中类型化、持久化且可编辑的区域。原始 MemGPT 论文定义了两种：

- **人类块** — 关于用户的事实（姓名、角色、偏好、目标）。
- **人格块** — 代理的自我概念（身份、语气、约束）。

Letta 将其泛化为任意用户定义块：一个用于当前目标的 `Task` 块，一个用于代码库事实的 `Project` 块，一个用于硬性约束的 `Safety` 块。每个块都有一个 `id`、`label`、`value`、`limit`（字符上限）、`description`（以便模型知道何时编辑它）。

记忆块可通过工具接口进行编辑：

- `block_append(label, text)`
- `block_replace(label, old, new)`
- `block_read(label)`
- `block_summarize(label)` — 压缩接近其限制的块。

### 睡眠时间计算

这是 Letta 在 2025 年的新增功能：在后台运行第二个代理，不在关键路径上。睡眠时间代理处理对话记录和代码库上下文，将 `learned_context` 写入共享记忆块，并整合或使归档记录失效。

由此产生的特性：

- **无延迟开销。** 主代理的响应无需等待内存操作。
- **允许使用更强模型。** 睡眠时间代理可以是更昂贵、更慢的模型，因为它不受延迟约束。
- **自然的整合窗口。** 当用户未等待时，进行去重、摘要、使被推翻的事实失效。

这种模式符合人类的工作方式：你完成任务，睡一觉，长期记忆在夜间沉淀。

### Letta V1 与原生推理

Letta V1（`letta_v1_agent`，2026 年）弃用了 `send_message`/心跳和内联 `Thought:` token，转而支持原生推理。Responses API（OpenAI）和带有扩展思考的 Messages API（Anthropic）在单独的通道上发出推理过程，这些过程会在轮次间传递（在生产环境中跨提供商加密）。控制循环仍然是 ReAct 思维链。思维轨迹是结构性的，而非提示词形式的。

### 此模式出错的地方

- **块膨胀。** 无限追加 `block_append` 会很快触及限制。在写入超过上限之前，接入一个块摘要器。
- **静默漂移。** 睡眠时间代理重写了一个块，而主代理从未察觉。对块进行版本控制，并在追踪日志中显示差异。
- **整合投毒。** 睡眠时间代理将攻击者可触及的内容处理到核心记忆中。第 27 课的内容同样适用于睡眠时间接口。

## 动手构建

`code/main.py` 实现了：

- `Block` — id、标签、值、限制、描述。
- `BlockStore` — CRUD 操作 + `near_limit(label)` 辅助函数。
- 两个脚本化的代理 — `PrimaryAgent` 服务一个轮次，`SleepTimeAgent` 在轮次之间进行整合。
- 一个追踪日志，展示一个包含块写入操作的三轮对话，以及一次睡眠时间处理，其中对一个块进行了摘要，并使一个过时的事实失效。

运行它：

```
python3 code/main.py
```

输出日志显示了这种分离：主代理的轮次快速且产生原始写入；睡眠过程则进行压缩和清理。

## 应用

- **Letta** (letta.com) 作为参考实现。可自托管或使用托管云服务。
- **Claude Agent SDK 技能** 作为块状知识 — 一个技能是一个命名的、版本化的、可检索的指令块，代理按需加载。
- **自定义构建** 适用于希望控制存储后端的团队。使用 Letta API 规范以便日后迁移。

## 部署

`outputs/skill-memory-blocks.md` 可为任何运行时生成一个具有睡眠时间钩子的 Letta 风格的块系统，包括安全规则和引用链接。

## 练习

1.  添加一个 `block_summarize` 工具，当 `near_limit` 返回 true 时，用模型生成的摘要替换块值。哪种触发阈值能最小化摘要调用次数和块溢出？
2.  实现归档层的睡眠时间去重：两个文本 token 重叠度超过 90% 的记录合并为一个。仅在睡眠过程中执行，绝不放在关键路径上。
3.  对记忆块进行版本控制。每次写入时记录旧值和差异。暴露 `block_history(label)` 接口，以便运维人员调试“为什么代理忘记了 X”。
4.  将睡眠时间代理视为不可信的写入者。当它们操作人格块或安全块时，需要第二个代理审查后才能提交。
5.  将示例移植到使用 Letta API（`letta_v1_agent`）。块的模式会有什么变化？原生推理如何改变追踪日志的形状？

## 关键术语

| 术语 | 人们通常说 | 其实际含义 |
|------|------------|------------|
| 记忆块 | “可编辑的提示词部分” | 核心记忆中类型化、持久化、LLM 可编辑的段落 |
| 人类块 | “用户记忆” | 关于用户的事实，固定在核心层 |
| 人格块 | “代理身份” | 自我概念、语气、约束，固定在核心层 |
| 睡眠时间计算 | “异步记忆工作” | 在关键路径外进行整合的第二个代理 |
| 核心层 / 回忆层 / 归档层 | “层级” | 三层内存划分：始终可见 / 对话 / 外部 |
| 块限制 | “容量上限” | 每个块的字符限制；强制进行摘要 |
| 原生推理 | “思考通道” | 提供商级别的推理输出，而非提示级别的 `Thought:` |
| 学习到的上下文 | “睡眠输出” | 睡眠时间代理写入共享块中的事实 |
