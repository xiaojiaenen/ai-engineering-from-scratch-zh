# 评估驱动的智能体开发

> Anthropic 的指导：“从简单提示开始，通过全面评估优化它们，仅在必要时添加多步骤智能体系统。”评估不是最后一步。它是驱动第14阶段所有其他选择的外部循环。

**类型：** 学习 + 实践
**语言：** Python（标准库）
**前提条件：** 完成第14阶段全部内容
**时间：** 约60分钟

## 学习目标

- 说出三个评估层——静态基准测试、自定义离线评估、在线生产评估——以及每个的目的。
- 解释评估器-优化器紧密循环。
- 描述2026年最佳实践：评估代码紧邻主代码，在CI中运行，作为PR合并门控。
- 将第14阶段的每堂课与其生成的评估案例关联。

## 问题所在

智能体在演示中表现良好，却在生产环境中以演示无法预测的方式失败。基准测试回答“此模型是否具有广泛能力？”，而非“此智能体是否为我的产品提供了正确的补丁？”解决方案：三层评估持续运行，每个防护栏和学习规则都映射到一个评估案例。

## 概念详解

### 三层评估

1. **静态基准测试** —— 代码评估用SWE-bench Verified（第19课），浏览/桌面评估用WebArena/OSWorld（第20课），通用评估用GAIA（第19课），工具使用评估用BFCL V4（第06课）。用于跨模型比较和回归测试门控。数据污染真实存在：SWE-bench+发现32.67%的解答泄露。务必报告Verified或+-审计后的分数。

2. **自定义离线评估** —— 贴合你的产品形态：
   - LLM作为评判（Langfuse、Phoenix、Opik —— 第24课）。
   - 基于执行（运行补丁，检查测试）。
   - 基于轨迹（将动作序列与标准答案对比；OSWorld-Human显示顶级智能体达到标准答案的1.4-2.7倍）。

3. **在线评估** —— 生产环境：
   - 会话重放（Langfuse）。
   - 防护栏触发警报（第16、21课）。
   - 每步骤成本/延迟跟踪（第23课OTel spans）。

### 评估器-优化器（Anthropic方法）

紧密循环：

1. 提议者生成输出。
2. 评估器判断。
3. 迭代优化直至评估器通过。

这是自优化（第05课）的泛化。任何你关注的智能体工作流都可以用评估器-优化器包裹以提升可靠性。

### 2026年最佳实践

- 评估代码与主代码紧邻。
- 在每次PR的CI中运行。
- 以评估分数作为合并门控（例如“相对main分支无>5%的退化”）。
- 每个防护栏映射到一个评估案例。
- 每条学习规则（Reflexion、专业工作流学习规则）映射到一个失败案例。

### 整合第14阶段

第14阶段的每堂课都会生成评估案例：

| 课程 | 生成的评估案例 |
|------|----------------|
| 01 智能体循环 | 预算耗尽、无限循环防护 |
| 02 ReWOO | 工具失败时规划器能正确重新规划 |
| 03 Reflexion | 学习的反思在重试时应用 |
| 05 自优化/CRITIC | 评判器通过优化后的输出 |
| 06 工具使用 | 参数强制转换有效；未知工具被拒绝 |
| 07-10 记忆 | 检索引用匹配来源；过时事实失效 |
| 12 工作流模式 | 每种模式产生正确输出 |
| 13 LangGraph | 恢复后能精确重现状态 |
| 14 AutoGen参与者 | DLQ捕获崩溃的处理器 |
| 16 OpenAI智能体SDK | 防护栏在正确输入上触发 |
| 17 Claude智能体SDK | 子智能体结果返回协调器 |
| 19-20 基准测试 | SWE-bench Verified分数、WebArena成功率、OSWorld效率 |
| 21 计算机使用 | 每步安全检查捕获注入的DOM |
| 23 OTel | Spans发射所需属性 |
| 26 失败模式 | 检测器标记已知失败 |
| 27 提示注入 | PVE拒绝受污染的检索结果 |
| 28 协调 | 主管路由到正确的专家 |
| 29 运行时形态 | DLQ处理N%的失败 |

如果你的评估套件包含以上每个案例，你就覆盖了第14阶段。

### 评估驱动开发失败之处

- **无基线。** 无上次已知良好版本的评估难以解读。存储基线。
- **无依据的LLM评判。** 评判器也会幻觉。采用CRITIC模式（第05课）—— 评判基于外部工具依据。
- **过度拟合评估。** 为优化评估而偏离生产实用性。轮换案例。
- **不稳定的评估。** 非确定性案例导致误报。固定种子，快照状态。

## 构建

`code/main.py`是一个标准库评估框架：

- 带分类（基准、自定义、在线）的案例注册表。
- 受测的脚本化智能体。
- 评估器-优化器循环：提议、判断、优化直至通过或达最大轮次。
- CI门控：聚合通过率 + 相对基线的退化检测。

运行它：

```
python3 code/main.py
```

输出：每个案例的通过/失败、退化标志、CI门控结论。

## 使用它

- 在智能体代码同一仓库中编写评估案例。
- 通过CI在每次PR上运行它们。
- 退化时使构建失败。
- 随时间跟踪通过率。
- 将每个生产失败关联到一个新案例。

## 部署

`outputs/skill-eval-suite.md`为智能体产品构建三层评估套件，包含CI门控和退化跟踪。

## 练习

1. 选取你的一个生产失败案例。编写一个能复现它的评估案例。你的智能体现在能通过吗？
2. 为你的领域构建一个包含三个维度（事实性、语气、范围）的LLM评判评分标准。对50个会话评分。
3. 将评估套件接入CI。退化>=5%时使构建失败。
4. 添加轨迹效率指标：智能体相比标准轨迹用了多少步？
5. 将第14阶段每堂课映射到你套件中的一个评估案例。有缺失吗？那就是需要弥补的缺口。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 静态基准测试 | “开箱即用评估” | SWE-bench, GAIA, AgentBench, WebArena, OSWorld |
| 自定义离线评估 | “领域评估” | 针对你的产品形态的LLM-as-judge / 执行 / 轨迹评估 |
| 在线评估 | “生产评估” | 会话重放、防护栏警报、成本/延迟跟踪 |
| 评估器-优化器 | “提议-判断-优化” | 迭代直至评判器通过 |
| CI门控 | “合并阻塞器” | 评估退化时使构建失败 |
| 基线 | “上次已知良好版本” | 用于检测退化的参考分数 |
| 轨迹效率 | “超越标准的步数” | 智能体步数除以人类专家最小步数 |
