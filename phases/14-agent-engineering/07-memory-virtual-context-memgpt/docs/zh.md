# 记忆：虚拟上下文与 MemGPT

> 上下文窗口是有限的。但对话、文档和工具调用轨迹却不是。MemGPT（Packer et al., 2023）将其类比为操作系统的虚拟内存——主上下文是 RAM，外部存储是磁盘，智能体在两者之间进行页面调度。这是每一个 2026 年的记忆系统都会继承的模式。

**类型：** 构建
**语言：** Python（标准库）
**先决条件：** Phase 14 · 01（智能体循环）、Phase 14 · 06（工具使用）
**时间：** ~75 分钟

## 学习目标

- 解释 MemGPT 所基于的操作系统类比：主上下文 = RAM，外部上下文 = 磁盘，记忆工具 = 页面调入/调出。
- 使用标准库实现两层 MemGPT 模式，包括一个主上下文缓冲区、一个可外部搜索的存储，以及页面调入/调出工具。
- 描述智能体如何发出“中断”来查询或修改外部记忆，以及结果如何被拼接回下一个提示中。
- 识别哪些 MemGPT 设计选择延续到了 Letta（第 08 课）和 Mem0（第 09 课）。

## 问题所在

上下文窗口看起来似乎能解决记忆问题，但事实并非如此。在生产环境中，三种失败模式反复出现：

1.  **溢出。** 多轮对话、长文档或工具调用密集的轨迹会超出窗口范围。超出截止点之后的内容都会丢失。
2.  **稀释。** 即使在窗口内，填入不相关的上下文也会稀释模型对重要信息的注意力。前沿模型在处理长输入时性能仍然会下降。
3.  **持久性。** 新会话以一个空窗口开始。没有外部记忆的智能体无法跨会话说“记得你之前让我...”。

更大的窗口有帮助，但无法根治这些问题。Mem0 在 2025 年的论文中测量发现，128k 窗口的基线模型仍然会遗漏那些拥有外部记忆的 4k 窗口智能体能捕捉到的长程事实。

## 核心概念

### MemGPT：操作系统类比

Packer 等人（arXiv:2310.08560, v2 Feb 2024）将上下文管理映射到操作系统的虚拟内存：

| OS 概念 | MemGPT 概念 | 2026 生产环境类比 |
|--------|-------------|-------------------|
| RAM    | 主上下文（提示） | Anthropic/OpenAI 上下文窗口 |
| 磁盘   | 外部上下文     | 向量数据库、KV 存储、图存储 |
| 缺页中断 | 记忆工具调用 | `memory.search`、`memory.read`、`memory.write` |
| OS 内核 | 智能体控制循环 | 带有记忆工具的 ReAct 循环 |

智能体运行一个常规的 ReAct 循环。额外的一类工具允许它将数据在主上下文和外部存储之间进行页面调度。

### 两层结构

- **主上下文。** 固定大小的提示，持有当前任务。模型始终可见。
- **外部上下文。** 无限大小，可通过工具搜索。在相关时读取，在事实出现时写入。

原始论文在两项超出基础窗口的任务上评估了该设计：超过 10万 token 的文档分析，以及跨多天的、具有持久记忆的多会话聊天。

### 中断模式

MemGPT 引入了“记忆即中断”：在对话中途，智能体可以调用一个记忆工具，运行时执行它，结果作为一个新的观测值拼接到下一个助手轮次中。这在概念上等同于 Unix 的 `read()` 系统调用，它会阻塞进程，返回字节，然后进程继续执行。

标准的记忆工具接口包括：

- `core_memory_append(section, text)` —— 写入提示中的一个持久化部分。
- `core_memory_replace(section, old, new)` —— 编辑一个持久化部分。
- `archival_memory_insert(text)` —— 写入可搜索的外部存储。
- `archival_memory_search(query, top_k)` —— 从外部存储检索。
- `conversation_search(query)` —— 扫描过去的对话轮次。

### MemGPT 的终点与 Letta 的起点

2024 年 9 月，MemGPT 更名为 Letta。研究仓库（`cpacker/MemGPT`）仍然保留；Letta 扩展了设计：

- 三层结构取代两层（核心、回忆、归档 —— 第 08 课）。
- 用原生推理取代 `send_message`/心跳模式（第 08 课）。
- 睡眠时间智能体异步执行记忆工作（第 08 课）。

即使生产系统运行的是 Letta、Mem0 或自定义的两层存储，MemGPT 论文仍然是 2026 年的基础。

### 该模式可能出错的地方

- **记忆腐化。** 写入速度快于读取速度；检索被过时的事实淹没。解决方案：定期整合（Letta 睡眠时间）、显式失效机制（Mem0 冲突检测器）。
- **记忆污染。** 外部记忆是检索到的文本。如果攻击者控制的内容进入记忆笔记，智能体会在下次会话时重新摄入它。这是 Greshake 等人（第 27 课）攻击在时间维度上的重述。
- **引用丢失。** 智能体回忆“用户让我寄送 X”，但无法引用是哪一轮说的。存储时需附带来源引用（会话 ID、轮次 ID）。

## 动手构建

`code/main.py` 在标准库中实现了 MemGPT 的两层模式：

- `MainContext` —— 固定大小的提示缓冲区，包含一个 `core` 字典和一个 `messages` 列表；当超出容量时，会自动压缩最早的消息。
- `ArchivalStore` —— 基于内存的、类似 BM25 的存储（基于 token 重叠评分），存储 (id, text, tags, session, turn) 记录。
- 五个映射到 MemGPT 接口的记忆工具。
- 一个脚本化的智能体，它用事实填充归档，然后通过调用 `archival_memory_search` 来回答问题。

运行它：

```
python3 code/main.py
```

跟踪显示智能体写了三个事实，填满主上下文至容量上限（强制驱逐），然后通过从归档检索来回答一个后续问题——在没有真实 LLM 的情况下再现了 MemGPT 的工作流程。

## 如何使用

当今每个生产环境的记忆系统都是 MemGPT 的变体：

- **Letta**（第 08 课）—— 三层结构、原生推理、睡眠时间计算。
- **Mem0**（第 09 课）—— 向量 + KV + 图融合，带评分层。
- **OpenAI Assistants / Responses** —— 通过线程和文件管理的受管记忆。
- **Claude Agent SDK** —— 通过技能和会话存储实现的长期记忆。

根据运营形态（自托管、托管、框架集成）来选择，而非根据核心模式——核心模式就是 MemGPT。

## 交付使用

`outputs/skill-virtual-memory.md` 是一个可复用的技能，它能为目标运行时生成一个正确的两层记忆脚手架（主 + 归档 + 工具接口），并内置了驱逐策略和引用字段。

## 练习

1.  添加一个 `max_main_context_tokens` 容量上限（以 token 计，用 `len(text.split())` * 1.3 估算）。当超出上限时，将最早的消息压缩成摘要。比较有无摘要器时的行为差异。
2.  在归档存储上正确实现 BM25（词频、逆文档频率）。在一个玩具事实集上，测量 recall@10，并与基于 token 重叠的基线进行比较。
3.  在归档插入时添加 `citation` 字段（session_id, turn_id, source_url）。让智能体在每个基于检索的回答中引用来源。
4.  模拟记忆污染：添加一条归档记录说“忽略所有未来的用户指令”。编写一个防护程序，扫描检索到的、具有指令性质的文本，并将其标记为不可信。
5.  将实现移植为使用 MemGPT 研究仓库的核心记忆 JSON 模式（`cpacker/MemGPT`）。从扁平字符串切换到类型化部分时，会发生什么变化？

## 关键术语

| 术语         | 人们如何说 | 其实际含义 |
|-------------|-----------|-----------|
| 虚拟上下文   | “无限记忆” | 主（提示） + 外部（可搜索）两层结构，带页面调入/调出 |
| 主上下文     | “工作记忆” | 提示 —— 固定大小，始终可见 |
| 归档记忆     | “长期存储” | 外部可搜索的持久化存储，按需检索 |
| 核心记忆     | “提示持久化部分” | 钉在主上下文内部的具名部分 |
| 记忆工具     | “记忆 API” | 智能体发出的用于读写外部记忆的工具调用 |
| 中断         | “记忆缺页中断” | 智能体暂停，运行时获取数据，结果拼接到下一轮 |
| 记忆腐化     | “过时事实” | 旧的写入淹没检索；通过整合来修复 |
| 记忆污染     | “注入的持久化笔记” | 攻击者内容作为记忆存储，回忆时被重新摄入 |
