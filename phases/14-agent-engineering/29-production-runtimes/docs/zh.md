# 生产运行时：队列、事件、定时任务

> 生产环境中的智能体运行在六种运行时形态上：请求-响应、流式输出、持久执行、基于队列的后台任务、事件驱动和定时调度。先选定形态，再挑选框架。在每种形态下可观测性都至关重要。

**类型：** 学习  
**语言：** Python（标准库）  
**前置要求：** 第14期·13（LangGraph），第14期·22（语音）  
**时间：** 约60分钟

## 学习目标

- 列举六种生产运行时形态，并将每种形态与对应的框架/产品模式相匹配。
- 解释持久执行（LangGraph）对于长周期任务的重要性。
- 描述事件驱动运行时，以及Claude Managed Agents的适用场景。
- 解释“可观测性至关重要”这一论断在多步骤智能体中的意义。

## 问题所在

生产智能体以Jupyter笔记本无法体现的方式失败：网络在第37步超时、用户在通话中途挂断、定时任务因机器重启而终止、后台工作进程内存耗尽。运行时形态决定了哪些故障是可恢复的。

## 核心概念

### 请求-响应

- 同步HTTP请求。用户需等待完成。
- 仅适用于短时任务（<30秒）。
- 技术栈：Agno（Python + FastAPI），Mastra（TypeScript + Express/Hono/Fastify/Koa）。
- 可观测性：标准HTTP访问日志 + OpenTelemetry追踪跨度。

### 流式输出

- 使用SSE或WebSocket实现渐进式输出。
- LiveKit将其扩展到WebRTC以支持语音/视频（第22课）。
- 技术栈：任何支持流式输出的框架 + 处理SSE/WS的前端。
- 可观测性：每块数据的计时、首token延迟、尾部延迟。

### 持久执行

- 每个步骤后检查点保存状态；故障后自动恢复。
- AutoGen v0.4的Actor模型将故障隔离到单个智能体（第14课）。
- LangGraph的核心优势（第13课）。
- 当步骤数未知且恢复成本高时至关重要。

### 基于队列/后台任务

- 任务进入队列，工作进程拾取，结果通过webhooks或发布/订阅返回。
- 对长周期智能体必不可少（根据Anthropic的计算机使用公告，每项任务需数十到数百个步骤）。
- 技术栈：Celery（Python）、BullMQ（Node）、SQS + Lambda（AWS）、自定义方案。
- 可观测性：队列深度、每项任务的延迟分布、死信队列大小。

### 事件驱动

- 智能体订阅触发器：新邮件、PR创建、定时任务触发。
- Claude Managed Agents开箱即用地覆盖此场景（第17课）。
- CrewAI Flows（第15课）用于构建事件驱动的确定性工作流。
- 可观测性：触发源、事件到启动的延迟、智能体延迟。

### 定时调度

- 以定时任务形式运行的周期性智能体。
- 结合持久执行，使失败的夜间任务能在下次触发时恢复。
- 技术栈：Kubernetes CronJob + 持久框架；托管服务（Render cron、Vercel cron）。

### 2026年部署模式

- **CrewAI Flows**：用于事件驱动的生产环境。
- **Agno**：无状态FastAPI，适用于Python微服务。
- **Mastra**：服务器适配器（Express、Hono、Fastify、Koa），便于嵌入。
- **Pipecat Cloud / LiveKit Cloud**：托管语音服务（第22课）。
- **Claude Managed Agents**：托管长时间运行的异步任务。

### 可观测性至关重要

没有OpenTelemetry GenAI跨度（第23课）加上Langfuse/Phoenix/Opik后端（第24课），你将无法调试一个在第40步失败的多步骤智能体。这对生产环境而言不是可选项。这决定了“我们能快速调试”与“我们只能通过增加日志从头重放”之间的差异。

### 生产运行时失败之处

- **形态选择错误**。为一个5分钟的任务选择请求-响应模式。用户挂断；工作进程堆积；重试加剧。
- **无死信队列**。队列工作进程缺少死信队列。失败的任务消失。
- **后台工作不透明**。后台智能体运行时未导出追踪数据。故障在用户报告前不可见。
- **跳过持久状态**。任何超过30秒且不能承受重启代价的运行都需要持久执行。

## 动手实践

`code/main.py`是一个基于标准库的多形态演示：

- 请求-响应端点（普通函数）。
- 流式处理器（生成器）。
- 带死信队列的基于队列的工作进程。
- 事件触发器注册表。
- 定时任务调度器。

运行它：

```bash
python3 code/main.py
```

输出：五个追踪记录，展示同一任务在每种形态下的行为。相同的智能体逻辑，不同的外部封装。持久执行（第六种形态）特意在第13课中通过LangGraph的检查点机制讲解。

## 应用场景

- **请求-响应**：用于类聊天用户体验。
- **流式输出**：用于渐进式响应。
- **持久执行**：用于长周期任务。
- **队列**：用于批处理/异步/长时间运行任务。
- **事件驱动**：用于智能体反应性。
- **定时任务**：用于维护工作（记忆整合、评估、成本报告）。

1. 将你的第01课ReAct循环移植到你的技术栈中的所有六种形态。哪种形态适合哪种产品界面？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|---------|----------|
| 请求-响应 | “同步” | 用户等待；仅限短任务 |
| 流式输出 | “SSE / WS” | 渐进式输出；更好的用户体验；每块数据延迟可观察 |
| 持久执行 | “从故障恢复” | 检查点保存状态；从最后一步重启 |
| 基于队列 | “后台任务” | 生产者/工作进程池/死信队列 |
| 事件驱动 | “基于触发” | 智能体对外部事件做出反应 |
| 死信队列 | “死信队列” | 失败任务的暂存区 |
| Claude Managed Agents | “托管运行框架” | Anthropic托管的长周期异步服务，带缓存和压缩 |
