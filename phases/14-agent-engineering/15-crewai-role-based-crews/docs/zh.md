# CrewAI：基于角色的团队与流程

> CrewAI 是 2026 年基于角色的多智能体框架。四个基本原语：智能体、任务、团队、流程。两种顶层结构：团队（自主、基于角色的协作）和流程（事件驱动、确定性）。文档直言不讳："对于任何生产就绪的应用，请从流程开始。"

**类型：** 学习 + 构建  
**语言：** Python（标准库）  
**前置知识：** 阶段 14·12（工作流模式），阶段 14·14（演员模型）  
**时间：** 约 75 分钟

## 学习目标

- 说出 CrewAI 的四个基本原语（智能体、任务、团队、流程）及其各自的所有者。
- 区分顺序、层级和共识流程；为每种工作负载选择合适的流程。
- 区分团队（自主、基于角色）和流程（事件驱动、确定性），并解释文档中的生产建议。
- 使用 `@tool` 装饰器和 `BaseTool` 子类连接工具；推导结构化输出与自由文本的区别。
- 说出 CrewAI 的四种记忆类型及其各自的适用场景。
- 使用标准库实现一个三人团队（研究员、撰稿人、编辑），产出简报。
- 识别 CrewAI 的三种失败模式：提示膨胀、管理器 LLM 开销、脆弱的交接。

## 问题所在

采用多智能体框架的团队会遇到同样的障碍。"自主协作"在演示中听起来很棒。但当客户提交 Bug 时，你需要确定性重放。或者财务部想知道一个 LLM 路由的团队每次运行的成本。或者值班人员需要知道哪个智能体在凌晨 3 点卡住了。

自由形式的 LLM 路由团队无法清晰地回答这些问题。纯 DAG 能回答所有问题，但却失去了头脑风暴智能体所需的探索性形态。

CrewAI 的划分坦诚地面对了这种权衡。团队用于协作性、基于角色、探索性的工作。流程用于事件驱动、代码可控、可审计的生产环境。同一框架，两种形态，按场景选择。

## 核心概念

### 四个基本原语

CrewAI 的接口很小。记住这些，其余都是配置。

- **智能体。** `role + goal + backstory + tools + (optional) llm`。背景故事至关重要。它塑造了语气、判断力以及智能体何时停止。工具是智能体可以调用的函数（详见下文）。
- **任务。** `description + expected_output + agent + (optional) context + (optional) output_pydantic`。可复用的工作单元。`expected_output` 是契约。`context` 列出了其输出将被传入的上游任务。`output_pydantic` 强制要求结构化形式。
- **团队。** 容器。拥有 `agents` 列表、`tasks` 列表、`process`，以及可选的 `memory` + `verbose` + `manager_llm` 设置。
- **流程。** 执行策略。顺序、层级、共识。决定运行的形态。

智能体之间不能直接相互看见。任务引用智能体。团队排序任务。流程决定谁选择下一个任务。这就是整个心智模型。

### 顺序、层级与共识

- **顺序。** 任务按声明顺序运行。任务 N 的输出可作为 `context` 提供给任务 N+1。成本最低。可预测性最高。当顺序固定时使用。
- **层级。** 一个管理器智能体（独立的 LLM 调用）在专家之间进行路由。CrewAI 会根据您的 `manager_llm` 配置或默认设置生成管理器。管理器每轮选择下一个任务，并且可以拒绝或重新路由。当您有四个或更多专家，并且顺序真正依赖于先前输出时使用。
- **共识。** 测试版。智能体对下一步进行投票。除了研究用途，很少值得付出额外的往返开销。

层级处理会在每次专家调用之上增加一轮 LLM 调用（管理器）。在五步运行中，代币成本可能翻三倍。仅在您需要路由时才使用它。

### 团队与流程

这是文档在 2026 年的主打框架。

- **团队。** LLM 驱动的自主性。框架在运行时选择形态。适用于：研究、头脑风暴、初稿，以及路径本身是答案一部分的任何场景。难以重放。难以测试。原型制作成本低。
- **流程。** 您所拥有的事件驱动图。`@start` 标记入口。`@listen(topic)` 标记一个步骤，当另一个步骤发出该主题时触发。每个步骤都是普通 Python（可以在内部调用团队）。适用于：生产。可观察。可测试。确定性。

文档 2026 年的生产建议：从流程开始。当自主性物有所值时，将团队作为 `Crew.kickoff()` 调用折叠到流程步骤内部。流程为您提供审计跟踪，团队为您提供探索能力。组合，而非选择。

### 工具集成

为智能体提供工具的三种方式。选择最简单合适的一种。

1.  **`@tool` 装饰器。** 纯函数变成工具。签名是模式；文档字符串是 LLM 看到的描述。最适合一次性助手。

    ```python
   from crewai.tools import tool

   @tool("Search the web")
   def search(query: str) -> str:
       """Return top results for the query."""
       return run_search(query)
   ```

2.  **`BaseTool` 子类。** 基于类的工具，具有显式参数模式、异步支持、重试。当工具具有状态（客户端、缓存）或需要结构化参数时使用。

    ```python
   from crewai.tools import BaseTool
   from pydantic import BaseModel

   class SearchArgs(BaseModel):
       query: str
       limit: int = 10

   class SearchTool(BaseTool):
       name = "web_search"
       description = "Search the web and return top results."
       args_schema = SearchArgs

       def _run(self, query: str, limit: int = 10) -> str:
           return self.client.search(query, limit=limit)
   ```

3.  **内置工具包。** CrewAI 提供第一方适配器：`SerperDevTool`、`FileReadTool`、`DirectoryReadTool`、`CodeInterpreterTool`、`RagTool`、`WebsiteSearchTool`。只需一次导入即可连接。

结构化输出使用 Pydantic。在任务上传递 `output_pydantic=MyModel`。CrewAI 会根据模型验证 LLM 的响应，并进行强制转换或重试。将此与严格的 `expected_output` 字符串配对。自由文本输出适用于初稿；结构化输出是下游流程可以消费的内容。

### 记忆钩子

CrewAI 开箱即用地提供四种记忆类型。它们可以组合：一个团队可以同时启用所有四种。

- **短期。** 单次运行中的对话缓冲区。结束时清除。
- **长期。** 跨运行持久化。存储在向量数据库中（默认 Chroma，可替换）。通过与当前任务的相似性进行检索。
- **实体。** 每个实体的事实。"客户 X 在企业计划中。"按键为实体，而非相似性。跨运行存在。
- **上下文。** 组装时检索。在智能体需要记忆的时刻拉取相关记忆，而不是预加载。

在团队上使用 `memory=True` 或按类型配置来启用。由您配置的嵌入提供程序支持（默认为 OpenAI，可替换为本地）。记忆是 CrewAI 相对于更轻量级框架赢得优势的地方之一；纯 LangGraph 需要您自己连接这些功能。

### 何时使用 CrewAI

- 三到六个具有命名角色和协作工作流的智能体。起草、审查、规划、头脑风暴。
- 路由中 LLM 对下一步的判断本身就是价值（层级处理）。
- 在任何团队更喜欢阅读 `role + goal + backstory` 而不是阅读图定义的地方。

### 何时不使用 CrewAI

- 具有严格顺序的确定性 DAG。使用 LangGraph（课程 13）。图形态是正确的抽象；CrewAI 的角色框架会带来摩擦。
- 亚秒级延迟预算。层级处理会增加往返次数。即使是顺序处理，也会序列化包含背景故事和先前输出的提示。
- 单智能体循环。跳过框架；智能体循环（课程 1）加工具注册表更简短。

课程 17（智能体框架权衡）在矩阵中展示了这一点。简而言之：CrewAI 位于"协作性基于角色"的角落。

### 依赖形态

独立于 LangChain。Python 3.10 到 3.13。使用 `uv`。2026 年初 GitHub 星标超过 3 万。文档中记录了与 AWS Bedrock 的集成；他们的基准测试引用在 QA 任务上比 LangGraph 快 5.76 倍。将框架供应商的数字视为方向性参考。

### 此模式出错的地方

- **背景故事导致的提示膨胀。** 每个智能体 2000 字的背景故事和一个五智能体团队会在第一次工具调用之前就耗尽上下文预算。将背景故事控制在 200 字以内。跨智能体复用短语；不要重复五次相同的写作风格。
- **管理器 LLM 代币税。** 层级处理会在每次专家调用之前增加一个管理器 LLM 调用。在一个五任务团队中，这是六次 LLM 调用而不是五次，并且管理器调用携带完整的任务列表以及先前的输出。除非路由依赖于输出，否则切换为顺序处理。
- **脆弱的交接。** 任务 N 的 `expected_output` 是"一个提纲"。任务 N+1 将其作为 `context` 读取并尝试解析三个部分。LLM 生成了四个部分。下游智能体即兴发挥。通过在任务 N 上设置 `output_pydantic` 来修复，这样任务 N+1 就读取一个类型化对象，而不是自由文本。
- **团队即生产。** 没有流程包装器的自由形式团队被推到生产环境。输出变异性高；重放不可能；值班人员无法将一次错误的运行与一次好的运行进行比较。用流程包装。

## 构建它

`code/main.py` 实现了两种形态的标准库版本以及一个三人团队。

形态：

- `Agent`、`Task` 数据类，匹配 CrewAI 的接口。
- `SequentialCrew.kickoff(inputs)` 按声明顺序运行任务，将输出作为 `context` 传递。
- `HierarchicalCrew.kickoff(topic)` 添加一个管理器智能体，每轮选择下一个专家，在"完成"时停止。
- `Flow` 配合 `@start` 和 `@listen(topic)` 装饰器、一个小型事件循环和一个跟踪。
- `tool(name)` 装饰器，镜像 CrewAI 的 `@tool` 形态。
- `Memory` 配合 `short_term`、`long_term`、`entity` 存储；模拟的相似性使用 numpy。
- 模拟的 LLM 响应是硬编码的字符串，按键为角色加上输入前缀。无网络。确定性。

具体演示：研究员、撰稿人、编辑团队，产出关于"2026 年智能体工程"的简报。研究员拉取（模拟的）来源。撰稿人起草。编辑精简。同一团队通过流程运行，以展示确定性形态。

运行它：

```bash
python3 code/main.py
```

跟踪覆盖：顺序团队通过 `context` 传递输出、具有管理器选择的层级团队（研究员、撰稿人、编辑，然后"完成"）、使用显式主题（`researched`、`drafted`、`edited`）运行相同三个步骤的流程、通过 `@tool` 路由的工具调用，以及跨两次启动存在的长期记忆。

团队的跟踪是流动的；管理器原则上可以重新排序。流程的跟踪是固定的。这个选择就是本课的要点。

## 使用它

- **CrewAI 流程** 用于生产。即使流程是一个调用 `Crew.kickoff()` 的单一步骤。流程提供审计边界。
- **CrewAI 团队（顺序）** 用于顺序清晰的协作工作，特别是初稿和审查循环。
- **CrewAI 团队（层级）** 当路由依赖于输出，并且您有四个或更多专家时使用。
- **LangGraph**（课程 13）用于显式状态机、持久化恢复、严格顺序。
- **AutoGen v0.4**（课程 14）用于演员模型并发和故障隔离。
- **OpenAI 智能体 SDK**（课程 16）用于 OpenAI 优先的产品，具有交接和防护栏。
- **Claude 智能体 SDK**（课程 17）用于 Claude 优先的产品，具有子智能体和会话存储。

## 部署它

`outputs/skill-crew-or-flow.md` 为任务选择团队或流程，并搭建最小实现。严格拒绝：没有背景故事的团队、没有显式主题的流程、少于三个专家的层级处理。

## 陷阱

- **背景故事作为风味。** 它影响输出。为每个智能体测试三个变体；方差是真实的。选择一个，固定下来。
- **跳过 `expected_output`。** 如果每个任务没有契约，下游任务就会拾取 LLM 生成的任何内容。团队运行了；审计失败了。
- **记忆始终开启。** 每次运行都写入长期记忆。向量数据库增长。检索变得嘈杂。将写入范围限定在事实持续存在的任务上。
- **管理器提示漂移。** 层级处理的管理器提示是隐式的。如果路由变得奇怪，请在详细模式下转储它并阅读。
- **团队中的工具副作用。** 一个团队可能比预期更频繁地调用一个工具。POST、DELETE、支付属于流程步骤，永远不应成为团队工具。

## 练习

1.  将顺序团队转换为流程。计算变异性降低的接触点。注意可读性降低的地方。
2.  为团队添加实体记忆：关于客户的事实跨启动存在。验证检索是否拉取了正确的实体。
3.  实现一个层级处理，其中管理器拒绝路由到编辑，直到撰稿人的输出至少有三个段落。跟踪重试。
4.  为一个（模拟的）网页搜索连接一个 `BaseTool` 子类。比较跟踪形状与 `@tool` 装饰器版本。
5.  在编辑任务上添加 `output_pydantic=Brief`，其中 `Brief` 具有 `title`、`summary`、`sections`。让撰稿人任务输出一次格式错误的 JSON；在跟踪中验证 CrewAI 的重试行为。
6.  阅读 CrewAI 文档介绍。将玩具移植到真实的 `crewai` API。标准库版本跳过了哪些保证？
7.  将 AgentOps 或 Langfuse（课程 24）连接到一次真实的运行。在标准库版本中您错过了哪些跟踪？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|------------|----------|
| 智能体 | "人设" | 角色 + 目标 + 背景故事 + 工具 |
| 任务 | "工作单元" | 描述 + 预期输出 + 执行者 + 可选的结构化输出 |
| 团队 | "智能体团队" | 智能体 + 任务 + 流程的容器 |
| 流程 | "执行策略" | 顺序 / 层级 / 共识 |
| 流程 | "确定性工作流" | 事件驱动、代码可控、可测试 |
| 背景故事 | "人设提示" | 塑造智能体语气和判断力的因素 |
| `@tool` | "函数工具" | 将函数转换为智能体可调用工具的装饰器 |
| `BaseTool` | "类工具" | 基于类的工具，具有参数模式、重试、异步支持 |
| 实体记忆 | "每实体事实" | 限定于客户/账户/问题的记忆 |
| 长期记忆 | "跨运行记忆" | 在启动之间存在的向量支持记忆 |
| 上下文记忆 | "即时检索" | 在智能体需要的时刻拉取的记忆 |
| 管理器 LLM | "路由器智能体" | 层级处理中用于选择下一个任务的额外 LLM |
| `expected_output` | "任务契约" | 告知智能体（和审计）返回何种形状的字符串 |
