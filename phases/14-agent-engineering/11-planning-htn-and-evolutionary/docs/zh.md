# 结合HTN与进化搜索的规划

> 符号规划适用于可证明方案正确性的场景。进化代码搜索适用于适应度函数可机器验证的场景。ChatHTN（2025）与AlphaEvolve（2025）展示了二者与LLM结合后各自解锁的能力。

**类型：** 构建  
**语言：** Python（标准库）  
**前置条件：** Phase 14 · 02（ReWOO与Plan-and-Execute）  
**时间：** 约75分钟

## 学习目标

- 解释分层任务网络：任务、方法、算子、前置条件、效果。
- 描述ChatHTN的混合循环——符号搜索与LLM后备分解机制。
- 解释AlphaEvolve的进化循环及其仅适用于可编程评估器的原因。
- 使用标准库实现玩具级HTN规划器与玩具级进化搜索。

## 问题背景

ReWOO（第02课）、Plan-and-Execute和ReAct覆盖了大多数智能体规划场景。但两类情况处理不足：

1. **可证明正确性的方案。** 调度、航路规划、合规流程——方案必须在构造上保持可靠。流畅但偶尔幻觉的LLM方案难以接受。
2. **具备机器可验证适应度函数的优化问题。** 矩阵乘法、调度启发式、编译器优化——目标不是“正确方案”而是“最优方案”。

HTN规划与AlphaEvolve分别解决这两类问题。二者均将LLM作为放大器而非替代品。

## 核心概念

### 分层任务网络

HTN包含：

- **任务** —— 复合型（待分解）与基元型（可直接执行）。
- **方法** —— 将复合任务分解为子任务的方式，附带前置条件。
- **算子** —— 具备前置条件和效果的基元操作。
- **状态** —— 事实集合。

规划过程：给定目标任务和初始状态，寻找满足序列化前置条件的基元算子分解方案。

HTN技术早于LLM出现，至今仍是可证明正确方案的基准方法。

### ChatHTN（Gopalakrishnan等，2025）

ChatHTN（arXiv:2505.11814）将符号化HTN与LLM查询交替执行：

1. 尝试用现有方法分解当前复合任务。
2. 若无适用方法，询问LLM：“在状态`s`下如何分解`task`？”
3. 将LLM响应转换为候选子任务。
4. 通过算子模式验证；拒绝无效分解。
5. 递归执行。

论文核心主张：每个生成方案均具备可证明可靠性，因LLM建议仅作为候选分解输入，而非直接修改方案。符号层保证正确性；LLM扩展方法库。

在线方法学习（OpenReview`gwYEDY9j2x`，2025后续研究）新增学习器，通过回归泛化LLM生成的分解方案，最高可减少75%的LLM查询频率。

### AlphaEvolve（Novikov等，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025年6月）是另一技术路线：由Gemini 2.0 Flash/Pro集成模型编排的进化代码搜索。

循环流程：

1. 以种子程序和可编程评估器（返回适应度分数）开始。
2. LLM集成体提出变异方案。
3. 通过评估器运行变异程序。
4. 保留最优结果；再次变异。

已公布成果：

- 56年来首次突破Strassen算法实现4×4复数矩阵乘法优化（48次标量乘法）。
- 通过Borg调度启发式提升Google 0.7%计算资源回收率。
- 在前沿工作负载中实现FlashAttention 32%加速。

硬性约束：适应度函数必须可机器验证。对文本答案的进化搜索无法收敛。

### 适用场景选择

| 问题类别 | 选用技术 | 原因 |
|---------|---------|------|
| 带硬约束的调度问题 | HTN + ChatHTN | 可证明可靠性 |
| 编译器优化 | AlphaEvolve | 机器可验证适应度 |
| 多步骤任务执行 | ReAct / ReWOO | LLM参与循环，无形式化保证 |
| 带测试用例的代码优化 | AlphaEvolve | 测试用例即评估器 |
| 策略约束自动化 | HTN | 前置条件编码策略 |

### 模式失效场景

- **HTN缺少算子定义。** 若没有前置条件/效果模式，可靠性主张将失效。ChatHTN的“LLM建议分解”需要模式来拒绝无效操作。
- **AlphaEvolve缺乏真实评估器。** “询问LLM代码是否更优”并非适应度函数。评估器必须确定且高效。
- **过度工程化。** 多数智能体任务无需二者。优先考虑ReAct或ReWOO。

## 动手实现

`code/main.py`实现两个玩具示例：

- 标准库HTN规划器：包含算子、方法、前置条件、效果，当无方法匹配复合任务时触发`LLMFallback`。“LLM”采用脚本分解器实现离线运行。
- 标准库进化搜索：在算术程序上搜索使测试集`|f(x) - target|`最小的表达式。评估器具备确定性。

## 实践应用

- **HTN规划器** —— `pyhop`、`SHOP3`或为特定领域策略强制自建。
- **ChatHTN** —— 研究代码；其模式（符号化+LLM后备）可移植至任意HTN规划器。
- **AlphaEvolve** —— DeepMind论文；其模式（集成体+评估器）具备可复现性。OpenEvolve等开源分支正在涌现。
- **智能体框架** —— 目前均无一等公民支持HTN或AlphaEvolve。可作为子智能体或后台任务构建。

## 部署方案

`outputs/skill-hybrid-planner.md`生成混合规划器脚手架（HTN或进化型），明确限定LLM的作用范围。

## 实践练习

1. 为HTN规划器添加回溯机制：当运行时算子后置条件失败时，回滚并尝试下一个方法。
2. 为ChatHTN添加LLM方法缓存：当LLM在状态模式`P`下分解任务`T`时，存储结果。下次调用时优先检查方法库。
3. 将进化搜索评估器替换为真实测试套件。演化通过20个测试用例的排序函数；报告达到收敛所需的迭代代数。
4. 阅读AlphaEvolve的评估器设计文档。为你关心的领域设计评估器（SQL查询优化、测试套件最小化、部署YAML）。
5. 组合方案：使用HTN将复合任务分解为子任务，再对每个子任务的基元算子进行进化搜索。分析其优势场景与过度工程化场景。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| HTN | “分层规划器” | 带算子、前置条件、效果的任务分解 |
| 方法 | “分解规则” | 将复合任务拆分为子任务的方式 |
| 算子 | “基元操作” | 带前置条件和效果的具体步骤 |
| ChatHTN | “LLM + HTN” | 当无方法匹配时询问LLM的符号规划器 |
| AlphaEvolve | “进化代码搜索” | LLM集成体变异代码；确定性评估器筛选 |
| 适应度函数 | “评估器” | 对输出的确定性机器可验证评分 |
| 在线方法学习 | “缓存LLM分解” | 存储并泛化LLM方案以降低查询成本 |
