# 编排模式：监督者、群集、层级结构

> 2026年的AI框架中普遍出现四种编排模式：监督者-工作者、群集/点对点、层级结构、辩论。Anthropic的指导原则是：“关键在于构建符合您需求的系统。”从简单开始；仅当单一代理配合五种工作流模式仍不满足需求时，才引入拓扑结构。

**类型：** 学习 + 实践
**语言：** Python (标准库)
**前置要求：** 第14期 · 第12节（工作流模式），第14期 · 第25节（多代理辩论）
**时间：** 约60分钟

## 学习目标

- 识别四种常见编排模式及其适用场景。
- 阐述2026年LangChain的建议：基于工具调用的监督与监督库的对比。
- 解释Anthropic的“构建合适系统”原则及其对拓扑选择的约束。
- 使用Python标准库，针对一个预设的脚本化LLM实现全部四种模式。

## 问题所在

团队往往在真正需要之前就急于采用“多代理”方案。四种模式在框架中反复出现；一旦你能清楚命名它们，就能做出正确选择——或完全跳过拓扑结构。

## 核心概念

### 监督者-工作者模式

- 一个中央路由LLM调度任务给专家代理。
- 决策：循环回到自身、移交给专家、或终止任务。
- 专家之间不直接通信；所有路由均通过监督者进行。

**相关框架：** LangGraph `create_supervisor`、Anthropic的编排器-工作者模式、CrewAI的层级流程。

**2026年LangChain建议：** 通过直接工具调用进行监督，而非使用 `create_supervisor`。这能提供更精细的上下文工程控制——您可以精确决定每个专家能看到什么。

### 群集 / 点对点模式

- 代理通过共享的工具接口直接进行任务移交。
- 没有中央路由器。
- 延迟低于监督者模式（跳转次数更少）。
- 更难推理（无单一控制点）。

**相关框架：** LangGraph的群集拓扑、OpenAI Agents SDK的移交机制（当所有代理均可互相移交时）。

### 层级结构模式

- 监督者管理子监督者，子监督者再管理工作者。
- 在LangGraph中实现为嵌套子图；在CrewAI中实现为嵌套团队。
- 以操作复杂性为代价，支持大规模代理群体。

**适用时机：** 当单个监督者的上下文预算无法容纳所有专家描述时。

### 辩论模式

- 并行提案者 + 迭代式交叉批评（见第25课）。
- 严格来说不是编排模式——更偏向验证——但在框架中常作为一种拓扑选择出现。

### CrewAI中的团队与流程

CrewAI定义了两种部署模式：

- **流程**：用于确定性的事件驱动自动化（生产环境推荐的起点）。
- **团队**：用于基于角色的自主协作。

这与上述四种模式正交，但可映射到拓扑结构：流程通常对应监督者或层级结构；团队通常对应带有LLM路由器的监督者模式。

### Anthropic的指导原则

“在LLM领域的成功不在于构建最复杂的系统，而在于构建符合您需求的系统。”

**决策顺序：**

1.  **单一代理 + 工作流模式**（第12课）——从这里开始。
2.  **监督者-工作者模式**——当您有2-4个专家时。
3.  **群集模式**——当延迟比推理清晰度更重要时。
4.  **层级结构模式**——仅当监督者的上下文预算不足时。
5.  **辩论模式**——当准确性比成本更重要时。

### 此模式可能失效的场景

- **拓扑优先思维。** 在明确多代理要解决什么问题之前，就决定“我们需要多代理”。
- **群集中的任务反弹。** A -> B -> A -> B。使用跳转计数器。
- **虚假的层级结构。** 为了“企业级”而设三层，实际只有两个团队。应合并。

## 动手构建

`code/main.py` 使用Python标准库，针对一个脚本化LLM实现了全部四种模式：

- `Supervisor` —— 中央路由器。
- `Swarm` —— 具有直接移交机制的点对点模式。
- `Hierarchical` —— 监督者的监督者。
- `Debate` —— 并行提案者 + 批评。

每种模式处理相同的三意图任务（退款 / 故障 / 销售）。追踪轨迹的形状各不相同。

**运行方式：**

```
python3 code/main.py
```

**输出：** 每种模式的轨迹 + 操作次数。监督者模式最清晰；群集模式最短；层级结构最深；辩论模式最昂贵。

## 练习

1.  移除路由器，将监督者-工作者模式转换为群集模式。什么会失效？什么会改善？
2.  为群集模式添加一个跳转计数器：超过3次移交则拒绝。这能否阻止 A->B->A 的任务反弹？
3.  为一个包含12个专家的领域构建一个两级层级结构系统。如果没有嵌套，上下文预算会在何处不足？
4.  在一个模拟生产环境的工作负载上，对四种模式进行性能分析。在延迟、成本、准确性、可调试性等指标上，哪种模式最优？
5.  阅读Anthropic的“构建高效代理”文章。将您的每个生产流程映射到四种模式之一。是否有流程无法清晰映射？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 监督者-工作者 | "路由器 + 专家" | 中央LLM调度任务给专家；专家之间不直接通信 |
| 群集 | "点对点" | 通过共享工具直接移交任务；无中央路由器 |
| 层级结构 | "监督者的监督者" | 针对大规模代理群体的嵌套子图 |
| 辩论 | "提案者 + 批评" | 并行提案者，交叉批评（第25课） |
| 基于工具调用的监督 | "无库的监督" | 将监督者实现为直接的工具调用以控制上下文 |
| 团队 | "自主团队" | CrewAI的基于角色的协作模式 |
| 流程 | "确定性工作流" | CrewAI的事件驱动生产模式 |
