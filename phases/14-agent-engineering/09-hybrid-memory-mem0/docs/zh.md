# 混合记忆：向量 + 图 + KV (Mem0)

> Mem0 (Chhikara 等人，2025) 将记忆视为三个并行存储——向量存储用于语义相似性，键值存储用于快速事实查找，图存储用于实体关系推理。一个评分层在检索时融合这三者。这是2026年外部记忆的生产标准。

**类型：** 构建
**语言：** Python (标准库)
**前置要求：** 阶段 14 · 07 (MemGPT)， 阶段 14 · 08 (Letta Blocks)
**时间：** 约75分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅KV）不足以满足智能体记忆需求。
- 列出Mem0的三个并行存储及其各自优化的方面。
- 描述Mem0的融合评分机制——相关性、重要性、时效性——以及为什么它是加权求和，而非层级结构。
- 使用标准库实现一个包含 `add()`（写入所有三个存储）和 `search()`（融合结果）的玩具级三存储记忆系统。

## 问题所在

单个存储在三类查询中总有两类会出错：

- **语义相似性** —— “我们上周关于智能体漂移讨论了什么？” 向量存储胜出；KV和图存储无法匹配。
- **事实查找** —— “用户的电话号码是什么？” KV存储胜出；向量存储浪费资源，图存储小题大做。
- **关系推理** —— “哪些客户共享同一个账单实体？” 图存储胜出；向量和KV存储无法回答。

生产环境中的智能体在一次会话中会发出所有三类查询。单存储记忆系统总有两类查询会出错。Mem0的贡献在于将三者整合到一个 `add`/`search` 接口之后，并用一个评分函数进行融合。

## 核心概念

### 三个并行存储

Mem0 (arXiv:2504.19413, 2025年4月) 在 `add(text, user_id, metadata)` 阶段：

1. 从文本中提取候选事实（一个由LLM驱动的步骤）。
2. 将每个事实写入向量存储（嵌入），用于语义搜索。
3. 将每个事实写入KV存储，以 (user_id, fact_type, entity) 为键，实现 O(1) 查找。
4. 将每个事实写入图存储 (Mem0g)，作为类型化边用于关系查询。

在 `search(query, user_id)` 阶段：

1. 向量存储根据嵌入余弦相似度返回 top-k 结果。
2. KV存储根据查询派生的 (user_id, type, entity) 键直接命中返回。
3. 图存储返回从查询实体可达的子图。
4. 一个评分层融合三者的结果。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性** —— 向量余弦相似度、KV精确匹配、图路径权重。
- **重要性** —— 在写入时标记或学习得到（某些事实更重要：姓名、ID、政策）。
- **时效性** —— 自上次写入或读取以来时间的指数衰减。

权重根据产品特性进行调整。聊天智能体侧重 `w_recency`；合规智能体侧重 `w_importance`；检索智能体侧重 `w_relevance`。

### Mem0g 与时序推理

Mem0g 增加了一个冲突检测器。当新事实与现有边矛盾时，现有边被标记为无效但不删除。时序查询（“用户三月份所在的城市是什么？”）会遍历在指定时间有效的子图。

这正是Letta无效化模式所泛化的合规级行为。

### 基准测试数据

Mem0 论文报告 (2025)：

- **LoCoMo** (长对话记忆): 91.6
- **LongMemEval** (长视野情景记忆): 93.4
- **BEAM 1M** (百万token记忆基准): 64.1

对比基线（全上下文128k LLM、扁平向量存储、扁平KV存储）均落后10分以上。仅凭基准测试不足以证明选择的合理性——运营形态更重要——但这些数字表明融合设计并非微不足道的误差。

### 作用域分类法

Mem0 按作用域划分记忆：

- **用户记忆** —— 跨会话持久化，以 `user_id` 为键。
- **会话记忆** —— 在单个对话线程内持久化。
- **智能体记忆** —— 每个智能体实例的状态。

每次写入选择一个作用域。检索时可以跨作用域查询，并使用每个作用域的权重。不加思考地混合作用域是导致“助手告诉Alice关于Bob的项目”这类事故的原因。

### 这种模式的潜在问题

- **嵌入漂移。** 向量结果在头一百次查询时看起来正确，但随着语料库增长会变差。应定期对使用频率最高的top-N记录重新嵌入。
- **KV模式膨胀。** `(user_id, type, entity)` 看起来简单，直到每个团队都添加自己的 `type`。应季度审计类型集合。
- **图爆炸。** 一个嘈杂的提取器每条消息添加50条边。应限制每次 `add` 调用的图写入量；丢弃低置信度的边。

## 构建它

`code/main.py` 在标准库中实现了三存储模式：

- `VectorStore` —— 使用朴素的token重叠相似度作为嵌入的替代。
- `KVStore` —— 以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore` —— 类型化边 (主体， 关系， 客体， 有效时间)。
- `Mem0` —— 顶层外观接口，包含 `add()`, `search()`, 融合评分以及作用域感知检索。
- 一个基于多用户、多会话对话的完整运行轨迹。

运行它：

```
python3 code/main.py
```

输出显示了三个独立的召回路径以及融合后的top-k结果。在 `main()` 的顶部翻转评分权重，观察排名的变化。

## 使用它

- **Mem0 (Apache 2.0)** —— 生产就绪。使用 Postgres + Qdrant + Neo4j 自托管，或使用托管云服务。
- **Letta** —— 核心/召回/归档三层架构；自带向量和图后端。
- **Zep** —— 商业替代方案，具有时序知识图谱和事实提取功能。
- **自定义构建** —— 当你需要精确控制提取器（合规性）或融合权重（语音智能体中时效性主导）时。

## 交付它

`outputs/skill-hybrid-memory.md` 生成一个包含融合评分器、作用域分类法和时序无效化功能的三存储记忆脚手架。

## 练习

1. 将玩具向量相似度替换为真实的嵌入模型（sentence-transformers, Ollama, OpenAI embeddings）。在合成的长对话上测量 recall@10。在1000次写入后排名是否发生漂移？
2. 添加一个时序查询：`search(query, as_of=timestamp)`。仅返回在该时间点或之前有效的记录。哪个存储需要最多改进？
3. 实现一个冲突检测器：如果传入的事实与图的一条边矛盾，则将旧边标记为无效并记录两者。在“用户住在柏林” -> “用户住在里斯本”的案例上进行测试。
4. 扩展融合评分器以包含 `user_feedback` 维度（对检索到的记录点赞）。如何防止博弈（智能体只返回它已经喜欢过的记录）？
5. 阅读Mem0文档 (`docs.mem0.ai`)。将玩具实现移植到使用 `mem0` 客户端调用。在相同的20个测试查询上比较检索质量。

## 关键术语

| 术语 | 人们常称 | 实际含义 |
|------|----------|----------|
| 混合记忆 | “向量加图加KV” | 三个并行写入的存储，在检索时融合 |
| 事实提取 | “记忆摄入” | LLM步骤，将文本拆分为 (实体， 关系， 事实) 元组 |
| 融合评分 | “相关性排名” | 相关性、重要性、时效性的加权求和 |
| 作用域 | “记忆命名空间” | 用户 / 会话 / 智能体 —— 决定谁能看到什么 |
| Mem0g | “记忆图” | 具有时序有效性的类型化边，用于关系查询 |
| 时序无效化 | “软删除” | 将矛盾的边标记为无效；永不删除 |
| 廖嵌入漂移 | “检索退化” | 向量质量随语料库增长而下降；需定期重新嵌入 |
