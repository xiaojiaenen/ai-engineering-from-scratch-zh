# 自我优化与CRITIC：迭代式输出改进

> Self-Refine（Madaan等人，2023）使用一个大语言模型在循环中扮演三个角色——生成、反馈、优化。平均增益：在7项任务上绝对提升+20分。CRITIC（Gou等人，2023）通过将验证步骤路由至外部工具来强化反馈环节。到2026年，此模式以"评估器-优化器"（Anthropic）或防护栏循环（OpenAI智能体SDK）的形式融入每个框架。

**类型：** 构建
**语言：** Python（标准库）
**前置条件：** 第14阶段 · 01（智能体循环），第14阶段 · 03（Reflexion）
**时间：** 约60分钟

## 学习目标

- 阐述Self-Refine的三个提示词（生成、反馈、优化），并解释为何历史记录对优化提示词至关重要。
- 解释CRITIC的核心洞见：缺乏外部依据时，大语言模型在自我验证方面不可靠。
- 使用标准库实现一个带历史记录和可选外部验证器的Self-Refine循环。
- 将此模式映射到Anthropic的"评估器-优化器"工作流和OpenAI智能体SDK的输出护栏。

## 问题所在

智能体生成一个几乎正确的答案。可能是一行代码存在语法错误。可能是一份摘要过长。可能是一个计划遗漏了边缘情况。你想要的是：智能体批评其自身输出，然后修正它。

Self-Refine证明这通过单个模型即可实现，无需训练数据，无需强化学习。但存在一个隐患：大语言模型在硬事实方面的自我验证能力很差。CRITIC指出了修正方向——通过外部工具（搜索、代码解释器、计算器、测试运行器）路由验证步骤。

这两篇论文共同定义了2026年迭代改进的默认范式：生成、验证（尽可能使用外部工具）、优化，当验证器通过时停止。

## 概念解析

### Self-Refine (Madaan等人，NeurIPS 2023)

单个大语言模型，三个角色：

```
generate(task)            -> output_0
feedback(task, output_0)  -> critique_0
refine(task, output_0, critique_0, history) -> output_1
feedback(task, output_1)  -> critique_1
refine(task, output_1, critique_1, history) -> output_2
...
stop when feedback says "no issues" or budget exhausted.
```

关键细节：`refine` 会看到完整的历史记录——所有先前的输出和批评——因此它不会重复犯错。论文对此进行了消融实验：去除历史记录后，质量急剧下降。

亮点：在包括GPT-4在内的7项任务（数学、代码、缩写、对话）上平均绝对提升+20分。无需训练，无需外部工具，单一模型。

### CRITIC (Gou等人，arXiv:2305.11738, v4 2024年2月)

Self-Refine的弱点：反馈步骤是大语言模型对自身进行评分。对于事实性主张，这是不可靠的（幻觉对于产生它的模型来说通常看起来很有说服力）。CRITIC将 `feedback(task, output)` 替换为 `verify(task, output, tools)`，其中 `tools` 包括：

- 用于事实性主张的搜索引擎。
- 用于代码正确性的代码解释器。
- 用于算术的计算器。
- 特定领域的验证器（单元测试、类型检查器、代码检查工具）。

验证器产生基于工具结果的结构化批评。优化器随后根据此批评进行条件优化。

亮点：CRITIC在事实性任务上优于Self-Refine，因为其批评是有依据的。在缺乏外部验证器的任务（创意写作、格式化）上，CRITIC退化为Self-Refine。

### 停止条件

两种常见形式：

1. **验证器通过。** 外部测试返回成功。在可用时优先（单元测试、类型检查器、护栏断言）。
2. **未发出反馈。** 模型表示"输出良好"。成本较低但不可靠；需配合最大迭代次数限制。

2026年默认方式：结合两者。"如果验证器通过，或者模型表示良好且迭代次数 >= 2，或者迭代次数 >= 最大迭代次数，则停止。"

### 评估器-优化器（Anthropic，2024）

Anthropic 2024年12月的帖子将此列为五种工作流模式之一。两个角色：

- 评估器：对输出进行评分并生成批评。
- 优化器：根据批评修改输出。

循环直到评估器通过。这是Anthropic框架下的Self-Refine/CRITIC。Anthropic补充的关键工程细节是：评估器和优化器的提示词应有显著差异，以避免模型只是走过场。

### OpenAI智能体SDK输出护栏

OpenAI智能体SDK将此模式实现为"输出护栏"。护栏是运行在智能体最终输出之上的验证器。如果护栏触发（引发 `OutputGuardrailTripwireTriggered`），则输出被拒绝，智能体可以重试。护栏可以调用工具（CRITIC风格）或作为纯函数（Self-Refine风格）。

### 2026年的潜在陷阱

- **走过场循环。** 同一个模型使用相同的提示词风格进行生成和批评，会收敛于"看起来不错"。使用结构上不同的提示词，或使用更小、更便宜的模型进行批评。
- **过度优化。** 每次优化都会增加延迟和token消耗。预算1-3次；之后升级为人工审核。
- **在简单任务上使用CRITIC。** 如果没有外部验证器，CRITIC会退化为Self-Refine；不要为虚拟的验证器支付延迟成本。

## 动手构建

`code/main.py` 在一个玩具任务上实现了Self-Refine和CRITIC：根据主题生成一个简短的项目符号列表。验证器检查格式（3个要点，每个少于60个字符）。CRITIC增加了一个外部"事实验证器"，用于惩罚已知的幻觉。

组件：

- `generate` —— 脚本化的生产者。
- `feedback` —— 大语言模型风格的自我批评。
- `verify_external` —— CRITIC风格的基于依据的验证器。
- `refine` —— 根据历史记录重写输出。
- 停止条件 —— 验证器通过或最多4次迭代。

运行它：

```
python3 code/main.py
```

比较Self-Refine和CRITIC的运行结果。CRITIC捕捉到了一个Self-Refine遗漏的事实性错误，因为外部验证器拥有自我批评所没有的依据。

## 实际应用

Anthropic的评估器-优化器是此模式在Claude友好语言中的体现。OpenAI智能体SDK的输出护栏是CRITIC形态的（护栏可以调用工具）。LangGraph提供了一个读起来像Self-Refine的反思节点。Google的Gemini 2.5 Computer Use增加了一个逐步安全评估器，这是CRITIC的一个变体：每个操作在提交前都会被验证。

## 部署上线

`outputs/skill-refine-loop.md` 根据任务形态、验证器可用性和迭代预算，配置一个评估器-优化器循环。输出生成器、评估器/验证器和优化器的提示词，以及停止策略。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| Self-Refine | "自我修正的大语言模型" | 单个模型内，带历史记录的生成 -> 反馈 -> 优化循环 |
| CRITIC | "工具依据的验证" | 用外部验证器（搜索、代码、计算、测试）替代反馈 |
| 评估器-优化器 | "Anthropic工作流模式" | 两个角色——评估器评分，优化器修订——循环至收敛 |
| 输出护栏 | "事后检查" | OpenAI智能体SDK的验证器，在智能体生成输出后运行 |
| 验证步骤 | "批评阶段" | 承重决策点：基于依据还是自我评分 |
| 优化历史 | "模型已尝试的内容" | 优化提示词前附加的先前输出 + 批评；移除则质量崩溃 |
| 走过场循环 | "自我一致失败" | 相同提示词的批评返回"看起来不错"；用结构不同的提示词修复 |
| 停止条件 | "收敛测试" | 验证器通过或 无反馈且迭代次数达上限；切勿单一条件 |
