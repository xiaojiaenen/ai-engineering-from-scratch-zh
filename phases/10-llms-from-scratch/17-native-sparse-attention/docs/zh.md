# 原生稀疏注意力 (DeepSeek NSA)

> 在64k token的场景下，注意力机制占据了70-80%的解码延迟。每个开源模型实验室都有计划来解决这个问题。DeepSeek的NSA（ACL 2025最佳论文）是其中一个成功落地方案：它包含三个并行的注意力分支——压缩的粗粒度token、选择性保留的细粒度token以及用于局部上下文的滑动窗口——通过一个可学习的门控机制进行组合。它硬件友好（对内核友好）、原生可训练（在预训练中使用，而非在推理时附加），并且在64k解码中，其运行速度比FlashAttention更快，同时匹配或超越全注意力的质量。本课程将端到端地构建这三个分支，并展示为何这种稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**先修课程：** 阶段7 · 12（KV缓存，Flash-Attention），阶段7 · 15（注意力变体），阶段10 · 16（差分注意力）
**时间：** ~60分钟

## 学习目标

- 陈述NSA的三个注意力分支及其各自捕捉的内容。
- 解释为何NSA是“原生可训练的”，而之前的稀疏注意力方法仅用于推理。
- 作为压缩块大小和选择top-k的函数，计算NSA相对于全注意力在64k上下文下的注意力计算节省量。
- 使用标准库Python在一个短合成序列上实现三路分支组合，并验证门控行为。

## 问题所在

在序列长度为N时，全注意力的时间复杂度为`O(N^2)`，每层的KV缓存为`O(N)`。在64k token时，计算和内存带宽的数字是灾难性的。NSA论文中的理论估计：在64k时，注意力占总解码延迟的70-80%。下游所有指标——首token时间、每秒token数、每百万token成本——都由注意力成本主导。

稀疏注意力是显而易见的答案。先前的尝试分为两类。固定模式稀疏性（滑动窗口、跨步、块局部）会丢弃信息，在长程召回任务中表现不佳。推理时稀疏性（KV缓存剪枝、H2O、StreamingLLM）应用于在密集注意力上预训练的模型，只能恢复一小部分潜在加速，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan et al., DeepSeek + PKU + UW, ACL 2025最佳论文, arXiv:2502.11089）兼顾了两者：一种在预训练期间模型学习到的稀疏模式，实现为一种内核对齐的算法，在推理时真正实现了计算节省。两年后，NSA或其直接变体将成为每个前沿长上下文模型的默认注意力机制。

## 概念解析

### 三个并行分支

对于每个查询，NSA针对KV缓存的三个不同视图运行三次注意力：

1.  **压缩分支。** token被分组为大小为`l`的块（通常为32或64）。每个块通过一个小型可学习MLP被压缩为一个单一的摘要token。查询在这些压缩后的token上计算注意力，获得整个序列的粗粒度视图。

2.  **选择分支。** 利用来自压缩分支的注意力分数，识别出与当前查询最相关的top-k个块。从这些块中读取细粒度（未压缩的）token，查询在所有这些token上计算注意力。可以将压缩分支的注意力视为选择过程的路由信号。

3.  **滑动窗口分支。** 查询在最近的`W`个token（通常为512）上计算注意力，以获取局部上下文。此分支捕获结构丰富的短程模式（语法、局部共指），而这些可能被其他两个分支遗漏。

三个分支的输出通过一个可学习的逐位置门控进行组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win`是来自查询的小型MLP的门控权重。它们不必和为1——可以独立地对各分支进行加权。

### 为何这是“原生可训练的”

选择步骤（top-k块）是离散的。离散操作会阻断梯度流动。先前的稀疏注意力工作要么跳过选择过程的反向传播（限制了训练），要么使用在推理时无法实现真正稀疏性的连续松弛方法。

NSA规避了这个问题：压缩分支的注意力本身就是一种在全局序列上的可微粗粒度注意力。top-k操作仅仅是复用压缩分支中最高的注意力分数，来选择加载哪些细粒度块。梯度流经压缩分支的分数（它同时影响压缩输出和选择逻辑），并且所选块对最终输出的贡献也是可微的。不可微的`top_k`操作在前向计算图上是一个空操作——它仅控制从内存中加载哪些块。

这就是NSA能够在预训练中端到端使用的原因。模型学习联合地通过三个分支路由信息，产生的稀疏模式在推理时能真正实现所承诺的加速。

### 硬件对齐的内核

NSA的内核为现代GPU内存层次结构而设计。内核按GQA组（外循环）加载查询，为每个组获取相应的稀疏KV块（内循环），并在SRAM上运行注意力。因为每个查询组看到的是相同的选择块（选择是按查询组进行的，而非按查询头），所以KV加载在组内被分摊。算术强度保持较高水平。

论文报告称，Triton内核在64k解码上比FlashAttention快9倍，且加速比随序列长度增长而增加。提供了前向和反向内核。

### 计算预算

设`N`为序列长度，`l`为压缩块大小，`k`为top-k选择数量，`w`为滑动窗口大小，`b`为选择块大小（通常等于`l`）。

- 压缩分支：每个查询`O(N/l)`个键，总计`O(N * N / l)`。
- 选择分支：每个查询`O(k * b)`个键，总计`O(N * k * b)`。
- 滑动分支：每个查询`O(w)`个键，总计`O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当`N = 64k, l = 64, k = 16, b = 64, w = 512`时：每个查询的代价是`1000 + 1024 + 512 = 2536 keys`。全注意力是`64000 keys`。计算减少25倍。

当`N = 128k, l = 64, k = 16, b = 64, w = 512`时：每个查询的代价是`2000 + 1024 + 512 = 3536 keys`。全注意力是`128000 keys`。减少36倍。收益随序列长度增长而增长，这正是其核心价值所在。

### 如何比较

| 方法 | 可微性 | 实际推理加速 | 长程召回能力 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 跨步/块稀疏 | 是 | 是 | 部分 |
| KV剪枝 (H2O, StreamingLLM) | 不适用（推理时） | 是 | 部分 |
| MoBA (Moonshot) | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k时9倍） | 匹配全注意力 |

MoBA (Moonshot, arXiv:2502.13189) 同期发表，采用了类似的“三合一”方法，将MoE原理应用于注意力块。NSA和MoBA是2026年长上下文预训练需要了解的两种架构。

## 动手实现

`code/main.py` 在一个短合成序列上实现了三个分支，并展示了：

- 压缩MLP（为教学清晰使用了简单的平均池化基线；实际NSA使用可学习的MLP）。
- 由压缩分支分数驱动的top-k块选择。
- 在最后`w`个token上的滑动窗口注意力。
- 门控组合。
- 与全注意力相比的计算量打印输出。

### 步骤1：将token压缩成块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤2：压缩分支注意力

对查询与压缩键运行softmax注意力。压缩分支的分数同时作为top-k选择的信号。

### 步骤3：top-k块选择

选取得分最高的`k`个压缩块的索引。从这些块中加载原始的未压缩token，并在其上运行注意力。

### 步骤4：滑动窗口注意力

取最后`w`个token，并对其运行标准注意力。

### 步骤5：门控 + 组合

一个小型MLP基于查询产生三个门控权重。最终输出是三个分支输出的加权和。

### 步骤6：计算量统计

打印每个分支及总计的每个查询所关注的键数量。与`N`（全注意力）进行比较。在一个1024 token的合成序列上，使用`l = 32, k = 4, w = 128`，NSA每个查询处理`32 + 128 + 128 = 288`个键，而全注意力需要1024个——减少了3.5倍。

## 使用场景

NSA已用于DeepSeek自身的长上下文预训练流程。截至2026年4月，公开推理栈中的集成状态如下：

- **DeepSeek内部**：原生使用，发布的权重采用NSA或其后继者DSA（DeepSeek稀疏注意力）。
- **vLLM**：针对DeepSeek-V3.x权重的实验性NSA支持正在开发中。
- **SGLang**：已发布NSA基准；生产路径追随vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在CPU吞吐量下不划算。

何时使用NSA：

- 针对64k及以上上下文，且有严肃计算预算的预训练或继续训练运行。
- 推理DeepSeek自身的长上下文检查点。这些权重是原生NSA的。

何时不使用：

- 服务现有的密集注意力预训练模型。不经过继续训练无法改装NSA。
- 上下文低于16k。三路分支的开销超过了节省的计算量。
- 单请求交互式聊天。延迟敏感的解码会受益，但仅在长上下文中。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 压缩分支 | “粗粒度视图” | 在块平均键上的注意力，以每个查询O(N/l)个键提供全局上下文 |
| 选择分支 | “top-k块” | 在得分最高的`k`个块上的细粒度注意力 |
| 滑动窗口 | “局部上下文” | 在最后`W`个token上的注意力，用于短程模式 |
| 原生可训练性 | “带着稀疏性预训练” | 稀疏模式在预训练期间学习，而非推理时附加 |
| 压缩块大小 l | “粗粒度视图的分组大小” | 多少个token合并为一个摘要；通常为32-64 |
| Top-k | “要保留的块数” | 其未压缩token会被读取的压缩块数量；通常为16 |
| 滑动窗口 W | “局部注意力半径” | 通常为512；更短会损害局部连贯性，更长则浪费计算 |
| 分支门控 | “如何混合三路” | 逐位置的MLP输出，用于加权三个分支的贡献 |
| 硬件对齐 | “内核友好的稀疏性” | 选择的稀疏模式使得实际GPU内核能实现理论加速 |
| DSA | “NSA的后继者” | DeepSeek稀疏注意力，是DeepSeek谱系中接续NSA的架构 |
