# DualPipe 并行

> DeepSeek-V3 在 2,048 块 H800 GPU 上进行训练，MoE 专家分布在各个节点之间。跨节点的专家全连接（all-to-all）通信，每消耗 1 GPU-小时的计算就会产生 1 GPU-小时的通信开销。GPU 有一半时间处于空闲状态。DualPipe（DeepSeek，2024年12月）是一种双向流水线，通过将前向和反向计算与它们触发的全连接通信重叠来工作。气泡减少，吞吐量提升，并且在专家并行已经将专家分布到各个等级的前提下，保留两份模型参数副本（这就是其名称中“Dual”的由来）的开销并不大。本课程是一次学习型演练，讲解 DualPipe 实际上做了什么，以及为什么海人工智能实验室的 DualPipeV 改进方案能以略微增加气泡的代价来消除 2 倍的参数成本。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**先修知识：** 第10阶段 · 05（分布式训练，FSDP，DeepSpeed），第10阶段 · 14（开源模型架构与MoE）
**时间：** 约60分钟

## 学习目标

- 列举 DualPipe 前向-反向分块的四个组成部分，并说明每个部分为何需要其独立的重叠窗口。
- 解释大规模下的流水线气泡问题，以及在实践中“零气泡”的真正含义与营销说辞的区别。
- 手动追踪一个 8 个流水线并行（PP）等级、16 个微批次的 DualPipe 调度表，并确认前向和反向流如何填充彼此的空闲时段。
- 阐述 DualPipeV（海人工智能实验室，2025）所做的权衡：在专家并行未激活时，以稍微更大的气泡为代价，消除了 2 倍的参数复制。

## 问题所在

在 2k 块 H800 GPU 上训练一个 671B 的 MoE 模型，会遇到三个相互叠加的瓶颈：

1. **内存压力。** 每块 GPU 仅持有模型的一部分。在 8k 序列长度下，跨越 61 层、128 个注意力头产生的激活内存是巨大的。
2. **流水线气泡。** 传统的流水线并行（GPipe，1F1B）在 GPU 等待其阶段的输入或梯度时会使其空闲。在 8 个阶段下，即使使用 1F1B 调度，GPU 时间也有约 12% 可能被浪费为气泡。
3. **跨节点全连接通信。** 具有专家并行的 MoE 会将专家分散到各个节点。每次前向传播都会触发一次全连接通信来将 token 分发给其专家，以及另一次来合并结果。在 2k 块 GPU 的规模下，这很容易导致计算与通信比达到 1:1。

针对这些问题，分别有现成的解决方案：梯度检查点用于缓解内存压力，零气泡（海人工智能实验室，2023）用于解决流水线气泡，专家并行通信内核用于优化全连接通信。DualPipe 所做的是让它们协同工作。该调度在单个前向-反向分块内将计算与通信重叠，同时从流水线两端注入微批次，并利用生成的调度来将全连接通信隐藏在计算窗口内。

报告结果：在 DeepSeek-V3 的 14.8T-token 训练运行中，近乎消除了流水线气泡，GPU 利用率超过 95%。

## 核心概念

### 流水线并行复习

将一个 N 层模型拆分到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。一个微批次从设备 0 到 P-1 向前流动，然后从 P-1 到 0 向后流动。每个设备只有在接收到前一设备发来的输出后才能开始其前向阶段，也只有在接收到下游设备发来的上游梯度后才能开始反向阶段。

GPipe（Huang 等人，2019）一次只调度一个微批次，浪费了大量 GPU 时间。1F1B（Narayanan 等人，2021）为多个微批次交错安排前向和反向传播。零气泡（Qi 等人，2023）将反向传播拆分为两部分——输入反向（B）和权重反向（W）——并调度它们以填补气泡。在零气泡之后，流水线几乎完全紧凑。

DualPipe 是更进一步的方案。它增加了两个想法：

### 想法一：分块分解

每个前向分块被拆分为四个组成部分：

- **注意力。** Q/K/V 投影、注意力计算、输出投影。
- **全连接分发。** 跨节点通信，将 token 发送给其对应的专家。
- **MLP。** MoE 专家的计算。
- **全连接合并。** 跨节点通信，将专家输出收集回来。

一个反向分块也增加了每一部分的梯度版本。DualPipe 的调度使得全连接分发与下一个分块的注意力计算并行执行，而全连接合并则与下一个分块的 MLP 计算并行执行。

### 想法二：双向调度

大多数流水线调度从阶段 0 注入微批次，流向阶段 P-1。DualPipe 则从**两端**注入微批次。阶段 0 看到起源于此的前向微批次；阶段 P-1 也看到起源于此的前向微批次。两股流在中间相遇。

为实现这一点，设备 `i` 必须同时持有流水线前部层 `i` 和流水线后部层 `P - 1 - i`。这就是 DualPipe 中“双”（Dual）的含义：每个设备都保留两份其需要服务的模型层副本（一个用于每个方向）。在 DeepSeek-V3 的规模下，这意味着 2 倍的参数复制成本。这是可负担的，因为专家并行已经将 MoE 专家分散得足够稀疏，以至于将非专家层复制两次的开销微不足道。

至关重要的是，一个方向的前向流和另一个方向的反向流，恰好在单向调度会产生气泡的地方重叠。气泡消失了。

### 手动追踪调度

考虑 P = 4 个等级，8 个微批次，分为 4 个正向 / 4 个反向。时间从左向右推进；行表示设备等级。

```
           Time →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

解读“F4/F5R”记法：等级 1 在同一个时间槽内，既在运行微批次 4 的前向（在流水线中从左到右），又在运行微批次 5 的前向（在流水线中从右到左）。这就是“双向”在操作层面的含义。

在等级 2 处，两股流更早地重叠；在等级 0 和 P-1 处，重叠最晚。在调度的稳定阶段中间，每个等级都运行 X 方向的前向与 Y 方向的反向重叠。计算处于忙碌状态。前向传播的全连接分发隐藏在反向计算内部。全连接合并隐藏在前向计算内部。气泡被挤压掉了。

### 气泡核算

标准 1F1B 流水线气泡（每个等级浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

零气泡改进将其降低，但未降至零。DualPipe 在稳定阶段，如果微批次总数可被 2 倍流水线深度整除，则气泡为零。在稳定阶段之外（预热和冷却阶段），仍存在一些气泡，但该气泡不会随微批次数量增加而增长——这是论文强调的一个关键特性。

用营销术语来说：“零气泡”。用技术术语来说：气泡不会随微批次数量增长。海人工智能实验室的后续分析（DualPipeV / Cut-in-half）表明，只有当专家并行不是瓶颈时才能实现完全的零气泡；在由专家并行驱动全连接通信的情况下，总会存在一些调度上的妥协。

### DualPipeV — 改进方案

海人工智能实验室（2025）观察到，当与专家并行通信重叠不是重点时，2 倍的参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠成一个“V 形”调度，在单一的参数副本上运行。其气泡比 DualPipe 略大，但内存节省相当可观。DeepSeek 在其开源 DualPipe 实现中采用了 DualPipeV 作为关闭专家并行的模式。

权衡比较：

| 特性 | DualPipe | DualPipeV | 1F1B | 零气泡 |
|---------|---------|-----------|------|------------|
| 每设备参数副本数 | 2 | 1 | 1 | 1 |
| 气泡与微批次的关系 | 恒定 | 缓慢增长 | 增长 | 增长 |
| 计算-通信重叠 | 完全 | 部分 | 极小 | 部分 |
| 使用场景 | 专家并行为主的 MoE | 密集模型或专家并行较轻 | 基线 | 任意流水线 |

### 对 14.8T-token 运行意味着什么

DeepSeek-V3 的预训练在 2,048 块 H800 GPU 上消耗了 14.8T 个 token，大约花费了 2.8M GPU-小时。如果使用朴素的 1F1B，他们将因流水线气泡损失其中 12-15% 的时间——34-42万 GPU-小时，足够训练一个完整的 70B 模型。DualPipe 恢复了其中的大部分。在没有内部日志的情况下，很难直接量化其贡献，但论文中声称训练期间平均 GPU 利用率超过 95%。

对于较小的运行规模（少于 1k 块 GPU），DualPipe 是过度设计的——流水线气泡相对于总成本占比较小，且密集模型训练很少遇到全连接通信瓶颈。对于前沿的 MoE 模型在数千块 GPU 规模下的训练，它实际上是必需的。

### 在技术栈中的位置

- 与 **FSDP**（第10阶段 · 05）互补。FSDP 在各等级间分片模型参数；DualPipe 调度各等级间的计算。它们可以结合使用。
- 兼容 **ZeRO-3** 梯度分片。两份副本复制的簿记需要与 ZeRO 的分片梯度协调。
- 需要针对特定集群拓扑调优的**定制全连接通信内核**。DeepSeek 的开源内核是参考实现。

## 使用它

`code/main.py` 是一个流水线调度模拟器。它接收 `(P, n_micro_batches, schedule)` 并分别打印 1F1B、零气泡、DualPipe 和 DualPipeV 在稳定阶段的利用率。它是一个教学工具——这些数字与论文中的定性描述相符，并非关于生产环境实测加速比的断言。

该模拟器的价值在于：用不同的 P 和微批次数量运行它，观察气泡分数在 1F1B 下如何增长，而在 DualPipe 下则不然。

集成到实际训练运行中的注意事项：

- 选择能被微批次数量整除的流水线并行深度。
- 确保您的专家并行网格支持双向全连接通信。DeepSeek 的内核是参考实现。
- 预计首次调试调度本身会花费一周时间。簿记工作非常繁琐。
- 监控每个等级的 GPU 利用率，而非仅仅是总体利用率。DualPipe 的收益来自于消除掉队者。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 流水线气泡 | "每个等级的闲置时间" | 由于流水线阶段等待其输入或梯度而浪费的 GPU 周期 |
| 1F1B | "默认流水线调度" | 一次前向/一次反向交错调度；DualPipe 要超越的基线 |
| 零气泡 | "海人工智能实验室 2023" | 将反向传播拆分为 B（输入梯度）和 W（权重梯度）；几乎完全收紧流水线 |
| DualPipe | "DeepSeek-V3 调度" | 双向流水线 + 计算-通信重叠；气泡不随微批次数量增长 |
| DualPipeV | "减半"（Cut-in-half） | V 形改进方案，以略微更大的气泡为代价，消除了 2 倍的参数复制 |
| 分块 | "流水线工作的单元" | 一个微批次在一个流水线阶段的一次前向或反向传播 |
| 全连接分发 | "将 token 发送给专家" | 将 token 路由到其分配的 MoE 专家的跨节点通信 |
| 全连接合并 | "将专家输出收集回来" | MLP 后收集专家输出的跨节点通信 |
| 专家并行 | "专家分布在 GPU 上" | 将 MoE 专家在各等级间分片，使得不同 GPU 持有不同的专家 |
| 流水线并行 | "层分布在 GPU 上" | 将模型层在各等级间分片；DualPipe 所调度的维度 |
| 气泡分数 | "浪费的 GPU 时间" | (气泡时间 / 总时间)；DualPipe 试图驱动至接近零的比例 |
