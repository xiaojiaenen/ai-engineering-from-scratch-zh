# 任意分辨率视觉：Patch-n'-Pack与NaFlex

> 真实图像并非224x224的正方形。收据是9:16，图表是16:9，医学影像可能是4096x4096，手机截图是9:19.5。2024年前视觉语言模型的解决方案——将所有图像调整为固定正方形——丢失了OCR、文档理解和高分辨率场景解析所需的有效信息。NaViT（Google，2023年）证明可以通过分块对角掩码将可变分辨率的图像块打包进单个Transformer批次中。Qwen2-VL的M-RoPE（2024年）完全抛弃了绝对位置编码表。LLaVA-NeXT的AnyRes将高分辨率图像分割为基础图像和子图像。SigLIP 2的NaFlex变体（2025年）现已成为开源视觉语言模型的默认编码器，它能使用单个模型检查点服务各种宽高比需求。本课程将端到端实现patch-n'-pack技术。

**类型：** 构建
**语言：** Python（标准库、patch打包器 + 分块对角掩码）
**前置课程：** 阶段12 · 01（ViT图像块）、阶段12 · 05（LLaVA）
**时长：** 约120分钟

## 学习目标

- 将一批可变分辨率图像的图像块打包成一个序列，并构建分块对角注意力掩码。
- 为特定任务选择AnyRes分块（LLaVA-NeXT）、NaFlex（SigLIP 2）或M-RoPE（Qwen2-VL）方案。
- 在不进行缩放的情况下，为OCR、图表和摄影计算token预算。
- 指出方形缩放的三种失败模式：文字变形、内容被裁剪、填充token浪费计算资源。

## 问题所在

Transformer期望处理一个序列。批次是多个等长序列的堆叠。如果图像都是224x224，每次都会得到196个图像块token，无需填充，任务完成。在224上训练，在224上推理，无需再考虑分辨率问题。

但现实世界并非如此。文档是纵向的（8.5x11英寸，约2:3）。图表截图是横向的（16:9）。收据是细长的（1:3）。医学影像尺寸为2048x2048或更大。移动设备截图是1170x2532（0.46:1）。

2024年前的三种方案及其各自失败的原因：

1.  **缩放至固定正方形（224x224或336x336）**：拉伸变形会扭曲文字和人脸。降低分辨率会破坏图表标签和OCR内容。这是LLaVA-1.5之前的通用做法。
2.  **裁剪为固定宽高比**：你会丢弃大部分图像内容，而选择裁剪位置本身就是一个计算机视觉问题。
3.  **填充至最长边**：解决了变形问题，但对纵向图像而言，50%以上的token被浪费在填充上。所有填充token都会带来二次方的注意力计算开销。

2024-2025年的解决方案：让Transformer在图像原始分辨率下处理图像块，并想办法将异构批次打包进一个序列而不浪费计算资源。

## 核心概念

### NaViT与patch-n'-pack

NaViT（Dehghani等人，2023年）是首个证明此方法大规模有效的论文。其思路很直接：

1.  对于批次中的每张图像，根据选定的图像块大小（例如14），计算其原始的图像块网格。
2.  将每张图像的图像块展平成其独有的变长序列。
3.  将批次中所有图像的图像块连接成一个长序列。
4.  构建一个分块对角注意力掩码，使得图像A的图像块只能关注图像A内部的图像块。
5.  携带每个图像块的位置信息（2D RoPE或分数位置编码）。

一个包含三张图像（336x336 -> 576个token，224x224 -> 256个token，448x336 -> 768个token）的批次，会变成一个1600个token的序列，并附带一个1600x1600的分块对角掩码。无填充，无计算浪费。Transformer能处理任意宽高比。

NaViT还引入了训练时的分数图像块丢弃——在批次中随机丢弃50%的图像块——这既起到了正则化作用，也加速了训练。SigLIP 2继承了这一特性。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT的AnyRes是一种务实的替代方案。给定一张高分辨率图像和一个固定编码器（如336分辨率的CLIP或SigLIP），对图像进行分块：

1.  从预定义的布局集合中——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2)等——选择一个最匹配图像宽高比的网格布局。
2.  将完整图像分割成网格；每个图块成为一个336x336的裁剪区域。
3.  同时生成一个缩略图：将整张图像缩放至336x336作为全局上下文token。
4.  将每个图块通过冻结的336编码器进行编码。将图块token和缩略图token连接起来。

对于一张672x672的图像，采用2x2网格加缩略图：4 * 576 + 576 = 2880个视觉token。计算开销大但效果显著——大语言模型同时看到了局部细节和全局上下文。

当你的编码器被冻结且仅支持单一分辨率时，AnyRes是首选方案。对于大图像，其token数量会爆炸式增长（一张1344x1344图像，4x4网格会产生9216 + 576 ≈ 9800个token，这几乎填满了一个8k上下文的大语言模型）。

### M-RoPE（Qwen2-VL）

Qwen2-VL引入了多模态旋转位置编码。与NaViT的分数位置或AnyRes的图块+缩略图方案不同，每个图像块携带一个3D位置（时间、高度、宽度）。查询/键的旋转处理任意的H、W和时间长度。

M-RoPE在不进行重训练的情况下原生支持动态分辨率。在推理时，你输入任意HxW的图像，图像块嵌入器产生H/14 x W/14个token，每个token获得其(t=0, r=行, c=列)位置，RoPE以正确的频率旋转注意力，完成。Qwen2.5-VL和Qwen3-VL延续了这一思路。InternVL3的V2PE是相同的理念，但对不同模态采用可变的编码。

与AnyRes不同，M-RoPE在原始分辨率下是O(H x W / P^2)个token——没有乘法级的图块开销。与NaViT不同，它仍然要求每次前向传播只处理单张图像。跨分辨率的批次打包仍然需要在其之上使用patch-n'-pack技术。

### NaFlex（SigLIP 2）

NaFlex是SigLIP 2模型检查点的原生灵活模式。单个模型在推理时服务多种序列长度（256、729、1024个token）。训练内部使用NaViT风格的patch-n'-pack，每个图像块使用绝对的分数位置。其卖点是：一个检查点，根据任务在推理时选择你的token预算。

对于语义任务（分类、检索），使用256个token。对于OCR或图表理解，使用1024个token。无需重训练。

### 打包掩码

分块对角掩码是大多数实现容易出错的地方。对于一个总长度为`N_total`、涵盖图像`i=0..B-1`且各图像序列长度为`n_i`的打包序列，形状为`(N_total, N_total)`的掩码`M`在如果两个索引都落在同一图像的块内时为1，否则为0。你可以通过一个累积长度列表来构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 iff there exists b where offsets[b] <= i < offsets[b+1] and offsets[b] <= j < offsets[b+1]
```

在PyTorch中，使用`torch.block_diag`或显式的gather操作，这可以是一行代码。FlashAttention的变长路径（`cu_seqlens`）完全跳过掩码，直接使用累积长度张量在序列内进行注意力计算——对于典型批次，这比密集掩码快约10倍。

### Token预算

根据任务选择策略：

-   **OCR / 文档**：1024-4096个token。使用1024分辨率的SigLIP 2 NaFlex，或3x3网格加缩略图的AnyRes。
-   **图表与UI**：在384-448原始分辨率下，729-1024个token。使用最大像素数限制的Qwen2.5-VL动态分辨率。
-   **自然照片**：256-576个token通常足够。下游大语言模型能看到足够信息。在内容密度高的地方投入更多token。
-   **视频**：空间池化后每帧64-128个token，2-8 FPS。课程12.17将涵盖此内容。

2026年的生产实践规则：为每个任务选择最大像素数限制，在该限制内按原始宽高比编码，打包批次，跳过填充。Qwen2.5-VL暴露了`min_pixels`和`max_pixels`参数，正是用于此调节。

## 动手实践

`code/main.py`为具有整数像素坐标的异构图像批次实现了patch-n'-pack。它：

-   接受一个(H, W)图像尺寸列表。
-   计算每个图像在图像块大小14下的序列长度。
-   将它们打包成一个总长度为`sum(n_i)`的序列。
-   构建分块对角注意力掩码（密集形式，便于理解）。
-   比较打包方案与方形缩放、AnyRes分块方案的计算成本。
-   为混合批次（收据、图表、截图、照片）打印token预算表。

运行它。输出的数字解释了为何2026年所有开源视觉语言模型都采用patch-n'-pack。

## 部署应用

本课程产出`outputs/skill-resolution-budget-planner.md`。给定一个混合宽高比的工作负载（OCR、图表、照片、视频帧）和一个总token预算，它会选择正确的策略（NaFlex、AnyRes、M-RoPE或固定正方形）并为每个请求生成配置。当你在为产品选择视觉语言模型规格时，请使用此技能——它可以防止那种无声的10倍token膨胀，这种膨胀会拖垮延迟预算。

## 练习

1.  一张收据图像尺寸为600x1500（1:2.5）。在图像块大小为14时，原始分辨率下有多少个token？缩放到336正方形后有多少？在实际应用中，哪种方式对OCR精度损失更大？

2.  为长度分别为256、576、729、1024的四张图像构建分块对角掩码。验证注意力矩阵大小为2585x2585，并且恰好有`256^2 + 576^2 + 729^2 + 1024^2`个非零元素。

3.  对于一张1792x896、图像块大小为14的图像，比较：(a) 缩放至336正方形后编码，(b) AnyRes 2x1 + 缩略图，(c) M-RoPE原始分辨率编码。哪种使用的token最少？哪种保留的细节最多？

4.  实现分数图像块丢弃：给定一个打包序列，均匀随机地丢弃50%的token，并相应地更新分块对角掩码。测量掩码稀疏度的变化。

5.  阅读Qwen2-VL论文的3.2节（arXiv:2409.12191）。用两句话描述`min_pixels`和`max_pixels`分别控制什么，以及为什么两者都需要限制。

## 关键术语

| 术语 | 人们常说 | 其实际含义 |
|------|----------|------------|
| Patch-n'-pack | "NaViT风格打包" | 将来自不同图像的变长图像块序列连接到一个批次维度中 |
| 分块对角掩码 | "打包掩码" | 一种注意力掩码，将每个图像的图像块限制为仅关注自身，而非批次中的邻居 |
| AnyRes | "LLaVA-NeXT分块" | 将高分辨率图像分割成固定大小图块的网格，外加一个全局缩略图；使用固定编码器编码每个图块 |
| NaFlex | "SigLIP 2原生灵活模式" | 单个SigLIP 2检查点，在推理时无需重训练即可服务256/729/1024个token的预算 |
| M-RoPE | "多模态RoPE" | 3D旋转位置编码（时间、行、列），无需位置表即可处理任意的H、W、T |
| cu_seqlens | "FlashAttention打包" | FlashAttention变长路径使用的累积长度张量，替代了密集的分块对角掩码 |
| min_pixels / max_pixels | "分辨率上下限" | Qwen2.5-VL的每请求参数，用于限制极小或极大输入的token数量 |
| 视觉token预算 | "每张图像多少token" | 每张图像产生的图像块token的大致数量；决定了大语言模型的提示词预算和注意力计算成本 |
