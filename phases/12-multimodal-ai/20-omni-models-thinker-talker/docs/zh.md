# 全模态模型：Qwen2.5-Omni 与思考者-说话者分离架构

> GPT-4o 在2024年5月的产品演示之所以具有颠覆性，并非源于底层模型，而是因其产品形态——一个语音界面：你说话，模型能看到摄像头画面，并能在250毫秒内回话。2024至2025年间，开源生态系统一直在竞相复现这一产品体验。Qwen2.5-Omni（2025年3月）是开源领域的参考设计：一个思考者（大型文本生成Transformer）加上一个说话者（并行的语音生成Transformer），通过流式语音词元连接。Mini-Omni 对其进行了简化，Moshi 匹配了其延迟，GLM-4-Voice 则将其扩展至中文。本课将解读思考者-说话者架构，以及使流式实时对话得以实现的延迟预算。

**类型:** 构建
**语言:** Python（标准库、流式管道延迟模拟器 + VAD循环）
**先修课程:** 阶段12 · 19（音频-LLM），阶段12 · 16（任意到任意）
**时间:** ~180分钟

## 学习目标

- 将推理管道拆分为思考者（文本推理）和说话者（语音合成），并解释并行流式处理为何有效。
- 逐一计算对话交互中“首音频字节时间”的预算。
- 描述TMRoPE如何在思考者内部，跨视觉、音频和文本实现时间对齐的位置编码。
- 说出三种实时对话模式：半双工、轮次、全双工。

## 问题

实时语音助手需要快速完成多项任务：

1.  听用户说话。实时语音词元化，通过语音活动检测（VAD）知晓用户何时说完。
2.  可选地看。以2-4 FPS输入的摄像头画面，与音频一起流式传入思考者。
3.  思考。基于对话历史，构思一个回复。
4.  说话。合成语音词元，解码为波形，并流式传输到用户的扬声器。

每一步都会增加延迟。对话感要求总往返延迟低于500毫秒——低于此值，用户便开始忽略延迟。GPT-4o声称约250毫秒。Moshi约160毫秒。Qwen2.5-Omni约350-500毫秒。

每个组件都需要流式处理。不能“先批量处理所有内容再解码”。

## 概念

### 思考者与说话者

Qwen2.5-Omni 的分解结构：

-   **思考者：** 一个7B-80B的文本生成Transformer。消费交错的文本、图像、音频词元。输出表示“该说什么”的文本词元。
-   **说话者：** 一个较小的语音生成Transformer（200M-1B）。消费思考者的文本输出词元以及近期的语音上下文词元。输出离散的语音词元（残差-VQ索引）。
-   **语音解码器：** 一个流式波形解码器（SNAC、MoVQGAN系列），实时将语音词元转换为音频样本。

这种分离很重要。思考者必须足够大才能有良好的推理能力。说话者可以很小，因为其任务是局部性的——将文本转换为语音词元。更大的说话者并不会更具表现力；只会更慢。

两者并行运行：

1.  思考者发出文本词元 t_i。
2.  说话者（通过流式）消费 t_i，并发出语音词元 s_i, s_{i+1}, ..., s_{i+k}。
3.  语音解码器消费到达的语音词元，并发出音频样本。
4.  当思考者进行到文本词元 t_{i+3} 时，说话者已经为 t_0..t_{i+2} 流式输出了音频。

### TMRoPE — 时间对齐的多模态位置编码

思考者需要整合图像帧（假设以4 FPS到达）、音频帧（以50帧/秒到达）以及来自对话历史的文本。朴素的序列顺序（所有图像，然后所有音频，然后文本）会丢失时间对齐性。

TMRoPE为每个词元分配绝对时间戳。视觉词元在t=2.3秒。音频词元在t=2.32秒。用户文本词元“停”在t=2.35秒。RoPE根据时间戳旋转注意力；模型将它们视为时间上并发的。

这是实现“他在说你好时挥手”的基础设施——模型在概念上同一时刻看到视频帧和音频。

### 流式语音合成

语音词元必须流式处理。Mini-Omni（Xie & Wu, 2024）提出了“语言模型可以边听边思考边流式说话”：思考者输出词元和说话者输出词元在同一个序列中交错。思考者一旦提交下一个文本词元，说话者就立即触发。没有批处理边界。

Moshi（Défossez 等人，2024年10月）是最快的开源实现。在单个A100上实现160毫秒的TTFAB。架构：单个7B Transformer在交替位置发出文本和语音词元，并有一个“内心独白”将思考流与说话流分开。这实际上是经过精心训练后，将思考者和说话者融合进一个模型。

### VAD 和轮次管理

语音活动检测在输入端运行。两种模式：

-   **半双工：** 用户说话，模型倾听。模型说话，用户倾听。通过VAD静默检测（~200毫秒）进行清晰的交接。
-   **全双工：** 双方可以同时说话。模型可以附和（“嗯哼”）或打断。难度大得多。Moshi支持此模式。

Qwen2.5-Omni默认支持半双工，通过静默阈值实现轮次管理。全双工需要应用层处理。

### Qwen3-Omni（2025年11月）

后继者。Qwen3-80B思考者，更大的说话者，改进的TMRoPE-v2。延迟接近GPT-4o的250毫秒。开放权重。在OmniBench上的基准测试与Gemini 2.0 Live竞争力相当。

### 生产环境延迟预算

对于典型的流式交互：

-   麦克风 -> 音频词元：40-80毫秒。
-   预填充（提示+历史记录）：7B模型100-200毫秒，70B模型更长。
-   首个思考者文本词元：40毫秒。
-   说话者处理首个文本词元：20毫秒。
-   首批语音词元提交：40毫秒。
-   残差-VQ解码：30毫秒。
-   语音波形解码：50-80毫秒。

总TTFAB：7B模型为320-510毫秒，70B模型为600-900毫秒。前沿质量通常意味着70B+；因此存在前沿延迟差距。

### 词元速率计算

对于16kHz语音，基础语音词元速率为50Hz，即每秒输出需要50个语音词元。说话者必须发射≥50 tok/s才能跟上。在H100上典型的LLM吞吐量为30-80 tok/s，一个小型（200-300M）说话者足够快；而一个7B的说话者会跟不上。

这就是为什么存在小型专用说话者模型，而不是“直接用主模型”。

## 练习

1.  你的目标TTFAB是300毫秒。对于一个7B思考者和300M说话者，写出每个组件的延迟。
2.  Qwen2.5-Omni使用TMRoPE。描述对于这样一个提示——用户在t=1秒开始说话，摄像头在t=1.2秒捕捉到一个手势——模型看到了什么。
3.  全双工支持要求模型在倾听的同时发出音频。提出一种能教会模型这一点的训练数据格式。
4.  阅读Moshi论文第4节。描述“内心独白”分离以及它如何避免了思考者-说话者的拆分。
5.  计算吞吐量预算：要跟上16kHz语音、50基础层词元/秒，说话者必须以多快的速度发射词元？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 思考者 | “推理大脑” | 产生“该说什么”的大型文本生成Transformer |
| 说话者 | “发声的嘴” | 从思考者的文本产生离散语音词元的小型Transformer |
| TTFAB | “延迟预算” | 首音频字节时间：从用户语音结束到第一个音频样本输出 |
| TMRoPE | “时间对齐RoPE” | 使用绝对时间戳跨视觉、音频、文本的位置编码 |
| 半双工 | “轮次交谈” | 用户和模型交替发言；VAD静默检测用户说完 |
| 全双工 | “同时进行” | 模型可以同时说话和倾听；支持附和 |
| 内心独白 | “Moshi分离” | 单模型设计，其中思考流和说话流交错 |
