# Janus-Pro：统一多模态模型的解耦编码器

> 统一多模态模型存在一个不可避免的矛盾：理解任务需要语义特征——富含概念级信息的SigLIP或DINOv2输出向量；生成任务需要利于重建的编码——能组合成清晰像素的VQ标记。这两个目标无法在单一编码器中兼容。Janus（深度求索，2024年10月）和Janus-Pro（深度求索，2025年1月）提出解决方案是放弃尝试：解耦两个编码器。在任务间共享Transformer主体，但通过SigLIP处理理解任务，通过VQ分词器处理生成任务。在70亿参数规模下，Janus-Pro在GenEval基准上超越DALL-E 3，同时在MMMU基准上匹配LLaVA。本节将解析为什么两个编码器能成功而单一编码器会失败。

**类型：** 构建  
**语言：** Python（标准库，双编码器路由+共享主体信号）  
**前置知识：** 第12·13节（Transfusion），第12·14节（Show-o）  
**时长：** 约120分钟

## 学习目标

- 解释为什么单一共享编码器会损害理解或生成质量。
- 描述Janus-Pro的路由机制：理解任务使用SigLIP特征作为输入，生成任务在输入和输出端均使用VQ标记。
- 追踪使Janus-Pro在Janus失败处成功的数据混合扩展策略。
- 比较解耦（Janus-Pro）、耦合连续（Transfusion）和耦合离散（Show-o）架构。

## 问题所在

统一模型在理解与生成任务间共享Transformer主体。先前尝试（Chameleon、Show-o、Transfusion）均使用单一视觉分词器处理两个方向。分词器是一种折衷：

- 为重建优化（生成）：VQ-VAE捕捉精细像素细节，但生成的标记语义连贯性较弱。
- 为语义优化（理解）：SigLIP嵌入将"猫"图像聚集在"猫"标记附近，但不利于高质量重建。

Show-o和Transfusion为此付出了明显质量代价，某一方向表现受限。Janus-Pro提出质疑：当任务需求不同时，为何必须使用单一标记器？

## 核心概念

### 解耦视觉编码

Janus-Pro架构分离两个编码器：

- **理解路径**：输入图像 → SigLIP-SO400m → 2层MLP → Transformer主体。
- **生成路径**：输入图像（若基于现有图像条件生成）→ VQ分词器 → 标记ID → Transformer主体。
- **输出生成**：Transformer预测的图像标记 → VQ解码器 → 像素。

Transformer主体被共享。主体上游和下游的所有组件均按任务定制。

通过提示格式区分输入：`<understand>`标签路由至SigLIP；`<generate>`路由至VQ。路由也可由任务隐式决定。

### 为何有效

- 理解损失获得SigLIP特征，其CLIP式预训练已针对语义相似性优化。模型感知基准测试表现优于Show-o/Transfusion，因为输入特征更适合该任务。
- 生成损失获得VQ标记，分词器已针对重建优化。图像质量优于Show-o，因为VQ编码能清晰地重构为像素。

共享Transformer主体接触两种输入分布（SigLIP和VQ）并学习处理两者。其主张：足够数据+足够参数，主体能适应切换。

### 数据扩展——Janus与Janus-Pro对比

Janus（原版，arXiv 2410.13848）引入解耦但规模较小（13亿参数，数据有限）。Janus-Pro（arXiv 2501.17811）进行了扩展：

- 70亿参数（对比13亿）。
- 第一阶段（对齐）使用9000万图像-文本对（原7200万）。
- 第二阶段（统一）使用7200万（原2600万）。
- 第三阶段新增20万图像生成指令样本。

结果：Janus-Pro-7B在MMMU基准上匹配LLaVA（60.3 vs ~58），在GenEval基准上击败DALL-E 3（0.80 vs 0.67）。单一开源模型在统一谱系的两侧均具竞争力。

### JanusFlow——整流流变体

JanusFlow（arXiv 2411.07975）将VQ生成路径替换为整流流生成路径（连续型）。分裂变为：理解用SigLIP + 生成用整流流。质量上限进一步提升。架构保持解耦编码器-共享主体。

### 共享主体的职责

Transformer主体处理统一序列，但接触两种输入分布。其职责是：

- 对于理解：消费SigLIP特征+文本标记 → 自回归输出文本。
- 对于生成：消费文本标记+（可选图像VQ标记）→ 自回归输出图像VQ标记。

主体各模块无模态特定权重。它是Qwen或Llama内部典型的文本式Transformer，外加两个输入适配器。

有趣的是，这意味着Janus-Pro的主体可从预训练LLM初始化。Janus-Pro确实从DeepSeek-MoE-7B初始化。这一选择很重要：LLM贡献的推理能力是纯粹从头开始的统一模型难以达到的。

### 与InternVL-U对比

InternVL-U（第12.10节）是2026年的后续工作。它结合了：

- 原生多模态预训练（InternVL3骨干）。
- 解耦编码器路由（输入用SigLIP，输出用VQ+扩散头）。
- 统一理解+生成+编辑。

InternVL-U将Janus-Pro的架构选择整合进更大框架。解耦编码器理念现已成为大规模统一模型的默认选择。

### 局限性

解耦编码器增加架构复杂性。需训练两个标记器，维护两条输入路径，处理两套失败模式。对于无需生成的产品，Janus-Pro过度设计——应选择LLaVA家族的理解模型。

对于无需理解的产品，Janus-Pro性能过剩——应选择Stable Diffusion 3/Flux模型。

对于需要两者的产品，Janus-Pro现为参考开源架构。

## 关键术语

| 术语 | 人们如何说 | 实际含义 |
|------|------------|----------|
| 解耦编码 | "两个视觉编码器" | 每个方向使用独立分词器或编码器：理解用语义型，生成用重建型 |
| 共享主体 | "一个Transformer" | 单一Transformer处理任一编码器的输出；无模态特定权重 |
| SigLIP用于理解 | "语义特征" | CLIP家族视觉塔，提供丰富概念特征但重建能力差 |
| VQ用于生成 | "重建编码" | 向量量化标记，能清晰解码回像素 |
| JanusFlow | "整流流变体" | Janus-Pro的变体，用连续流匹配生成头替代VQ |
| 路由标签 | "任务标签" | 提示标记（`<understand>`/`<generate>`），用于选择输入编码器 |
