# MIO 与任意模态到任意模态的流式多模态模型

> GPT-4o 发布了一个大多数开源模型无法复制的产品：一个能实时听声音、看视频并语音回应的智能体。开源生态在 2024 年底给出的答案是 MIO (Wang et al., September 2024)。MIO 对文本、图像、语音和音乐进行分词，在交错序列上训练一个因果 Transformer，并能实现任意模态到任意模态的生成。AnyGPT (Zhan et al., February 2024) 是概念验证；MIO 是规模化版本；Unified-IO 2 (Allen AI, December 2023) 则是增加了视觉和动作对接的表亲。本课程解析任意模态到任意模态的模式——四个分词器、一个 Transformer、流式友好的解码。

**类型：** 学习
**语言：** Python (标准库，四模态分词分配器 + 流式解码循环)
**先修要求：** 阶段 12 · 11 (Chameleon)，阶段 6 (语音与音频)
**时间：** ~120 分钟

## 学习目标

- 设计一个共享词表，容纳文本、图像、语音和音乐的词元，且互不冲突。
- 比较 SEED-Tokenizer (用于图像) 和 SpeechTokenizer 残差 VQ (用于语音) 在压缩与重建权衡方面的表现。
- 阐述构建任意模态到任意模态生成的四阶段课程。
- 列举三种开源的任意模态到任意模态方案及其主要权衡：MIO、AnyGPT、Unified-IO 2。

## 问题所在

统一的多模态模型说起来容易，规模化构建却很难。2024 年之前的大多数“任意模态到任意模态”系统都是流水线式的：视觉模型 → 文本表示 → 语音模型 → 音频。每一次转换都会丢失信息、增加延迟并使训练复杂化。GPT-4o 的演示视频展示了一个具有亚秒级响应的单模型替代方案；开源系统落后了数月。

工程挑战：

- 必须为每种模态都存在分词器，以足够无损的方式进行压缩以实现重建，并生成 Transformer 能处理速率的词元。
- 单一词表必须为文本 (32k+)、图像 (16k+)、语音 (4k+)、音乐 (8k+) 分配空间。至少需要四万多个条目。
- 训练数据必须覆盖每种输入-输出组合 (文本→图像，图像→语音，语音→图像等)，或者模型必须具备组合能力。
- 推理必须足够快地流式输出词元，以满足对话延迟要求 (<500ms 从输入到第一个音频字节的时间)。

## 核心概念

### 面向四种模态的四个分词器

MIO 的分词器栈：

- 文本：标准 BPE，词表大小 ~32000。
- 图像：SEED-Tokenizer (2023) —— 量化 VAE，使用离散码本，4096 个条目，每张图像 32x32 个词元。
- 语音：SpeechTokenizer 残差 VQ (2023) —— 将 16kHz 波形编码到 8 个分层码本中；第一层是粗糙内容，后续层添加韵律和说话人身份信息。
- 音乐：类似的残差 VQ (Meta 的 MusicGen / Encodec 家族)，4-8 个码本。

每种模态产生整数词元。这些词元在共享词表中获得互不重叠的 ID 范围：

```
text:   0..31999
image:  32000..36095  (4096 image tokens)
speech: 36096..40191  (4096 speech base tokens, plus residual layers)
music:  40192..48383  (8192 music tokens)
sep:    48384..48390  (<image>, <speech>, <music>, </...>, etc.)
```

总计：~48k 词表。输入嵌入和输出投影覆盖所有这些词元。

### 流式解码

语音生成使用残差 VQ。Transformer 预测基础层（第 0 层）语音词元；一个并行解码的残差量化器预测后续层。每个第 0 层词元在 16kHz 下大约对应 50ms 的音频。

流式模式：

1. 用户对着麦克风说话；实时音频分词器每 50ms 发射一次语音词元。
2. MIO 在词元到达时消费它们（提示预填充 + 增量前向传播）。
3. 生成时，输出词元流式发出；一个并行的语音解码器以约 50-150ms 的延迟将其转换为音频样本。
4. 从输入到第一个音频字节的时间：在 MIO 论文中约为 300-500ms，接近 GPT-4o 的 ~250ms。

Mini-Omni (arXiv:2408.16725)、GLM-4-Voice (arXiv:2412.02612) 和 Moshi (arXiv:2410.00037) 是互补的流式语音 LLM 设计。Moshi 尤其在单 GPU 上实现了 160ms 的往返延迟。

### 四阶段课程

MIO 的训练课程：

1. 阶段 1 —— 对齐。大规模模态对语料库：文本-图像、文本-语音、文本-音乐。每对使用其自己的词表分段进行训练。训练共享词表。
2. 阶段 2 —— 交错。多模态交错文档（带图像和视频的博客、带转录的播客等）。训练跨模态上下文。
3. 阶段 3 —— 语音增强。额外的音频数据以提升语音质量，同时不丧失文本能力。
4. 阶段 4 —— SFT。跨模态指令微调：VQA、字幕生成、叙述、语音到语音对话。

缺少某个阶段会损害特定能力：跳过阶段 2，模型将失去跨模态上下文；跳过阶段 3，语音效果会很差。

### 视觉思维链

MIO 引入了视觉思维链：模型将中间图像词元作为推理步骤发出。对于“猫正在爬树吗？”这个问题，模型会：

1. 发出 `<image>` 词元，渲染场景（来自输入图像或草图）。
2. 发出文本分析草图。
3. 发出最终答案。

渲染出的中间图像充当草稿纸。在空间推理任务上的基准测试有所提高。这个想法模仿了文本推理的思维链。

### 任意模态到任意模态的竞争对手

- AnyGPT (arXiv:2402.12226)：4 种模态（文本、图像、语音、音乐），设计相似。
- Unified-IO 2 (arXiv:2312.17172)：增加了视觉动作输出、深度图、法线图。任务多样性更高，规模更小。
- NExT-GPT (arXiv:2309.05519)：LLM + 特定模态扩散解码器。不是单模型方法。
- CoDi (arXiv:2305.11846)：可组合扩散；通过共享潜在空间实现任意模态到任意模态。

MIO 是最接近纯词元任意模态到任意模态的模型。AnyGPT 是其概念上的前身。

### 延迟预算

对于对话产品，每个组件的延迟都很重要：

- 麦克风到音频词元：~50ms。
- 预填充（音频词元 + 历史）：在 8B 模型上约 ~100ms。
- 第一个输出词元：~50ms。
- 并行残差 VQ + 语音解码器：~100-150ms。

总时间到第一个音频字节：最少约 ~300ms。GPT-4o 声称 ~250ms。Moshi 声称 160ms。MIO/AnyGPT 根据公开基准测试处于 400-600ms 范围内。

### 为什么任意模态到任意模态仍然困难

即使在 2026 年，开源的任意模态到任意模态模型在两个轴上仍落后于闭源模型：

- 语音质量。残差 VQ 分词器是有损的；与 ElevenLabs 级别的声音相比，对话式语音听起来很机械。
- 跨模态推理。要求模型“为你看到的东西唱歌”仍然比纯视觉任务更容易失败。

这些是开放的研究问题。Qwen3-Omni（课程 12.20）是 2025 年最先进的开源尝试。

## 使用它

`code/main.py`：

- 定义四模态词表分配并打印它。
- 将多模态输入列表（文本、图像、音频片段、音乐）路由到分词器路由器。
- 模拟文本到语音响应的流式解码，并计数延迟。
- 根据编码器、预填充和解码器延迟，计算预期的从输入到第一个音频字节的时间。

## 练习

1. 你的产品接受语音输入并返回语音输出。端到端的延迟预算目标是多少？列出耗费时间的组件。
2. SpeechTokenizer 残差 VQ 使用 8 个码本。解释为什么并行解码残差层是必要的（相对于顺序解码），以及它带来了多少延迟节省。
3. 你的词表有 32k 文本 + 4k 图像 + 4k 语音。添加 8k 音乐和 ~10 个分隔符。在隐藏维度为 4096 时，嵌入矩阵的参数成本是多少？
4. 视觉思维链发出中间图像。哪些类型的问题受益？哪些类型的问题会因为额外的词元而受损？
5. 阅读 Moshi (arXiv:2410.00037)。描述其“内心独白”技术，并与 MIO 的视觉思维链进行比较。

## 关键术语

| 术语 | 人们怎么说 | 它的实际含义 |
|------|-----------------|------------------------|
| 任意模态到任意模态 | "多模态输入输出" | 一个单一模型，能够接受并产生文本、图像、语音和音乐，且方向任意 |
| 残差 VQ | "语音分词器栈" | 多码本分词，每一层添加信息；基础层是内容，后续层是韵律 |
| SEED-Tokenizer | "图像编码" | MIO 使用的离散图像分词器，码本大小为 4096 |
| 视觉思维链 | "视觉草稿纸" | 模型在生成最终答案之前，作为推理步骤生成中间图像 |
| 从输入到第一个音频字节的时间 | "TTFAB" | 从用户语音到第一个音频输出的延迟；<500ms 才能有对话感 |
| 四阶段课程 | "训练配方" | 对齐 -> 交错 -> 语音增强 -> SFT，按此顺序进行 |
