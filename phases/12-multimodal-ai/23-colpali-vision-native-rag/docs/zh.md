# ColPali 与视觉原生文档 RAG

> 传统 RAG 将 PDF 解析为文本，分割成块，嵌入块并存储向量。每一步都丢失信号：OCR 丢弃图表数据，分块破坏表格行，文本嵌入忽略图形。ColPali (Faysse et al., July 2024) 提出了一个更简单的问题：为何要提取文本？直接通过 PaliGemma 嵌入页面图像，使用 ColBERT 风格的后期交互进行检索，并保留文档携带的所有布局、图形、字体和格式信号。已发布的基准测试显示：在视觉丰富文档上，端到端准确率比文本 RAG 高出 20-40%。ColQwen2、ColSmol 和 VisRAG 扩展了这一模式。本课程将阐述视觉原生 RAG 理念，并构建一个小型类 ColPali 索引器。

**类型：** 构建
**语言：** Python (标准库，多向量索引器 + MaxSim 评分器)
**先决条件：** 第11阶段 (LLM 工程 — RAG 基础)，第12阶段 · 05 (LLaVA)
**时间：** 约180分钟

## 学习目标

- 解释双编码器检索（每个文档一个向量）与后期交互检索（每个文档多个向量）之间的区别。
- 描述 ColBERT 的 MaxSim 操作以及 ColPali 如何将其从文本 token 推广到图像块。
- 构建一个小型类 ColPali 索引器：页面 → 块嵌入 → 查询词嵌入的 MaxSim → top-k 页面。
- 在发票/财务报告用例中，比较 ColPali + Qwen2.5-VL 生成器与文本 RAG + GPT-4。

## 问题所在

PDF 上的文本 RAG 丢弃了文档的大部分内容。财务报告的 Q3 收入增长通常存在于图表中；医疗报告的发现存在于注释图像中；法律合同的签名区块是布局事实，而非文本事实。

文本 RAG 流水线：

1. PDF → 通过 OCR / pdftotext 转为文本。
2. 文本 → 300-500 token 的块。
3. 块 → 双编码器嵌入（一个向量）。
4. 用户查询 → 嵌入 → 余弦相似度 → top-k 块。
5. 块 + 查询 → LLM。

五个有损步骤。图表未捕获。表格跨块断裂。多栏布局被展平。图形注释消失。

ColPali 的解决方案：跳过 OCR，直接嵌入页面图像。使用 ColBERT 风格的后期交互进行检索，以便模型在查询时关注细粒度的图像块。

## 概念详解

### ColBERT (2020)

ColBERT (Khattab & Zaharia, arXiv:2004.12832) 是一种文本检索方法。它不为文档生成一个向量，而是为每个 token 生成一个向量。在查询时：

- 查询 token 得到各自的嵌入（N_q 个向量）。
- 文档 token 得到嵌入（N_d 个向量，通常被缓存）。
- 评分 = 对所有查询 token 求和，每个查询 token 取其与所有文档 token 余弦相似度的最大值：Σ_i max_j cos(q_i, d_j)。

这就是 MaxSim 操作。每个查询 token “选择”其最匹配的文档 token。最终分数是这些最大值的总和。

优点：召回率强，能处理术语级语义。缺点：每个文档有 N_d 个向量，存储成本高。

### ColPali

ColPali (Faysse et al., arXiv:2407.01449) 将 ColBERT 模式应用于图像。

- 每页由 PaliGemma (ViT + 语言模型) 编码成图像块嵌入：每页 N_p 个向量。
- 每个用户查询（文本）被编码成查询 token 嵌入：N_q 个向量。
- 评分 = Σ_i max_j cos(q_i, p_j)，即查询文本 token 与页面图像块之间的 MaxSim。
- 根据总分检索 top-k 页面。

文档摄取时：用 PaliGemma 嵌入每一页，存储所有图像块嵌入。查询时：嵌入查询 token，对所有存储的页面嵌入计算 MaxSim，返回 top-k 页面。

优点：在视觉丰富的文档上，端到端性能比文本 RAG 高 20-40%。每个图像块向量捕获了局部布局和内容。

缺点：每页 N_p 个图像块 × 4 字节浮点数 × D 维向量 → 存储增长很快。通过 PQ / OPQ 量化可缓解。

### ColQwen2 和 ColSmol

ColQwen2 (illuin-tech, 2024-2025) 将 PaliGemma 替换为 Qwen2-VL。更好的基础编码器，更好的检索效果。

ColSmol 是用于本地/边缘场景的更小规模变体。约 10 亿参数的 ColSmol 检索器可在消费级 GPU 上运行。

### VisRAG

VisRAG (Yu et al., arXiv:2410.10594) 是一个不同的变体：它不在图像块上使用 MaxSim，而是用视觉语言模型将每页池化成单个向量，然后用双编码器检索。索引速度更快、存储空间更小，但召回率较弱。

质量与成本的权衡：追求质量选 ColPali，追求规模选 VisRAG。

### M3DocRAG

M3DocRAG (Cho et al., arXiv:2411.04952) 将多模态检索扩展到多页多文档推理。跨文档检索页面，为视觉语言模型组合多页上下文。

### ViDoRe — 基准测试

ColPali 的配套基准测试。视觉文档检索评估。任务包括财务报告、科学论文、行政文件、医疗记录、手册。指标：nDCG@5。

ColPali-v1 在 ViDoRe 上得分约 80% nDCG@5；相同文档上的文本 RAG 得分约 50-60%。

### 端到端 RAG 流水线

对于视觉原生 RAG：

1. 摄取：PDF → 页面图像 → PaliGemma 编码 → 存储所有图像块嵌入。
2. 查询：用户文本 → 查询 token 嵌入 → 对所有索引页面进行 MaxSim → top-k 页面。
3. 生成：top-k 页面图像 + 查询 → 视觉语言模型 (Qwen2.5-VL 或 Claude) → 答案。

无需任何 OCR。图形、图表、字体、布局全部流入答案。

### 存储计算

一个 50 页的财务报告，每页 729 个图像块，128 维嵌入：

- ColPali：50 * 729 * 128 * 4 字节 ≈ 18 MB 原始，PQ 后 ≈ 4 MB。
- 文本 RAG：50 个块 * 768 维 * 4 字节 ≈ 150 kB。

ColPali 每个文档的存储量约大 30 倍。大规模应用时，OPQ / PQ 可将其降至 5-10 倍，通常可以接受。

### 文本 RAG 仍然胜出的场景

- 没有布局信号的纯文本文档（维基文章、聊天记录）。文本 RAG 更简单且存储成本更低。
- 数百万页的档案，存储成本主导总成本。
- 有严格的法规要求，需要可提取的 OCR 文本与检索结果共存。

对于 2026 年的其他一切——财务报告、科学论文、法律合同、医疗记录、用户体验文档——视觉原生 RAG 胜出。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------|----------|
| 后期交互 | “ColBERT 风格” | 使用每 token 或每图像块嵌入 + MaxSim 的检索，而非单文档向量 |
| MaxSim | “Max-over-patches” | 对于每个查询 token，选择最相似的文档 token；对所有查询求和 |
| 双编码器 | “单向量” | 每个文档一个向量；速度更快但损失粒度 |
| 多向量 | “每文档多向量” | 每个文档/页面存储 N_p 个向量；存储成本增加但召回率提升 |
| 图像块嵌入 | “页面特征” | 视觉语言模型编码器为每个图像块生成的向量，每页缓存 |
| ViDoRe | “视觉文档基准” | ColPali 用于视觉文档检索的基准测试套件 |
| PQ 量化 | “乘积量化” | 在缩小存储约 8 倍的同时保持向量相似性的压缩方法 |
