# 百万token上下文下的长视频理解

> 一段1小时的4K视频（24 FPS）经分块嵌入后，会产生约6000万token。一集2小时的播客转录文本约3万token。一部完整蓝光电影，即使经过激进的池化压缩，也有数十万token。谷歌的Gemini 1.5（2024年3月）以1000万token上下文开启了这一时代，能在小时级视频中实现可靠的“大海捞针”式召回。LWM（Liu等人，2024年2月）展示了环注意力的扩展路径。LongVILA和Video-XL进一步扩展了视频摄入能力。VideoAgent则用智能体检索替代了原始上下文。每种方法都在计算量、召回率和工程复杂度上做出了不同的权衡。本课程将它们并列解读。

**类型：** 构建  
**语言：** Python（标准库，大海捞针模拟器 + 智能体检索路由器）  
**前置要求：** 第12阶段 · 第17节（视频时序token）  
**时间：** 约180分钟

## 学习目标

- 计算不同帧率（FPS）和池化策略下长视频的视觉token总量。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、环注意力（LWM）、token压缩（LongVILA / Video-XL）。
- 在准确率和延迟方面，比较原始上下文视频VLM与智能体检索视频VLM（VideoAgent）。
- 为一段30分钟的视频设计一个“大海捞针”测试，并测量特定分钟的召回率。

## 问题所在

一个Qwen2.5-VL规模的patch、原始分辨率为384的单帧图像约含729个token。经过3x3池化后，每帧约81个token。一段30分钟的视频，以1 FPS采样 = 1800帧 = 145,800 token。这对2025年的开源VLM来说勉强可行。若以2 FPS采样，则为291,600 token——只有最大的上下文才能容纳。

一部2小时的电影以1 FPS采样则为58.3万token。这超出了2026年大多数开源模型的能力范围；需要Gemini 2.5 Pro或更激进的池化策略。

于是出现了三种扩展路径。

## 核心概念

### 路径1：暴力上下文（Gemini 1.5，Claude Opus）

用硬件砸问题。将上下文扩展到数百万token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro上线时支持100万token；Gemini 1.5 Ultra支持1000万token；2026年的Gemini 2.5 Pro能可靠处理数小时视频。其论文（arXiv:2403.05530）记录了在约950万token时仍能保持99.7%的“大海捞针”召回率。

工程实现：一种定制的注意力机制，采用内存层次结构（局部 + 全局 + 稀疏），并结合MoE专家路由以提高长上下文效率。具体细节未完全公开。非开源。

### 路径2：环注意力（LWM，LongVILA）

环注意力将长序列分布在一个“环”结构的设备上，每个设备持有一个序列块。通过设备间以环形模式传递其序列块、计算局部注意力并聚合，来实现对完整序列的注意力计算。

LWM（Liu等人，2024年）使用此方法训练了一个100万token上下文的模型。训练计算量随上下文长度线性增长，而非二次增长——注意力机制的二次计算开销被环中的多个设备分摊了。

LongVILA（arXiv:2408.10188）将此模式应用于VLM。以每帧192 token处理1400帧视频 = 26.8万上下文，并使用8路并行的环注意力进行训练。

### 路径3：token压缩（Video-XL，LongVA）

比暴力上下文成本更低：在LLM看到序列之前进行激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要token：每段由N帧组成的视频片段生成一个单一的“摘要”token，该token对这N帧进行注意力计算。在推理时，LLM每段只看到一个摘要token，从而大幅缩减上下文。

LongVA通过一种“长上下文迁移”技术，将LLM的上下文从20万扩展到200万。先在长上下文文本上训练，再通过共享表示迁移到长上下文视频。

token压缩以牺牲特定时间戳的召回率为代价换取可扩展性。模型大致知道发生了什么，但有时会错过具体的帧。

### 路径4：智能体检索（VideoAgent）

不将整个视频输入LLM。而是将视频视为数据库，并用LLM对其进行查询。

VideoAgent（arXiv:2403.10517）：

1.  LLM阅读问题。
2.  LLM请求检索工具获取相关片段（“给我看有猫的片段”）。
3.  工具返回匹配片段的时间戳。
4.  LLM通过VLM读取这些片段。
5.  LLM组织答案或提出后续查询。

这是将LLM作为智能体模式应用于长视频。推理成本更低（只编码相关片段），工程更复杂（检索质量成为瓶颈）。

### “大海捞针”基准测试

标准的长上下文测试：在视频的随机位置插入一个独特的视觉或文本标记，然后提出一个需要回忆该标记的问题。

指标：跨视频长度和标记位置的Recall@k。

Gemini 2.5 Pro在长达90分钟的视频中召回率 >99%。开源72B模型（Qwen2.5-VL-72B, InternVL3-78B）在30分钟时召回率约85-90%，超过60分钟后性能下降。

VideoAgent在2小时以上的视频中可以匹配甚至超越原始上下文模型，前提是检索工具足够好，能够命中“针”。

### 如何选择路径

对于15分钟片段且要求前沿准确率：开源72B模型 + 原生上下文通常可行。选择Qwen2.5-VL-72B。

对于30分钟至1小时内容：开源选LongVILA或Video-XL；闭源选Gemini 2.5 Pro。质量门槛很重要——前沿技术属于闭源。

对于2小时以上内容：VideoAgent或类似的检索模式。或者，将内容总结成更小的块，并输入层次化的摘要。

### 2026年生产模式

在实践中，生产级的长视频处理流程是混合式的：

1.  对整个视频运行动态FPS采样 + 激进池化（得到一个10万token的全局表示）。
2.  输入一个72B VLM生成全局摘要。
3.  如果用户提出细节问题，则运行智能体检索，使用摘要作为索引。

这结合了暴力上下文用于全局理解和检索用于局部细节。

- 包含一个智能体检索路由器模拟器，用于选择特定片段输入下游VLM。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 暴力上下文 | “只是更多token” | 将LLM上下文扩展到数百万token；在一次处理中处理所有内容 |
| 环注意力 | “LWM风格的并行” | 一种分布式注意力模式，每个设备持有一个序列块并轮换 |
| token压缩 | “摘要token” | 在LLM处理前，通过学习型压缩器减少每个片段的token数 |
| 大海捞针 | “NIH测试” | 在随机点插入一个独特标记，要求模型在测试时回忆它 |
| 智能体检索 | “LLM作为查询规划器” | LLM请求检索工具获取相关片段，通过VLM读取它们，组织答案 |
| VideoAgent | “视频的检索模式” | 典型的智能体检索设计：问题 -> 工具 -> 片段 -> 答案 |
