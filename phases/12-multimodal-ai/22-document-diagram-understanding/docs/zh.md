# 文档与图表理解

> 文档并非普通图片。PDF、科学论文、发票或手写表单具有版式、表格、图表、脚注、页眉和语义结构，这些是纯图像理解无法捕捉的。前VLM技术栈是一个流水线：Tesseract OCR + LayoutLMv3 + 表格提取启发式规则。VLM浪潮用无需OCR的模型取代了这一方案——Donut（2022）、Nougat（2023）、DocLLM（2023）——这些模型可直接输出结构化标记。到2026年，前沿技术只需“将页面图像以2576像素原始分辨率输入Claude Opus 4.7”，即可免费获得结构化标记输出。本课程将解读文档AI的三个时代历程。

**类型：** 实践构建  
**语言：** Python（标准库，布局感知文档解析器骨架）  
**前置课程：** 阶段12 · 05 (LLaVA)，阶段5 (NLP)  
**时间：** 约180分钟

## 学习目标

- 阐述文档AI的三个时代：OCR流水线、无需OCR、VLM原生。
- 描述LayoutLMv3的三路输入流：文本、布局（边界框）、图像块，及其统一掩码机制。
- 比较Donut（无需OCR，图像→标记）、Nougat（科学论文→LaTeX）、DocLLM（布局感知生成式）、PaliGemma 2（VLM原生）。
- 为新任务（发票、科学论文、手写表单、中文小票）选择合适的文档模型。

## 问题本质

“理解这份PDF”听起来简单实则困难。信息分布于：

- 文本内容（90%的信息量）。
- 版式布局（页眉、页脚、侧边栏、双栏排版）。
- 表格（行、列、合并单元格）。
- 图形与图表。
- 手写注释。
- 字体与排版（标题与正文）。

原始OCR仅提取文本，丢失其余信息。需要处理发票的系统必须知道“总计：$1,245”来自右下角，而非脚注。

## 核心概念

### 时代一 — OCR流水线（2021年前）

经典技术栈：

1. PDF → 每页图像。
2. Tesseract（或商用OCR）提取文本及逐词边界框。
3. 版式分析器识别区块（页眉、表格、段落）。
4. 表格结构识别器解析表格。
5. 领域规则+正则表达式提取字段。

适用于清晰印刷文本。但在手写、倾斜扫描、复杂表格、非英文文本上失效。每种故障模式都需要定制异常处理路径。

### TrOCR（2021）

TrOCR（Li等人，arXiv:2109.10282）用在合成+真实文本图像上训练的Transformer编码器-解码器取代了Tesseract的经典CNN-CTC架构。在手写和多语言文本上取得显著优势。仍是流水线（检测器→TrOCR→版式），但OCR步骤大幅提升。

### 时代二 — 无需OCR（2022-2023）

首批无需OCR模型主张：完全跳过检测，直接将图像像素映射到结构化输出。

Donut（Kim等人，arXiv:2111.15664）：
- 编码器-解码器Transformer，编码器为Swin-B。
- 输出为JSON（表单理解）、Markdown（摘要）或任何任务特定模式。
- 无OCR、无版式分析、无检测。

Nougat（Blecher等人，arXiv:2308.13418）：
- 专为科学论文训练。
- 输出为LaTeX/Markdown。
- 处理数学公式、多栏排版、图形。
- 所有arXiv解析器调用的模型。

这些是专用模型，非通用模型。Donut处理科学论文会失败；Nougat处理发票也会失败。

### LayoutLMv3（2022）

另一技术路线。LayoutLMv3（Huang等人，arXiv:2204.08387）保留OCR但增加了布局理解：

- 三路输入流：OCR文本token、逐token二维边界框、图像块。
- 跨所有模态的统一掩码训练目标（掩码文本、掩码图像块、掩码布局）。
- 下游任务：分类、实体提取、表格问答。

LayoutLMv3是基于OCR的文档理解的巅峰。在表单和发票上表现强劲。需要上游OCR。在标准化文档基准上达到VLM前最佳精度。

### DocLLM（2023）

DocLLM（Wang等人，arXiv:2401.00908）是LayoutLM的生成式姊妹模型。根据布局token生成自由格式答案。更适用于文档问答；仍依赖OCR输入。

### 时代三 — VLM原生（2024+）

2024年，VLM已足够强大，可完全取代流水线。将高分辨率完整页面图像输入VLM，提出问题，获得答案。

- LLaVA-NeXT 336-tile AnyRes适用于小型文档。
- Qwen2.5-VL动态分辨率原生处理2048+像素。
- Claude Opus 4.7支持2576像素文档。
- PaliGemma 2（2025年4月）专为文档+手写训练。

VLM原生与OCR流水线的差距迅速缩小。到2026年，VLM原生在以下场景胜出：

- 场景文本（手写+印刷，混合文字）。
- 含合并单元格的复杂表格。
- 嵌入文本的数学公式。
- 带文本注释的图形。

OCR流水线仍在以下场景胜出：

- 大规模纯扫描工作负载，对每页延迟敏感。
- 流水线可靠性（确定性故障 vs VLM幻觉）。
- 需要可审计OCR输出的受监管环境。

### Claude 4.7 / GPT-5前沿

在2576像素原生输入下，前沿VLM以接近人类的精度进行文档理解。2026年初的基准测试数据：

- DocVQA：Claude 4.7 ~95.1，PaliGemma 2 ~88.4，Nougat ~77.3，流水线化LayoutLMv3 ~83。
- ChartQA：Claude 4.7 ~92.2，GPT-4V ~78。
- VisualMRC：Claude 4.7 ~94。

闭源模型的差距主要在于分辨率和基础LLM规模。7B开源模型落后几个百分点但正在追赶。

### 数学公式与LaTeX输出

科学论文需要精确的LaTeX公式输出。Nougat为此训练。以LaTeX为目标训练的VLM（Qwen2.5-VL-Math、Nougat衍生模型）可产生可用的LaTeX。若无显式LaTeX训练，VLM产生可读但不精确的转录。

2026年科学论文处理流程：在PDF上使用Nougat处理数学公式，再在复杂页面上使用VLM处理图形。

### 手写识别

仍是最困难的子任务。印刷+手写混合（医生笔记、填写表单）是OCR流水线在成本上仍优于VLM的场景。纯手写识别的VLM正在改进（Claude 4.7、PaliGemma 2）。

### 2026年方案

针对新文档AI项目：

- 大规模纯印刷发票：LayoutLMv3+规则，成本高效。
- 混合文档（科学+手写+表单）：VLM原生（PaliGemma 2或Qwen2.5-VL）。
- 完整arXiv收录：Nougat处理数学公式，VLM处理图形。
- 受监管场景：OCR流水线+VLM验证器进行交叉检查。

## 练习题

1. 你的项目需每日处理1000万张发票。在不损失精度的前提下，哪种技术栈能最小化每页成本？

2. 为什么LayoutLMv3在表单QA上优于纯CLIP-VLM，但在场景文本上表现不佳？边界框流舍弃了什么？

3. Nougat生成LaTeX。请提出一个测试案例，证明VLM原生输出在LaTeX保真度上优于Nougat，再提出一个Nougat胜出的案例。

4. 阅读PaliGemma 2论文（Google，2024）。相比PaliGemma 1，哪项关键训练数据的添加提升了文档精度？

5. 设计一个监管安全的混合方案：OCR流水线为主，VLM为辅进行交叉检查。如何解决两者不一致的问题？

## 核心术语

| 术语 | 常见称呼 | 实际含义 |
|------|----------|----------|
| OCR流水线 | "Tesseract式" | 分阶段技术栈：检测→OCR→版式→规则；确定性，脆弱 |
| 无需OCR | "Donut式" | 跳过显式OCR的图像到输出Transformer；单模型方案 |
| 布局感知 | "LayoutLM" | 输入包含逐token边界框坐标；跨模态统一掩码 |
| VLM原生 | "前沿VLM" | 将页面图像直接以高分辨率输入Claude/GPT/Qwen VLM；无流水线 |
| DocVQA | "文档基准" | 文档视觉问答标准；引用率最高的分数 |
| 标记输出 | "LaTeX/MD" | 结构化输出格式而非自由文本；支持下游自动化 |
