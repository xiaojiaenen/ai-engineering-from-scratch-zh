# 安全 — 密钥管理、API密钥轮换、审计日志、防护措施

> 通过集中化保险库（HashiCorp Vault、AWS Secrets Manager、Azure Key Vault）消除密钥蔓延。切勿将凭证存储在配置文件、版本控制系统中的环境文件或电子表格中。优先使用IAM角色而非静态密钥；在CI/CD中使用OIDC。AI网关模式是2026年的标准解决方案：应用程序 → 网关 → 模型提供商，网关在运行时从保险库拉取凭证。在保险库中轮换密钥，所有应用在几分钟内自动获取——无需重新部署，也无需在Slack上询问“谁有新密钥”。轮换策略不超过90天；使用TruffleHog / GitGuardian / Gitleaks在每次提交时进行扫描。实施零信任：多因素认证、单点登录、基于角色/属性的访问控制、短期令牌、设备合规状态检查。PII脱敏使用实体识别在转发前掩码PHI/PII；一致的令牌化（Mesh方法）将敏感值映射为稳定的占位符，使大语言模型保持代码/关系语义。网络出口：将大语言模型服务部署在专用的VPC/VNet子网中，仅允许连接`api.openai.com`、`api.anthropic.com`等地址；阻止所有其他出站连接。2026年安全事件驱动因素：Vercel供应链攻击通过泄露的CI/CD凭证，在数千个客户部署中窃取环境变量。

## 问题描述

一名实习生提交了包含API密钥的`.env`。他们迅速删除了提交，但密钥已存在于git历史中——GitGuardian扫描发现了它，而你们的轮换流程是“在Slack上通知团队，更新40个配置文件，重新部署所有服务”。8小时后，一半服务已上线，一半还在等待部署窗口。

另外，用户提示中包含“我的社会安全号码是123-45-6789”。该提示被发送到OpenAI。虽然你们有业务关联协议，但内部政策要求在转发前掩码PII。你们没有这样做。

另外，你们的EKS集群中的大语言模型Pod可以访问任何互联网主机。有人通过DNS查询攻击者控制的域名窃取数据。没有任何机制阻止此行为。

大语言模型服务的安全必须解决所有三个攻击向量。基于保险库的凭证管理。PII脱敏。网络出口过滤。审计日志。

## 核心概念

### 集中化保险库 + IAM角色拉取

**保险库**：HashiCorp Vault、AWS Secrets Manager、Azure Key Vault、GCP Secret Manager。单一可信数据源。

**IAM角色**：应用/网关通过其IAM身份进行认证，而非使用静态密钥。保险库在令牌生命周期内返回密钥。

**AI网关模式**：网关在请求时从保险库拉取`OPENAI_API_KEY`。在保险库中轮换密钥；下一个请求即可获取新密钥。无需重新部署。

### 轮换策略 ≤ 90天

适用于所有API密钥、保险库根令牌、CI/CD凭证。尽可能实现自动化轮换。手动轮换需记录并跟踪。

### 密钥扫描

- **TruffleHog** — 正则表达式 + 提交内容的熵值分析。
- **GitGuardian** — 商业工具，准确率高。
- **Gitleaks** — 开源工具，在CI中运行。

在每次提交时运行。检测到新密钥时阻止合并请求。

### 零信任态势

- 所有账户要求多因素认证。
- 通过SAML/OIDC实现单点登录。
- RBAC（基于角色）或ABAC（基于属性）进行细粒度访问控制。
- 短期令牌（小时级，而非天级）。
- 设备合规状态——仅允许具有磁盘加密的公司设备。

### PII/PHI脱敏

在提示离开您的基础设施之前：

1.  实体识别（spaCy NER、Presidio、商业工具）。
2.  掩码匹配到的实体：`"My SSN is 123-45-6789"` → `"My SSN is [SSN_TOKEN_A3F]"`。
3.  一致的令牌化（Mesh方法）：相同值映射到相同的占位符，使大语言模型保持关系。
4.  可选：对大语言模型响应进行反向映射。

静态正则表达式过滤器能捕获基本模式；NER能捕获更多。两者结合使用。

### 输入/输出防护措施

输入：阻止已知的越狱提示、禁止话题；按用户进行速率限制。

输出：使用正则表达式清除泄露的密钥（API密钥模式、拒绝响应中的电子邮件模式），使用分类器检测策略违规。

### 网络出口白名单

将大语言模型服务部署在专用子网中：
- 白名单：`api.openai.com`、`api.anthropic.com`、向量数据库端点、保险库端点。
- 其他所有连接：丢弃。
- DNS通过仅允许名单的解析器（避免DNS隧道数据窃取）。

### 审计日志

记录每次大语言模型调用的不可变日志，包含：
- 时间戳。
- 用户/租户。
- 提示哈希值（出于隐私考虑，不记录原始提示）。
- 模型 + 版本。
- token计数。
- 成本。
- 响应哈希值。
- 任何触发的防护措施。

根据监管要求保留（SOC 2：1年，HIPAA：6年）。

### 2026年Vercel事件

供应链攻击：泄露的CI/CD凭证导致数千个客户部署的环境变量被窃取。教训：CI/CD凭证等同于生产环境凭证。存储在保险库中。严格限制权限范围。积极进行轮换。

### 需要记住的数字

- 轮换策略：≤ 90天。
- 每次提交扫描：TruffleHog / GitGuardian / Gitleaks。
- Vercel 2026事件：CI/CD凭证泄露 → 数千个客户环境变量泄露。
- 审计日志保留期限：SOC 2 = 1年，HIPAA = 6年。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 保险库 | “密钥存储” | 集中式凭证管理服务 |
| IAM角色 | “基于身份的认证” | 应用程序承担的角色；返回短期凭证 |
| CI/CD中的OIDC | “云颁发的令牌” | CI中不使用静态密钥——通过OIDC进行身份验证 |
| TruffleHog / GitGuardian / Gitleaks | “密钥扫描器” | 提交时密钥检测工具 |
| RBAC / ABAC | “访问控制” | 基于角色 vs 基于属性的访问控制 |
| PII脱敏 | “数据掩码” | 移除或令牌化敏感实体 |
| 一致令牌化 | “稳定占位符” | 相同值 → 每次获得相同的令牌 |
| Mesh方法 | “Mesh令牌化” | 保持语义的令牌化模式 |
| 出口白名单 | “出站允许列表” | 仅允许访问指定的域名 |
| 审计日志 | “不可变历史记录” | 用于合规的仅追加记录 |
