# Serverless LLMs 的冷启动缓解策略

> 一个 20 GB 的模型镜像从冷启动到提供服务需要 5-10 分钟 (7B) 到 20 分钟以上 (70B)。在真正的无服务器世界中，这不是预热——这是中断。缓解措施在五个层面运作：预置节点镜像 (AWS 上的 Bottlerocket，双卷架构)、模型流式传输 (NVIDIA Run:ai Model Streamer，vLLM 原生支持)、GPU 内存快照 (Modal checkpoints，重启速度提升高达 10 倍)、热池 (`min_workers=1`)、分层加载 (ServerlessLLM 的 NVMe→DRAM→HBM 流水线，延迟降低 10-200 倍)，以及实时迁移——仅移动输入 tokens (KB) 而非 KV cache (GB)。Modal 的冷启动时间为 2-4 秒作为基准；Baseten 默认为 5-10 秒，预热后可达亚秒级。本课程将教你如何测量、规划并组合运用这五个层面。

**类型：** 学习  
**语言：** Python (标准库，简易冷启动路径模拟器)  
**先决条件：** 阶段 17 · 02 (推理平台经济学)，阶段 17 · 03 (GPU 自动扩缩容)  
**时间：** 约 60 分钟

## 学习目标

- 列举冷启动缓解的五个层面，并为每个层面命名一个工具或模式。
- 计算 70B 模型的总冷启动时间，它是 (节点配置) + (权重下载) + (权重加载至 HBM) + (引擎初始化) 的总和。
- 解释为什么实时迁移传输的是输入 tokens (KB) 而非 KV cache (GB)，以及其代价是什么 (重新计算)。
- 说明热池的权衡 (为空闲 GPU 付费或接受冷启动尾部延迟)，以及 `min_workers > 0` 变为强制性要求时的 SLA 阈值。

## 问题所在

你的无服务器 LLM 端点在夜间缩容至零。上午 8 点流量飙升。第一个请求在等待期间：
1.  Karpenter 配置一个 GPU 节点：45-60 秒。
2.  容器拉取一个包含权重的 30 GB 镜像：120-300 秒。
3.  引擎将权重加载到 HBM 中：45-120 秒，取决于模型大小和存储速度。
4.  vLLM 或 TRT-LLM 初始化 CUDA 图、KV 缓存池、分词器：10-30 秒。

总计：220-510 秒 (大约 3-8 分钟) 才能返回第一个 token。你的 SLA 是 2 秒。你部署了一个热池 (`min_workers=1`)，问题似乎消失了——但现在你需要 24x7 为一个空闲 GPU 付费。如果你的服务有 5 个产品，每个都有一个热副本，那就是 5 × 24 × 30 = 3,600 GPU-小时/月，无论是否有用户调用。

冷启动缓解就是在保持无服务器经济学优势的同时，接近永不停机的延迟水平。

## 核心概念

### 层级 1 — 预置节点镜像 (Bottlerocket)

在 AWS 上，Bottlerocket 的双卷架构将操作系统与数据分离。使用预拉取容器镜像的快照保存数据卷；在你的 `EC2NodeClass` 中引用该快照 ID。新节点启动时，权重已在本地 NVMe 上——步骤 2 和部分步骤 3 被消除。可与 Karpenter 原生配合使用。对于大型模型，通常可节省每个冷启动 2-4 分钟。

在 GCP 上的等效方案：包含预烘焙容器层的自定义 VM 镜像。在 Azure 上：使用相同模式的托管磁盘快照。

### 层级 2 — 模型流式传输 (Run:ai Model Streamer)

不必在回答第一个请求前加载整个文件，而是将权重逐层流式传输到 GPU 内存，并在第一个 Transformer 块驻留后立即开始处理。NVIDIA Run:ai Model Streamer 在 vLLM 2026 中原生提供。支持 S3、GCS 和本地 NVMe。对于大型模型，通过重叠 I/O 与计算设置，可将权重加载时间缩短约一半。

### 层级 3 — GPU 内存快照 (Modal)

Modal 在首次加载后对 GPU 状态 (权重、CUDA 图、KV 缓存区域) 进行检查点保存。后续重启时直接反序列化到 HBM 中——比重新初始化快 10 倍。这是最接近 "2 秒启动一个热 GPU" 的方案。权衡：快照与特定 GPU 拓扑绑定，因此如果 Karpenter 将你迁移到不同的 SKU，你需要重新建立检查点。

### 层级 4 — 热池 (min_workers=1)

最简单的缓解措施：始终保持一个副本就绪。成本是一个 GPU 每小时的费率 24x7。对于小模型，算术很残酷 (你支付 $0.85-$1.50/小时 以避免 30 秒的冷启动)，对于大模型则很友好 (支付 $4/小时 以避免 5 分钟的冷启动)。热池变为强制性的 SLA 阈值：通常是在 70B+ 模型上 TTFT P99 < 60 秒。

### 层级 5 — 分层加载 (ServerlessLLM)

ServerlessLLM 将存储视为一个层级体系：NVMe (速度快但容量大)、DRAM (中等速度但层级化)、HBM (容量小但即时访问)。权重预加载到 DRAM 中；按需加载到 HBM。论文报告称，与朴素的磁盘到 HBM 方式相比，冷加载延迟降低 10-200 倍。生产采用尚处早期，但已存在与 vLLM 的集成。

### 层级 6 — 实时迁移 (附加模式)

当节点不可用 (Spot 抢占、节点排空) 时，传统模式是冷启动另一个副本并排空请求队列。实时迁移将输入 tokens (千字节) 移动到一个已加载模型的目标节点，并在目标节点重新计算 KV 缓存。重新计算比通过网络传输 GB 级的 KV 缓存成本更低。适用于分解式部署。

### 热池算术

对于 P99 TTFT SLA 为 2 秒的服务，问题不在于 "是否需要热池"，而在于 "需要多少个热副本，以及为哪些路径配备"。

- 高价值交互路径 (实时聊天、语音助手)：`min_workers=1-2`。
- 后台批处理路径 (夜间分类任务)：接受缩容至零，5-10 分钟的冷启动可容忍。
- 高级层级：`min_workers` 每租户配备专用容量。

### 优化前先测量

新节点上 70B 模型的冷启动分解 (示例)：

| 阶段 | 时间 | 缓解措施 |
|-------|------|-----------|
| 节点配置 | 50 秒 | Bottlerocket + 预置种子镜像，热池 |
| 镜像拉取 | 180 秒 | 预置数据卷 (消除) |
| 权重加载到 HBM | 75 秒 | 模型流式传输器 (减半)；GPU 快照 (消除) |
| 引擎初始化 | 20 秒 | 持久化 CUDA 图缓存 |
| 第一次前向传播 | 3 秒 | 最小固有延迟 |
| **总冷启动时间** | **328 秒** | |
| **采取缓解措施后总时间** | **约 15 秒** | 降低 22 倍 |

### 应该记住的数字

- Modal 冷启动：2-4 秒 (使用 GPU 快照)。
- Baseten 默认冷启动：5-10 秒；预热后可达亚秒级。
- 原始 70B 冷启动：3-8 分钟。
- Run:ai Model Streamer：权重加载速度提升约 2 倍。
- ServerlessLLM 分层加载：延迟降低 10-200 倍 (论文数据)。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 冷启动 | "大停顿" | 从请求到在新副本上返回第一个 token 的时间 |
| 热池 | "始终在线的最低配置" | `min_workers >= 1` 以保持至少一个副本就绪 |
| 预置种子镜像 | "烘焙 AMI" | 包含预加载容器权重的节点镜像 |
| Bottlerocket | "AWS 节点操作系统" | AWS 容器优化操作系统，支持双卷快照 |
| 模型流式传输器 | "流式加载" | 重叠权重 I/O 与计算设置 |
| GPU 快照 | "检查点到 HBM" | 序列化加载后的 GPU 状态；重启时反序列化 |
| 分层加载 | "NVMe + DRAM + HBM" | 存储层级体系；按需加载 |
| 实时迁移 | "移动 tokens" | 传输输入 (KB)，在目标节点重新计算 KV |
| `min_workers` | "热副本" | 无服务器最低保活计数 |
| 缩容至零 | "完全无服务器" | 空闲时无成本；接受完全冷启动税 |
