# LLM 功能的 A/B 测试 — GrowthBook、Statsig 与"感觉"问题

> 传统的 A/B 测试并非为非确定性的 LLM 所设计。关键区别在于：评估（evals）回答的是"模型能否完成任务？"，而 A/B 测试回答的是"用户是否在乎？"两者都不可或缺；仅凭"感觉不错"就上线已经过时了。2026 年需要测试的方面：提示工程（措辞）、模型选择（GPT-4 vs GPT-3.5 vs 开源；准确率 vs 成本 vs 延迟）、生成参数（温度、top-p）。实际案例：一个聊天机器人的奖励模型变体带来了 +70% 的对话长度和 +30% 的留存率；Nextdoor AI 通过优化奖励函数使主题行的点击率提升了 1%；Khan Academy 的 Khanmigo 在延迟与数学准确率之间进行了迭代权衡。平台选择：**Statsig**（2025年9月被 OpenAI 以 11 亿美元收购）— 序贯检验，CUPED，一体化方案。**GrowthBook** — 开源，仓库原生，贝叶斯 + 频率派 + 序贯引擎，CUPED，SRM 检查，Benjamini-Hochberg + Bonferroni 校正。你的选择取决于对仓库 SQL 的偏好以及"被 OpenAI 收购"对你的组织是否重要。

**语言：** Python（标准库，简易序贯测试模拟器）
**前置知识：** 第 17 阶段 · 第 13 课（可观测性），第 17 阶段 · 第 20 课（渐进式部署）

## 学习目标

- 区分评估（"模型能否完成任务"）与 A/B 测试（"用户是否在乎"）。
- 枚举三个可测试的维度（提示、模型、参数），并为每个维度选择合适的指标。
- 解释 CUPED、序贯检验以及 Benjamini-Hochberg 多重比较校正。
- 根据仓库 SQL 方针和公司收购立场，选择 Statsig 或 GrowthBook。

## 问题所在

你手动调整了一个系统提示。感觉更好了。你把它发布上线。转化率变化了一些噪声。你责怪指标。或者你发布了一个新模型，但转化率没有变化——是模型退化了，还是变化太小无法检测？你不知道，因为你发布时没有进行 A/B 测试。

评估回答的是模型在标注数据集上能否完成任务。它们不能回答用户是否更喜欢该输出。只有受控的在线实验才能回答这个问题，并且只有在实验有足够功效、控制了非确定性并校正了多重比较的情况下才行。

## 核心概念

### 评估 vs A/B 测试

**评估** — 离线，标注数据集，评判者（评分标准或 LLM-as-judge 或人类）。回答："在这个固定分布上，输出是否正确/有帮助/安全？"

**A/B 测试** — 在线，真实用户，随机化。回答："新变体是否提升了重要的用户级指标？"

两者都必需。评估在用户接触前捕获回归；A/B 测试在之后确认产品影响。

### 测试什么

1.  **提示工程** — 措辞、系统提示结构、示例。指标：任务成功率、用户留存率、单次请求成本。
2.  **模型选择** — GPT-4 vs GPT-3.5-Turbo vs Llama 开源。指标：准确率（任务）+ 单次请求成本 + P99 延迟。多目标优化。
3.  **生成参数** — 温度、top-p、max_tokens。指标：取决于具体任务（输出多样性 vs 确定性）。

### CUPED — 方差缩减

使用实验前数据进行受控实验。在比较实验后数据之前，回归掉实验前数据的方差。典型的方差缩减：30-70%。有效样本量（免费地）增加了。

实现：Statsig 和 GrowthBook 都实现了。

### 序贯检验

经典的 A/B 测试假设固定样本量。序贯检验（"边看边定"）在重复查看下控制假阳性率。始终有效的序贯程序（mSPRT，Howard 置信序列）允许你在出现明显赢家时提前停止。

### 多重比较校正

以 95% 置信度运行 20 个 A/B 测试，会凭空产生一个假阳性。Bonferroni 校正收紧每个测试的 α；Benjamini-Hochberg 控制错误发现率。GrowthBook 两种都实现了。

### SRM — 样本比例不匹配

分配哈希将用户随机分配到变体。如果是 50/50 分配，却出现了 47/53 的结果，说明出了问题——SRM 检查会标记出来。两个平台都实现了。

### Statsig vs GrowthBook

**Statsig**：
- 于 2025 年 9 月被 OpenAI 以 11 亿美元收购。托管型 SaaS。
- 序贯检验，CUPED，保留人群。
- 一体化：功能开关 + 实验 + 可观测性。
- 最适合：团队已经想要捆绑产品，不介意 OpenAI 的所有权。

**GrowthBook**：
- 开源 (MIT)；仓库原生（直接从 Snowflake/BigQuery/Redshift 读取）。
- 多种引擎：贝叶斯、频率派、序贯。
- CUPED，SRM，Bonferroni，BH 校正。
- 自托管或托管云。
- 最适合：仓库 SQL 团队，数据团队控制指标层，希望使用开源。

### 非确定性使功效计算复杂化

相同的提示会产生不同的输出。传统的功效计算假设 IID（独立同分布）观测值。由于 LLM 的非确定性，有效样本量低于名义样本量。将所需样本量乘以约 1.3-1.5 倍作为安全边际。

### 实际案例结果

- 聊天机器人奖励模型变体：+70% 对话长度，+30% 留存率。
- Nextdoor 主题行：优化奖励函数后点击率提升 1%。
- Khan Academy Khanmigo：在延迟与数学准确率之间进行迭代权衡。

### 反模式：凭感觉上线

每个资深工程师都能说出一个因为"感觉更好"而没有进行 A/B 测试就上线的功能。其中大多数都对团队几个月都没注意到的指标造成了回归。A/B 测试是强制性的校验。

### 你应该记住的数字

- Statsig 被 OpenAI 收购：11 亿美元，2025 年 9 月。
- GrowthBook：开源 MIT；贝叶斯 + 频率派 + 序贯。
- CUPED 方差缩减：30-70%。
- LLM 非确定性 → +30-50% 的样本量缓冲。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|------------|----------|
| 评估 | "离线测试" | 模型能力的标注集评估 |
| A/B 测试 | "实验" | 基于用户的实时随机化比较 |
| CUPED | "方差缩减" | 通过实验前回归来减少方差 |
| 序贯检验 | "看一眼没事的测试" | 允许提前停止的始终有效程序 |
| 多重比较 | "整体错误" | 运行多个测试会增加假阳性 |
| Bonferroni | "严格校正" | 将 α 除以测试数量 |
| Benjamini-Hochberg | "BH FDR" | 错误发现率控制，不那么保守 |
| SRM | "比例不对" | 样本比例不匹配；分配错误 |
| Statsig | "OpenAI 旗下的" | 商业一体化方案，2025 年被收购 |
| GrowthBook | "那个开源的" | MIT 仓库原生平台 |
| mSPRT | "序贯概率比检验" | 经典的序贯检验程序 |
