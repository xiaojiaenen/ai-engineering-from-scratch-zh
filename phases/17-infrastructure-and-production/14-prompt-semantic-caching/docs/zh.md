# 提示缓存与语义缓存经济学

> **定价快照截至2026年4月。** 下文数值反映本课程发布时的供应商价目表；引用前请核对链接文档。

> 缓存发生在两个层级。L2（供应商级）提示/前缀缓存复用重复前缀的注意力KV值——Anthropic提示缓存文档宣称长提示最高可降低90%成本、85%延迟；Claude 3.5 Sonnet缓存读取价格为每百万token 0.30美元，新输入为3.00美元，TTL为5分钟，1小时TTL选项写入溢价为2倍（docs.anthropic.com，2026年4月）。OpenAI提示缓存对≥1024 token的提示自动生效，缓存输入价格约为新输入的10%（platform.openai.com，2026年4月）；具体每模型缓存价格取决于实时价目表。L1（应用级）语义缓存在嵌入相似度命中时完全跳过LLM调用。供应商宣称的“95%准确率”指匹配正确率而非命中率——已报告的生产环境命中率从10%（开放式聊天）到70%（结构化FAQ）不等；两家供应商均未公布官方基线，故视作社区遥测数据而非保证。生产环境陷阱：并行化会破坏缓存效果（在首次缓存写入前发出N个并行请求可能导致开销增长数倍），且前缀中的动态内容会完全阻止缓存命中。ProjectDiscovery报告通过将动态文本移出可缓存前缀，命中率从7%提升至74%（2025年11月）。

## 学习目标

- 区分L2提示/前缀缓存（供应商侧KV复用）与L1语义缓存（相似提示跳过LLM）。
- 解释Anthropic的`cache_control`显式标记及两种TTL选项（5分钟与1小时）及其价格倍数。
- 根据命中率、提示/响应混合比例及token价格计算预期月度节省。
- 识别导致账单膨胀5-10倍的并行化反模式，以及使命中率归零的动态内容反模式。

## 问题场景

您为RAG服务添加了提示缓存。账单保持不变。测量命中率仅为7%。提示看似静态实则不然——系统提示包含按分钟格式化的当前时间、请求ID及为多样性随机排列的示例顺序。每个请求都写入新缓存条目，读取为零。

另有一个代理对每个用户问题并行发出十个工具调用。所有十个请求在首次缓存写入完成前就已到达供应商。十次写入，零次读取。账单是“启用缓存”预期成本的5-10倍。

缓存是一种协议而非开关。两个层级，两种不同的失败模式。

## 核心概念

### L2 — 供应商提示/前缀缓存

供应商存储可缓存前缀的注意力KV值，并在下一个匹配该前缀的请求中复用。您支付一次写入成本，后续读取几乎免费。

**Anthropic（Claude 3.5/3.7/4系列）**：请求中需显式添加`cache_control`标记。您标记哪些内容块可缓存。TTL：5分钟（写入成本为基础费用的1.25倍）或1小时（写入成本为基础费用的2倍）。缓存读取：Claude 3.5 Sonnet每百万token 0.30美元 vs 新输入3.00美元——便宜10倍（docs.anthropic.com，截至2026年4月）。不同模型费率有异（Opus/Haiku单独公布）；务必交叉核对实时定价页面。

**OpenAI**：对≥1024 token的提示自动缓存（platform.openai.com，2026年4月）。无需显式标记。在当前gpt-4o/gpt-5价目表上，缓存输入价格约为新输入的10%。文档和发布说明均未公布官方命中率基线；社区报告在精心设计提示下命中率约30-60%。通过监控`usage.cached_tokens`测量您自己的命中率。

**Google（Gemini）**：通过显式API进行上下文缓存；支持100万token上下文意味着缓存收益更大。

**自托管（vLLM、SGLang）**：第17章06节涵盖RadixAttention——在您自己的计算资源上实现相同模式。

### L1 — 应用级语义缓存

在完全调用LLM之前，对提示进行哈希和嵌入，查找相似的缓存请求（余弦相似度超过阈值，通常为0.95+）。命中则返回缓存响应；未命中则调用LLM并缓存结果。

开源方案：Redis向量相似度搜索、GPTCache、Qdrant。商业方案：Portkey Cache、Helicone Cache。

供应商的准确率声明指返回的缓存响应在语义上合适的频率——而非命中率。生产环境命中率：

- 开放式聊天：10-15%。
- 结构化FAQ/客服支持：40-70%。
- 代码问题：20-30%（微小变体破坏命中）。
- 语音代理重复提示：50-80%（语音归一化固定语料）。

### 并行化反模式

您的代理并行发出10个工具调用。全部10个包含相同的4K token系统提示。Anthropic的缓存写入是按请求进行的；首次缓存写入在供应商看到提示后约300毫秒完成。请求2-10在同一毫秒窗口到达，每个都遇到缓存未命中。您支付了10次写入溢价，享受了0次读取折扣。

修复方案：采用“顺序优先”批处理——单独发出请求1，待请求1缓存填充后再发出2-10。为首个工具调用增加300毫秒延迟；节省5-10倍账单。

### 动态内容反模式

您的系统提示如下所示：

```
You are a helpful assistant. The current time is 14:32:17.
User ID: abc123. Today is Tuesday...
```

每个请求都是唯一的。每个请求都需写入。命中率为零。

修复方案：将所有真正静态的内容移至可缓存前缀；将动态内容追加到缓存边界之后：

```
[cacheable]
You are a helpful assistant. [rules, examples, instructions]
[/cacheable]
[dynamic, not cached]
Current time: 14:32:17. User: abc123.
```

ProjectDiscovery通过此方法将缓存命中率从7%提升至74%并公开了具体实施细节。

### 组合批处理与缓存用于隔夜工作负载

批处理API（第17章15节）以24小时周转时间为基准提供50%折扣。在此基础上叠加缓存输入可进一步获得约10倍收益。隔夜分类、标注和报告生成工作负载通过组合使用可将成本降至同步未缓存成本的约10%。

价格要点截至2026年4月，摘自链接的供应商文档，每数月可能变动——依赖前请重新核查。

- Anthropic缓存读取：Claude 3.5 Sonnet每百万token 0.30美元，约为新输入价格的10%（docs.anthropic.com）。
- Anthropic缓存写入溢价：1.25倍（5分钟TTL）或2倍（1小时TTL）。

## 练习题

1. 运行`code/main.py`。切换并行化标志。账单变化有多大？
2. 您的系统提示包含日期。将其移出。展示移动前后的命中率计算。
3. 根据您的请求到达率，计算1小时TTL（2倍写入）与5分钟TTL（1.25倍写入）的盈亏平衡点。
4. 0.95阈值的语义缓存命中率为20%。0.85阈值时命中率为50%，但出现错误的缓存响应。选择合适阈值并说明理由。
5. 您将每个用户问题的10个并行子查询进行批处理。重写为缓存友好形式，且不增加端到端延迟。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| L2提示缓存 | "前缀缓存" | 供应商存储重复前缀的KV值 |
| `cache_control` | "Anthropic缓存标记" | 显式标记可缓存内容块的属性 |
| 缓存写入溢价 | "写入税" | 首次未命中到缓存的额外成本（1.25倍或2倍） |
| L1语义缓存 | "嵌入缓存" | 调用LLM前的应用层哈希与嵌入 |
| GPTCache | "LLM缓存库" | 流行的开源L1缓存库 |
| 缓存命中率 | "命中数/总数" | 由缓存提供服务的请求比例 |
| 并行化反模式 | "N次写入陷阱" | N个并行请求错过缓存N次 |
| 动态内容陷阱 | "时间戳入提示陷阱" | 前缀中的动态字节破坏命中率 |
| RadixAttention | "副本内缓存" | SGLang的前缀缓存实现 |
