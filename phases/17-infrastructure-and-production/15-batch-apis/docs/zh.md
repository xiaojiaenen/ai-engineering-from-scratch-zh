# 批量 API —— 50% 折扣成为行业标准

> 每个主要提供商都推出带有 50% 折扣和约 24 小时处理时间的异步批量 API。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 批量层级、Together 批量）都实现了相同的模式。将批量与提示缓存结合使用，夜间处理管道的成本可降至同步未缓存成本的约 10%。规则极其简单：如果不是交互式的，就应该使用批量。内容生成管道、文档分类、数据提取、报告生成、批量标注、目录标签 —— 任何能容忍 24 小时延迟的应用场景，如果不迁移到批量，就是在浪费资金。2026 年的生产模式是将每个新的 LLM 工作负载分为三类：交互式（带缓存的同步）、半交互式（带回退的异步队列）、批量（夜间处理，缓存输入叠加）。那些伪装成交互式但能容忍数分钟延迟的工作负载是最浪费的。

**类型：** 学习
**语言：** Python（标准库，演示性批量与同步成本模拟器）
**先决条件：** 阶段 17 · 14（提示与语义缓存）
**时间：** 约 45 分钟

## 学习目标

- 列举三大提供商的批量 API（OpenAI、Anthropic、Google）及其通用的 50% 折扣 + 24 小时处理时间保证。
- 计算在一个夜间分类工作负载上叠加使用批量和缓存输入的成本，并与同步未缓存基线进行比较。
- 将工作负载分类为交互式 / 半交互式 / 批量，并说明理由。
- 指出两个陷阱：部分交互性（用户期望快于 24 小时）和输出模式漂移（批量文件格式因提供商而异）。

## 问题所在

你的团队推出一个夜间报告生成管道。50,000 份文档，每份都要总结，对总结进行聚类，然后起草一份执行摘要。同步运行需要 4 小时，每晚成本 2,000 美元。你听说了批量 API。

批量能给你 50% 的折扣。你还在系统提示上启用了提示缓存（在所有 50,000 次调用中共享）。叠加使用后，账单降至每晚 180 美元 —— 约为基线的 9%。相同的管道，只需修改三个配置。

批量是 LLM 成本工具箱中最廉价的杠杆，却很少被使用。原因主要是组织性的：团队想着“实时”，而实际的服务等级协议是“到早上完成”。本课程的目标是别把 90% 的账单留在桌上。

## 核心概念

### 三大批量 API

**OpenAI 批量 API**：上传包含请求列表的 JSONL 文件。承诺 24 小时内处理完成（实际通常为 2-8 小时）。输入和输出 token 均享受 50% 折扣。`/v1/batches` 端点。符合缓存条件的输入还能叠加享受缓存输入定价。

**Anthropic 消息批量处理**：上传 JSONL 文件。24 小时处理时间。50% 折扣。支持 `cache_control` —— 缓存写入是显式的，在批量内自动进行读取。

**Google Vertex AI 批量预测**：输入来自 BigQuery 或 GCS。Gemini 模型同样享受 50% 折扣。与 Vertex 管道集成。

### 语义：异步，而非缓慢

批量是“我保证在 24 小时内返回结果” —— 不是“这需要 24 小时来完成”。典型的 P50 处理时间是 2-6 小时。提供商会在 GPU 资源利用率较低的非高峰时段安排执行你的批量任务。

### 与缓存叠加使用

对一份包含相同 4K-token 系统提示的 50k 文档进行摘要处理：

- 同步未缓存：按全价计算 50000 × ($输入成本 × 4000 + $输出成本 × 200)。
- 同步缓存：系统提示在首次写入后被缓存；其余 49,999 次调用的输入成本降低 10 倍。
- 批量缓存：以上所有，且读写操作均享受 50% 折扣。

叠加效果：批量 + 缓存 ≈ 同步未缓存账单的 10%。任何夜间运行且包含共享系统提示的工作负载都应使用此方法。

### 工作负载分类

**交互式** —— 用户等待响应。首 token 时间很重要。使用带提示缓存的同步调用。不能批量。

**半交互式** —— 用户提交任务，几分钟后查看结果。异步队列，若批量不可用则回退到同步。例如中等规模的 RAG 索引。

**批量** —— 用户期望结果“到早上”或“下一小时”获得。内容管道、大规模分类、离线分析。始终使用批量，并叠加缓存。

常见错误：因为管道是生产环境，就将所有任务归类为交互式。生产环境不是延迟规格 —— 服务等级协议才是。

### 部分交互性陷阱

某些功能看似交互式，但能容忍 5-10 分钟的延迟。例如：带有“刷新”按钮的夜间客户健康报告。用户点击刷新；等待 10 分钟是可以接受的。团队却将其实现为同步调用。50 个并发刷新的成本，是批量处理并通过电子邮件发送方案的 10 倍。

要问的问题：“24 小时对这个用户意味着什么？”如果答案是“他们不会注意到”，就应该批量处理。

### 输出模式陷阱

批量文件格式因提供商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入其中。
- Vertex：BigQuery 表或带有 TFRecord 的 GCS 前缀。

为所有提供商编写“一个批量客户端”意味着需要为每个提供商编写适配器代码。那些宣传支持多提供商批量处理的网关（如 Portkey，LiteLLM 的某些层级）也只是对原始格式做了薄封装。

### 你应该记住的数字

- 各提供商的批量折扣：输入 + 输出均统一 50% 折扣。
- 处理时间 SLA：保证 24 小时，典型 P50 为 2-6 小时。
- 叠加批量 + 缓存输入：约等于同步未缓存成本的 10%。
- 工作负载分类规则：如果 24 小时延迟可接受，始终使用批量。

## 关键术语

| 术语 | 人们常说的 | 其实际含义 |
|------|----------------|------------------------|
| 批量 API | “异步折扣” | 50% 折扣，24 小时处理时间 |
| JSONL | “批量格式” | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| 消息批量处理 | “Anthropic 批量” | Anthropic 批量 API 的产品名称 |
| 批量预测 | “Vertex 批量” | Vertex AI 批量 API 的产品名称 |
| 处理时间 SLA | “24 小时承诺” | 保证，非典型；典型为 2-6 小时 |
| 工作负载分类 | “交互性决策” | 交互式 / 半交互式 / 批量的路由决策 |
| 输出模式 | “响应格式” | 各提供商不同的 JSONL 布局；不可移植 |
| 叠加折扣 | “批量 + 缓存” | 当两者均适用时，约为未缓存同步账单的 10% |
