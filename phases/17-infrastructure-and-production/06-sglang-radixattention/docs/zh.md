# 面向前缀密集型工作负载的SGLang与RadixAttention

> SGLang将KV缓存视为存储在基数树中的首要可复用资源。当vLLM以先来先服务（FCFS）方式调度请求时，SGLang的缓存感知调度器会优先处理具有更长共享前缀的请求——这实际上是一种深度优先的基数遍历，确保热分支常驻于HBM内存中。在Llama 3.1 8B模型上使用类ShareGPT的1K提示词时，SGLang达到约16,200 tok/s，而vLLM约为12,500，优势约29%。对于前缀密集型的RAG工作负载，优势可达6.4倍。在语音克隆类工作负载中，缓存命中率超过86%。截至2026年，已在xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS的超过400,000个GPU上部署。需注意，当提示词前缀顺序不一致时，6.4倍的优势将不复存在——顺序是工程师的关键调节手段。

**类型：** 学习
**语言：** Python（标准库，简易基数树缓存 + 缓存感知调度器）
**先决条件：** Phase 17 · 04（vLLM 服务内部原理）、Phase 14（智能体RAG）
**时间：** 约75分钟

## 学习目标

- **图解RadixAttention**：前缀如何存储在基数树中，以及KV块如何在同一分支下的序列间共享。
- **解释缓存感知调度**：为什么FCFS不适合前缀密集型流量。
- **计算预期加速比**：给定工作负载的提示词缓存命中率和提示词长度分布。
- **命名关键规则**：了解使6.4倍优势得以实现（或丧失）的提示词排序规范。

## 问题所在

传统服务将每个请求的提示词视为黑盒。即使5,000个RAG请求都以相同的2,000个token的系统提示加上相同的检索序言开始，vLLM也会对这2,000个token的前缀进行5,000次预填充。GPU反复执行相同的工作。

观察发现：在智能体和RAG工作负载中，提示词几乎总是共享长前缀。系统提示、工具模式、少样本示例、检索头信息、对话历史——这些内容在请求间重复出现。如果你只存储一次该前缀的KV缓存并复用，就无需再次进行预填充。

RadixAttention正是这样实现的。token被索引在基数树中；每个节点拥有从根节点到该节点路径上token序列对应的KV块。新请求遍历树：任何匹配token的节点都会复用该节点的KV块。预填充成本仅与“新”后缀成正比，而非整个提示词。

挑战在于调度。如果两个请求共享2,000个token的前缀，而第三个请求只共享同一前缀中的200个token，你希望优先处理这两个长共享请求，以便长前缀能常驻在HBM中。FCFS的做法恰恰相反——它优先处理先到达的请求，可能在下一个长前缀请求到来前就驱逐了热分支。

## 核心概念

### 基数树作为KV索引

基数树（紧凑型字典树）存储token序列。每个节点拥有一个token范围及为该范围计算的KV块。子节点在序列上扩展一个或多个token。

```
root
 |- "You are a helpful assistant..."  (2,000 tokens, 124 KV blocks)
      |- "Context: <doc A>..."        (500 tokens, 31 blocks)
           |- "Question: Alice..."    (80 tokens, 5 blocks)
           |- "Question: Bob..."      (95 tokens, 6 blocks)
      |- "Context: <doc B>..."        (520 tokens, 33 blocks)
```

一个新请求进入，包含系统提示 + “Context: <doc A>” + “Question: Carol”。调度器遍历：系统前缀匹配（复用124个KV块），doc-A分支匹配（复用31个KV块），然后仅为“Question: Carol”分配新的KV块（4个）。预填充成本：4个新token对应的KV块。若无此树结构：需要160个KV块。预填充节省约40倍。

### 缓存感知调度

基于基数树的复用若缓存频繁失效则无意义。两个关键策略：

1. **深度优先调度**。从队列中选择下一个请求时，优先选择与当前运行请求共享同一分支根节点的请求。这能保持热分支常驻。
2. **分支级LRU，而非块级LRU**。驱逐整个分支（从最近最少使用的叶子节点开始），而非单个KV块，使缓存形状与基数树形状匹配。

FCFS违反这两条原则。一个共享2,000个token的请求排在一个仅共享50个token的请求后面，导致2,000个token的分支被驱逐以接纳那50个token的请求。

### 需要牢记的基准测试数据

- Llama 3.1 8B，H100，ShareGPT 1K提示词：SGLang 约16,200 tok/s 对比 vLLM 约12,500（优势约29%）。
- 前缀密集型RAG（相同系统 + 相同文档，不同问题）：SGLang最高可达6.4倍优势。
- 语音克隆工作负载：前缀缓存命中率86.4%。
- SGLang客户的生产环境命中率：根据提示词规范程度，范围在50%-99%。
- 截至2026年已部署于超过400,000个GPU。

### 顺序陷阱

6.4倍的优势依赖于一致的提示词模板顺序。如果你的客户端在部分请求中构造提示词为 `[system, tools, context, history, question]`，而在其他请求中为 `[system, context, tools, history, question]`，树结构将无法找到共享前缀。在人眼看来是共享前缀的内容，对基数树而言是两个不同的序列。

工程师的调节手段：你的提示词模板就是一个缓存键。固定顺序。将所有不可变内容（系统提示、工具、模式）放在最前。接着是检索上下文。最后是用户问题。不要将动态内容穿插到前缀中。

研究中的真实案例：将动态内容移出可缓存前缀，一次修改就将某部署的缓存命中率从7%提升至74%。

### RadixAttention的适用与不适用场景

**适用场景：**
- RAG（相同的检索序言，不同的问题）。
- 智能体（相同的工具模式，不同的查询）。
- 具有长系统提示的聊天场景。
- 具有重复序言的语音/视觉工作负载。

**不适用场景（性能回落至vLLM水平）：**
- 单次生成且提示词唯一（代码补全、无系统提示的开放式聊天）。
- 动态提示词，每个请求都在前缀中穿插独特内容。

### 为什么这是调度问题，而不仅仅是内核问题

你可以将KV复用实现为一种内核技巧。SGLang的洞察在于，只有当调度器保持热分支常驻时，复用才有收益。在混合负载下，朴素的“若可用则复用”策略会导致缓存频繁失效。基于基数树索引的调度器正是将内核技巧转化为29%生产优势的关键。

### 与vLLM的交互

这两个系统并非严格的竞争关系。2026年，vLLM增加了前缀缓存（`--enable-prefix-caching`）和一个缓存感知路由器（用Rust编写的vLLM Router）。差距缩小但并未完全消失——SGLang的整个技术栈以基数树为核心；vLLM是后期整合。对于以前缀复用为主的工作负载，SGLang仍是默认选择。对于缺乏明显前缀模式的通用服务，vLLM表现相当或更优。

## 动手实践

`code/main.py` 实现了一个简易的基数树KV缓存以及支持两种策略（FCFS和缓存感知）的调度器。通过同一工作负载运行两者，报告前缀缓存命中率和吞吐量差异。然后运行一个“打乱顺序”的工作负载，展示6.4倍优势如何消失。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| RadixAttention | “SGLang的那个技术” | 将KV缓存索引为基数树，使共享前缀复用KV块 |
| 基数树 | “紧凑型字典树” | 每个节点拥有一个token范围及其KV块的树结构 |
| 缓存感知调度器 | “热分支优先” | 优先处理共享当前常驻分支请求的调度器 |
| 前缀缓存命中率 | “你的提示词有多少是免费服务的” | 提示词token中通过复用KV块服务的比例 |
| FCFS | “先来先服务” | 破坏前缀局部性的默认调度策略 |
| 分支级LRU | “驱逐叶子节点” | 与基数树形状匹配的驱逐策略 |
| 提示词模板顺序 | “缓存键” | 提示词组件的顺序决定了树能共享什么 |
| 系统提示固定 | “常驻前缀” | 保持不可变的系统部分常驻以避免驱逐抖动 |
