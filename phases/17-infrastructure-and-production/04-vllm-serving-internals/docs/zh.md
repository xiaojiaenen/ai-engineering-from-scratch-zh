# vLLM 服务内部机制：分页注意力、连续批处理与分块预填充

> vLLM 在 2026 年的主导地位基于三个相互增强的默认配置，而非单一技巧。分页注意力始终开启。连续批处理在解码迭代之间将新请求注入活跃批次。分块预填充将长提示词切片，确保解码 token 永远不会“饥饿”。三者同时开启时，在单张 H100 SXM5 上运行的 Llama 3.3 70B FP8 模型，在 128 并发下可实现 2,200-2,400 tok/s 的吞吐量——比 vLLM 自身的默认设置高出约 25%，是朴素 PyTorch 循环的 3-4 倍。本课将深入到足以绘制图表的层次，解读调度器和注意力内核，并以一个玩具级的连续批处理器 `code/main.py` 结尾，该批处理器按 vLLM 的方式调度预填充和解码。

## 学习目标

- 解释分页注意力作为 KV 缓存分配器的原理：块、块表以及在生产负载下碎片率为何低于 4%。
- 在迭代层面绘制连续批处理的图表：已完成的序列如何离开批次，新序列如何在不中断的情况下加入。
- 用一句话描述分块预填充，并指出它保护的是哪个延迟指标（提示：是 TTFT 尾部，而非平均吞吐量）。
- 指出 2026 年 vLLM v0.18.0 中，一次性启用所有优化时会让团队踩坑的问题。

## 问题所在

朴素的 PyTorch 服务循环一次只处理一个请求：分词、预填充、解码直到 EOS、返回。面对单个用户时，这没问题。面对一百个用户时，它就变成了一个需要耐心等待的队列。显而易见的改进——静态批处理——会将每个请求填充到当前批次中最长提示词的长度，将每次解码填充到预期最长输出的长度，并因最慢的序列而拖慢整个批次。你为你根本用不到的填充付费，而快速请求在等待慢速请求。

vLLM 同时解决了三个问题。分页注意力阻止了 KV 缓存碎片像经典连续分配那样吞噬 60-80% 的 GPU 内存。连续批处理让请求在每次解码迭代之间加入和离开批次，因此批次始终充满实际工作。分块预填充将一个 32k token 的提示词切成约 512 token 的切片，与解码交错执行，从而一个长提示词不会冻结 GPU 上的所有解码 token。

2026 年的生产默认配置是三者全开。你需要理解每一项的作用，因为故障模式都出在调度器上，而非模型本身。

## 核心概念

### 分页注意力作为虚拟内存系统

一个 KV 缓存大小为 `num_layers × 2 × num_heads × head_dim × seq_len × bytes_per_element` 每序列。对于 8192 token 的 Llama 3.3 70B 模型，在 BF16 下，每个序列的缓存大约为 1.25 GB。如果你为每个请求预先保留 8192 个槽位，但平均请求只使用 1500 个 token，那么你大约浪费了所预留 HBM 的 82%。经典批处理需要承受这种浪费。

分页注意力借鉴了操作系统虚拟内存的思想。KV 缓存不是每个序列连续分配的。它以固定大小的块（默认 16 个 token）进行分配。每个序列都有一个块表，将其逻辑 token 位置映射到物理块 ID。当一个序列超出其已分配的块时，会添加一个新块。当序列结束时，其块返回到池中。

碎片率从 60-80%（经典方式）下降到低于 4%（分页注意力）。你无需通过一个标志来启用分页注意力——它是 vLLM 唯一提供的分配器。调节旋钮是 `--gpu-memory-utilization`（默认 0.9），它告诉 vLLM 在加载权重和激活后，为 KV 块保留多少 HBM。

### 迭代层面的连续批处理

旧的“动态批处理”会等待一个时间窗口（例如 10 毫秒）来填满一个批次，然后运行预填充 + 解码 + 解码 + 解码，直到每个序列完成。快速序列会提前离开并在 GPU 完成慢序列时闲置。

连续批处理在每个解码步骤之间运行。将正在运行的序列集合称为 `RUNNING` 列表。在每次迭代中：

1. 任何刚刚达到 EOS 或 max_tokens 的 `RUNNING` 中的序列会被移除。
2. 调度器查看等待队列。如果有空闲 KV 块，它会接收新序列（预填充或恢复的）。
3. 前向传播在当前的 `RUNNING` 序列上运行，每个序列输出一个新的 token。

批次大小不会填充到固定数字。处于输出不同位置的序列共享一次融合前向传播。在 2026 年的 vLLM 中，这称为 `V1 scheduler`。关键不变性：调度器每个解码迭代运行一次，而不是每个请求一次。

### 分块预填充保护 TTFT 尾部

预填充是计算密集型的。在单张 H100 上，对 Llama 3.3 70B 处理一个 32k token 的提示词需要约 800 毫秒的纯预填充时间。在预填充运行期间，批次中其他所有序列的解码 token 都在等待。在服务循环中，一个长提示词的首 token 延迟（TTFT）会变成其他几十个用户的令牌间延迟（ITL）尖峰。

分块预填充将预填充拆分成固定大小的块（默认 512 个 token），并将每个块作为一个单元进行调度。在块之间，调度器可以将解码序列推进一个 token。你牺牲了一小点绝对预填充延迟（每块几毫秒），换来了大幅降低的解码时抖动。在已发布的基准测试中，混合负载下的 P99 ITL 从约 50 毫秒下降到约 15 毫秒。

### 三个默认配置的相互作用

这三个特性都相互依赖。分页注意力为调度器提供了细粒度的 KV 资源进行权衡。连续批处理需要这种细粒度资源，以便在接收新序列时无需强制进行全局重排。分块预填充是调度器在同一个 `RUNNING` 列表上做出的决定——它是另一个调度策略，而非独立的系统。

你不需要知道每个标志。你需要知道调度器优化的是什么：在 KV 块预算约束下，在分块预填充切片限制下的有效吞吐量。

### 2026 年 v0.18.0 的陷阱

在 vLLM v0.18.0 中，你无法将 `--enable-chunked-prefill` 与草稿模型投机解码（`--speculative-model`）结合使用。文档记录的例外情况是 V1 调度器中的 N-gram GPU 投机解码。那些不看发布说明就开启所有标志的团队，会在启动时遇到运行时错误，而非性能软下降。如果你的投机收益值得为启用分块预填充付出代价，请重新考虑这个选择——2026 年的正确答案通常是使用 EAGLE-3 而不启用分块预填充，而不是一个无法编译的草稿模型加分块预填充。

### 你应该记住的数字

- Llama 3.3 70B FP8，H100 SXM5，128 并发，三项全开：2,200-2,400 tok/s。
- 同模型，默认 vLLM（无分块预填充）：约 1,800 tok/s。
- 同模型，朴素 PyTorch 前向循环：约 600 tok/s。
- 生产负载下，使用分页注意力的 KV 碎片浪费率：<4%。
- 混合负载下的 P99 ITL：使用分块预填充约 15 毫秒，不使用约 50 毫秒。

### 调度器的样子

```
while True:
    finished = [s for s in RUNNING if s.is_done()]
    for s in finished: release_blocks(s); RUNNING.remove(s)

    while WAITING and have_free_blocks_for(WAITING[0]):
        s = WAITING.pop(0)
        allocate_initial_blocks(s)
        RUNNING.append(s)

    # schedule prefill chunks + decode in one batch
    batch = []
    for s in RUNNING:
        if s.in_prefill:
            batch.append(next_prefill_chunk(s))   # e.g. 512 tokens
        else:
            batch.append(decode_one_token(s))     # 1 token

    run_forward(batch)                            # one fused GPU call
```

`code/main.py` 正是这个循环用标准 Python 实现，使用了伪造的 token 计数和伪造的前向传播延迟。运行它可以看到分块预填充如何在长预填充期间保持解码序列存活。

## 动手实践

`code/main.py` 模拟了一个 vLLM 风格的调度器，支持可切换的功能。运行它可以观察：

- `NAIVE` 模式：一次一个请求，无批处理。
- `STATIC` 模式：填充并等待，经典批处理。
- `CONTINUOUS` 模式：迭代级准入和释放。
- `CONTINUOUS + CHUNKED` 模式：预填充切片与解码交错执行。

输出显示总吞吐量（每虚拟秒 token 数）、平均 TTFT 和 P99 ITL。在混合流量下，`CONTINUOUS + CHUNKED` 行应该占据主导地位。

## 关键术语

| 术语 | 人们怎么说 | 它的实际含义 |
|------|------------|--------------|
| 分页注意力 | “那个 KV 技巧” | KV 缓存的固定大小块分配器；碎片率 <4% |
| 块表 | “页表” | 每个序列从逻辑 token 位置到物理 KV 块的映射 |
| 连续批处理 | “动态批处理，但做对了” | 每次解码迭代都做出准入/释放决策 |
| 分块预填充 | “预填充切片” | 将长预填充切成 512 token 切片，与解码交错 |
| TTFT | “首 token 时间” | 预填充 + 队列 + 网络；长提示词时受预填充主导 |
| ITL | “令牌间延迟” | 连续解码 token 之间的时间；受批次大小主导 |
| 有效吞吐量 | “满足 SLO 的吞吐量” | 每个请求都达到 TTFT 和 ITL 目标时的 token/s |
| V1 调度器 | “新调度器” | vLLM 的 2026 调度器；N-gram 投机解码是兼容分块预填充的路径 |
| `--gpu-memory-utilization` | “内存旋钮” | 加载权重和激活后，为 KV 块保留的 HBM 比例 |
