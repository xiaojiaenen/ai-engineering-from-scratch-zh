# 大语言模型的影子流量、金丝雀发布与渐进式部署

> 大语言模型的部署结合了软件发布中最棘手的部分：没有单元测试、故障模式分散、信号延迟。标准流程是：(1) 影子模式——将生产环境请求复制到候选模型，记录日志并与生产输出对比，对用户零影响；可捕获明显的分布问题，但不能保证质量；(2) 金丝雀发布——渐进式流量切换：10% → 25% → 50% → 75% → 100%，每一步设置准入门槛；追踪延迟百分位数、单次请求成本、错误/拒绝率、输出长度分布、用户反馈率；(3) 在稳定性确认后，为不同替代方案进行A/B测试。非确定性无法消除——由于GPU浮点运算非结合性和批量大小差异，相同输入在不同运行中可能产生高达15%的准确率波动。成本是变量而非常数——性能提升20%的模型，单次调用成本可能增加3倍。回滚速度至关重要：如果回滚需要重新部署，那速度就太慢了。策略配置在config/flags中；模型存储在注册表中并锁定摘要；回滚 = 切换策略 + 恢复阈值 + 锁定旧模型，整个过程以秒计。

**类型：** 学习
**语言：** Python（标准库，简易金丝雀进度模拟器）
**先决条件：** 第17阶段 · 13（可观测性），第17阶段 · 21（A/B测试）
**时长：** 约60分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀发布（实时流量渐进式发布）和A/B测试（稳定性确认后的对比）。
- 列举五项大语言模型特定的金丝雀指标（延迟、单次请求成本、错误/拒绝率、输出长度分布、用户反馈）。
- 解释为什么大语言模型的非确定性（高达15%）改变了发布过程中“稳定”的含义。
- 设计一条可在秒级完成的回滚路径（策略切换），而非耗时数小时（重新部署）。

## 问题背景

你发布了一个新模型。离线评估显示准确率提升了3%。你在生产环境中启用它。24小时内，成本上升了40%，用户差评率上升了8%，三张客户工单报告“回答很奇怪”。你进行了回滚。重新部署耗时3小时。你的周末泡汤了。

上述所有问题本可避免。影子模式会在用户看到任何问题前，就捕获到40%的成本飙升。金丝雀发布会在差评率上升时，在10%流量阶段就停止。基于策略标志的回滚只需30秒。这种规范填补了“离线评估看起来不错”与“真实用户满意”之间的鸿沟。

## 核心概念

### 影子模式

候选模型接收与生产模型相同的请求；输出被记录，但不会返回给用户。对用户零影响。记录内容包括：

- 输出内容（与生产模型输出的差异）。
- Token数量（成本差异）。
- 延迟。
- 拒绝与错误。

可捕获：成本飙升、长度回退、明显的拒绝变化、硬性错误。**无法捕获**：用户会感知到的质量差异。影子模式是冒烟测试，不是质量测试。

### 金丝雀发布

带有准入门槛的渐进式流量切换。典型进度：1% → 10% → 25% → 50% → 75% → 100%。每一步基于五项指标设置门槛：

1.  **延迟百分位数** — P50、P95、P99。门槛：金丝雀的P99超过基线的1.5倍。
2.  **单次请求成本** — 综合成本。门槛：高于基线超过20%。
3.  **错误/拒绝率** — 5xx错误加上明确的拒绝。门槛：超过基线的2倍。
4.  **输出长度分布** — 均值和P99。门槛：分布发生偏移。
5.  **用户反馈率** — 差评/工单提交。门槛：超过基线的1.5倍。

### 非确定性是新的方差

相同的输入会产生不同的输出。原因包括：

- GPU浮点运算非结合性（浮点数归约顺序随批量变化）。
- 批量大小差异（相同提示词在128个批量与16个批量中的表现不同）。
- 采样（temperature > 0）。

测量结果：在相同评估集上，不同运行间的准确率差异可达15%。发布中的“稳定”意味着指标在预期的方差范围内，并非与基线完全一致。门槛应设置在噪声水平之上。

### 成本是变量

性能提升20%的模型，单次调用成本可能增加3倍。单次请求成本是五项门槛之一。发布一个“更好”但破坏单位经济效益的模型，是回滚的用例。

### 回滚是关键武器

- 策略标志（功能标志系统）：在配置中切换百分比；耗时秒级。
- 模型锁定（注册表摘要）：锁定的模型不会自动升级。
- 回滚 = 恢复标志 + 将锁定摘要设为旧版本。以秒计，而非小时。

如果你的技术栈回滚需要重新部署，请在发布前修复此问题。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由** — 服务网格层面的流量分割。

**KServe / Seldon Core** — 内置金丝雀发布功能的模型服务。

**功能标志** — LaunchDarkly, Flagsmith, Unleash。策略层面的切换，无需重新部署。

### 指标检查频率

金丝雀门槛检查频率为每5-15分钟一次，具体取决于流量大小。1%的流量，10次请求/分钟，每个窗口可提供50-150个数据点——足够用于延迟分析，但对用户反馈数据可能噪声较大。10%的流量提供约10倍的数据。进度应暂停足够时间，以便在每一步积累足够的样本。

### A/B测试步骤是可选的

如果新模型差异显著（行为不同、成本曲线不同、风格不同），在金丝雀通过后，在50%流量下进行A/B测试。如果只是改进版本，在金丝雀通过后可直接切换到100%。

### 你需要记住的数字

- 金丝雀进度：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入不同运行间差异可达15%。
- 五项金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本门槛：高于基线超过20%为越界。
- 回滚：以秒计，而非小时。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|------------|----------|
| 影子模式 | "复制到新模型" | 零影响地将流量发送到候选模型以进行记录 |
| 金丝雀 | "渐进式流量" | 带有准入门槛的渐进式用户暴露发布 |
| 门槛 | "发布检查" | 阻止进度推进的指标阈值 |
| 非确定性 | "大语言模型方差" | 不可消除的运行间差异 |
| 策略标志 | "标志切换回滚" | 配置层面的回滚，以秒计 |
| 模型锁定 | "注册表摘要" | 模型版本的不可变引用 |
| Argo Rollouts | "K8s渐进式" | Kubernetes原生的金丝雀/回滚控制器 |
| KServe | "推理K8s" | 具有金丝雀发布原语的模型服务 |
| Istio加权 | "网格分割" | 服务网格流量分割器 |
