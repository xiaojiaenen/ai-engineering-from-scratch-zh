# LLM FinOps — 单位经济性与多租户归因

> 传统FinOps在LLM支出管理上失效。成本基于token交易而非资源运行时间。标签无法映射——API调用是交易而非资产。工程决策（提示设计、上下文窗口、输出长度）即财务决策。2026年运营手册包含三个需在初期就实施的归因维度：按用户（`user_id`）用于席位定价与扩展，按任务（`task_id` + `route`）用于产品界面成本与优先级，按租户（`tenant_id`）用于单位经济性与续费。四层token结构——提示词、工具、记忆、响应——单一成本桶会掩盖支出。多租户产品的执行梯度：每租户速率限制（预期峰值的2-3倍，返回429状态码+重试头）；每日支出上限（合同上限的1.5-3倍；触发速率收紧+警报）；当支出z分数>4时启用熔断机制（自动暂停+呼叫值班人员）。归因模式：标签聚合、遥测关联器（追踪ID→计费；最高精度）、抽样推算、基于模型分配、事件溯源、实时流处理。单位指标：每解决查询成本、每生成制品成本——而非每百万token成本。追溯标记总会遗漏；应在请求创建时埋点。

**类型：** 学习
**语言：** Python（标准库，简易成本归因模拟器含熔断机制）
**前置知识：** 第17阶段·13（可观测性），第17阶段·14（缓存）
**时间：** 约60分钟

## 学习目标

- 解释为何传统FinOps（标签+层级）在LLM支出管理上失效，并说明三个新归因维度
- 枚举四层token结构（提示词、工具、记忆、响应）以及单一成本桶计费如何隐藏真实成本
- 为多租户产品设计执行梯度（速率限制→支出上限→熔断机制）
- 选择单位指标（每解决查询/制品成本）而非每百万token成本

## 问题所在

您的账单显示$40,000，但您无从知晓：
- 哪个租户产生了该支出
- 哪个产品功能驱动了该支出
- 是否有单个用户存在滥用行为
- 是否因提示词膨胀、工具调用或记忆放大导致

供应商端的标签聚合适用于云资源（EC2、S3），其标签可传递至计费项。LLM API调用不会自动标记——必须在调用点标注用户/任务/租户信息并贯穿整个流程。追溯归因总会遗漏边缘情况。

## 核心概念

### 三个归因维度

**按用户**（`user_id`）：谁产生了什么成本。驱动席位定价、扩展对话、识别核心用户。

**按任务**（`task_id` + `route`）：哪个产品界面成本几何。驱动功能优先级、砍掉高成本功能的决策。

**按租户**（`tenant_id`）：哪个客户具有盈利性。驱动单位经济性、续费定价、层级阈值。

务必在初期就在调用点实现这三维度埋点。追溯归因效果永远更差。

### 四层token结构

| 层级 | 示例 | 通常占比 |
|------|------|----------|
| 提示词 | 系统+用户输入 | 40-60% |
| 工具 | 工具调用结果反馈 | 20-40%（智能体工作负载） |
| 记忆 | 历史对话/检索文档 | 10-30% |
| 响应 | 模型输出 | 10-30% |

将四层混为一谈会使优化陷入盲目。应在归因架构中将其分离。

### 执行梯度

1. **速率限制**按租户。设为预期峰值的2-3倍。返回429状态码并携带`Retry-After`。租户会感受到限制；避免意外账单。

2. **每日支出上限**按租户。设为合同上限的1.5-3倍。触发条件：收紧速率限制+通知客户成功团队。

3. **熔断机制**当支出z分数超过租户基线4倍时触发。自动暂停租户；呼叫值班人员；升级至运营与客服团队。

### 归因模式

- **标签聚合**：通过元数据头打标；后续聚合。简单但粗糙。
- **遥测关联器**：通过追踪ID关联追踪与计费。精度最高。成熟团队的做法。
- **抽样+推算**：抽样5-10%后乘以系数。适用于粗略支出估算；会遗漏长尾数据。
- **基于模型分配**：回归分析推断成本驱动因素。适用于无标签的遗留数据。
- **事件溯源**：将成本视为流中的事件（Kafka/Kinesis）。实时处理。
- **实时流处理**：仪表板亚秒级更新。

### X的单位成本才是正确指标

每百万token成本是供应商话语。产品指标应为：

- 每解决支持工单成本
- 每生成文章成本
- 每成功智能体任务成本
- 每用户会话分钟成本

将成本与产品产出挂钩。否则优化将失去锚点。

### 成本归因追踪结构

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用时输出。存储在数据湖中。按维度聚合。第17阶段·13可观测性技术栈正是为此而生。

### 复合优化栈

技术栈：缓存+批处理+路由+网关。四者结合时：
- L2缓存（第17阶段·14）：输入成本降低约10倍
- 批处理（第17阶段·15）：节省50%
- 路由至低成本模型（第17阶段·16）：降低60%成本
- 网关效率（第17阶段·19）：冗余+重试机制

最佳情况下的复合优化：约为基线成本的5-10%。多数团队仅采用2-3个优化杠杆；少有团队能叠加全部四个。

### 关键数据记忆点

- 归因维度：按用户、按任务、按租户
- 四层token结构：提示词、工具、记忆、响应
- 熔断机制：支出z分数>4时触发
- 单位指标：每解决查询成本，而非每百万token成本
- 复合优化：可能达到基线成本的5-10%
