# AI SRE — 多智能体事件响应、运行手册、预测性检测

> AI SRE 利用基于基础设施数据（日志、运行手册、服务拓扑）的 LLM，通过 RAG 自动化调查、文档记录和协调阶段。2026 年的架构范式是多智能体编排 —— 专业智能体（日志、指标、运行手册）由监督者协调；AI 提出假设和查询，人类审批决策判断。Datadog Bits AI 和 Azure SRE Agent 已将此作为托管产品发布。运行手册正在演进：NeuBird Hawkeye 采用对抗性评估（两个模型分析同一事件；一致 = 确信，分歧 = 不确定）；操作记忆在团队变动中保持持久性。自动修复保持谨慎：AI 建议，人类批准。完全自主操作范围狭窄（重启 Pod、回滚特定部署）且有严格约束 —— 任何宣扬“设置后无需干预”的说法都言过其实。新兴前沿：事件前预测。MIT 研究报告称，一个基于历史日志、GPU 温度和 API 错误模式训练的 LLM，提前 10-15 分钟预测了 89% 的故障。展望：到 2026 年底，95% 的企业 LLM 将实现自动故障转移。

**类型：** 学习
**语言：** Python（标准库，简易多智能体事件分诊模拟器）
**前置知识：** 阶段 17 · 13（可观测性），阶段 17 · 24（混沌工程）
**时间：** 约 60 分钟

## 学习目标

- 绘制多智能体 AI SRE 架构图：监督者 + 专业智能体（日志、指标、运行手册）+ 人类审批关口。
- 解释为何自动修复范围狭窄（重启 Pod、回滚部署）而非宽泛（重新设计服务架构）。
- 说明对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 确信；不一致 = 上报。
- 引用 MIT 的 89% 早期检测结果及操作约束：没有执行的预测只是仪表盘。

## 问题所在

凌晨 3 点，值班工程师收到告警：“结账环节错误率高”。他们检查 Datadog、Loki、三份运行手册、部署日志。30 分钟后，他们发现根本原因是 KV 缓存激增导致的 vLLM 内存不足。他们重启了 Pod；错误消除。

在 2026 年，那 20 分钟的调查是可以自动化的。按服务分组日志、关联近期部署、匹配运行手册 —— 这些都是 RAG + 工具使用。一个受监督的智能体可以在人类打开 Datadog 之前完成初步分诊并提出假设。

完全自主修复是另一个问题。重启 Pod：安全。扩缩 GPU 池：策略允许则安全。重新设计服务架构：绝对不行。学科要求在于划定那条狭窄的界线。

## 核心概念

### 多智能体架构

```
          Incident
             │
             ▼
        Supervisor
        /    |    \
       ▼     ▼     ▼
  Log agent  Metric agent  Runbook agent
       │     │     │
       └─────┴─────┘
             │
             ▼
        Hypothesis + evidence
             │
             ▼
        Human approval
             │
             ▼
        Action (narrow set)
```

监督者将事件分解为子查询。专业智能体拥有工具访问权限（日志搜索、PromQL、文档检索）。监督者进行综合，向人类呈现假设及证据。人类批准或重新指导。

### 自动修复范围

**安全（狭窄）**：重启 Pod、回滚特定部署、在预批准范围内扩缩资源池、启用预批准的功能标志。

**不安全（宽泛）**：更改服务拓扑、修改资源限制、部署新代码、更改 IAM、变更数据库。

任何宣扬“设置后无需干预”的说法都言过其实。随着 AI SRE 成熟，安全范围会扩大，但边界是真实存在的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们就根本原因达成一致，则确信度高。如果存在分歧，则将两个假设都呈现给人工进行升级。简单模式，有效过滤幻觉生成的根本原因。

### 操作记忆

团队更替是传统 SRE 的隐形杀手 —— 隐性知识流失。AI SRE 将运行手册和事后分析存储在向量数据库中；智能体在每次新事件中进行检索。当新工程师加入时，AI 拥有完整历史记录。

### 事件前预测

MIT 2025 年研究：基于历史日志、GPU 温度和 API 错误模式训练的 LLM，在测试集上提前 10-15 分钟预测了 89% 的故障。

现实检验：没有执行的预测只是仪表盘。操作性问题是“当我们预测到时，我们该做什么？”预先排空流量？发送寻呼通知？自动扩容？答案取决于具体策略。

### 2026 年产品

- **Datadog Bits AI** —— Datadog 内的托管 SRE 副驾驶。
- **Azure SRE Agent** —— Azure 原生。
- **NeuBird Hawkeye** —— 对抗性评估 + 操作记忆。
- **PagerDuty AIOps** —— 分诊 + 去重。
- **Incident.io Autopilot** —— 事件指挥官 + 协调。

### 代码化运行手册

运行手册从 Confluence 页面演进为带有结构化部分（症状、假设、验证、行动）的版本控制 Markdown。结构化运行手册能提供更好的 RAG 检索效果。开始任何 AI-SRE 推广的第一步，就是将非结构化运行手册转为结构化格式。

### 应记住的数字

- MIT 早期检测：89% 的故障，提前 10-15 分钟。
- 多智能体分诊：监督者 +（日志、指标、运行手册）+ 人类。
- 安全自动修复集合：重启 Pod、回滚部署、在约束范围内扩缩。
- 对抗性评估：两个模型独立；一致 = 确信。

## 使用它

`code/main.py` 模拟多智能体分诊：日志智能体发现错误，指标智能体发现 CPU 峰值，运行手册智能体匹配到已知问题。监督者对假设进行排序。

## 部署它

本课程产出 `outputs/skill-ai-sre-plan.md`。基于当前值班安排、事件数量、团队成熟度，设计一个 AI SRE 推广方案。

## 练习

1.  运行 `code/main.py`。如果日志智能体和指标智能体意见不一致怎么办？监督者如何解决？
2.  为你的服务定义三个“安全”的自动修复操作。为每个操作提供理由。
3.  编写一份结构化运行手册模板：包含部分、必填字段、验证命令。
4.  预测性检测提前 12 分钟触发。你的策略是什么 —— 发送寻呼通知、预先排空流量，还是两者都做？
5.  论述一个 3 人团队在 2026 年是否应采用 AI SRE 还是等待。考虑成熟度、事件量、风险因素。
